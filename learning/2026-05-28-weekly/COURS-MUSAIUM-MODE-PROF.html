1 La carte de l’app

La carte de l'app — du tap au pixel : cycle de vie d'une requete reelle (mobile -> backend hexagonal -> LLM/PG)

L’idée maîtresse — Une requete traverse 3 couches concentriques — l'app cliente (mobile/web) parle HTTP au backend, qui passe par un mur de middlewares globaux dans app.ts, puis route /api -> module hexagonal (route primary -> useCase -> domain -> adapter secondary -> PG/LLM) et renvoie la reponse.

Pourquoi ça compte — Une fois cette carte memorisee, Tim peut situer N'IMPORTE QUEL fichier du monorepo dans le flux en 2 secondes (est-ce une porte d'entree, un cerveau metier, ou un adaptateur vers le monde exterieur ?) — c'est la cle pour debugger et etendre l'app sans se perdre.

Le flux réel, étape par étape

  1. 1. Le client mobile construit l'appel HTTP
    L'app RN/Expo n'appelle jamais fetch() en dur dans un ecran. Une feature (ex chat) a une couche infrastructure : postMessage() construit le body (JSON, ou multipart FormData si imageUri) puis delegue a httpRequest() — le wrapper axios partage. C'est LA frontiere FE->BE.
    museum-frontend/features/chat/infrastructure/chatApi/send.ts:162
  2. 2. httpRequest -> axios attache l'auth et part vers /api
    httpRequest pose Content-Type, gzip eventuel des gros bodies (W1-GZIP), puis httpClient.request(). Le token Bearer + base URL sont injectes par l'intercepteur axios (requiresAuth:true par defaut). Cote web admin c'est l'equivalent api.ts (apiGet/Post) qui ajoute X-CSRF-Token + credentials:include car l'auth y passe par cookie HttpOnly, pas Bearer.
    museum-frontend/shared/api/httpRequest.ts:103
  3. 3. Le backend recoit : mur de middlewares globaux (app.ts)
    applyGlobalMiddleware monte dans CET ordre precis : requestId -> helmet (tot, pour que meme un 429 ait les headers) -> tracePropagation -> CORS -> rate-limit IP -> compression -> [timeout] -> decompression gzip -> express.json -> cookieParser -> csrf -> acceptLanguage -> dataMode. L'ordre est load-bearing (ex csrf APRES cookieParser pour lire le cookie).
    museum-backend/src/app.ts:179
  4. 4. Montage des routes sous /api (apiRouter)
    createApiRouter expose /health + /health/deep puis mountDomainRouters branche chaque module sur son prefixe : /chat, /auth, /museums, /daily-art, /admin, /reviews, /leads, /telemetry... Chaque sous-routeur vient du dossier adapters/primary/http/routes d'un module.
    museum-backend/src/shared/routers/api.router.ts:349
  5. 5. Route PRIMARY du module : parse + middlewares route-specifiques
    Ex POST /chat/sessions/:id/messages : le handler lit req.file / req.body, parse le contexte (locale, museumMode...), valide via Zod (parsePostMessageRequest), passe par isAuthenticated + dailyChatLimit + llmCostGuard + idempotency, puis appelle chatService.postMessage(). La route NE contient PAS de logique metier — elle traduit HTTP -> appel useCase.
    museum-backend/src/modules/chat/adapters/primary/http/routes/chat-message.route.ts:81
  6. 6. Use-case = le cerveau metier (orchestration)
    chat.service.ts est la facade ; il delegue a chat-message.service.ts qui orchestre la sequence reelle : guardrail input -> persist message user -> cache LLM lookup -> generation -> guardrail output -> persist reponse. Le use-case ne connait QUE des interfaces (ports), jamais TypeORM ni LangChain directement.
    museum-backend/src/modules/chat/useCase/message/chat-message.service.ts:268
  7. 7. Adapter SECONDARY : le use-case touche le monde exterieur

    museum-backend/src/modules/chat/adapters/secondary/llm/langchain.orchestrator.ts:240

    Le DTO remonte useCase -> route -> errorHandler (museum-backend/src/app.ts:353) qui transforme toute AppError en JSON propre + status. Sentry capte les 500. Le body redescend par axios -> ensureContract() valide la forme cote FE avant de toucher l'UI.

À retenir

DEUX styles de module, meme contrat hexagonal. Regarde un index.ts : museum/index.ts est un BARREL PUR (re-exporte juste les useCases depuis ./useCase). chat/index.ts est une COMPOSITION-ROOT (expose buildChatService qui instancie tout). Dans les deux cas la structure interne est identique : adapters/{primary,secondary} + domain + useCase. primary = qui m'appelle (HTTP), secondary = qui j'appelle (DB, LLM, S3). Si tu vois 'adapters/primary/http/routes' = porte d'entree ; 'adapters/secondary/llm' = sortie vers l'exterieur ; 'useCase' = metier pur ; 'domain' = types + interfaces (ports) + entites.
LES ALIAS D'IMPORT SONT TA BOUSSOLE. @modules/* (= src/modules) pointe un module, @shared/* (= src/shared) le code transverse (middlewares, cache, observability), @data/* (= src/data) la DB/migrations, @src/* la racine. Donc des qu'un fichier importe @shared/middleware/csrf.middleware tu sais que c'est de l'infra HTTP, et @modules/chat/domain/ports/... que c'est un contrat (interface) que le useCase consomme sans connaitre l'implem. Regle d'or hexagonale visible dans le code : un useCase n'importe JAMAIS depuis adapters/secondary directement — il recoit le port en parametre (DI dans chat-module.ts build()).
L'INVERSION DE DEPENDANCE EST CONCRETE, pas theorique. chat-message.service.ts:337 fait this.orchestrator.generate(...) ou orchestrator est de type ChatOrchestrator (un port/interface dans domain/ports). Le useCase ne sait pas que c'est LangChain derriere. C'est exactement pour ca que les tests peuvent injecter un faux orchestrator (createApp accepte chatService en option, app.ts:46). Pour brancher Deepseek au lieu d'OpenAI : tu touches l'adapter secondary, jamais le useCase.
L'ORDRE DES MIDDLEWARES EST UN PIEGE A BUGS. Dans app.ts les commentaires expliquent POURQUOI chaque position : helmet AVANT rate-limit (sinon un 429 partait sans headers de securite, TD-HEL-01), decompression gzip AVANT express.json (sinon le parser lit des octets compresses), csrf APRES cookieParser (sinon pas de cookie a lire). Quand tu ajoutes un middleware, sa POSITION dans applyGlobalMiddleware est une decision, pas un detail. Le mount /api vient APRES tout ce mur, donc chaque route herite gratuitement de CORS+csrf+rate-limit+trace.

2 Le pipeline chat

Le pipeline chat — defense en profondeur (du message user a la reponse TTS)

L’idée maîtresse — Un seul tour de chat traverse 6 couches de garde-fous independantes (chacune couvrant un mode d echec different) avant ET apres l appel LLM, le tout cable pour de vrai au composition root — ce n est pas de la theorie.

Pourquoi ça compte — C est le coeur produit de Musaium : si une seule couche cede (injection, off-topic, fuite PII, sidecar down, juge lent), les autres tiennent. Comprendre l ordre exact des couches et leur politique de panne (fail-CLOSED vs fail-OPEN vs fail-SOFT) te permet de modifier le chat sans rouvrir un trou de securite.

À retenir

POURQUOI defense-en-profondeur, couche par couche : V1 keyword (etape 2) attrape l evident en 5ms sans cout LLM (insultes, jailbreak connus) ; le sidecar (etape 3, fail-CLOSED) attrape l injection/PII subtile que les keywords ratent — fail-CLOSED car un scanner de securite muet = danger ; le juge LLM (etape 6, fail-OPEN) juge le SEMANTIQUE off-topic — fail-OPEN car un juge lent ne doit JAMAIS casser la dispo voice-first ; l isolation structurelle (etape 5) defend meme si tout le reste cede (le LLM voit clairement systeme vs user). Chaque couche = un mode d echec distinct, et la politique de panne est choisie par couche.
La regle d or des politiques de panne dans ce code : sidecar = fail-CLOSED (bloque en cas de doute, c est de la securite) ; juge = fail-OPEN (laisse passer en cas de doute, c est de la dispo) ; friction store Redis = fail-SOFT (panne = ni 500 ni faux bloc, juste pas d escalade). Confondre ces 3 te ferait soit refuser des users legitimes, soit ouvrir un trou. Le code le repete a chaque catch (v2-layers.helper.ts:75-82 vs guardrail-escalation.ts:108-111).
Le passage inline -> parallele du juge (hybrid-gravity, 2026-06-01) est la decision la plus subtile : avant, le juge tournait AVANT generate (latence serie sur CHAQUE message long). Maintenant runParallel lance juge + generate ensemble via Promise.allSettled (guardrail-escalation.ts:131-139). Un message on-topic paie ~0 latence supplementaire. Cout : si off-topic franchit le seuil, la reponse est deja generee donc payee en tokens — le commentaire chat-message.service.ts:357-360 l assume honnetement (abort non cable, AbortSignal non supporte).
La friction n est PAS un simple compteur off-topic : c est 2 scopes (session + principal) avec des poids par gravite (securite=2, off-topic=1, env.ts:448-449) et 2 seuils (session=3, user/IP=10). Un spammeur d injections cumule des strikes poids-2 sur son IP hashee a travers TOUTES ses sessions (recordSecurityStrike) jusqu a armer un cool-down global. C est ce qui transforme un hard-block ponctuel en rate-limit progressif sans bannir un user qui derape une fois. L IP est toujours hashee (hashIp) — RGPD.

Geo & voix — le contexte du monde réel (LocationResolver per-message + pipeline voix STT→LLM→TTS)

L’idée maîtresse — Musaium injecte le « où » (géo per-message, jamais les coordonnées brutes au LLM) et le « comment » (voix STT→LLM→TTS) dans la conversation, avec deux protections de coût distinctes : un breaker global observe-only sur le voice spend et une clé de cache TTS voice-aware.

4 Auth, consentement & RGPD

Auth, consentement & RGPD — la conformité câblée dans Musaium

L’idée maîtresse — L'auth Musaium est une rotation de refresh-token "banking-grade" à familles révocables sur réutilisation, posée sur des tokens device-bound côté FE ; la conformité RGPD est exécutée par du code (consent gate fail-closed, erasure multi-store, audit hash-chain) — pas par une politique sur papier.

Pourquoi ça compte — Tim vend un compagnon IA qui envoie des données utilisateur à OpenAI/Google : la moindre faille auth ou un trou RGPD (consentement, droit à l'effacement, registre inviolable) est un risque légal direct et un signal de sérieux pour des futurs musées B2B. Comprendre où la conformité est réelle vs honor-system est vital avant le launch V1.

Le flux réel, étape par étape

  1. 1. Login (BE) — vérif identité + portes 403
    AuthSessionService.login() normalise l'email, applique le rate-limiter en mémoire, bcrypt.compare le mot de passe, puis 4 portes 403 dans un ordre précis : deletedAt > suspended > email_verified (ADR-052 : le signal opérateur prime sur la bannière de vérification). Messages d'erreur volontairement génériques (INVALID_CREDENTIALS) pour ne pas révéler si un compte existe ou utilise le social sign-in.
    museum-backend/src/modules/auth/useCase/session/authSession.service.ts:89-162
  2. 2. Émission de la paire JWT à familles
    SessionIssuerService.issueSession() génère accessJti + refreshJti + familyId (3 UUID). L'access token (HS256, TTL court) porte sub/role/museumId ; le refresh porte familyId. On ne stocke JAMAIS le refresh en clair : seul son sha256 va en base (tokenHash), donc une fuite de la table refresh_tokens ne donne pas de tokens utilisables. iss+aud épinglés par famille (musaium-access vs musaium-refresh) → un access token ne peut pas se faire passer pour un refresh.
    museum-backend/src/modules/auth/useCase/session/session-issuer.service.ts:57-111
  3. 3. Persistance device-bound (FE)
    AuthContext reçoit la réponse, écrit refresh+access dans authStorage qui délègue à expo-secure-store avec keychainAccessible = WHEN_UNLOCKED_THIS_DEVICE_ONLY : les tokens n'entrent jamais dans une sauvegarde iCloud/iTunes et ne migrent pas vers un autre appareil.     museum-frontend/features/auth/infrastructure/authTokenStore.ts:55-67

Sinon expiry, puis fenêtre d'inactivité glissante (lastRotatedAt). rotate() est atomique : insère le nouveau + marque l'ancien rotatedAt dans une seule transaction.

À retenir

LE PIÈGE JSON.stringify replacer-array (cas d'école, audit-chain.ts:43-46) : computeRowHash fait JSON.stringify(input.metadata, Object.keys(input.metadata).sort()). Le 2e argument de JSON.stringify n'est PAS qu'un ordonnanceur de clés — quand c'est un tableau, c'est une ALLOWLIST de propriétés appliquée RÉCURSIVEMENT à tous les niveaux. Donc pour un metadata imbriqué comme {action:'notify', cnil:{ref:'X', severity:'high'}}, seules les clés top-level (action, cnil) sont dans l'allowlist ; les clés internes (ref, severity) n'y sont pas → l'objet cnil est sérialisé comme {} vide. Deux notifications CNIL avec des contenus internes DIFFÉRENTS produisent le MÊME hash = collision. Le contenu nested est invisible à la chaîne d'intégrité. La leçon : pour canonicaliser un objet de profondeur arbitraire, le replacer-array est le mauvais outil ; il faut un sérialiseur à clés triées récursif (replacer-FONCTION, ou tri récursif manuel) avant le hash.
Pourquoi la rotation est 'banking-grade' et pas juste 'un refresh token' : la combinaison familyId + sha256-only + revokeFamily-sur-reuse implémente le pattern OWASP de détection de vol. Si un attaquant vole un refresh et le rejoue après que le vrai user a déjà tourné (session-issuer.service.ts:124-127), le hash ne matche plus OU la ligne est déjà rotated → toute la famille est tuée. Conséquence pédagogique importante : le vrai user est déconnecté aussi (c'est voulu — le réveil forcé est le signal). C'est pour ça que la fenêtre d'inactivité glissante existe en plus (ligne 134-141) : sans rotation pendant X, on force la re-auth même sans vol.
Le logout doit gérer DEUX surfaces, pas une (authSession.service.ts:226-250) : révoquer le refresh token ne suffit pas, car l'access token reste cryptographiquement VALIDE jusqu'à son TTL (≤15 min). D'où la denylist Redis sur le access-jti, avec un TTL = exp - now (on n'inscrit rien si déjà expiré). Le logout est volontairement idempotent et fail-OPEN : un token invalide est avalé silencieusement (fiabilité côté FE), et un échec d'écriture denylist ne casse pas le logout. C'est l'inverse exact du consent gate qui, lui, est fail-CLOSED.
Fail-OPEN vs fail-CLOSED, choisis selon l'enjeu : le consent gate RGPD est fail-CLOSED (consent-gate.ts + checker:42-44 : dans le doute on REFUSE, car laisser passer = violation Art.7). Le logout/denylist est fail-OPEN (dans le doute on continue, car bloquer = user coincé connecté). L'erasure Art.17 (deleteAccount) est best-effort par étape externe MAIS la cascade DB s'exécute toujours, avec une outbox durable pour Brevo : on accepte qu'une étape tierce rate temporairement, jamais qu'elle bloque la suppression légale ni qu'elle laisse de la PII résiduelle non retentée.

Glossaire du module
Token family (familyId)
UUID partagé par toutes les générations successives d'un même refresh token. Permet de révoquer d'un coup toute la lignée quand un vol/replay est détecté.
Refresh-token rotation
À chaque refresh, l'ancien token est invalidé et un nouveau émis (rotate atomique). Un token n'est utilisable qu'une fois ; le réutiliser = signal de vol.
Fail-closed / fail-open
En cas d'incertitude (erreur, donnée manquante), fail-closed REFUSE (consent gate RGPD) ; fail-open AUTORISE (logout/denylist). On choisit selon que le risque est de laisser passer ou de bloquer à tort.
Device-bound token (WHEN_UNLOCKED_THIS_DEVICE_ONLY)
Classe d'accessibilité keychain iOS : le secret ne sort pas dans les sauvegardes iCloud/iTunes et ne migre pas vers un autre appareil.
Hash chain (registre en chaîne)
Chaque ligne d'audit contient le hash de la précédente (prev_hash) + son propre hash. Altérer une ligne casse tous les hash suivants → falsification détectable par rejeu (verifyAuditChain).
JSON.stringify replacer-array
2e argument de JSON.stringify sous forme de tableau : c'est une ALLOWLIST de propriétés appliquée à TOUS les niveaux d'imbrication. Toute clé absente (y compris dans un sous-objet) est omise — source du bug de collision.
pg_advisory_xact_lock
Verrou applicatif PostgreSQL au niveau transaction, relâché au COMMIT/ROLLBACK. Ici il sérialise les INSERT d'audit pour que la chaîne ne forke pas sous concurrence.
Outbox durable (Brevo)
Quand la suppression du contact marketing échoue, on persiste l'intention (lead brevo_erasure) qu'un cron retente — au lieu d'avaler l'échec et laisser de la PII tierce survivre.

À retenir

Le contrat de parité FE↔BE est une astuce élégante : au lieu d'un package partagé, on duplique volontairement les données mais on garde un sentinel sha256 qui hash UNIQUEMENT les octets entre les marqueurs NETWORK_PROFILES_DATA_REGION_START/END. C'est pourquoi le commentaire dit que ces marqueurs sont 'LOAD-BEARING' — déplacer un commentaire ou ajouter un import à l'intérieur de la région casse le hash. Le BE peut ainsi importer les profils sans tirer react/netinfo, et la dérive est impossible à merger.
Le ~16× n'est pas magique, c'est de l'arithmétique vérifiable : NORMAL_UPLOAD = 1600px JPEG ciblant 2_700_000 octets ; lowUploadWebp = 1024px WebP q0.60 ciblant 150_000 octets (compressionDecision.pure.ts:43 et :56). 2.7 Mo / 150 Ko ≈ 18×. Et c'est une décision PURE testable sans toucher au manipulateur expo — d'où le suffixe .pure.ts. Le dérivé local plus grand (320 Ko) sert le carnet post-visite, pas le réseau.
La doctrine '2 min local > 15 min CI rouge' se voit littéralement dans la migration KVM : avant, Maestro ne tournait JAMAIS par-PR (macOS arm64 software-only trop lent), donc les régressions e2e n'étaient catchées qu'en nightly — ou jamais (faux-vert). KVM rend le smoke per-PR possible en ~12 min, transformant un blocker local-green-only en garde-fou pré-merge. Le reviewer a deux fois sauvé le chantier en chopant des faux-verts (APK :3000 non shapé, toxic mono-direction) — preuve que le pipeline fresh-context /team fonctionne.
Le fix V2 guardrail illustre le pire piège de la confiance aveugle : le code SUPPORTAIT les deux couches V2 (sidecar fail-CLOSED, judge fail-OPEN) mais buildChatTestService les laissait tomber silencieusement, donc chaque test passait en V1-only — un faux sentiment de sécurité. Pire, la suite live a révélé que le timeout judge par défaut (500 ms) était inférieur à la latence réelle de gpt-4o-mini (670-1050 ms) → le judge fail-open sur CHAQUE appel prod, ne bloquant jamais rien. Corrigé à 1500 ms. On ne peut prouver une couche de sécurité qu'en l'exécutant réellement, sans mock.

Glossaire du module
Profil réseau (NetworkProfile)
Une condition réseau ratifiée décrite par données pures (latence, jitter, bande passante up/down en kbps, perte %, type NetInfo, data mode attendu). 6 existent : offline/2g/edge/3g-lossy/flapping/normal.
data region (parité FE↔BE)
Le bloc d'octets entre les marqueurs NETWORK_PROFILES_DATA_REGION_START/END, hashé par un sentinel sha256 pour garantir que la copie BE du registre est byte-identique à la vérité FE.
Toxiproxy (L3)
Proxy TCP qui injecte des 'toxics' (latence, bande passante, timeout) sur le trafic réel. Service dev-only gaté profiles:[weaknet] dans docker-compose ; listener shapé :3100, admin :8474. Jamais en prod (sentinel).
3 couches de test (L1/L2/L3)
L1 = simulateur FE pur (logique, déterministe). L2 = middleware BE d'injection de faute (HTTP-level, headers). L3 = Toxiproxy device-track (transport-level réel sur émulateur). Les trois sont pilotées par le MÊME registre keystone.
KVM (ubuntu Maestro)
Kernel-based Virtual Machine : virtualisation nested exposée par /dev/kvm sur les runners ubuntu-latest, qui rend un émulateur Android x86_64 accéléré matériellement — impossible sur macOS hébergé (HVF indispo).
smoke vs full (shards)
smoke = subset rapide de happy-paths critiques, ~12 min, 1 boot émulateur, joué par-PR. full = 4 shards (auth/chat/museum/settings), ~45 min/shard, joué nightly cron + push main.
fail-CLOSED vs fail-OPEN
Politique de dégradation. Sidecar LLM-Guard = fail-CLOSED (sidecar down → on BLOQUE, sécurité d'abord). Judge LLM = fail-OPEN (timeout/budget épuisé → on LAISSE PASSER, on retombe sur V1 pour ne pas casser le voice-first).
zip-bomb cap
Garde anti-déni-de-service : un petit corps gzip peut s'inflater en gigaoctets. Le middleware BE compte les octets décompressés au fil de l'eau et détruit le flux + renvoie 413 dès le dépassement, AVANT de tout matérialiser en mémoire.
