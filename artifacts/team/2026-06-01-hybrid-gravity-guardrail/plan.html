Spec — Guardrail hybride par gravité (judge parallèle + compteur de friction 2 niveaux)

RUN_ID 2026-06-01-hybrid-gravity-guardrail — phase spec (UFR-022 fresh-context). Source de vérité design : docs/superpowers/specs/2026-06-01-hybrid-gravity-guardrail-design.md (validé Tim 2026-06-01). Statut : enrichissement défense-en-profondeur (PAS un blocker launch). Le bug aigu judge fail-open silencieux est déjà corrigé (commit d769e2df, timeout 1500ms).

1. Stakeholders

RôleIntérêt
Visiteur B2C (utilisateur final, FR + 7 autres locales)Réponse fluide voice-first ; un écart hors-sujet occasionnel doit être recentré gracieusement, pas hard-bloqué.
Visiteur abuseur / spammeur off-topicDoit être escaladé (hard-block + cool-down) sans pénaliser le visiteur légitime.
Tech Lead / sécurité (Tim)Les invariants sécurité §3 du design restent non-négociables ; pas de PII/injection envoyée au LLM de réponse ; pas de fail-closed dû à une panne infra.
OpsKill-switch env sans redeploy ; observabilité (compteurs de strikes, escalades).
FinOpsOff-topic sous-seuil paie answer+judge ; abort = optimisation coût, pas une condition de correction.

2. Glossaire

  • Friction strike : incrément pondéré enregistré dans le GuardrailFrictionStore quand un message déclenche une friction (off-topic via judge = poids w_offtopic ; sécurité via V1/sidecar = poids w_security). Stocké par scope (session / user / ip).
  • Gravity band (bande de gravité) : classe de sévérité d'un message — security (injection/toxicité/PII, hard-block inline), offtopic (judge block, soft-redirect par défaut), on-topic (réponse normale, poids 0).
  • Soft-redirect : laisser la génération répondre ; le section prompt recentre gracieusement l'off-topic occasionnel SANS court-circuiter ni émettre policy:off_topic.
  • Cool-down : message de recentrage localisé honnête (i18n) renvoyé à la place de la réponse, sans 500, sans emoji. Deux déclencheurs distincts : (a) escalade session par-message off-topic, (b) cool-down temporisé global user/IP.
  • Escalade session (par-message ciblée off-topic) : à partir de FRICTION_SESSION_THRESHOLD strikes session, un message off-topic est hard-bloqué (cool-down) AU LIEU d'être soft-redirigé, POUR LE RESTE de la session. Les messages on-topic continuent normalement.
  • Cool-down user/IP (temporisé global) : à partir de FRICTION_USER_THRESHOLD strikes sur le scope user (ou IP hashée si anon), pendant FRICTION_COOLDOWN_MS, TOUS les messages de cet user/IP sont rate-limités (cool-down), même en changeant de session.
  • Fail-soft : sur indisponibilité du store de friction, count() renvoie 0 → pas d'escalade ; le soft-redirect + la sécurité inline tiennent. Jamais de blocage dur dû à une panne infra.
  • Fail-OPEN (judge) : judge timeout/erreur/budget → null → repli sur le verdict V1 (allow). Vérifié judgeWithLlm → null (llm-judge-guardrail.ts:175-249).
  • Fail-CLOSED (sidecar) : sidecar muet/timeout → policy:service_unavailable (ADR-047, inchangé).
  • Suppress (abort dégradé) : si l'orchestrateur ne propage pas l'AbortSignal externe, on laisse finir la génération puis on jette/remplace la réponse par le cool-down. Correct, juste moins économe.

3. Requirements (EARS)

3.1 Sécurité inline (invariants durs — design §3, non négociables)

  • R1 — WHEN un message est reçu, THE SYSTEM SHALL évaluer V1 keyword guardrail puis sidecar LLM-Guard AVANT tout appel au LLM de réponse. (Inchangé : evaluateInput reste appelé avant orchestrator.generate, prepare-message.pipeline.ts:283 puis chat-message.service.ts:285.)
  • R2 — IF la couche sécurité (V1 ou sidecar) renvoie un block, THE SYSTEM SHALL hard-bloquer le message immédiatement, persister l'échange bloqué (handleInputBlock), enregistrer un friction strike de poids FRICTION_WEIGHT_SECURITY sur les scopes session ET user/IP, et vérifier le plancher user/IP.
  • R3 — THE SYSTEM SHALL NEVER envoyer du contenu classé PII/injection/toxicité au LLM de réponse. (Seul le judge — fail-OPEN, off-topic — peut s'exécuter en parallèle de la génération.)
  • R4 — WHILE le sidecar est muet ou en timeout, THE SYSTEM SHALL répondre policy:service_unavailable (fail-CLOSED, ADR-047, inchangé).

3.2 Judge parallèle (off-topic, latence)

  • R5 — WHEN un message ≥ LLM_GUARDRAIL_JUDGE_MIN_LENGTH (50c) passe V1+sidecar ALLOW, THE SYSTEM SHALL exécuter le judge sémantique EN PARALLÈLE de la génération de la réponse (Promise.allSettled), et non plus en série avant elle.
  • R6 — WHILE le judge s'exécute en parallèle d'une réponse on-topic, THE SYSTEM SHALL n'ajouter aucune latence mesurable au cas allow (latence totale ≈ latence de la génération seule).
  • R7 — IF le judge renvoie allow (ou null = fail-OPEN), THE SYSTEM SHALL renvoyer la réponse générée.
  • R8 — IF le judge timeout/erreur/budget épuisé (null), THE SYSTEM SHALL faire fail-OPEN (repli verdict V1 allow) ; un judge lent NE DOIT JAMAIS bloquer la disponibilité.

3.3 Friction + escalade

  • R9 — WHEN le judge renvoie block (off_topic) ET le scope session est sous FRICTION_SESSION_THRESHOLD ET le scope user/IP est sous FRICTION_USER_THRESHOLD, THE SYSTEM SHALL enregistrer un strike FRICTION_WEIGHT_OFFTOPIC sur session ET user/IP, et renvoyer la réponse générée (soft-redirect — le section prompt recentre). Aucune émission de policy:off_topic.
  • R10 — WHEN le judge renvoie block (off_topic) ET le scope session atteint FRICTION_SESSION_THRESHOLD, THE SYSTEM SHALL hard-bloquer ce message off-topic (cool-down localisé policy:off_topic) au lieu de le soft-rediriger, POUR LE RESTE de la session. Les messages on-topic suivants de la même session SHALL continuer de répondre normalement.
  • R11 — WHEN le scope user (ou IP hashée si anon) atteint FRICTION_USER_THRESHOLD sur la fenêtre FRICTION_USER_TTL_MS, THE SYSTEM SHALL armer un cool-down global de FRICTION_COOLDOWN_MS ; WHILE ce cool-down est armé, TOUS les messages de cet user/IP (toutes sessions) SHALL recevoir le cool-down localisé.
  • R12 — THE SYSTEM SHALL utiliser le scope user (par userId) quand l'utilisateur est authentifié, et le scope ip (par sha256(ip)) quand anonyme. L'IP NE DOIT JAMAIS être stockée ni loggée en clair (RGPD).
  • R13 — WHILE GUARDRAIL_FRICTION_ENABLED=false, THE SYSTEM SHALL retomber sur le comportement legacy (judge inline qui hard-block tout off-topic) — kill-switch sans redeploy.

3.4 Fail-soft store

  • R14 — IF le GuardrailFrictionStore (Redis) est indisponible ou throw, THE SYSTEM SHALL traiter count() comme 0 (pas d'escalade), laisser le soft-redirect + la sécurité inline opérer, et NE JAMAIS fail-closed le chat ni renvoyer un 500 dû à la panne infra.
  • R15 — THE SYSTEM SHALL appliquer un TTL par scope : session FRICTION_SESSION_TTL_MS (déf 6h), user/IP FRICTION_USER_TTL_MS (déf 24h glissant). Le cool-down user/IP utilise un flag dédié à TTL FRICTION_COOLDOWN_MS.

3.5 Abort (best-effort, optimisation)

  • R16 — WHEN une escalade (R10/R11) est décidée pendant que la génération est en cours, THE SYSTEM SHALL, SI l'orchestrateur propage un AbortSignal externe, abort la génération pour ne payer qu'une fraction des tokens.
  • R17 — IF l'orchestrateur ne propage PAS d'AbortSignal externe (état actuel vérifié — voir design §AbortSignal), THE SYSTEM SHALL laisser la génération finir puis supprimer/remplacer la réponse par le cool-down. L'abort est une optimisation, pas une condition de correction (R16 et R17 produisent le MÊME résultat user-visible).

3.6 Sortie & message de cool-down

  • R18 — THE SYSTEM SHALL renvoyer le cool-down comme message assistant localisé (8 locales : en, fr, es, de, it, ja, zh, ar), honnête, SANS emoji unicode, avec citation policy:off_topic dans metadata.citations.

4. NFR (non-fonctionnels)

  • NFR-Latence-1 — Sur message on-topic ≥50c, la latence ajoutée par le judge SHALL être ≈ 0 (parallélisme). Le judge (~0,6–1,2s) finit avant la réponse (3–9s).
  • NFR-Latence-2 — Le socket ceiling chat (chatRouteSocketCeilingMs, chat-route.helpers.ts:39) NE DOIT PAS augmenter : le judge ne s'additionne plus en série (serialGuardrailBudgetMs, chat-route.helpers.ts:24-25). L'invariant R2 du I-OPS4 reste vrai ; le budget peut baisser. Vérifier honnêtement et documenter le delta.
  • NFR-Sécurité-1 — Les invariants §3.1 (R1–R4) sont testés explicitement en réel (test §7.8 du design).
  • NFR-Fail-soft-1 — Aucune panne du store de friction ne peut produire un 500 ni un fail-closed (test §7.6).
  • NFR-Coût-1 — Off-topic sous-seuil paie answer+judge (judge ~0,06¢ réel, compté 1¢ conservateur, llm-judge-guardrail.ts:89). Acceptable : off-topic = minorité.
  • NFR-RGPD-1 — IP jamais en clair (hash sha256), cohérent avec le pattern RedisRateLimitStore existant.
  • NFR-Observabilité-1 — Les fail-soft du store SHOULD être comptés (métrique/log) façon guardrailBudgetRedisFallbackTotal (guardrail-budget.ts:128).
  • NFR-Test-1 — Suite tests/ai/guardrail-v2-live.ai.test.ts prolongée (réel, no mock LLM) ; plancher MIN_TOTAL_AI_TESTS (scripts/sentinels/ai-tests-count.mjs:72) bumpé dans le même commit que les nouveaux it().

5. Critères d'acceptation

  1. Off-topic isolé < seuil → soft-redirect : réponse présente, PAS de policy:off_topic, recentrage par le section prompt. (R9)
  2. Off-topic répété ≥ FRICTION_SESSION_THRESHOLD (3×) → 3ᵉ message hard-block policy:off_topic + cool-down ; un on-topic ensuite répond normalement. (R10)
  3. Latence parallèle : message on-topic jugé → latence totale ≈ latence réponse seule (pas la somme answer+judge ; marge tolérante). (R6, NFR-Latence-1)
  4. Plancher user cross-session : strikes répartis sur 2 sessions du même userId → cool-down à FRICTION_USER_THRESHOLD. (R11, R12)
  5. Abort : judge bloque pendant la génération → génération avortée OU supprimée (pas de réponse complète user-visible ; cool-down rendu). (R16/R17)
  6. Fail-soft store : GuardrailFrictionStore qui throw → pas d'escalade, chat répond, pas de 500. (R14)
  7. Kill-switch : GUARDRAIL_FRICTION_ENABLED=false → comportement legacy (judge inline hard-block). (R13)
  8. Sécurité inchangée : injection/PII toujours hard-block immédiat + strike sécurité poids FRICTION_WEIGHT_SECURITY. (R1, R2, R4)
  9. Unitaires GuardrailFrictionStore : recordStrike (incrBy + TTL), count (0 si store down), reset, backends memory/redis, __setStoreForTest. (façon guardrail-budget tests)
  10. i18n : nouveau champ refocus/cool-down présent et non vide dans les 8 locales ; golden test guardrail-refusals.test.ts étendu. (R18)
  11. tsc 0, pnpm lint 0, lint:test-discipline 0, suite guardrail-v2-live + nouveaux tests verts en réel.

Design — Guardrail hybride par gravité

RUN_ID 2026-06-01-hybrid-gravity-guardrail — phase plan (UFR-022 fresh-context). Ancré au code RÉEL vérifié 2026-06-01 (path:line). Lit spec.md + docs/superpowers/specs/2026-06-01-hybrid-gravity-guardrail-design.md.

0. Constat de faisabilité AbortSignal (UFR-013 — vérifié, honnête)

Vérification du code réel :

  • LangChainChatOrchestrator.generate(input: OrchestratorInput) prend un seul argument (langchain.orchestrator.ts:290). Le port ChatOrchestrator.generate(input) aussi (chat-orchestrator.port.ts:99).
  • OrchestratorInput (chat-orchestrator.port.ts:11-95) n'a AUCUN champ signal. Aucun AbortSignal externe n'y entre.
  • L'orchestrateur construit son PROPRE AbortSignal.timeout() en interne, par section (langchain.orchestrator.ts:424 run: async (signal: AbortSignal) via le section-runner ; et chemin walk langchain.orchestrator.ts:493 AbortSignal.timeout(env.llm.totalBudgetMs)).
  • La couche modèle SUPPORTE bien un signal : ChatModel.invoke(messages, { signal }) (langchain-orchestrator-support.ts:82,85,114) et mergeInvokeOpts({ signal }, ...) (langchain.orchestrator.ts:65-72,245,495).

Conclusion honnête : l'abort externe n'est PAS supporté aujourd'hui au niveau du port. Le câbler exigerait : (a) ajouter signal?: AbortSignal à OrchestratorInput (ou un 2ᵉ arg à generate), (b) combiner ce signal externe avec le signal interne par section via AbortSignal.any([internalTimeoutSignal, externalSignal]) dans invokeSection (langchain.orchestrator.ts:243-247) ET generateWalk (langchain.orchestrator.ts:493-497).

Décision design (conforme spec R16/R17, design §4.1 « abort = optimisation ») :

  • Phase 1 (ce run) = SUPPRESS-only. On NE modifie PAS le port orchestrateur. On parallélise judge ‖ generate ; si une escalade est décidée, on ignore/remplace la réponse générée par le cool-down. Résultat user-visible identique à l'abort. Coût : on paie les tokens de la génération escaladée (minorité de cas, judge ~0,06¢, off-topic répété rare).
  • Abort réel = explicitement HORS périmètre de ce run, noté en tech-debt (OrchestratorInput.signal + AbortSignal.any). Le tenter ici toucherait le port partagé + walk path + tous les fakes de test → blast-radius large à J-6 pour un gain coût marginal. UFR-019 : ne pas sur-scoper.

⚠️ Pour l'éditeur : NE PAS prétendre câbler un abort. Implémenter le SUPPRESS. Si une tâche « abort » apparaît, elle est OPTIONNELLE et documentée comme tech-debt, pas requise pour les critères d'acceptation.

1. Vue d'ensemble du flux cible

État actuel (série) :

prepare() [pipeline]
  → evaluateInput(text)  // V1 + sidecar + JUDGE (tous inline, AVANT génération)  -- guardrail-evaluation.service.ts:122-212
      ├─ block → handleInputBlock → refused                                       -- :215-251
      └─ allow → ready
chat-message.service.postMessage:285 → orchestrator.generate(input)  // génération
  → commitResponse → evaluateOutput  // output guardrail

État cible (gravité) :

prepare() [pipeline]
  → securityGate(text)  // = evaluateInput SANS le judge : V1 + sidecar inline    -- evaluateInput allégé
      ├─ block (sécurité) → recordStrike(security, w_security) on session+user/IP
      │                    → check plancher user/IP
      │                    → handleInputBlock → refused
      └─ allow → ready  (+ flag « judge à faire en parallèle »)
chat-message.service.postMessage:
  → cool-down pré-check : si user/IP en cool-down armé (R11) → renvoyer cool-down (skip génération)
  → PARALLÈLE { semanticJudge(text, scope, frictionState)  ‖  orchestrator.generate(input) }   via Promise.allSettled
      ├─ judge allow|null  → réponse = answer généré
      └─ judge block(offtopic) :
            recordStrike(offtopic, w_offtopic) on session+user/IP
            escalate = (sessionCount ≥ S_soft) || (userCount ≥ U_floor)
            escalate ? réponse = cool-down (suppress answer)  : réponse = answer (soft-redirect)
  → commitResponse → evaluateOutput  (inchangé)

2. (a) Extraction du judge hors de evaluateInput

Fichier : guardrail-evaluation.service.ts

Aujourd'hui evaluateInput (:122-212) enchaîne : V1 keyword (:131), sidecar provider (:147), redaction log (:168-176), judge (:182 runLlmJudge), preClassified short-circuit (:196). Le judge est la seule partie à extraire.

  • evaluateInput allégé : garde V1 (:131-143) + sidecar (:147-163) + redaction (:165-176) + le retour allow (avec redactedText). On retire le bloc judge :178-194. Sémantique : evaluateInput ne fait plus QUE la sécurité inline (gravity band security). Son contrat de retour InputGuardrailResult (guardrail-evaluation.types.ts:9-20) est inchangé.
  • Nouvelle méthode evaluateInputSemantic(text, scope) sur GuardrailEvaluationService : encapsule l'appel judge (le code retiré :178-194 : runLlmJudge(text, this.judgeDeps(), scope)), retourne un résultat structuré { verdict: 'allow' | 'offtopic'; reason?; } (ne hard-block pas elle-même ; c'est l'orchestration qui décide soft vs hard selon la friction). Gère le gating longueur ≥ judgeMinMessageLength et le llmJudgeEnabled (sinon allow direct — fail-OPEN legacy).
  • Audit : le block judge enregistrait logBlock (:185-192). Dans le modèle cible, le logBlock n'est émis que si l'off-topic escalade vers hard-block (sinon soft-redirect = pas un block, pas d'entrée chaîne d'audit). À câbler dans l'orchestration (voir §3) ou via un paramètre escalated passé à une méthode handleOffTopicEscalation.
  • Kill-switch legacy (R13) : quand GUARDRAIL_FRICTION_ENABLED=false, evaluateInput RÉINTÈGRE le judge inline (comportement legacy hard-block). Implémentation : garder le bloc judge dans evaluateInput gardé par if (!env.guardrails.frictionEnabled), et l'orchestration n'appelle evaluateInputSemantic en parallèle QUE si frictionEnabled. Ainsi un seul code path par mode, pas de double évaluation.

Risque : evaluateInput est appelé depuis prepare-message.pipeline.ts:283. Impact analysis à lancer en GREEN (gitnexus_impact evaluateInput) avant édition. Le contrat de retour ne change pas → callers inchangés hors pipeline.

3. (b) Restructuration prepare-message.pipeline.ts + chat-message.service.ts

Interface (spec §4.2 du design)

export type FrictionScope =
  | { kind: 'session'; sessionId: string }
  | { kind: 'user'; userId: number }
  | { kind: 'ip'; ipHash: string };

export interface IGuardrailFrictionStore {
  recordStrike(scope: FrictionScope, weight: number): Promise<void>; // incrBy + TTL
  count(scope: FrictionScope): Promise<number>;                      // 0 si store down (FAIL-SOFT)
  armCoolDown(scope: FrictionScope): Promise<void>;                  // SET flag TTL = FRICTION_COOLDOWN_MS
  isCoolingDown(scope: FrictionScope): Promise<boolean>;             // false si store down (FAIL-SOFT)
  reset(scope: FrictionScope): Promise<void>;                        // tests / cron défensif
}

Différence clé vs budget : FAIL-SOFT (renvoie 0/false sur store down) — l'OPPOSÉ du budget qui est FAIL-CLOSED (Infinity sur down, guardrail-budget.ts:120-142). Bien recopier la STRUCTURE, mais INVERSER la politique de panne (spec R14). Pas de ping-gate fail-closed : sur erreur Redis, catch → 0/false + métrique + log warn (façon guardrailBudgetRedisFallbackTotal.inc() :128, créer un guardrailFrictionRedisFallbackTotal).

5. (d) Politique d'escalade 2 niveaux

Implémentée dans l'orchestration (§3 step 3), pure logique sur les counts :

  • Niveau 1 — session par-message off-topic : escalateSession = sessionCount >= env.guardrails.frictionSessionThreshold. Si vrai ET judge=offtopic → hard-block CE message (cool-down policy:off_topic). On-topic continue (le judge ne bloque pas l'on-topic → pas d'escalade).
  • Niveau 2 — cool-down user/IP global : après recordStrike user/IP, userCount = count(user|ip). Si userCount >= env.guardrails.frictionUserThreshold → armCoolDown(user|ip). Le pré-check (§3 step 1) isCoolingDown court-circuite les messages suivants pendant FRICTION_COOLDOWN_MS.
  • Strike sécurité (R2) : poids frictionWeightSecurity (2) sur session+user/IP → un spammeur d'injections atteint U_floor plus vite (2 vs 1) → cool-down dur. Câblé dans le pipeline securityGate block branch (prepare-message.pipeline.ts:291-302).

SUPPRESS-only ce run. Pas de modification du port. La génération escaladée est jetée/remplacée. Tâche abort = tech-debt optionnelle (non requise par les critères d'acceptation 5 : « avortée OU supprimée »).
