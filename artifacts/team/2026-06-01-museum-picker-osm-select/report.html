Spec — Museum picker : exposer l'id DB des entrées locales + rendre les entrées OSM sélectionnables

Run: 2026-06-01-museum-picker-osm-select Author: architect (opus-4.8) · Created: 2026-06-01 Phase: spec (UFR-022 fresh-context) — spec.md uniquement, pas de design/tasks/code.

EARS-format requirements. Source of truth for this run. Editor + verifier read this before touching code.

1. Problem statement

Le museum-picker (museum-frontend/features/museum/ui/MuseumPickerScreen.tsx) affiche toujours « Aucun musée trouvé » même quand GET /api/museums/search renvoie 200 OK avec des lignes réelles (géoloc accordée incluse). Cause racine vérifiée (Read 2026-06-01) : le contrat FE↔BE est cassé sur le champ id.

  • BE : l'interface SearchMuseumEntry (museum-backend/src/modules/museum/useCase/search/searchMuseums.useCase.ts:30-46) ne déclare aucun champ id. La projection des entrées locales perd l'id DB : LocalMuseumWithCoords (l.62-68) et fetchLocalMuseumsWithCoords (l.70-94) lisent repository.findAll() (renvoie des Museum[] portant id) mais ne propagent pas m.id ; mergeAndDedupe (l.196-239) construit les entrées source:'local' (l.205-215) et source:'osm' (l.218-235) sans id. Le contrat OpenAPI GET /api/museums/search (museum-backend/openapi/openapi.json:5856-5893) n'expose pas non plus d'id (absent de required et de properties).
  • FE : toPickable() (MuseumPickerScreen.tsx:53-62) retourne null dès que l'entrée n'a pas d'id entier > 0 (l.54-55). Les listes nearby (l.136) et search (l.176) appliquent museums.map(toPickable).filter(m !== null). Le type FE MuseumSearchEntry (museum-frontend/features/museum/infrastructure/museumApi.ts:27) est généré depuis l'OpenAPI (commenté « includes OSM results without id/slug ») → 'id' in entry est statiquement faux → 100 % des lignes sont filtrées, d'où l'écran vide systématique.
  • handlePick (l.193-199) fait addFavourite(museum.id) puis onSelect({ id: number, name }) ; le contrat onSelect (l.40) exige id: number.

Décision produit TRANCHÉE par l'utilisateur (Option B) — encodée ci-dessous, NON rediscutée :

  1. Les entrées source:'local' doivent porter leur id DB → sélection démarre une conversation contexte-musée (museumId numérique).
  2. Les entrées source:'osm' (sans id DB) doivent être sélectionnables → sélection démarre une conversation générique sans contexte musée (museumId absent/null). POST /api/chat/sessions accepte déjà museumId optionnel (museum-backend/src/modules/chat/adapters/primary/http/routes/chat-session.route.ts:116).

2. Scope

In scope :

  • BE : ajouter un champ id optionnel (présent uniquement pour source:'local') à SearchMuseumEntry ; propager m.id à travers LocalMuseumWithCoords → fetchLocalMuseumsWithCoords → mergeAndDedupe (branche local uniquement) ; mettre à jour le contrat OpenAPI search (openapi/openapi.json + tout schéma Zod de réponse si présent) pour exposer cet id optionnel.
  • FE : régénérer les types OpenAPI (npm run generate:openapi-types) ; faire en sorte que toPickable accepte les entrées OSM (sans id) ; introduire un modèle PickableMuseum qui distingue local (museumId numérique) vs osm (pas de museumId) ; élargir le contrat onSelect pour accepter une sélection LOCAL ou OSM ; adapter le flow de sélection (handlePick) ; adapter les appelants (museum-frontend/app/(stack)/museums-picker.tsx:23-35 → startConversation).
  • Favoris OSM : décision Option B → OSM non favoritable (pas d'ajout aux favoris au tap pour une entrée OSM ; les favoris restent keyés par id DB numérique).
  • E2E : flow Maestro museum-frontend/.maestro/museum-picker-flow.yaml runtime-vert pour la sélection d'une ligne LOCALE + navigation vers le chat.

Out of scope :

  • Toute migration de schéma DB (changement de projection, pas de schéma — Museum.id existe déjà ; aucune colonne créée/modifiée).
  • Tout feature flag (UFR-015 — pre-launch, pas de flag).
  • Favoritage des entrées OSM via une clé alternative (slug/coords) — explicitement écarté par Option B.
  • La navigation/parcours guidé V2.
  • Réécriture du moteur de dedup mergeAndDedupe (on ne touche QUE la propagation d'id sur la branche local).
  • Le tri/ordre des résultats search (BE pré-trie déjà par distance ; inchangé).

Design — Museum picker : exposer l'id DB des entrées locales + rendre les entrées OSM sélectionnables

Run: 2026-06-01-museum-picker-osm-select · Author: architect (opus-4.8) · Created: 2026-06-01 Companion to: spec.md · discovery.md

Architecture decisions for this run. Editor reads this and ONLY this for the "what to build" view.

1. Overview

Le contrat GET /api/museums/search perd l'id DB des musées locaux (la projection SearchMuseumEntry n'a aucun champ id), et le FE toPickable() filtre toute entrée sans id entier > 0 → 100 % des lignes droppées → écran « Aucun musée trouvé » systématique (cause racine vérifiée Read 2026-06-01, cf. discovery.md). On corrige en deux couches additives, non-breaking :

  1. BE — propager m.id de repository.findAll() jusqu'à la projection, n'exposer id que sur les entrées source:'local' (les entrées source:'osm' n'ont aucune ligne DB → pas d'id). On ajoute id au schéma OpenAPI search en optional (jamais dans required).
  2. FE — régénérer les types (id?: number apparaît sur MuseumSearchEntry), remplacer le modèle PickableMuseum plat par une union discriminée local | osm, faire en sorte que toPickable mappe les deux kinds (et garde le filtrage null pour le chemin favoris/directory), élargir onSelect en union discriminée, et router la sélection : LOCAL → conversation contexte-musée (museumId), OSM → conversation générique (sans museumId), OSM non favoritable.

KISS : on ne touche QUE la propagation d'id (branche local), le schéma de réponse, le picker FE et son seul appelant. Pas de refonte du moteur mergeAndDedupe, pas de migration DB (la colonne Museum.id existe), pas de feature flag (UFR-015, pre-launch).

9. Decisions

  • D1 — id optionnel local-only plutôt qu'un id|null ou un champ requis. required ne doit pas contenir id (les entrées osm n'ont pas de ligne DB). Optionnel = additive non-breaking, et l'invariant 'id' in entry === (source==='local') est porté structurellement par les deux chemins de construction distincts (local = spread de LocalMuseumWithCoords ; osm = littéral explicite). On évite id: number | null car null brouillerait le discriminant et obligerait un garde supplémentaire FE.
  • D2 (résout Q1) — onSelect = union discriminée { kind:'local' } | { kind:'osm' }, pas un objet plat.
  onSelect: (
    museum:
      | { kind: 'local'; museumId: number; name: string }
      | { kind: 'osm'; name: string; latitude: number; longitude: number }
  ) => void;

Justification : le caller (museums-picker.tsx) doit router LOCAL → startConversation({ museumMode:true, museumId }) vs OSM → startConversation (générique). Un objet plat { museumId?, latitude?, longitude? } forcerait le caller à un if (museumId != null) ambigu (rien n'empêcherait museumId absent ET coords absentes). L'union discriminée rend le switch (kind) exhaustif et type-safe (le compilateur garantit que la branche local a museumId, la branche osm a les coords). Conforme au domaine : LOCAL et OSM sont deux états mutuellement exclusifs.

  • D3 (résout Q2) — testID des lignes OSM dérivé d'une identité OSM stable, pas de l'index. item.id est undefined pour OSM → museum-picker-row-undefined collisionnerait. On dérive osmKey = osm:${latitude.toFixed(5)}:${longitude.toFixed(5)}` (lat/lng arrondis à 5 décimales ≈ 1 m, stable entre renders, unique par POI ; mergeAndDedupe` collapse déjà les doublons ≤100 m). testID :
    • LOCAL : museum-picker-row-${museumId} (inchangé, rétro-compatible avec le flow Maestro museum-picker-row-<id>).
    • OSM : museum-picker-row-osm-${osmKey} (déterministe, distinct, jamais undefined).

keyExtractor FlatList utilise la même clé (kind==='local' ? local-${museumId} : osmKey). On préfère lat/lng à l'index car l'index n'est pas stable au re-tri/refresh (R13 exige « stable, unique »). Coordonnées et non hash du nom : le nom peut dupliquer (« Musée d'art moderne » dans deux villes), les coords non.

  • D4 (résout Q3) — sélection OSM = conversation générique SANS museumId ; le nom n'est passé QUE comme libellé d'affichage, pas comme rattachement. Option B dit « sans contexte musée ». Le caller appelle startConversation({ skipSettings:true }) sans museumId ni museumMode ni museumName musée. Décision : ne PAS passer museumName (qui, côté useStartConversation, est destiné au contexte musée et s'affiche comme en-tête de session musée → trompeur sans museumId). Si un libellé du POI est souhaité plus tard, ce sera un initialPrompt/titre dédié — hors scope de ce run (YAGNI). → OSM ouvre une conversation propre, vide de contexte.
  • D5 — OSM non favoritable (Option B, déjà tranché spec §2). handlePick ne fait addFavourite que pour kind==='local'. Les favoris restent keyés par id DB numérique (storage inchangé). Le chemin favoris/directory (toPickable sur MuseumDirectoryEntry) garde son filtrage null (R8) car les favoris exigent un id.
  • D6 — toPickable reste une fonction pure exportée (ou testée via le screen). Pour permettre un unit test direct (R6/R7/R8), exporter toPickable depuis MuseumPickerScreen.tsx (named export) si pas déjà testable autrement. Signature : (entry: MuseumSearchEntry | MuseumDirectoryEntry) => PickableMuseum | null. Discrimination : si source === 'osm' (présent sur MuseumSearchEntry) → kind osm (pas besoin d'id) ; sinon exiger id entier > 0 → kind local ; sinon null. Les entrées directory (pas de source) tombent dans le chemin local et exigent id → null si absent (préserve R8).
