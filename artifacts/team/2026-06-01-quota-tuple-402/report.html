Spec — Fix quota free-tier 402 jamais émis (TypeORM UPDATE…RETURNING tuple mal lu)

Run: 2026-06-01-quota-tuple-402 Author: architect (opus-4.8) · Created: 2026-06-01 Phase: spec (UFR-022 fresh-context) — spec.md uniquement, ni design ni code.

EARS-format requirements. Source of truth for this run. Editor + reviewer read this before touching code.

1. Problem statement

Le cap mensuel de sessions pour les comptes free-tier (FREE_TIER_MONTHLY_SESSION_LIMIT, défaut 3, museum-backend/src/config/env.ts:279) ne bloque jamais. Un utilisateur free au-delà de sa limite peut continuer à créer des sessions de chat : POST /api/chat/sessions renvoie 201 au lieu de 402 QUOTA_EXCEEDED, et sessions_month_count n'est même pas incrémenté. Reproduit live le 2026-06-01 (user 38, free, count=3/limit=3 → 201, count reste 3, aucun log quota_check_hit_limit).

Cause racine confirmée code + runtime (discovery.md) : PgMonthlyQuotaRepo.tryConsume (museum-backend/src/shared/middleware/monthly-session-quota.repo.pg.ts:33-65) appelle this.dataSource.query("UPDATE …users… RETURNING …") (l.40-57) puis traite le résultat comme un tableau de rows (result.length === 0 l.59 ; result[0] lu comme une row l.60). Or TypeORM 0.3.28 renvoie pour un UPDATE…RETURNING un tuple [rows, affectedCount], pas rows[]. Conséquences observées en runtime (logs DBG_TRYCONSUME_RAW) :

  • quota épuisé (0 row mise à jour) : result = [[], 0] → result.length === 2 (jamais 0) → garde l.59

non déclenchée → result[0] = [] (tableau vide) traité comme row → row.sessions_month_count = undefined, new Date(undefined) = Invalid Date → objet truthy renvoyé → middleware if (consumed) next() (monthly-session-quota.middleware.ts:169-172) → 201.

  • 1 row : result = [[{…}], 1] → result[0] = le tableau de rows (pas la row) → row.sessions_month_count

= undefined également.

tryConsume ne renvoie donc jamais null (length toujours 2) → le quota n'a jamais bloqué dans cet environnement. Le SQL lui-même est correct (la clause WHERE … sessions_month_count < limit refuse bien 0 rows à count=limit, testé manuel UPDATE 0).

Design — Fix quota free-tier 402 jamais émis (TypeORM UPDATE…RETURNING tuple mal lu)

Run: 2026-06-01-quota-tuple-402 Author: architect (opus-4.8) · Phase: plan (UFR-022 fresh-context, zéro mémoire de la phase spec — spec.md/discovery.md lus du disque) Reads: spec.md (7 requirements R1-R6, 7 AC), discovery.md (cause racine confirmée code + runtime)

1. Overview

Bug runtime confirmé : PgMonthlyQuotaRepo.tryConsume (museum-backend/src/shared/middleware/monthly-session-quota.repo.pg.ts:33-65) lit le résultat de dataSource.query("UPDATE … RETURNING …") comme un tableau de rows (result.length === 0 l.59, result[0] comme row l.60). Or TypeORM 0.3.28 retourne pour un UPDATE…RETURNING un tuple [rows, affectedCount] (vérifié runtime : [[], 0] quand 0 row, [[{…}], 1] quand 1 row). Conséquence : result.length vaut toujours 2 → la garde === 0 ne tire jamais → tryConsume ne retourne jamais null → le cap free-tier ne bloque jamais (201 au lieu de 402, compteur non incrémenté).

Fix = correction de parsing en mémoire dans le SEUL adapter secondary tryConsume. Le SQL atomique (single UPDATE…WHERE…RETURNING) est correct et NE change PAS (préserve N13/R11, row-lock anti-race). Le middleware, la route, l'entité User, config/env.ts ne changent pas (spec §2 Out of scope, confirmé code).

Pas de migration (logique applicative pure). Pas de feature flag (UFR-015). Pas de changement du port MonthlyQuotaRepo (monthly-session-quota.middleware.ts:30-37) : signature tryConsume(...) → { sessionsMonthCount, sessionsMonthStart } | null inchangée.

6. Décision Q4 — forme du fix (tranchée)

Décision : garde défensive tuple-vs-flat, simple et explicite (PAS une heuristique opaque).

Forme retenue (pseudo-code ; l'éditeur green le concrétise) :

const result: unknown = await this.dataSource.query(`UPDATE … RETURNING …`, [...]);

// TypeORM 0.3.28 : UPDATE…RETURNING → tuple [rows, affectedCount] ; SELECT → rows[].
// On détecte le tuple via Array.isArray(result[0]) (le 1er élément est le tableau de rows).
// Sinon on traite `result` comme rows[] (robustesse future bump TypeORM v1.0, cf. CLAUDE.md §Dependency Monitoring).
const rows: ReadonlyArray<{ sessions_month_count: number; sessions_month_start: string }> =
  Array.isArray(result) && Array.isArray(result[0])
    ? (result[0] as …[])      // forme tuple observée : [[…rows], affected]
    : ((result as …[]) ?? []); // forme plate (SELECT-like / futur driver)

if (rows.length === 0) return null;            // 0 row updatée ⇒ quota épuisé ⇒ null ⇒ 402
const row = rows[0];
return {
  sessionsMonthCount: row.sessions_month_count,
  sessionsMonthStart: new Date(row.sessions_month_start),
};

Justification (pesée explicite spec §8 Q4) :

  • Pour la garde défensive : robustesse au bump TypeORM 0.3.28 → v1.0 (repo 0.3.x archivé, cf. CLAUDE.md

§Dependency Monitoring) où la forme de retour de query() pour les statements modifiants pourrait s'aligner sur rows[]. La garde Array.isArray(result[0]) distingue sans ambiguïté tuple [[…], n] de plat [{…}, {…}] : un tuple a son 1er élément = tableau ; une liste de rows a son 1er élément = objet row.

  • Contre le risque de masquage de régression (spec §8 Q4 "ne doit pas masquer une vraie régression") :

la garde NE fabrique JAMAIS de row truthy à partir de rien. Les DEUX branches convergent sur la même invariante rows.length === 0 ⇒ null. C'est précisément l'invariante que le bug actuel viole (il retournait truthy sur [[], 0]). Le fix la restaure quelle que soit la forme.

  • Contre la complexité (KISS) : 1 ternaire lisible + 1 commentaire qui cite la forme observée. Pas

d'heuristique « intelligente » (pas de sniffing de affectedCount, pas de réflexion sur le type de statement). Le test (T-RED) couvre les DEUX formes → la garde n'est pas du code mort non-testé.

Alternative écartée (strict-à-la-forme-observée) : const [rows] = result; if (rows.length === 0) return null;. Plus court, mais casse silencieusement si un futur bump renvoie rows[] (alors rows = 1ère row, rows.length = nombre de clés → comportement faux). Coût de la garde = 1 Array.isArray ⇒ on prend la robustesse.

Faits VÉRIFIÉS par le dispatcher (Read/Grep + reproduction runtime live le 2026-06-01). Backend dev Docker up: localhost:3000 (NODE_ENV=production, base museumAI, nodemon ts-node sur src/). Compte e2e: e2e-test-login@test.musaium.dev / TestPassword123! (userId 38, free, count=3, start=2026-06-01). Re-vérifie les path:line en spec, mais ils tiennent au 2026-06-01.

Symptôme

Un utilisateur free-tier au-delà de la limite mensuelle (FREE_TIER_MONTHLY_SESSION_LIMIT=3, config/env.ts:279) peut continuer à créer des sessions : POST /api/chat/sessions renvoie 201 au lieu de 402 QUOTA_EXCEEDED, et le compteur sessions_month_count n'est même pas incrémenté. Reproduit live : user 38 à count=3 → POST → 201, count reste 3, aucun log quota_check_hit_limit.

Hypothèses ÉLIMINÉES (ne pas y revenir)

  • « repo non câblé » (handoff initial) : FAUX. setMonthlyQuotaRepo(new PgMonthlyQuotaRepo(AppDataSource))

est appelé inconditionnellement au boot (museum-backend/src/index.ts:565, après AppDataSource.initialize()). 0 log monthly_quota_repo_unwired dans le conteneur.

  • !user?.id : FAUX. claims.id = Number(decoded.sub) (token-jwt.service.ts:92) → req.user.id=38 (number, log userId:38).
  • !row || premium : FAUX (instrumentation runtime : loadUser renvoie {id:38,tier:free,count:3,start:2026-06-01}).
  • middleware non monté / stale : FAUX (route chat-session.route.ts:107 + compilé identique ; nodemon sur src vivant).
  • limite haute : FAUX (env conteneur FREE_TIER unset → défaut 3 ; count ne s'incrémente jamais).

Cause racine (CONFIRMÉE par instrumentation runtime)

Fichier : museum-backend/src/shared/middleware/monthly-session-quota.repo.pg.ts, méthode tryConsume (l.33-65).

tryConsume fait :

const result = await this.dataSource.query(`UPDATE "users" SET ... WHERE ... RETURNING ...`, [...]);
if (result.length === 0) return null;   // <-- BUG
const row = result[0];                  // <-- BUG
return { sessionsMonthCount: row.sessions_month_count, sessionsMonthStart: new Date(row.sessions_month_start) };

dataSource.query() (TypeORM 0.3.28) pour un UPDATE…RETURNING renvoie un tuple [rows, affectedCount], PAS le tableau de rows directement (contrairement à un SELECT). Preuve runtime capturée (logs DBG_TRYCONSUME_RAW) :

  • quota épuisé (0 match) : result = [[], 0] → result.length === 2 (jamais 0) → garde NON déclenché →

result[0] = [] (tableau vide) traité comme row → row.sessions_month_count = undefined, new Date(undefined) = Invalid Date → renvoie un objet truthy → middleware if (consumed) next() → 201.

  • 1 match : result = [[{sessions_month_count:1, sessions_month_start:'...'}], 1] → result[0] = le TABLEAU de rows.

Conséquence : tryConsume ne renvoie JAMAIS null (length toujours 2) → le quota n'a jamais bloqué dans cet environnement. Le SQL lui-même est correct (testé manuel : UPDATE 0 à count=3/limit=3 — il refuse bien).

Fix attendu (minimal, à spécifier/coder en green)

Lire le tableau de rows depuis result[0] (pas result directement). Valider contre les DEUX formes observées :

  • const [rows] = result; if (!Array.isArray(rows) || rows.length === 0) return null; const row = rows[0];

→ [[],0]→rows=[]→null→402 ✓ ; [[{…}],1]→rows=[{…}]→consume ✓. Considérer une garde défensive (forme tuple vs forme plate) si on veut être robuste aux variations TypeORM, mais la forme [rows, affected] est celle observée en prod-like ici.

Gotcha généralisable (à documenter — CLAUDE.md § Pièges connus)

TypeORM dataSource.query() : un SELECT renvoie rows[] ; un INSERT/UPDATE/DELETE…RETURNING renvoie [rows, affectedCount]. Tout code qui fait .query('UPDATE…RETURNING') et lit result[0] comme une row est buggé. Auditer les autres usages de raw query() avec RETURNING (hors scope de ce run, mais à signaler).
