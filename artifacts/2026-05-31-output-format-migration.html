Doctrine & migration

« Output format » — ce que l'on pourrait changer

La nouvelle règle Read = HTML, Edit = markdown ajoutée à CLAUDE.md ne change rien rétroactivement. Ce rapport recense les surfaces qui produisent des artefacts « à lire » et où la doctrine s'appliquera désormais.

InnovMind / Musaium 2026-05-31 Source : CLAUDE.md § Output format Premier artefact conforme à la doctrine

En une ligne. La doctrine est forward-looking : elle s'applique aux nouveaux artefacts à lire, pas à une migration de masse des ~116 fichiers existants.

Action la plus rentable. Patcher les 5 skills qui génèrent des rapports (/recap, /security-scan, /test-routes, /test-writer, /team) pour qu'ils émettent du HTML self-contained dans artifacts/ — le reste suit naturellement.

Ne pas toucher. Specs, configs, READMEs, spec/design/tasks.md, learning/, docs/ — matériel d'édition, reste markdown.

1La décision : Read vs Edit

La doctrine repose sur une seule question : « est-ce que quelqu'un va le lire, ou est-ce que quelqu'un va l'éditer ? » La réponse tranche le format. Voici la classification appliquée au repo.

CatégorieExemples dans le repoVerdict
Rapports d'exécution SDLC code-review.json, verify.json, security.json, STORY.md → HTML
Audits ponctuels & QA audit-state/**/REPORT.md, QA-NOTES.md → HTML
Récaps / status / briefings sortie /recap, /test-routes, /test-writer → HTML
Specs & plans éditoriaux spec.md, design.md, tasks.md (Spec Kit) markdown
Notes & knowledge base learning/**, docs/**, LESSONS.md markdown
Données brutes / signaux state.json, *.patch, red-test-manifest.json ni l'un ni l'autre

Cas limite — team-knowledge/lessons/*.md : auto-capturés et « lus », mais consommés surtout par des agents (machine-read) et adossés à LESSONS.md human-edited. Faible valeur HTML ; je les laisse en markdown sauf demande contraire.

2Les surfaces à modifier (les 5 skills)

Le vrai levier n'est pas de convertir des fichiers, c'est de changer ce que les générateurs émettent désormais. Chaque skill ci-dessous écrit aujourd'hui du markdown ou du JSON destiné à la lecture.

/recap — récap quotidien

  • Aujourd'hui : markdown libre, affiché en chat, non persisté (.claude/skills/recap/SKILL.md).
  • Changement : optionnellement écrire un artifacts/<date>-recap.html self-contained quand l'utilisateur veut le garder / partager.

/security-scan — audit sécurité léger

  • Aujourd'hui : rapport JSON structuré (findings + verdict).
  • Changement : conserver le JSON (donnée, machine-read) et rendre un artifacts/<date>-security-scan.html lisible pour les humains qui consultent le verdict.

/test-routes & /test-writer — rapports de validation

  • Aujourd'hui : markdown (table de coverage / synthèse).
  • Changement : émettre la version « à lire » en HTML self-contained ; garder le markdown si l'output sert d'entrée à une autre étape.

/team — orchestrateur SDLC (impact principal)

  • Aujourd'hui : STORY.md + code-review.json + verify.json + security.json par run, dans .claude/skills/team/team-reports/<RUN_ID>/.
  • Changement : ajouter une étape de rendu HTML (Step 9 finalize) qui agrège STORY + les JSON en un seul report.html self-contained — c'est l'artefact « à lire » pour l'humain ; les JSON restent la source machine.
Recommandation transverse. Plutôt qu'un template HTML copié dans 5 skills, créer un helper unique (ex. scripts/render-artifact.mjs : markdown/JSON → HTML self-contained avec le CSS de ce rapport inliné). Une seule source de style → la doctrine « restrained design » reste cohérente partout.

5Honnêteté & limites

Ce rapport est forward-looking, pas un plan de migration de masse. Convertir 116 fichiers d'archive en HTML brûlerait des tokens pour des artefacts que personne ne relit. La doctrine s'applique aux nouveaux artefacts ; l'historique reste markdown (git le garde de toute façon).
  • Les comptes (§3) sont vérifiés par find au 2026-05-31, pas estimés — un inventaire d'agent antérieur donnait des chiffres différents, j'ai pris les vrais.
  • Les classifications Read/Edit du §1 sont mon jugement appliqué à la règle, pas une vérité absolue : lessons/ en particulier est un cas limite assumé.
  • Aucun skill n'a encore été modifié — ce rapport décrit ce qu'on pourrait changer. Le seul changement réellement appliqué à ce stade est l'ajout de la section ## Output format à CLAUDE.md.
  • Le helper render-artifact.mjs n'existe pas encore ; c'est une proposition, pas un fait.
