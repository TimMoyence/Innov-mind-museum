Élagage des agents /team — analyse pour décision

Musaium · 2026-05-31 · fondé sur l'audit 360 (raw/07, raw/11) + lecture des 6 agents + données d'usage réelles sur 77 runs. Aucune suppression effectuée — décision en attente.

1. Le constat qui cadre tout

Deux lectures coexistent et ne se contredisent pas :

  • Les 9 agents sont disjoints entre eux. Lecture ligne-à-ligne : verifier = gates mécaniques (lint/tsc/test/build/mutation), security = threat-model (OWASP LLM/API, SAST, contrats auth), reviewer = sémantique (spec↔impl, KISS/DRY, a11y, frozen-test cross-check). Pas de doublon mutuel significatif.
  • L'appareil global duplique ~80 % du natif. L'audit (raw/11 §1.5) chiffre : « 5-phase spec→plan→red→green→review ≈ brainstorming + writing-plans + TDD + subagent-driven-dev + requesting-code-review → ~80 % conceptuel ». La valeur nette non-native = frozen-test + lib-docs/LESSONS.

Donc l'élagage ne se justifie pas par « ces agents font la même chose », mais par « cette couche refait du natif » et « cette machinerie n'a jamais servi en 77 runs ».

3. Recommandation par agent

FUSIONNER doc-fetcher + doc-curator → doc-cache

Le split en deux agents n'a qu'une justification : préserver le context-window des agents aval (ils lisent le PATTERNS.md de ~400 lignes, pas le snapshot de 50 KB). Mais cet argument concerne l'aval, pas fetcher/curator eux-mêmes — un seul agent peut fetch puis curer dans son propre contexte et n'émettre que PATTERNS.md. Le split ajoute 1 spawn + 1 handoff sans bénéfice aval.

Gain : −1 agent, −1 handoff. Conservé : la valeur lib-docs que l'audit dit explicitement de garder (P0). Ce n'est pas la suppression de lib-docs — c'est dé-sur-découper sa production.

SUPPRIMER learning-curator (→ V2 si jamais)

0 amendement produit en 77 runs. La boucle d'apprentissage capture 9 lessons mais n'a jamais rien proposé. C'est de la machinerie spéculative (T2.1 KR4) non amortie à J-7 du launch. Les 9 lessons restent lisibles et exploitables manuellement ; l'agent dédié qui n'a rien sorti est du pur overhead.

Gain : −1 agent (162 lignes) + le hook post-complete-lesson-capture.sh devient optionnel. N'affecte ni frozen-test ni lib-docs. Audit raw/11 §4 : « questionner … learning-curator face au coût solo ».

FUSIONNER verifier → reviewer + hooks existants

Le cœur de verifier = exécuter pnpm lint && tsc && test && build + ratchet + mutation. Ce sont des commandes déterministes, déjà couvertes par les hooks post-edit-lint.sh, post-edit-typecheck.sh, pre-complete-verify.sh + la CI. Un agent LLM pour lancer des commandes déterministes est le cas d'école du « ça devrait être un hook, pas un agent ». Sa seule part de jugement (scope-boundary, spot-check archi) recoupe le reviewer.

Gain : −1 agent (193 lignes). Conservé : le gate pre-complete-verify.sh (déterministe) + scope-boundary/spot-check repliés dans reviewer.

GARDER security — et pousser-back sur l'audit

L'audit suggère de « fusionner verifier/security dans le reviewer two-stage natif ». Je recommande de NE PAS fusionner security. Il est disjoint et domain-specific : OWASP LLM Top-10 (injection, untrusted_content wrapping, Presidio PII), contrats auth (rotation refresh-token, nonce social ID-token, rate-limit 30/min), SAST (semgrep/CodeQL), supply-chain. Pour une app IA culturelle avec surface d'injection LLM, cette profondeur n'est pas réductible à un reviewer générique sans perte.

GARDER reviewer, architect, editor, documenter

reviewer = valeur sémantique + frozen-test cross-check. architect/editor = le red→green fresh-context (l'audit dit P0-garder). documenter = scope d'écriture disjoint.
