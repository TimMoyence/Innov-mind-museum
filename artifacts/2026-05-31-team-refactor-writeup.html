1. Contexte & mandat

L'audit 360 du 2026-05-31 (audit-state/2026-05-31-cartographie-360/raw/07-machinerie-team.md et raw/11-team-vs-anthropic.md) a établi que le skill /team a une plus-value réelle mais marginale : ~80 % du pipeline 5-phase duplique le plugin officiel superpowers + la sémantique native des subagents Claude Code. La valeur non-native se réduit à deux mécaniques : le frozen-test et le lib-docs versionné.

La session avait quatre objectifs, sous trois contraintes : exempt /team (ce travail méta ne passe pas lui-même par le pipeline /team), UFR-013 (honnêteté — vérifier avant d'affirmer), et présenter les options d'élagage avant d'agir.

  1. lib-docs versionné — vérifier que le contrat sha256 tient réellement, qu'il n'est pas honor-system, et que le câblage des hooks existe.
  2. Drift modèle — les frontmatter d'agents pinnaient claude-opus-4-7 alors que l'environnement tourne Opus 4.8.
  3. Élagage — décision utilisateur sur doc-fetcher / doc-curator / learning-curator (sur-découpage ?) et verifier / security (chevauchent le two-stage review ?).
  4. Décision stratégique — plan d'alignement /team ↔ superpowers.

2. Pré-requis vérifié (session #3)

Avant tout, vérification que le câblage du hook frozen-test (traité en session #3) était bien en place, pour éviter une collision :

  • post-edit-green-test-freeze.sh est bien câblé dans .claude/settings.json (PostToolUse, matcher Edit|Write).
  • lint-on-edit.sh ne fait pas un skip glob *.test.* naïf : il skippe les fichiers listés dans un red-test-manifest.json actif — donc il ne gèle que les tests réellement manifestés par une phase red. Supérieur au glob (les tests non-gelés restent formatés). Non retouché.
Insight : deux hooks PostToolUse sur le même fichier doivent se coordonner. Un prettier --fix silencieux sur un test gelé changerait son sha256 et contournerait mécaniquement le frozen-test. Le skip manifest-based est ce qui rend cette coordination correcte.

3. Objectif 1 — lib-docs : fermer l'honor-system

Le problème

Le câblage des hooks était présent (pre-phase-doc-freshness.sh en Step 4.5, pre-phase-doc-reference-check.sh en Step 6 ; les deux self-tests passaient). Mais le contrat sha256 était honor-system : aucun hook ne re-hashait les fichiers sur disque contre l'INDEX.json.

Preuve, pas supposition : sur les 107 libs de l'INDEX, 43 hash ne correspondaient pas à leurs fichiers locaux (20 snapshots + 23 PATTERNS). Le champ patternsSha256 était purement décoratif.

Pourquoi c'était activement nuisible

pre-phase-doc-reference-check.sh comparait le hash déclaré par l'agent (patternsSha256AtConsult) contre l'INDEX — en faisant confiance à l'INDEX comme vérité-terrain. Or l'INDEX était faux. Donc : soit l'agent re-hashait honnêtement le vrai fichier → faux BLOCK « HASH DRIFT » sur ~23 libs ; soit l'agent recopiait le hash de l'INDEX → drift jamais détecté. Les deux lectures = cassé.

Le fix

  1. Réconciliation de l'INDEX : re-hash de tous les fichiers locaux → 18 drifts réels corrigés + 25 champs absents remplis (12 snapshot + 13 patterns). Résultat : 0 mismatch, vérifié par re-hash.
  2. Durcissement du hook : pre-phase-doc-freshness.sh passe d'un check 3-way à 4-way — il re-hashe le PATTERNS.md sur disque et le compare à l'INDEX ; un drift re-queue la lib (self-heal). Plus honor-system.
# nouvelle 4e branche du staleness check (pre-phase-doc-freshness.sh)
if [ ! -f "$patterns_path" ]; then
  REASONS+="patterns-missing-local "
elif ! patterns_hash_matches "$lib" "$patterns_path"; then
  REASONS+="patterns-hash-drift "      # <-- de-honor-system
fi

Self-test du hook : 7/7 (3 nouveaux scénarios : match, drift détecté, no-recorded-hash → pas de faux trigger).

5. Objectif 3 — élagage des agents (décision utilisateur)

Le cadrage qui change tout

Deux lectures coexistent sans se contredire :

  • Les agents sont disjoints entre eux. Lecture ligne-à-ligne : verifier = gates mécaniques, security = threat-model, reviewer = sémantique. Pas de doublon mutuel.
  • L'appareil global duplique ~80 % du natif. Le « redondant » de l'audit vise l'appareil vs superpowers/subagents, pas un agent vs un autre.

Donc l'élagage ne se justifie pas par « ils font la même chose », mais par « cette couche refait du natif » et « cette machinerie n'a jamais servi ».

Décisions actées (après présentation des options)

FUSION doc-fetcher + doc-curator → doc-cache
Le split en deux agents n'avait qu'une justification : préserver le context-window des agents aval (ils lisent un PATTERNS.md de ~400 lignes, pas le snapshot de 50 KB). Mais cet argument concerne l'aval, pas fetcher/curator eux-mêmes — un seul agent peut fetch puis curer dans son propre contexte et n'émettre que PATTERNS.md. Le split ajoutait 1 spawn + 1 handoff sans bénéfice. −1 agent, −1 handoff ; valeur lib-docs intacte.
SUPPRESSION learning-curator (+ mode /team learning:review)
0 amendement produit en 77 runs. Machinerie spéculative non amortie. Les lessons restent capturées (post-complete-lesson-capture.sh) pour lecture manuelle. N'affecte ni frozen-test ni lib-docs. −1 agent (162 lignes).
FUSION verifier → reviewer + hooks
Le cœur de verifier = lancer lint/tsc/test/mutation — des commandes déterministes déjà couvertes par les hooks + la CI. Un agent LLM pour ça était du gaspillage. Sa seule part de jugement (scope-boundary, spot-check) recoupe le reviewer. −1 agent ; gate verify reste (hooks), jugement → reviewer.
CONSERVÉ security — push-back sur l'audit
L'audit suggérait de fusionner security dans le reviewer. Refusé : OWASP LLM Top-10 (injection, <untrusted_content>, Presidio), contrats auth (rotation refresh, nonce social, rate-limit), SAST (semgrep/CodeQL), supply-chain — non réductible à un reviewer générique pour une app IA culturelle avec surface d'injection LLM.

Exécution : 9 → 6 agents, rewiring cohérent sur ~19 fichiers (création doc-cache.md, suppression des 4, mise à jour SKILL.md / agent-mandate.md / sdlc-index / sdlc-pipelines / enterprise / quality-gates / conflict-resolution / error-taxonomy / state.schema.json / user-feedback-rules.json / CLAUDE.md + hooks). Balayage final : 0 référence orpheline active. Commit 1535abe0 (27 fichiers, +525/−1025, 8 gates pre-commit verts).

6. Objectif 4 + absorption systematic-debugging

Décision stratégique (Q4) : plutôt que remplacer /team par les skills natifs, /team reste primaire et absorbe le bon de superpowers. « La team est la priorité d'évolution ». Conséquence : on ne bascule pas vers le natif ; on vendore (copie locale) les mécaniques utiles pour que /team reste self-contained.

Pourquoi systematic-debugging

L'absorption utile comble un trou méthodologique réel, pas un doublon. TDD ≈ red/green+frozen-test, writing-plans ≈ spec/plan, requesting-code-review ≈ reviewer loop — tout ça, /team l'a déjà. Ce qui manquait : aucune méthodologie de debug. Quand le gate verify FAIL ou que le green produit du code qui casse, l'editor « corrige » sans discipline root-cause (le cap 2 boucles = un compteur, pas une méthode).

La convergence exploitée

Le skill dit : « 3+ fixes échoués = questionne l'architecture, ne refixe pas ». Or /team a déjà ce signal : intraPhaseHookLoops cap=2 → STOP+escalade (REGLE 14). Ce qui manquait, c'est la méthode entre les tentatives. Donc l'absorption n'ajoute aucun nouveau point bloquant — elle enrichit le stop existant : au cap, l'escalade devient « questionne l'archi » avec un debug-log.md documentant les hypothèses échouées.

Ce que /team ajoute au skill (les « teeth »)

superpowers est un skill en prose. /team le rend enforce-able — fil rouge de toute la session (« pas d'honor-system ») : un nouveau hook pre-complete-debug-log-check.sh exige un debug-log.md complet (4 phases + section « Architecture question ») dès que intraPhaseHookLoops ≥ 2. Tiéré pour ne pas faux-bloquer un fix trivial. Self-test 4/4.

# enforcement tiéré
loops < 2  → PASS (une ligne de root-cause en STORY suffit)
loops ≥ 2  → debug-log.md MUST exist + 4 phase markers + ≥1 Hypothesis
             + section "Architecture question" (Phase 4.5)   sinon FAIL

Wiring : team-protocols/systematic-debugging.md (méthodo vendored, crédite Obra/superpowers) ; editor.md section DEBUG PROTOCOL (phase green) ; SKILL.md Step 6 (hook d) + REGLE 14 + changelog ; error-taxonomy.md E-RUNTIME ; state.schema.json (gate enum) ; sdlc-index + team-hooks/README. 7. Décisions d'architecture & principes transférables

1. « Agent LLM qui exécute du déterministe » = signal « ça devrait être un hook » Le verifier lançait lint/tsc/test. Le bon critère d'élagage n'est pas « est-ce redondant ? » mais « est-ce que ça demande un LLM ? ». Inversement, frozen-test (jugement de contrat) et security (threat-model) restent.
2. Un mécanisme honor-system ne tient pas — ajoute des dents Le patternsSha256 était dans un fichier tracké mais jamais re-vérifié → 43 divergences silencieuses. Même logique pour la discipline de debug : un protocole en prose ne suffit pas, il faut un artefact + un hook. Si une garantie compte, elle doit être vérifiable.
3. Sur-découper sans bénéfice aval = coût net doc-fetcher → doc-curator était justifié par l'économie de contexte… des agents aval, pas d'eux-mêmes. Fusionner ne perd rien et économise un spawn + un handoff.
4. La machinerie non-amortie se prune sur les données, pas l'intuition learning-curator : 0 sortie / 77 runs. Les données d'usage tranchent ; on ne garde pas « au cas où » avant un launch.
5. Absorber > remplacer — et vendorer pour rester self-contained Q4 : /team garde son orchestration et copie localement les mécaniques natives utiles (pas de dépendance au plugin). On exploite les convergences (le cap existant = le signal du skill) plutôt que d'empiler.
6. Honnêteté documentaire (UFR-013) Les changelogs historiques datés sont préservés (réécrire l'historique = mensonge). Les SHA/stats cités en mémoire ont été corrigés aux valeurs réelles après commit. Le ratio −1025/+525 raconte l'histoire : on a retiré deux fois plus de prose qu'ajouté.

10. Suites & non-faits

  • Push : rien n'est poussé. dev est 97 commits ahead de main. HEAD au moment de l'écriture = e9754d6f (voir ci-dessous) ; mes deux commits en sont des ancêtres vérifiés (git merge-base --is-ancestor).
  • Staging discipliné : mes deux commits n'ont stagé que leurs propres fichiers (jamais git add -A), pour éviter un commit composite. Les modifs pré-stagées sans rapport (CI workflows, maestro, cache-key-parity.test.ts) ont ensuite été committées séparément par les sentinelles event-driven du repo (hook Stop ssot-sentinel.sh) : 45d159a6 (ci deploy-prod gate) et e9754d6f (re-stamp ROPA cascade) — commits distincts, pas un bundle.
  • Direction Q4 ouverte : systematic-debugging est la 1re absorption ; le modèle (vendorer + ajouter un hook enforce) est reproductible pour d'autres skills natifs — candidat suivant naturel : receiving-code-review (discipline côté phase re-spawnée après un CHANGES_REQUESTED).
  • GitNexus : index marqué stale ; le hook PostToolUse sur git commit relance npx gitnexus analyze en arrière-plan.
