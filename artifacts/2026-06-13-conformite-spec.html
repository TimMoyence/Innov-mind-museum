Spec — Conformité réglementaire pré-launch : email RGPD + juridiction Langfuse

Phase SPEC (UFR-022 fresh-context). Aucun code, aucun test écrit ici. RUN_ID : 2026-06-13-conformite-email-subprocessors Toute affirmation ci-dessous = file:line vérifié ou commande exécutée (UFR-013).

1. Contexte + objectif

Avant le launch V1, deux corrections de conformité réglementaire :

  • A — Email de contact RGPD. L'email de contact du responsable de traitement publié dans tous les contenus légaux user-facing et docs légales est tim.moyence@gmail.com (boîte personnelle Gmail). Il doit devenir contact@musaium.com (boîte de marque). Décision actée : domaine canonique = musaium.com, email = contact@musaium.com.
  • B — Juridiction Langfuse. La doc docs/compliance/SUBPROCESSORS.md:52 affirme un transfert/hébergement US par défaut pour Langfuse (« US by default », « SCC + DPF », « US-hosted by default »). Décision actée : Langfuse = région EU (cloud.eu.langfuse.com), DPA auto-accepté au signup (click-accept = SCC Annexe I.A). La doc doit refléter EU et cesser d'affirmer un transfert US.

Constat de terrain dirimant (à trancher par l'humain — cf. §6 et résumé) : les copies canoniques légales (privacy-content.canonical.json BE + web, FR/EN) décrivent déjà Langfuse comme Germany/Allemagne, transferMechanism: "internal" (EU). Vérifié :

  • museum-backend/src/shared/legal/privacy-content.canonical.json:67,242-247,354,529-534
  • museum-web/src/lib/legal/privacy-content.canonical.json (byte-identique, cf. §INV-1).

Donc la route /subprocessors rend déjà EU. Le seul artefact qui affirme US pour Langfuse = docs/compliance/SUBPROCESSORS.md:52. La réconciliation B est donc essentiellement doc-only côté contenu legal — MAIS le défaut de code env.ts:290 pointe https://cloud.langfuse.com (US), ce qui rend l'ancienne phrase de la doc factuellement vraie pour le déploiement par défaut. Voir §6 « point ambigu ».

2. User stories

  • US-A1 — En tant qu'utilisateur final (visiteur RGPD), je veux que l'email de contact affiché dans la politique de confidentialité, les CGU et la page cookies soit contact@musaium.com, afin de pouvoir exercer mes droits (Art. 15-22 RGPD) auprès d'une adresse de marque pérenne et non d'une boîte Gmail personnelle.
  • US-A2 — En tant qu'autorité de contrôle / DPO externe, je veux que les docs légales internes (DPIA, ROPA, readiness) citent l'email de contact officiel contact@musaium.com, afin que la documentation Art. 30 (ROPA) et Art. 35 (DPIA) soit cohérente avec les surfaces publiques.
  • US-A3 — En tant que développeur, je veux que la source de génération (codegen-legal-content.mjs) et la copie générée (privacyPolicyContent.ts) portent le même email, afin qu'une régénération ne réintroduise pas l'ancien email (anti-drift).
  • US-B1 — En tant qu'utilisateur final / autorité, je veux que la page /subprocessors et la doc SUBPROCESSORS.md décrivent Langfuse comme hébergé en région EU (Allemagne, transfert interne UE), afin que l'information de transfert hors-EEE soit exacte (Art. 13(1)(f), Art. 28, Art. 44-46 RGPD).
  • US-B2 — En tant que responsable conformité, je veux que SUBPROCESSORS.md n'affirme plus un transfert US ni un mécanisme DPF pour Langfuse, afin de ne pas documenter un transfert hors-EEE qui n'a pas lieu (sur-déclaration = signal d'imprécision en audit).

3. Exigences EARS

Sous-objectif A — Email RGPD

  • EARS-A1 — WHEN un contenu légal user-facing (privacy, terms, cookies) est rendu sur n'importe quelle surface (BE canonical, web, mobile), THE SYSTEM SHALL afficher contact@musaium.com comme email de contact RGPD et SHALL NOT afficher tim.moyence@gmail.com.
  • EARS-A2 — WHILE codegen-legal-content.mjs régénère privacyPolicyContent.ts, THE SYSTEM SHALL émettre contactEmail: 'contact@musaium.com' (la valeur étant hardcodée dans le template du script, cf. codegen-legal-content.mjs:199, PAS lue depuis le JSON).
  • EARS-A3 — WHEN les copies canoniques privacy/terms BE et web sont comparées, THE SYSTEM SHALL les garder JSON-équivalentes (cf. INV-1), y compris sur les paragraphes contenant l'email.
  • EARS-A4 — IF une occurrence de tim.moyence@gmail.com subsiste dans un contenu légal user-facing ou une doc légale (hors exclusions §6), THE SYSTEM SHALL être considéré non conforme (critère d'acceptation AC-A1 échoue).
  • EARS-A5 — WHEN les docs légales internes (DPIA, ROPA, DPIA_ROPA_READINESS) référencent l'email du responsable, THE SYSTEM SHALL utiliser contact@musaium.com. Le mécanisme d'alias dpo@musaium.com → boîte de redirection SHALL pointer vers contact@musaium.com (et non plus vers tim.moyence@gmail.com) là où ce flux est décrit (DPIA.md:8, ROPA.md:9, DPIA_ROPA_READINESS.md:16).

Sous-objectif B — Juridiction Langfuse

  • EARS-B1 — WHEN docs/compliance/SUBPROCESSORS.md décrit Langfuse (ligne 52), THE SYSTEM SHALL indiquer une juridiction/hébergement région EU (cloud.eu.langfuse.com) et SHALL NOT affirmer « US by default » / « US-hosted by default » / un transfert hors-EEE comme état nominal.
  • EARS-B2 — WHEN la doc décrit le mécanisme de transfert Langfuse, THE SYSTEM SHALL refléter un DPA auto-accepté au signup (click-accept = SCC Annexe I.A) cohérent avec un hébergement EU, et SHALL NOT mentionner DPF (Data Privacy Framework, mécanisme US-only) pour Langfuse.
  • EARS-B3 — WHEN la route /subprocessors et la doc SUBPROCESSORS.md sont comparées sur la ligne Langfuse, THE SYSTEM SHALL être cohérent (les deux disent EU). (Constat : la route dit déjà EU via canonical JSON ; seule la doc diverge.)
  • EARS-B4 — IF la valeur de juridiction Langfuse dans le canonical JSON (Germany/Allemagne, internal) doit changer, THE SYSTEM SHALL appliquer le changement aux DEUX copies canoniques (BE + web) pour préserver INV-1. (Note : aucun changement JSON requis si on garde EU=Allemagne/internal ; cf. §6.)

4. NFR (Non-Functional Requirements)

  • NFR-1 (i18n FR/EN) — Les contenus légaux existent en FR et EN. Le remplacement de l'email DOIT couvrir les deux locales. L'email étant un littéral identique FR/EN, aucune traduction n'est requise, mais chaque occurrence localisée DOIT être traitée (ex. cookies/page.tsx:65 EN + :104 FR).
  • NFR-2 (no-drift 3 copies canoniques) — Les trois surfaces de contenu privacy/terms (BE canonical JSON, web canonical JSON, FE généré) DOIVENT rester cohérentes. Mécanique réelle :
    • BE canonical JSON = source de vérité (privacy-content-drift.mjs:62,80).
    • Web canonical JSON = copie JSON-équivalente vérifiée par privacy-content-drift.mjs:80-87 (whitespace-insensitive).
    • FE privacyPolicyContent.ts = généré par codegen-legal-content.mjs (header GENERATED-FROM-CANONICAL — DO NOT EDIT MANUALLY, privacyPolicyContent.ts:1). Le contactEmail est hardcodé dans le script, pas dérivé du JSON.
    • FE termsOfServiceContent.ts = NON généré (hand-maintained, pas de header GENERATED ; le codegen ne touche pas terms — vérifié : grep terms codegen-legal-content.mjs = 0).
  • NFR-3 (RGPD Art. 28 / 13-14) — Le registre des sous-traitants (Art. 28) et l'information sur les transferts (Art. 13(1)(f)/14) DOIVENT être exacts : pas de sur-déclaration d'un transfert US inexistant pour Langfuse.
  • NFR-4 (idempotence codegen) — Après édition, node museum-frontend/scripts/codegen-legal-content.mjs puis git diff --exit-code -- museum-frontend/features/legal/privacyPolicyContent.ts DOIT être vide (codegen et fichier généré alignés).
  • NFR-5 (sentinels verts) — privacy-content-drift.mjs et subprocessor-ledger-completeness.mjs DOIVENT rester verts après modifications.
