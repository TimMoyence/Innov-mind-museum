## Verdicts par Feature

> Pour chaque feature: **Intérêt** (pourquoi elle existe), **Critique** (ce qui va/va pas), **Verdict** (UTILE/INUTILE/PARTIEL)

### BACKEND

#### Auth (register, login, refresh, logout, verify-email, social, change-password, change-email, forgot/reset, API keys, delete account, onboarding)
- **Intérêt**: Fondamental. Pas d'app sans auth. 12 endpoints couvrant le lifecycle complet user.
- **Critique**: Excellente couverture. JWT rotation avec détection de réutilisation famille-based. Bcrypt. Rate limiting per-IP et per-user. Zod validation sur tous les endpoints. Tests unitaires + intégration + E2E + contrat OpenAPI. Factories partagées (makeUser, makeToken).
- **Ce qui sert**: Tout. Refresh token rotation protège contre le vol de token. Rate limiting bloque le bruteforce. La séparation use-case/route/repo est réelle et testable.
- **Ce qui ne sert pas**: Rien à signaler. Feature solide.
- **Verdict**: **UTILE** — production-grade

#### Chat (session, message, streaming, image, audio, TTS, feedback, report, user memory, art keywords)
- **Intérêt**: Core business. C'est la raison d'être de l'app — l'utilisateur pose une question, obtient une réponse IA.
- **Critique**: Le module le plus complexe (77 fichiers) et le mieux testé (55+ tests unitaires dans chat-message-service seul). Pipeline LLM avec sections parallèles, guardrail input/output, prompt sanitization, image enrichment, streaming SSE. Circuit breaker sur le LLM.
- **Ce qui sert**: Tout le pipeline est branché et fonctionnel. Le streaming est feature-flagged. L'art-topic guardrail filtre insults, prompt injection, off-topic avant l'appel LLM. La memory cross-session enrichit les réponses.
- **Ce qui ne sert pas**: `FallbackChatOrchestrator` (uniquement utilisé dans les tests, jamais en production — c'est un test double, pas du code mort mais pourrait être déplacé dans tests/helpers). `getUserMemoryService` exporté depuis chat/index.ts mais jamais importé.
- **Verdict**: **UTILE** — coeur de métier, bien implémenté

#### Admin (users, RBAC, audit logs, analytics, moderation, reviews, tickets)
- **Intérêt**: Nécessaire pour opérer l'app. CRUD users, modération contenu, analytics.
- **Critique**: 13 endpoints admin, tous testés. RBAC admin/moderator/museum_manager. Audit logging. Cependant, les tests analytics (4 use cases) sont des tests de délégation pure — ils vérifient que le mock a été appelé, pas la logique réelle (car les use cases SONT des passthroughs).
- **Ce qui sert**: RBAC empêche l'escalade de privilèges. Audit log trace les actions admin. Modération gère les signalements.
- **Ce qui ne sert pas**: Les 4 tests analytics use case sont du théâtre (mock → assert mock called → pass). La vraie logique SQL est dans admin-analytics-queries.ts et n'est testée que via mock repo. **Risque**: une erreur SQL en prod ne serait détectée par aucun test unitaire.
- **Verdict**: **UTILE** mais analytics testing est **DÉCORATIF** (4 tests)

#### Museum Directory, Support, Review, Daily Art
- **Intérêt**: Features secondaires mais nécessaires pour l'expérience utilisateur complète.
- **Critique**: Toutes implémentées, branchées, testées, avec migrations. Museum directory a des tests Overpass/Nominatim. Support a 9 fichiers de tests. Review a 4 fichiers.
- **Ce qui sert**: Museum directory permet de trouver les musées proches. Support tickets gèrent le SAV. Reviews créent de l'engagement.
- **Ce qui ne sert pas**: Rien à signaler.
- **Verdict**: **UTILE** — features complètes

#### Health Check, Rate Limiting, Error Handling, Request ID, Swagger
- **Intérêt**: Infrastructure middleware. Non visible par l'utilisateur mais essentielle pour l'opérationnel.
- **Critique**: Health check vérifie DB + Redis + LLM circuit breaker. Rate limiting multiple strategies. Error handler centralisé (AppError → JSON structuré + Sentry pour 5xx). Request ID pour traçabilité. Swagger servi en non-prod.
- **Ce qui sert**: Tout. Le health check est utilisé par les load balancers. Le rate limiting protège l'API. Le request ID permet de tracer un bug en production.
- **Ce qui ne sert pas**: 10 instances de `res.status(401).json(...)` inline dans les routes **bypass** le error handler centralisé. Incohérence d'observabilité: certains 401 sont loggés via middleware, d'autres non.
- **Verdict**: **UTILE** mais les 10 inline 401 sont un **ANTI-PATTERN** à corriger

## Architecture

### Hexagonale: 7/10 — Réelle mais impure

**Ce qui fonctionne:**
- Ports et Adapters sont correctement définis à la frontière service/repository
- Chaque repository a une interface (port) dans domain/ et une implémentation TypeORM dans adapters/secondary/
- Les use cases dépendent uniquement des ports (injection de dépendances)
- La composition root (module/index.ts) câble tout proprement
- Testabilité excellente: 2294 tests backend sans services externes

**Les violations (critiques à comprendre):**
1. **15 entités domain importent TypeORM** — `import { Entity, Column, ... } from 'typeorm'` dans CHAQUE fichier domain/*.entity.ts. Le domaine EST l'ORM. Si on change d'ORM, on réécrit TOUT le domain layer. C'est un compromis pragmatique (TypeORM impose ce pattern), mais c'est une violation structurelle de l'hexagonale pure.
2. **Couplage cross-module dans le domaine**: ChatSession importe User depuis auth/domain. UserMemory importe User. Support importe PaginationParams depuis admin/domain.
3. **Admin adapter importe des entités de 3 modules** (auth, chat, support) — acceptable pour un read-model mais crée du couplage.

**Verdict honnête**: C'est de l'hexagonale au niveau des ports/adapters, pas au niveau des entités. C'est la bonne approche pragmatique avec TypeORM. Appeler ça "hexagonal" est correct si on précise: "hexagonal at service boundary, coupled at entity level."

## Tests: Pertinence Réelle

### Score global: 8.5/10

### Les tests servent-ils à quelque chose ? **OUI, massivement.**

**~2270+ tests backend à haute valeur** (sur 2294):
- Vérifient des comportements réels: rejet d'input invalide avec bon status code, ownership access control, guardrail bloquant les injections, ordering des appels orchestrateur, format JWT, pagination cursor, rate limiting, conformité OpenAPI
- Les factories partagées (`makeUser`, `makeMessage`, `makeSession`, `buildChatTestService`) sont utilisées presque partout
- Les E2E (64 tests) utilisent testcontainer avec vrai PostgreSQL

**~15-20 tests décoratifs** (faible valeur):
1. `chat-service.test.ts`: "exposes all public facade methods" — vérifie `typeof === 'function'` pour 11 méthodes. Zero valeur comportementale.
2. `analytics-usecases.test.ts` (4 tests): délégation pure → assert mock called. Les use cases sont des passthroughs, le test vérifie que le passthrough passe.
3. `session-list.test.ts`: 1 seul test, 13 lignes. Couvre un seul chemin de rejet.

**63 tests skippés — TOUS légitimes:**
- 9 suites E2E (64 tests): gatés par `RUN_E2E=true`, nécessitent PostgreSQL testcontainer → design correct
- 4 suites AI (19 tests): gatés par flag env, nécessitent vraies clés API LLM → design correct
- 0 test obsolète identifié

**2 violations de factory discipline:**
- `chat-message-service.test.ts:26` et `chat-media.service.test.ts:15`: `makeMessage()` local au lieu d'importer depuis `tests/helpers/chat/message.fixtures.ts`

### Conclusion tests
Les tests ne sont PAS là juste parce qu'on les a demandés. Ils testent des comportements réels avec des assertions comportementales. Le ratio 2294 tests / ~5000 LOC (0.46 tests/ligne) est sain, pas excessif. La couverture à 93.69% est genuine (branches à 83.91% — pas du coverage gaming). Les ~15-20 tests décoratifs sont la seule vraie faiblesse.
