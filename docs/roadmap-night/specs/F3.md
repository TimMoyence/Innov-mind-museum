**Bug summary** : the modal renders `reason.resetAt` as the raw ISO-UTC string :

```tsx

  <Text style={styles.meta}>
    {t('paywall.resetsOn')} {reason.resetAt}
```

`reason.resetAt` is the raw ISO-8601 UTC timestamp emitted by `museum-backend/src/shared/middleware/monthly-session-quota.middleware.ts:81-84` via :

```ts
const firstOfNextUtcMonthIso = (): string => {
  const now = new Date();
  return new Date(Date.UTC(now.getUTCFullYear(), now.getUTCMonth() + 1, 1)).toISOString();
};
```

→ produces `'2026-06-01T00:00:00.000Z'` (or `'2026-07-01T00:00:00.000Z'` in June, etc.).

**User-visible** :

- EN locale : `Resets on 2026-06-01T00:00:00.000Z`
- FR locale : `Réinitialisation le 2026-06-01T00:00:00.000Z`
- The 6 other locales (es / de / it / ja / zh / ar — confirmed by `ls museum-frontend/shared/locales/`) inherit the same anti-pattern : their localized prefix concatenated with a developer-facing UTC timestamp.

**Spec violation** : `R1.md §3.4 D4 (line 302)` reads verbatim :

> `resetAt` — first-day-of-NEXT-UTC-month, ISO string. **Mobile renders "Resets on June 1" via `Intl.DateTimeFormat`.**

The spec **explicitly anticipated** a raw ISO wire format + a FE formatter step. The R1 implementation pinned the BE side (R8 / N5 — wire format is ISO and the test corpus at `chat-session-quota.integration.test.ts` asserts `resetAt: "2026-06-01T00:00:00.000Z"`) but **skipped the FE Intl.DateTimeFormat call** on the rendering side. Net result : the wire contract is correct ; the UI contract is violated.

**Why it matters (R1 §0.1 coupling)** :

- **User trust** — a paywall is the moment of highest friction in the entire funnel (axios 402 interceptor → `PaywallProvider.open` → modal). Showing `2026-06-01T00:00:00.000Z` to a non-technical visitor reads as a bug at exactly the wrong moment. The KR4 conversion analysis (`docs/ROADMAP_PRODUCT.md:194`) measures the funnel — a "looks broken" signal at the meta line corrupts the conversion-readiness data we're trying to collect.
- **Localization promise** — the brief ships 8 locales (en / fr / es / de / it / ja / zh / ar) deliberately. The `paywall.resetsOn` key is translated in all 8 (confirmed for en + fr by grep ; R29 sentinel test pins the en + fr pair). Rendering the date in dev-facing UTC format **across all 8 locales** silently nullifies the localization investment for that specific line. The Japanese / Chinese / Arabic users see Latin ISO timestamps mid-sentence — visual incoherence beyond just ugly.
- **GDPR DSAR coherence** — when the user later requests their data, the BE will emit ISO timestamps (which is correct in a JSON DSAR export). If they remember "Resets on 2026-06-01T00:00:00.000Z" from the modal and DON'T see the same exact string in the export, they may construct a complaint. Minor but real. Fix moves the modal to a human format that doesn't compete with DSAR semantics.

This is exactly the kind of drift the ultrareview pipeline exists to catch. Closing F3 closes the loop : spec → impl → reviewer → corrective → spec re-satisfied.

## 3. Design

### 3.1 Approach (D1) — Intl.DateTimeFormat via i18next i18n.language + dateStyle 'long'

The reviewer's recommendation (verbatim from the bug ticket) :

```tsx
const formattedReset = useMemo(() => {
  if (!reason?.resetAt) return '';
  try {
    return new Intl.DateTimeFormat(i18n.language, { dateStyle: 'long' })
      .format(new Date(reason.resetAt));
  } catch {
    return reason.resetAt; // defensive fallback
  }
}, [reason?.resetAt, i18n.language]);
```

**Why this exact shape** :

| Decision | Rationale |
|---|---|
| **`useMemo` vs inline** | The Intl.DateTimeFormat constructor does an ICU locale-data lookup ; rebuilding it on every keystroke in the email field is wasteful. The memo's dep array `[reason?.resetAt, i18n.language]` matches the only two inputs. R1 R5 (locale change mid-modal) is a real flow (user opens settings, switches language, returns to modal still open) ; the memo handles it correctly. |
| **`dateStyle: 'long'`** | Three choices : `'full'` (e.g. "Wednesday, June 1, 2026"), `'long'` ("June 1, 2026"), `'medium'` ("Jun 1, 2026"), `'short'` ("6/1/26"). `'long'` is the spec's verbatim example ("June 1") and the most user-friendly for the paywall context — `'short'` is too clipped (looks like a typo) ; `'full'` is too verbose (the weekday is irrelevant — the user cares about the date, not the day). `'medium'` is a reasonable alternative but mixes abbreviation with full year, looking inconsistent. Pick : `'long'`. Captured in Q2 for confirmation. |
| **No `timeStyle`** | The reset moment is 00:00 UTC. Displaying time would either show "00:00" (uninformative, possibly confusing — looks like a placeholder) or convert to local time (which would show 02:00 in Paris, 09:00 in Tokyo — also confusing because "the reset is at midnight" is the mental model). `dateStyle` alone is correct : we tell the user the DATE they're freed, not the exact second. |
| **No explicit `timeZone`** | Letting the device's default timezone apply. June 1 UTC in Tokyo IS June 1 JST (09:00 JST) ; in NY it's May 31 (20:00 EDT). `dateStyle: 'long'` extracts only the date components — so Tokyo sees "June 1" (correct, they're a day-of-month later in their time), NY sees "May 31" (correct, their reset is the END of May 31 / START of June 1 in NY time). Either is defensible — see Q1. Default : no timezone override. |
| **`new Date(reason.resetAt)`** | Standard ES parser handles ISO 8601 reliably (with 'Z' suffix it's UTC, which is what the BE emits). No moment.js, no date-fns — both would be 30KB+ of unused locale data for one date format call. |
| **`i18n.language` source** | The `useTranslation()` hook exposes `i18n.language` which is the active i18next language tag (e.g. `'en'`, `'fr'`, `'fr-FR'` if a region was set). Intl resolves BCP-47 tags itself — `'en'` becomes `'en-US'` (the JS engine's default region for `'en'`) or `'en-GB'` depending on the host. We do NOT resolve regions ourselves — let Intl handle the fallback chain. |
| **Defensive try/catch** | Two failure modes : (a) `new Date('garbage').toISOString()` doesn't throw but `getTime()` returns NaN ; `format(NaN_date)` throws RangeError. (b) `new Intl.DateTimeFormat('invalid-locale', ...)` throws RangeError. The try/catch returns the raw string on both, surfacing the bug visibly to QA without crashing the UX. |

**Why not** (alternatives considered) :

| Alternative | Why rejected |
|---|---|
| **`toLocaleDateString(locale, options)` on Date instance** | Equivalent functionality but creates an implicit Intl.DateTimeFormat instance per call → no memoization possible (the method does the lookup internally). Slower under repeated render. |
| **`date-fns` + `format(date, 'PPP', { locale })`** | 30KB+ for one date format. Overkill. Library dep increases bundle. |
| **Format string from i18next bundle** (`paywall.resetDateFormat: 'MMMM d, yyyy'`) | Would require maintaining 8 format strings × locales, each manually correct for that locale's date conventions. `Intl.DateTimeFormat` already encodes that data via ICU. Wasteful and error-prone. |
| **Move format step to BE** (return `resetAt` + `resetAtFormatted`) | Breaks the wire contract (R1 N5), requires BE to know the user's locale (currently it doesn't — `Accept-Language` parsing not implemented), forces a re-serialize on every locale switch. FE format is the spec's chosen design (R1 §3.4 D4). |
| **Polyfill via `@formatjs/intl-datetimeformat`** | Adds 200KB+ of ICU data. Hermes ships ICU natively — polyfill is redundant. Bundle bloat. |

### 3.2 Locale source (D2) — i18n.language semantics + unsupported-tag fallback

**`useTranslation()` returns `i18n.language`** — this is the i18next instance's active language tag. After `i18n.init({ lng: 'en' })` (which is what `shared/i18n/i18n.ts:27` does), `i18n.language === 'en'`. After `i18n.changeLanguage('fr')`, `i18n.language === 'fr'`.

**Intl.DateTimeFormat locale resolution** :

- `new Intl.DateTimeFormat('en', { dateStyle: 'long' }).format(new Date('2026-06-01'))` → `"June 1, 2026"` (en-US default).
- `new Intl.DateTimeFormat('fr', { dateStyle: 'long' }).format(...)` → `"1 juin 2026"`.
- `new Intl.DateTimeFormat('ja', { dateStyle: 'long' }).format(...)` → `"2026年6月1日"`.
- `new Intl.DateTimeFormat('ar', { dateStyle: 'long' }).format(...)` → `"١ يونيو ٢٠٢٦"` (Arabic numerals + month name, if Hermes ships Arabic ICU data).

**Edge case — `ar` Arabic numerals** : Intl's Arabic locale renders dates with Arabic-Indic digits by default. If the design wants Latin digits in the Arabic locale (for compatibility with the RTL layout effort tracked in `feedback_rtl_start_end`), the format options would need `numberingSystem: 'latn'`. F3 default is to let Intl's locale default apply — Arabic users see Arabic digits, consistent with their other locale conventions. Captured in Q3.

**Edge case — unsupported BCP-47 tags** : if `i18n.language` is something Intl doesn't support directly (e.g. `'xx'` test locale), Intl throws RangeError → the try/catch fallback fires → raw ISO surfaces. Acceptable.

**Edge case — region-suffixed tags** : if i18next is initialised with `'fr-CA'` (French Canada), Intl handles it (`"1 juin 2026"` with Canadian conventions, identical to fr-FR for this date). The codebase's `i18n.ts` initialises with just `'en'` (no region) — so no region drift today.

### 3.3 Defensive fallback (D3) — try/catch wraps Intl call, fallback to raw ISO

Three failure modes :

| Failure | Where | Behaviour with try/catch | Behaviour without |
|---|---|---|---|
| `reason.resetAt = ''` (BE emits empty string by mistake) | `new Date('')` returns Invalid Date ; `.format(Invalid)` throws RangeError | Fallback to `''` rendered (line shows "Resets on " with trailing space — graceful) | Component crash on the format call ; whole modal unmounts mid-render |
| `reason.resetAt = 'not-an-iso'` | `new Date('not-an-iso')` returns Invalid Date ; `.format(Invalid)` throws | Fallback to `'not-an-iso'` rendered visibly — QA sees the BE bug | Same crash |
| `Intl` itself undefined (Hermes-without-Intl edge config) | `new Intl.DateTimeFormat(...)` throws ReferenceError | Fallback to raw ISO rendered | Crash + Sentry reports the missing-Intl deployment misconfiguration |

**Why fallback to raw ISO instead of empty string** : the raw ISO is ugly but informative ; an empty string hides the bug. The metadata line stays visible : `"Resets on 2026-06-01T00:00:00.000Z"`. Worst case = pre-fix behaviour, which is the ground we already stand on. NOT worse than today.

**Why NOT silent return of `''`** : would hide a real BE schema regression. If a future BE refactor changes `resetAt` to `{date, time}` object, the FE would silently stop rendering — a P1 we'd take weeks to notice. Visible-but-ugly fallback flags it within hours.

**Why NOT a Sentry breadcrumb on fallback** : tempting (it'd capture the BE bug to monitoring) but introduces a side-effect inside `useMemo`, which violates React's "memo factory must be pure" contract. Could move the breadcrumb to a separate `useEffect` keyed on the memo result, but then we'd be 20+ LoC into a corrective for a defensive fallback. Out of F3 scope ; if it matters, V1.1.

### 3.4 Memo deps (D4) — `[reason?.resetAt, i18n.language]` to prevent re-format on every render

The modal re-renders on :

- Email typing → `setEmail` → render. Memo deps unchanged → cached `formattedReset` reused. CORRECT.
- Consent toggle → `setConsent` → render. Memo deps unchanged → cached reused. CORRECT.
- Submit state transitions (`'idle' → 'sending' → 'success'|'error'`) → 3 renders. Memo deps unchanged → cached × 3. CORRECT.
- `visible` flip (F2 reset path) → render. Memo deps unchanged → cached reused. CORRECT. (The F2 effect clears the four state slots ; it does NOT mutate `reason.resetAt` or `i18n.language`.)
- `reason` prop change (provider receives a new 402 while modal is open — Q2 of F2 acknowledged this edge case) → memo recomputes if `resetAt` differs. CORRECT.
- Locale change via `i18n.changeLanguage` → `useTranslation` re-renders consumers → memo recomputes. CORRECT.

**Dep array shape — `reason?.resetAt` vs `reason`** :

- Using `reason?.resetAt` (optional chain) means : if `reason === null`, the optional chain returns `undefined`, the memo dep is stable at `undefined`. If `reason` later becomes non-null with `resetAt = 'xyz'`, dep changes `undefined → 'xyz'` → recompute.
- Using `reason` directly as the dep would re-compute on every PROVIDER-state change (since `reason` is an object reference and the provider may emit a new object even with the same content). Wasteful.
- Pick : `reason?.resetAt` (the only field the memo actually consumes).

**Why not `[reason, i18n.language]`** : reference instability per above. The memo would invalidate every time `PaywallProvider.open()` is called even with same payload → re-format. Cheap but wasteful.

**Why not `[reason?.resetAt, i18n]`** : `i18n` itself is a stable reference (the i18next instance is module-singleton) ; it never changes. Memo would never invalidate on language change. WRONG.

**Why `i18n.language` is the right dep** : it's a string primitive that changes when `i18n.changeLanguage(x)` is called and i18next emits its `languageChanged` event (which causes `useTranslation` consumers to re-render — at that point, the new `i18n.language` value is read inside the memo factory).

## 4. Open Questions

### Q1. `dateStyle: 'long'` vs `'medium'` vs `'full'` — confirm spec's pick ?

The spec's R1 §3.4 D4 example reads `"Resets on June 1"` — note the YEAR is OMITTED. That suggests `{ month: 'long', day: 'numeric' }` (no year) rather than `dateStyle: 'long'` (which includes year).

**Default position** : pick `dateStyle: 'long'` (includes year) because :

1. The reset is at month boundary, but if a user opens the modal in late May, the reset is "June 1" of CURRENT year (clear). If they open in December, reset is "January 1" of NEXT year — without the year, the user might think "Jan 1, this past January, which is in the past, this is broken". Including year disambiguates.
2. Cross-year edge case is annual but not negligible (1/12 of paywall opens cross the calendar boundary).
3. ICU's `dateStyle: 'long'` is a well-known semantic preset — `{ month: 'long', day: 'numeric' }` is a custom combination that may render slightly differently across locales (e.g. some locales prefer `1 juin` not `juin 1` ; the preset handles ordering automatically).

**Alternative** : `{ month: 'long', day: 'numeric', year: 'numeric' }` explicit options (semantically equivalent to `dateStyle: 'long'` for date-only) — let Tim pick the format level. Default = `'long'`.

### Q2. Should the formatter respect device timezone OR force UTC ?

Scenario : user in Tokyo (UTC+9) hits paywall at "May 31, 23:00 UTC = June 1, 08:00 JST". The reset moment is "June 1, 00:00 UTC = June 1, 09:00 JST" (~1 hour from now in their time).

- **Device-default timezone** : modal shows "Resets on June 1" (date-only). User reads "tomorrow morning my time" → mental model matches reality. CORRECT.
- **Force UTC** (`timeZone: 'UTC'`) : modal shows "Resets on June 1" (same date in UTC). User reads "June 1 UTC" → mental load to convert. Marginally worse UX.

Both render "June 1" because `dateStyle: 'long'` is date-only. The difference materialises on edge cases :

- User in NY (UTC-4) opens at "May 31, 21:00 EDT = June 1, 01:00 UTC" (1 hour AFTER UTC reset). Device-default shows "May 31" (their local date) ; UTC-forced shows "June 1". Discrepancy : 0 hours, just date offset.
- Wait — if the user is opening the paywall WHEN their counter hit limit, they need to know "when does my counter reset". If the BE counter resets at "June 1, 00:00 UTC" and they're NY-time at "May 31, 21:00", the BE perspective is "your counter resets in 3 hours" — the date is technically "June 1" UTC.

**Default position** : device-default timezone. Aligns with the user's mental model of "the day my counter comes back". The minor drift across timezones (Tokyo user sees "June 1" tomorrow ; NY user sees "May 31" today for the same UTC reset moment) is HONEST about local time — the user resets when their local "month boundary" hits, which for the BE is UTC boundary, displayed in their local terms.

### Q3. Should Arabic locale render Latin digits (`numberingSystem: 'latn'`) ?

Default Intl renders Arabic dates with Arabic-Indic digits (`١ يونيو ٢٠٢٦`). Some apps force Latin digits in Arabic for accessibility / readability.

**Default position** : let Intl default. The codebase has RTL doctrine (`feedback_rtl_start_end`) which respects locale conventions. Arabic users seeing Arabic digits is locale-faithful. If a localization reviewer flags it, switch to `numberingSystem: 'latn'`.

### Q4. Should F3 emit a Sentry breadcrumb when the defensive fallback fires ?

The fallback is silent today. If a BE bug ever emits malformed ISO, F3 swallows it visibly to the user but invisibly to monitoring.

**Default position** : NO for V1 — out of F3 scope. The `useMemo` factory must stay pure ; adding a `Sentry.addBreadcrumb` inside violates that contract. A separate `useEffect` keyed on the memo result could do it (`useEffect(() => { if (formattedReset === reason?.resetAt) Sentry.addBreadcrumb(...); }, [formattedReset, reason?.resetAt])`) but that adds 5 LoC + 1 effect for a defensive guard we may never hit. V1.1 candidate. Captured in §7.

### Q5. Should F3 add a TECH_DEBT entry for "audit other ISO-in-UI render sites" ?

This bug is one instance of a class : raw ISO strings interpolated into UI text. Other surfaces likely exist (other dates in admin, chat-message timestamps, daily-art reset). A V1.1 audit pass could grep for `{[^}]*\.toISOString()|reason\.[^}]*At}` patterns across UI files.

**Default position** : YES — add a TECH_DEBT entry. Cheap, honest, prevents the doctrine bullet from being a one-off note. Tracked in T3.
