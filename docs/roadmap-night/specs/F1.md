**Bug summary** : `monthlySessionQuota.tryConsume` issues an atomic `UPDATE users SET sessions_month_count = ... RETURNING ...` BEFORE `validateBody(createSessionSchema)` runs. Any Zod 400 (non-object body, museumId ≤ 0, invalid lat/lng, intent not in enum, locale wrong type) OR handler 5xx (DB blip, FK miss inside `chatService.createSession`) burns one free-tier slot for a session that is never created.

**Verified attack** (free-tier user `id=42`, `sessions_month_count=0`) :

1. `POST /api/sessions` body `[]` → `monthlySessionQuota` UPDATE → count=1 → `validateBody` 400 "Payload must be an object". Session never created.
2. Repeat ×2 → count=3.
3. Genuine `POST /api/sessions` body `{}` → 402 QUOTA_EXCEEDED. User locked out for the month with **zero sessions actually created**.

**Why it matters (R1 §0.1 coupling)** : `sessions_month_count` IS the KR4 funnel signal that drives the Stripe go/no-go decision (`docs/ROADMAP_PRODUCT.md:194`). Inflated counters (a) unfairly lock free users out, (b) corrupt the conversion-readiness data — the exact thing R1 was built to emit cleanly.

## 1. Spec (EARS requirements)

- **F1.R1.** WHEN `POST /api/sessions` is received with a body that fails `createSessionSchema` validation (e.g. non-object `[]`, `museumId <= 0`, `coordinates.lat > 90`, `intent` not in `CHAT_SESSION_INTENTS`) THE SYSTEM SHALL respond `400 Bad Request` via `validateBody` AND MUST NOT mutate `users.sessions_month_count` or `users.sessions_month_start`.
- **F1.R2.** WHEN `POST /api/sessions` is received with a valid body AND `req.user.tier === 'free'` AND `sessions_month_count < limit` THE SYSTEM SHALL increment `sessions_month_count` by exactly 1 (atomic UPDATE, R11 preserved) AND create the session AND respond `201 Created`.
- **F1.R3.** WHEN `POST /api/sessions` is received with a valid body AND `req.user.tier === 'free'` AND `sessions_month_count >= limit` (same month) THE SYSTEM SHALL respond `402 Payment Required` with body `{ code: 'QUOTA_EXCEEDED', tier: 'free', currentCount, limit, resetAt, message }` AND MUST NOT execute the handler AND MUST NOT mutate the counter (over-limit by definition).
- **F1.R4.** WHEN the handler (`chatService.createSession`) throws **AFTER** the counter has been mutated (D1 = option (a) only — the counter is bumped before the handler runs) THE SYSTEM SHALL [D3 choice — see §3.3]. If D1 = option (b) reserve+commit, the counter MUST NOT have been mutated yet at the moment the handler throws ; the commit step is skipped on throw. If D1 = option (c) try/catch decrement, the counter MUST be decremented inside the catch block before the error propagates.
- **F1.R5.** WHEN two concurrent `POST /api/sessions` requests arrive for the same `userId` (multi-device) AND both are zod-valid AND the user's current count is `limit - 1` THE SYSTEM SHALL serialize the counter mutations such that exactly one request succeeds (201) and the other receives 402 (R11 invariant preserved, regardless of D1 choice).
- **F1.R6.** WHEN `POST /api/sessions` is received with body `{}` (empty object — zod-valid because all fields optional) AND tier is free AND under limit THE SYSTEM SHALL increment exactly once AND create the session (the pre-fix behaviour for valid-body requests is unchanged).
- **F1.R7.** WHEN `POST /api/sessions` is received without a JWT THE SYSTEM SHALL be rejected by `isAuthenticated` BEFORE either `validateBody` or `monthlySessionQuota` runs (R9 invariant). No counter touch, no audit, no log.
- **F1.R8.** WHEN `POST /api/sessions` is received with a JWT for a `tier='premium'` user THE SYSTEM SHALL bypass the quota mutation entirely (R5 invariant) regardless of body validity ; `validateBody` runs in its position per the new pipeline order, premium users still get 400 on invalid bodies (correct — that's not a paywall concern).
- **F1.R9.** WHEN the `quota_check_hit_limit` log fires (first 402 of (user, month)) THE SYSTEM SHALL only fire on a TRUE quota hit — i.e. a zod-valid request that genuinely hit the limit. Zod 400s MUST NOT trigger the log under any D1 option.
- **F1.R10.** WHEN F1's tests run THE SYSTEM SHALL include explicit coverage for : (a) Zod 400 leaves counter untouched ; (b) handler 5xx [behaviour per D3 — either prevented or compensated] ; (c) two concurrent valid-body requests serialize correctly (Risk1 mitigation re-verified post-fix).

## 3. Design

### 3.1 Approach selection (D1) — option (a) swap order vs (b) reserve+commit vs (c) try/catch decrement

Three options weighed honestly :

| Option | Pro | Con | Verdict |
|---|---|---|---|
| **(a) Swap order** — `validateBody` BEFORE `monthlySessionQuota`. Route line 34-51 becomes `isAuthenticated → validateBody(createSessionSchema) → monthlySessionQuota → handler`. | Smallest possible diff (3 lines moved in route file). Closes the entire Zod 400 hole (every zod-rejected request short-circuits before the UPDATE). No middleware changes. No port widening. No new race-window vector. Risk1 invariant (R11 atomic UPDATE) byte-unchanged. | Does NOT close the handler-5xx hole. If `chatService.createSession` throws AFTER the counter is bumped (DB blip mid-INSERT, FK miss on `museumId`, downstream timeout), the slot is still burned. Honesty : the residual hole is materially smaller than the Zod one (handler 5xx is rare ; Zod 400 is everyday client-bug territory) but it exists. Documented in §3.3 D3. CPU cost : negligible — `validateBody` is a sync `safeParse` of a parsed JSON object, no I/O. |
| **(b) Reserve+commit** — `monthlySessionQuota` becomes a read-only check (verify count < limit, return 402 if at limit, attach `monthStart` + `nextCount` to `res.locals.quotaReservation`) ; the handler calls `await monthlyQuota.commit(userId, monthStart)` AFTER `chatService.createSession` resolves successfully. The commit step is the existing atomic `UPDATE ... WHERE ... < limit ... RETURNING` (still single-SQL, still race-safe). | Most correct semantically — counter only moves when a session actually got created. Closes BOTH the Zod 400 AND the handler-5xx holes. Symmetric with the "billing-adjacent state" doctrine the tracker calls out. | TOCTOU window between the read-check and the commit (two parallel zod-valid requests both pass the read with count = limit - 1, both then try to commit — one succeeds, one needs to be rejected on the commit). The commit is still the atomic `UPDATE ... < limit ...` so the WHERE-clause refusal returns 0 rows for the loser → must convert that into a 402 from inside the handler, AFTER `chatService.createSession` already created the session. Either we surface the late 402 (and orphan the just-created session in `chat_sessions`) or we tear down the session in a follow-up DELETE (introduces a NEW failure mode where the DELETE itself can fail). +30 LoC, +2 new test cases, +1 port method. The "session already created but commit refused" branch is genuinely ugly. |
| **(c) try/catch decrement** — Wrap handler body in `try / catch`. On catch, call `repo.decrement(userId, monthStart)` (new method ; single SQL `UPDATE users SET sessions_month_count = sessions_month_count - 1 WHERE id = $1 AND sessions_month_start = $2 AND sessions_month_count > 0`). Then re-throw. | Closes BOTH Zod 400 hole (catch fires on zod throw if we put `validateBody` AFTER the quota — but then we'd need `validateBody` to throw a typed error the route can distinguish from handler 5xx) AND handler 5xx. | TOCTOU on decrement itself : if a parallel request from same user increments between our increment + decrement, we decrement THEIR slot. The conditional `... AND sessions_month_count > 0` doesn't help — both increments are valid SQL, the decrement would race-undo whichever is the latest write. Genuinely fragile. Also requires the catch-block decrement to be best-effort (what if the DB connection is the thing that failed?). |

**Decision — option (a) "swap order", with explicit acceptance of the residual handler-5xx hole**, documented in §3.3 D3 and `CLAUDE.md § Pièges connus`.

**Why (a) over (b)** despite reviewer preference for (b) :

1. **Diff size + risk** — option (a) is a 3-line route change. Option (b) is +30 LoC middleware split + a new port method + a new failure mode ("session created but commit refused"). For a PR that closes one finding on a feature already APPROVED, the smaller diff has better cost/risk.
2. **Frequency analysis** — the bug as filed lists Zod 400 attack vectors as the primary path (non-object body, museumId ≤ 0, invalid coords, invalid intent). These are everyday client-side bugs (older mobile build, fuzzer traffic, retries with stale state). Handler 5xx (DB blip mid-INSERT, FK miss) is rare in steady state.
3. **Honesty** — option (b)'s "session created but commit refused" branch is **not strictly more correct** than option (a)'s "counter inflated on handler 5xx" : (b) creates an orphan session row in `chat_sessions` that the user can't reach (because the FE thinks it was 402'd), which is its own corruption. Trading "occasional counter inflation" for "occasional orphan session" is not a clean win.
4. **Forward-compat for (b)** — if post-launch telemetry on `quota_check_hit_limit` reveals the handler-5xx tail is material (e.g. >1 % of free-tier monthly counts are inflated by 5xxs), V1.1 can promote (b) without breaking the API contract. The current fix is "remove the easy hole now, document the hard one".

Honesty disclosure : the reviewer's stated preference is (b). The above reasoning argues for (a). The user (Tim) makes the final call — if "do (b) anyway" is the answer, the spec re-pivots and the §3.2 / §3.3 sections below switch from "moot under (a)" to "implemented" — Q1 captures this.

### 3.2 Concurrent race (D2) — TOCTOU resolution

**Under D1 = option (a)** : the race is **unchanged** from R1 baseline. Two concurrent valid-body requests both reach `monthlySessionQuota` after `validateBody` passes ; both invoke `tryConsume` which is a single atomic UPDATE with `WHERE id = $1 AND sessions_month_count < $limit` ; PostgreSQL serialises the row lock during UPDATE ; the second UPDATE re-evaluates the WHERE on the post-first-update row state ; if first bumped count to `limit`, the second's WHERE is false → 0 rows returned → 402. Risk1 (R1 §6) invariant preserved byte-for-byte. **No new TOCTOU window introduced by F1 option (a).**

**Under D1 = option (b)** (moot but documented) : the middleware reads count via `loadUser` (no lock) ; both requests pass the read with `count = limit - 1` ; both proceed to handler ; both handlers eventually call `commit` ; the commit is the same atomic `UPDATE ... < limit ... RETURNING` SQL ; PG serialises ; one commit succeeds (count moves limit-1 → limit), the other returns 0 rows. The looser's handler must convert that into a 402 AND delete the just-created session row (the orphan tear-down). The session DELETE is itself idempotent (DELETE WHERE id = $sessionId) and the failure mode (DELETE fails → orphan session permanently in DB, queryable via the user's session list because the count column on `chat_sessions` is just primary-key + FK, not gated) is the new ugliness. Mitigation : `SELECT ... FOR UPDATE` on the user row during the read step would serialise the reads themselves, but that introduces a real session-row contention hotspot under multi-device load — exactly the V1.1 concern Risk3 (R1 §6) names. **Not worth opening that lock in a corrective fix.**

**Under D1 = option (c)** (moot but documented) : the decrement race is unfixable without an opaque "did this request own this increment" token threaded through the catch block. Not a clean design. Ruled out.

### 3.3 Handler error taxonomy (D3) — which errors trigger decrement vs not

**Under D1 = option (a)** : NO decrement step exists. The counter mutation already happened by the time the handler runs. Two distinct sub-cases :

| Handler error | Counter outcome | Acceptable ? |
|---|---|---|
| `chatService.createSession` throws `AppError(400, 'userId must be a positive integer')` (defensive — req.user.id should always be valid since `isAuthenticated` guarantees it ; this is unreachable in practice) | Counter bumped + 400 returned. | Unreachable, moot. |
| Handler throws `AppError(404, 'museum not found')` (museumId FK miss on `chatService.resolveMuseumInfo`) | Counter bumped + 404. | **Slot inflated.** Documented residual hole. Frequency : depends on client bug rate. |
| Handler throws on DB connection blip mid-`INSERT INTO chat_sessions ...` (TypeORM throws) | Counter bumped + 500. | **Slot inflated.** Documented residual hole. |
| Handler throws on Redis cache invalidation (`this.cache.delByPrefix` after the INSERT succeeded — session IS created, just the cache prefix wipe failed) | Counter bumped + 500, session row exists. | Counter is **correct** here — the session was actually created. The 500 surfaces a non-critical post-success error. Marginal UX issue : user sees 500 but the session exists. Out of F1 scope (different bug class). |

**Honest framing** : option (a) trades the everyday Zod-400 vector (closed) for a rare handler-throw-after-counter-bump tail (~handful per month at our scale, pre-launch). The reviewer's option (b) trades the same Zod-400 vector for an orphan-session-on-commit-race tail. Neither is fully clean. Option (a) is honestly the smaller residual.

**Funnel impact mitigation under (a)** : `quota_check_hit_limit` continues to fire on TRUE 402s only — handler-5xxs don't fire the log (the log is in the 402-response branch of the middleware). So the KR4 funnel signal stays clean even if the underlying count is occasionally inflated by handler 5xxs. The **count itself** is what's inflated, not the funnel event count. KR4 reads the funnel event, not the counter ; the counter is only consulted for the 402 decision. So a free user occasionally getting 402'd one session early is the only visible symptom. Acceptable for V1.

### 3.4 Audit / observability (D4) — log when counter would have been incremented but wasn't

**Under D1 = option (a)** : the question is moot — under option (a), the counter is incremented **after** `validateBody` passes, so by construction the increment is paired with a zod-valid request. There is no "would have been incremented but wasn't" branch in the new pipeline. `quota_check_hit_limit` (R12) keeps its existing semantics : fires once per (user, month) on TRUE 402.

**Forward-compat note** : if D1 ever pivots to (b), we'd want a `quota_reserved_but_handler_failed` debug-level log emitted by the handler's catch block — counts how many sessions were reserved-but-not-committed, which is the new tail to monitor. Not built in F1.

**No new audit-log row** for any F1 case. The audit chain stays for security + lifecycle events (R1 Q6 doctrine — quota events are NOT audited).

### 3.5 Test strategy (D5) — Zod 400 case + handler-throw case + concurrent race case

**Three new unit-test cases** added to `monthlySessionQuota.middleware.test.ts` :

1. **F1.T-zod400** — Build a route-level test (Supertest against a mini Express app composed at test scope OR call `validateBody` and `monthlySessionQuota` in the same composition order as the route). Send body `[]` → assert 400 returned + `repo.tryConsume` was NEVER called.
2. **F1.T-handler-throw** — Mock `chatService.createSession` to throw. Send valid body. Assert : (a) the middleware's `tryConsume` WAS called (under D1 = option (a), the increment runs before the handler) ; (b) handler throws ; (c) global error handler returns 5xx. Comment in the test names this as the "documented residual hole — option (a) accepts handler-throw inflation per F1 §3.3 D3". The test asserts current behaviour, not desired behaviour — the assertion is "counter bumped". This makes the trade-off visible in the test corpus and prevents accidental regression to option (b) without an ADR.
3. **F1.T-concurrent-race** — In an integration test (real DB, two Promise.all parallel `POST /api/sessions`), assert exactly one 201 + one 402, and the final `users.sessions_month_count` row == `limit` (NOT `limit + 1`). Re-verifies R1 Risk1 mitigation post-F1.

Plus **one new integration test** : `chat-session-quota-no-inflation.integration.test.ts` (or extension of `chat-session-quota.integration.test.ts`) covering the exact bug scenario : 3 × invalid body (400) + 1 × valid body → 201 (not 402).

## Appendix A — Spec drift uncovered during inspection

- **R1 §3.3 D3 (lines 271-281)** explicitly codifies the buggy order with the comment "// ← NEW, R1" between `isAuthenticated` and `validateBody`. The R1 inspection at the time treated middleware ordering as a free choice ; the bug emerged because the spec didn't consider invalid-body counter mutation as a failure mode. Captured in Q5.
- **Inline route comment** at `chat-session.route.ts:30-33` describes the buggy order as a feature ("short-circuit before zod runs"). Anti-documentation. T2 rewrites it.
- **`createSessionSchema`** has `intent` as `z.enum(CHAT_SESSION_INTENTS).optional()` — strict enum (no fallthrough). So `intent: 'not_a_valid_intent'` is one of the everyday Zod-400 vectors. Confirmed at `chat-session.schemas.ts:98`.
- **Handler error surface (`chat-session.service.ts:74-97`)** : the only handler-side throw I found is `badRequest('userId must be a positive integer')` at line 76, which is unreachable in practice (auth middleware guarantees a valid user id). Other failure modes (museum FK miss, DB blip, cache invalidation failure) are implicit via TypeORM / cache adapter throws — not surfaced as named `AppError`s in the service. So §3.3 D3's residual-hole frequency estimate ("rare") relies on TypeORM not flaking, which is consistent with the rest of the codebase's posture.
- **Existing unit test `monthlySessionQuota.middleware.test.ts`** uses jest-mocked repo (`setMonthlyQuotaRepo`) — does NOT exercise the route composition. The F1 zod400 cases require either a new test composing `validateBody → monthlySessionQuota` in-process, or a Supertest integration test. T1 chooses the in-process composition for unit-level speed.
- **`monthlySessionQuota.middleware.ts:107-114`** has a fail-OPEN defensive branch when `repo` is unwired. F1 doesn't touch this — orthogonal concern.
- **`monthlySessionQuota.middleware.ts:130-132`** computes `limit` from `env.freeTierMonthlySessionLimit` with a literal `3` fallback. Used in both the 402 body (R8) and the SQL parameter. F1 preserves.
- **`router.post('/sessions', isAuthenticated, monthlySessionQuota, validateBody(createSessionSchema), async (req, res) => {...})` at lines 34-39** — confirms the buggy order shipped in commit `b05b201a` (R1 main implementation commit). Easy to swap.
- **`tryConsume` SQL (`monthly-session-quota.repo.pg.ts:43-67`)** uses parameterised SQL with explicit columns, `WHERE id = $1 AND tier = 'free' AND (start IS NULL OR start <> $2 OR count < $3) RETURNING ...`. Single statement, atomic. F1 byte-unchanged.
- **`tryConsume`'s `WHERE tier = 'free'` clause** means a tier flip from free → premium between `loadUser` and `tryConsume` would race-skip the increment (UPDATE matches zero rows because `tier != 'free'` post-flip). Then the middleware would treat the zero-row return as a 402. That's a pre-existing race, not introduced by F1 — only relevant if option (b) is ever adopted (then the read-check + commit window widens this race). Out of F1 scope ; noted for V1.1.
