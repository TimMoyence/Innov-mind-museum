## 1. Spec (EARS requirements)

- **R1.** WHEN the smoke script has created a chat session (line 314 today) THE SYSTEM SHALL POST a short text message to `/api/chat/sessions/:id/messages` with body `{ text: "<fixed FR prompt ~40 chars about an artwork>", context: {} }` and assert HTTP 201 with `message.id` (UUID string) and `message.role === 'assistant'` and non-empty `message.text`.
- **R2.** WHEN the chat 201 response is received THE SYSTEM SHALL capture `message.id` for the subsequent TTS call. No assumption on metadata content (citations, suggestions, etc. — see chat-ux worktree for that scope).
- **R3.** WHEN `message.id` is known THE SYSTEM SHALL POST to `/api/chat/messages/:messageId/tts` (no body) with the same `Authorization: Bearer <accessToken>` and assert HTTP 200 with `Content-Type` starting with `audio/`.
- **R4.** WHEN TTS responds 200 THE SYSTEM SHALL read the response as a binary buffer (`response.arrayBuffer()` → `Buffer.from`) and validate it starts with one of the two MP3 signatures :
  - `ID3` (bytes `0x49 0x44 0x33`) — id3v2 header, the common case from OpenAI TTS, OR
  - MPEG frame sync `0xFF 0xFB` / `0xFF 0xF3` / `0xFF 0xF2` (frame-sync first nibble `0xFFE` masked) — id3-less MP3.
- **R5.** WHEN the audio buffer is validated THE SYSTEM SHALL assert `buffer.length >= 1024` (1 KB minimum — anything smaller is almost certainly an error envelope misrouted or a truncated stream). Upper bound not enforced (TTS can legitimately be large for long replies, kept defensive but unbounded).
- **R6.** WHEN the assistant reply is empty/whitespace (`null` returned from `synthesizeSpeech`, surfaced as HTTP 204 by `createTtsHandler` line 184-186) THE SYSTEM SHALL fail with `[smoke:api] FAIL: TTS returned 204 (empty assistant text)` — empty replies indicate a chat regression upstream, NOT a TTS issue, but it's still a smoke failure.
- **R7.** WHEN the TTS endpoint responds 501 with `error.code === 'FEATURE_UNAVAILABLE'` THE SYSTEM SHALL exit 1 with `[smoke:api] FAIL: TTS unavailable (501 FEATURE_UNAVAILABLE)`. Doctrine : prod must have `OPENAI_API_KEY`; if it doesn't, the deploy is broken.
- **R8.** WHEN any unexpected HTTP status is returned by the message or TTS POST (i.e. not 201 for chat, not 200 for TTS) THE SYSTEM SHALL exit 1 with the full status + truncated body, matching the existing `fetchJson` pattern.
- **R9.** WHEN the TTS stage completes successfully THE SYSTEM SHALL log `[smoke:api] tts OK (bytes=<N>, contentType=<X>, msgId=<UUID-prefix-8>)` — consistent with `[smoke:api] compare OK (status=200, matches=N, modelVersion=X)` shape at line 378.
- **R10.** WHEN the TTS stage finishes (success OR failure) THE SYSTEM SHALL still proceed to the cleanup `DELETE /api/chat/sessions/:id` step. Rationale : a smoke session left behind is mild noise but a recurring leak across 100 prod deploys becomes admin-panel clutter. Implementation: wrap TTS in `try { ... } catch (err) { /* delete session, then rethrow */ }` OR move delete into a `finally` block.
- **R11.** WHEN running locally (`docker-compose.dev.yml` stack + dev `OPENAI_API_KEY`) THE SYSTEM SHALL pass identically to prod runs (no environment-conditional code, no skip-when-local branch).
- **R12.** WHEN reading any new env var THE SYSTEM SHALL not introduce one — reuses `SMOKE_API_BASE_URL`, `SMOKE_TEST_EMAIL`, `SMOKE_TEST_PASSWORD`, `SMOKE_TIMEOUT_MS` only.

## 2. Acceptance Criteria

- **AC1.** Running `pnpm smoke:api` against local Docker stack with `OPENAI_API_KEY` set exits 0, and stdout contains in order : `health OK`, `auth OK`, `create session OK`, `list sessions OK`, `compare OK ...`, `tts OK (bytes=..., contentType=audio/..., msgId=...)`, `cleanup delete session OK`, `PASS`.
- **AC2.** Running `pnpm smoke:api` against a backend where `OPENAI_API_KEY` is intentionally unset (DisabledTextToSpeechService active) exits 1, last log lines include `[smoke:api] FAIL: TTS unavailable (501 FEATURE_UNAVAILABLE)`.
- **AC3.** Running `pnpm smoke:api` against a backend where the chat assistant reply is somehow empty (forced via test stub) exits 1 with `TTS returned 204` message.
- **AC4.** No new env variables introduced — `git diff museum-backend/scripts/smoke-api.cjs museum-backend/.env.example` shows the `.env.example` is unchanged.
- **AC5.** The smoke does NOT write any file under `tmp/`, `museum-backend/test-results/`, or the repo root. Verified by running smoke, then `git status` — clean.
- **AC6.** The TTS stage runs AFTER `compare OK` and BEFORE `cleanup delete session OK` so a compare regression does not mask TTS, and so the temporary session always carries a non-trivial conversation history for the TTS call.
- **AC7.** Total stage duration (chat post + TTS post) < `SMOKE_TIMEOUT_MS` (15s default) on a healthy backend — no new timeout env.
- **AC8.** When the script aborts due to TTS failure, the `DELETE /api/chat/sessions/:id` cleanup STILL runs (verified by checking the deleted-session log line appears in `--trace` or by running against a backend with debug logging).
- **AC9.** Log line `[smoke:api] tts OK` matches the regex `/^\[smoke:api\] tts OK \(bytes=\d+, contentType=audio\/[a-z0-9.+\-]+, msgId=[0-9a-f]{8}\)$/` (contract test in T1).
- **AC10.** No `console.error` is emitted on the happy path (matches existing convention — only failures hit `console.error` via the top-level catch).

## 3. Design

### 3.1 Approach

Add a single `if (ttsEnabled) { ... }` block? **No** — N1 forbids a feature flag. Add a plain sequential block between the `compare` stage (line 391) and the `DELETE` cleanup (line 393), guarded by `try/finally` so cleanup always runs.

Structure :

```
... existing compare block ends at line 391 ...

// R5 (C7.1) — TTS voice round-trip.
// Always runs. No env flag — see CLAUDE.md feedback_no_feature_flags_prelaunch.
let assistantMessageId = null;
try {
  // 1. Post a fixed text message to drive an assistant reply.
  const chatPost = await fetchJson({
    baseUrl, path: `/api/chat/sessions/${createdSessionId}/messages`,
    method: 'POST', token: accessToken,
    body: { text: 'Bonjour, parle-moi de la Joconde.' },
    timeoutMs, expected: 201,
  });
  assistantMessageId = chatPost.json?.message?.id;
  // assertions on message shape...

  // 2. POST /messages/:id/tts → binary audio.
  //    Use a custom fetch (not fetchJson) because response is audio/mpeg.
  const ttsResult = await fetchBinary({
    baseUrl, path: `/api/chat/messages/${assistantMessageId}/tts`,
    method: 'POST', token: accessToken, timeoutMs,
    expected: 200,
  });
  // magic-byte + length assertions...
  console.log(`[smoke:api] tts OK (bytes=${ttsResult.length}, contentType=${ttsResult.contentType}, msgId=${assistantMessageId.slice(0, 8)})`);
} finally {
  // Cleanup runs unconditionally (R10).
}

... existing DELETE block continues at line 393 ...
```

A new helper `fetchBinary` is added (parallel to `fetchJson` and `fetchMultipart`), localized to `smoke-api.cjs`. It mirrors `fetchJson` but returns `{ status, buffer, contentType }`.

### 3.2 Endpoint choice (D1)

**Decision** : two endpoints chained.

1. `POST /api/chat/sessions/:id/messages` — text path (no audio upload). Verified at `museum-backend/src/modules/chat/adapters/primary/http/routes/chat-message.route.ts:189`. Returns `PostMessageResponse` (chat.contracts.ts:86-114) with `message.id` + `message.role: 'assistant'` + `message.text`. **No audio inline.** Audio is fetched separately.
2. `POST /api/chat/messages/:messageId/tts` — synthesize endpoint. Verified at `museum-backend/src/modules/chat/adapters/primary/http/routes/chat-media.route.ts:254-260` mounting `createTtsHandler` (line 179-193). Response body is raw `audio/mpeg` buffer (`res.send(result.audio)` line 191), `Content-Type` set from `result.contentType`.

The audio-upload alternative (`POST /api/chat/sessions/:id/audio`, chat-media.route.ts:222) is **rejected** because :
- It requires a real audio fixture (MP3/M4A) → fixture management surface.
- It exercises STT + LLM + TTS = 3 paid calls/run vs 2 for text→TTS.
- STT is implicitly covered by C7.5 device smoke.

### 3.3 Audio validation (D2)

**Decision** : magic-byte check only. No `ffprobe`, no `libmp3lame` parse.

Rationale :
- `ffprobe` adds a system dep that's not currently required by CI runners (`.github/workflows/ci-cd-backend.yml`).
- libmp3 npm parsers are flaky and add a runtime dep just for one assertion.
- Magic-byte alone catches the only realistic failure modes : (a) TTS service returned an error envelope misrouted as binary, (b) the response was truncated to 0 bytes, (c) the response was actually JSON because the route bypassed the TTS handler.
- Duration assertion (R3 in user's draft EARS list) is **dropped** in favor of byte-length floor (R5 in section 1) — proxy for "non-empty audio" without needing a decoder.

The two acceptable MP3 signatures :
- `0x49 0x44 0x33` (`ID3`) — id3v2 header.
- `0xFF` followed by `0xF*` matching MPEG sync (`(buf[1] & 0xE0) === 0xE0`) — id3-less frames.

OpenAI `gpt-4o-mini-tts` returns ID3-prefixed MP3 in our observed traffic per `text-to-speech.openai.ts`, but the second form is checked defensively.

### 3.4 Smoke account TTS quota (D3)

**Decision** : do not bypass cost guards. Accept the cost.

Verified : the `/messages/:messageId/tts` route applies `llmCostGuard` middleware (chat-media.route.ts:262). The smoke user is `role: 'visitor'` (seed-smoke-account.ts:51) and TTS for a single 30-char assistant reply costs ~$0.0003 (AI_VOICE.md §Coût). One run per deploy → negligible.

`dailyChatLimit` is also wired on the chat post route (chat-message.route.ts:191). Visitor users have a per-day cap. The smoke runs once per deploy → headroom is fine. If two deploys happen in the same calendar day and the smoke chat post hits 429, that's a real signal worth a loud failure, not a bypass.

No `X-Smoke-Bypass` header. No middleware short-circuit. The smoke user goes through the same chokepoints as a real user — exactly the point of a prod smoke.

### 3.5 Logging (D4)

**Decision** : exact format

```
[smoke:api] tts OK (bytes=<N>, contentType=<X>, msgId=<first-8-of-uuid>)
```

Matches existing patterns :
- `[smoke:api] compare OK (status=200, matches=0, modelVersion=siglip-base-...)` (line 378)
- `[smoke:api] health OK` / `[smoke:api] auth OK` (single-token form for trivial stages)

`msgId` is truncated to first 8 hex chars for log brevity (full UUID would bloat each line by 30+ chars and isn't needed for grep-locating a smoke event in CI logs).

No new Prometheus metric, no new structured log — the smoke is a binary pass/fail at the deploy gate.

### 3.6 Helper function shape (D5)

`fetchBinary` (new local helper) :

```
async function fetchBinary({ baseUrl, path, method = 'POST', token, body, timeoutMs, expected }) {
  // mirror fetchJson, but:
  //   - response body = arrayBuffer → Buffer
  //   - on error status: try to parse text body for diagnostic (JSON error envelope is likely)
  //   - return { status, buffer, contentType }
}
```

Justification for not reusing `fetchJson` : `fetchJson` calls `JSON.parse` on the body which would throw on binary MP3 input. Refactoring `fetchJson` to handle both is out of scope (touches every existing call). Local addition is cleaner.

## 4. Open Questions

### Q1. Does the chat endpoint apply guardrails that might block the fixed prompt?
The fixed prompt `"Bonjour, parle-moi de la Joconde."` is a textbook art question and SHOULD pass the V1 keyword guardrail (`art-topic-guardrail.ts`) + V2 LLM Guard sidecar trivially. But the V2 LLM Guard sidecar is `fail-CLOSED` (CLAUDE.md §AI Safety, ADR-047). **If the prod sidecar is offline at smoke time, the chat POST returns 5xx and the smoke fails.** Is that desired (loud signal that the guardrail layer is down) or should the smoke include a fallback prompt? — **Default position : keep loud failure. Prod-safety stack health is part of what smoke MUST verify.** Confirm before T1.

### Q2. Is there a daily chat-quota carryover risk?
`dailyChatLimit` middleware is per-user-per-day. If multiple deploys land in the same UTC day (e.g. hotfix sequence), the smoke could hit 429 on the chat post. Acceptable signal, or should the smoke distinguish 429 from real chat failure? — **Default position : 429 = real failure, exit 1.** A hotfix burst exceeding daily chat quota = real prod issue worth waking someone. Confirm with user before red-test-agent.

### Q3. Should the smoke also verify `audioUrl` was persisted on the ChatMessage row?
After TTS synthesis, `ChatMediaService.synthesizeSpeech` writes `audioUrl` + `audioGeneratedAt` to the message (`chat-media.service.ts:320-324`). A follow-up `GET /api/chat/sessions/:id` returning the message with `audioUrl` set would verify the S3 persistence side-effect end-to-end. **But** : the persistence is `fire-and-forget` (line 326 comment), so it's racy — the TTS HTTP response may return before the DB write commits. Polling-with-retry would inflate stage complexity. — **Default position : DO NOT verify `audioUrl` in R5.** Scope to round-trip success. S3 persistence is covered by separate tests (`chat.repository.typeorm.spec.ts`). Confirm.

### Q4. What is the actual prod `OPENAI_API_KEY` budget headroom?
The user has a soft monthly OpenAI cap configured (referenced in `feedback_no_solo_dev_estimates` orientation but not explicit). 1 smoke / deploy × ~50 deploys/month × $0.001 = $0.05/month. Trivial. **No question to user, just stated for context.**

### Q5. Should TTS voice be pinned in the smoke?
The smoke user's `User.ttsVoice` defaults to `null`, which falls back to `env.tts.voice = 'alloy'` (chat-media.service.ts:290 + env.ts:236). If a future migration sets `ttsVoice` on the smoke user accidentally, the smoke would silently exercise a different voice. — **Default position : DO NOT pin voice in R5.** A voice drift on the smoke user is itself worth catching (seed-smoke-account.ts could be hardened separately). Confirm not to over-rotate.

### Q6. Is the `dailyChatLimit` middleware bypass-eligible for the smoke user?
Per AC4 + N1, no. But want to double-check there's no existing internal allow-list. — Grep target before T1 : `grep -r "SMOKE_TEST_EMAIL\|isSmokeUser" museum-backend/src/shared/middleware/`. **If grep finds an existing bypass we INHERIT it (no change). If not, no new bypass added.**

## 6. Risk register

- **R1 (cost)** — Daily quota carryover (Q2). Likelihood : low (smoke = 1 chat/deploy). Mitigation : let 429 surface as failure, real signal.
- **R2 (false negative — flaky LLM)** — OpenAI rare 5xx blip on the chat post stage drops smoke. Likelihood : medium. Mitigation : the existing `fetchJson` has no retry; this is consistent with all other smoke stages. Operator runs `pnpm smoke:api` manually to retry — acceptable for prod smoke discipline.
- **R3 (S3 leak)** — Every smoke run persists an audio MP3 to S3 via fire-and-forget side-effect of `ChatMediaService.synthesizeSpeech`. Over 1000 deploys = 1000 orphan MP3s. Mitigation : `s3-orphan-purge.job.ts` already exists in chat module and handles this; smoke session is deleted (R10) so the message rows are removed → orphan job cleans the S3 keys. **Verify** — Q3-adjacent, confirm purge job handles this case.
- **R4 (guardrail false positive)** — V2 LLM Guard or V1 keyword guardrail blocks the fixed prompt on a future tightening. Likelihood : low (the prompt is intentionally textbook-art). Mitigation : pick a deliberately unambiguous prompt (`"Bonjour, parle-moi de la Joconde."` — Mona Lisa in French = canonical art topic). If it breaks, that IS a smoke-worthy regression in guardrails.
- **R5 (smoke-user TTS quota)** — User has per-day TTS limit somewhere not yet identified. Q6 / grep before T2. If found, dispatcher decision : raise smoke-user limit or accept failure mode.
- **R6 (TTS endpoint contract change)** — A future refactor changes the response from raw `audio/mpeg` to JSON `{ audioUrl }`. Smoke would break. Mitigation : that's exactly what smoke is for — loud failure on contract drift.

- **STT round-trip** (audio upload → transcription → assistant reply). Requires fixture audio file. Tracked separately if needed post-V1. C7.5 device smoke covers this on real iPhone.

- **TTS cache hot-path verification** (Redis `tts:<msgId>` hit on second call). Would require two TTS calls per smoke run, doubling cost. Defer.
- **`audioUrl` persistence verification** (Q3). Race condition with fire-and-forget write. Covered by `chat.repository.typeorm.spec.ts`.
- **TTS voice rotation check** (Q5). Future hardening of `seed-smoke-account.ts` if `User.ttsVoice` drifts become a real incident.
- **Voice catalog drift** (e.g. `alloy` retired by OpenAI). Out-of-band monitoring problem, not a smoke responsibility.
- **WebRTC Realtime smoke** (V1.1 if pipeline migrates per AI_VOICE.md §Évolution V1.1). N/A for V1.
