## 1. Spec (EARS requirements)

### Landing section (museum-web)

- **R1.** WHEN un utilisateur navigue sur `/fr` ou `/en` THE SYSTEM SHALL render un nouveau bloc `<BetaSignupSection>` après `<LandingDownloadCTA>` (avant `<Footer>` injecté par le layout), avec un titre + sous-titre + form minimal.
- **R2.** WHEN le bloc `BetaSignupSection` rend THE SYSTEM SHALL exposer un `<section id="beta-signup" aria-labelledby="beta-signup-heading">` contenant exactement : un `<h2 id="beta-signup-heading">` (`dict.landing.beta.heading`), un `<p>` sous-titre (`dict.landing.beta.subheading`), un `<form>` avec UN input email + UNE checkbox consent + UN honeypot caché + UN bouton submit.
- **R3.** WHEN le `<form>` rend THE SYSTEM SHALL contenir : (a) `<input type="email" name="email" required autoComplete="email" maxLength={254}>` avec `<label>` visible (`dict.landing.beta.fieldEmail`), (b) `<input type="checkbox" name="consent" required>` avec label visible incluant un `<a href="/[locale]/privacy">` (`dict.landing.beta.consentPrivacyLink`), (c) honeypot `<input type="text" name="website" tabIndex={-1} autoComplete="off" aria-hidden="true">` positionné off-screen (mirror R4 `style={{position:'absolute', left:'-10000px',...}}` lignes 279-282), (d) `<button type="submit">` avec label `dict.landing.beta.submit` (état idle) / `dict.landing.beta.sending` (état pending, `disabled={submitting}`).
- **R4.** WHEN aucun field marketing additionnel (nom, prénom, préférences, musée d'intérêt) N'EST présent — friction maximalement basse. ZERO additional input.
- **R5.** WHEN any TSX file under `src/app/[locale]/BetaSignupSection.tsx` est parsé THE SYSTEM SHALL NOT contain de strings FR ou EN brutes hors `dict.landing.beta.*` (enforced par Vitest grep-test, voir §1 R-T1).

### Form behavior + submission

- **R6.** WHEN l'utilisateur submit le form avec email valide + consent coché + honeypot vide THE SYSTEM SHALL POST `application/json` vers `/api/leads/beta` avec body `{ email, consent: true, website: '' }` (le honeypot value est envoyé tel quel pour permettre BE silent-drop côté use case).
- **R7.** WHEN la réponse est 202 (accepted) THE SYSTEM SHALL afficher le message `dict.landing.beta.success` dans une région `<div role="status" aria-live="polite">` et masquer le form (mirror `B2bContactForm.tsx` lignes 86-95).
- **R8.** WHEN la réponse est non-2xx (network error, 4xx, 5xx, 429 rate-limit) THE SYSTEM SHALL afficher `dict.landing.beta.error` dans le `aria-live` region, garder les contents du form intacts, restaurer le bouton submit en idle.
- **R9.** WHEN le submit est en cours THE SYSTEM SHALL désactiver le bouton submit ET afficher `dict.landing.beta.sending` comme texte.
- **R10.** WHEN le honeypot field `website` est rempli (non-empty après trim) AND le form est submitted THE SYSTEM SHALL toujours afficher le success state à l'utilisateur (silent-accept au niveau UX) mais le BE handler DROP le payload sans appel Brevo. Prevent bot enumeration via response timing.
- **R11.** WHEN la consent checkbox est unchecked THE SYSTEM SHALL bloquer client-side via `required` attribute AND BE valide `consent === true` server-side et retourne 400 si absent — defense-in-depth (mirror R4 R11).

### BE endpoint

- **R12.** WHEN BE reçoit `POST /api/leads/beta` avec body valide THE SYSTEM SHALL appliquer un rate-limit middleware (5 req / 600s / IP) — mêmes paramètres que `b2bLeadLimiter` (`leads.route.ts:13-17`) — et répondre 202 `{ accepted: true }` sur success.
- **R13.** WHEN BE process une beta signup valide AND `BREVO_API_KEY` + `BREVO_BETA_LIST_ID` sont set THE SYSTEM SHALL appeler **directement** l'API Brevo `POST /v3/contacts` avec body `{ email, listIds: [<BREVO_BETA_LIST_ID>], updateEnabled: true, attributes: { OPT_IN: true } }` (voir D4 pour le shape exact et le rationale "PAS sendEmail").
- **R14.** WHEN BE n'a PAS `BREVO_API_KEY` OR PAS `BREVO_BETA_LIST_ID` configuré (local dev) THE SYSTEM SHALL fallback sur un `NoopBetaSignupNotifier` qui `logger.warn('beta_signup_notifier_noop', { emailDomain })` — l'email est dropped en local mais la submission renvoie 202 (success UX cohérent pour test E2E local).
- **R15.** WHEN Brevo répond non-2xx THE SYSTEM SHALL throw une erreur descriptive avec le status + body slice (mirror `BrevoEmailService` lignes 29-32 `brevo-email.service.ts`). Le route handler propage en 5xx générique (laisse le middleware d'erreur global gérer le format).
- **R16.** WHEN Brevo répond 400 avec `code: "duplicate_parameter"` (Brevo's signal qu'un contact existe déjà avec cet email) THE SYSTEM SHALL traiter le cas comme un succès idempotent : 202 retourné à l'utilisateur, log `beta_signup_already_subscribed` info-level. Voir Q3 pour le rationale.
- **R17.** WHEN BE valide un beta signup THE SYSTEM SHALL log structured `beta_signup_submitted` (logger.info, `requestId + emailDomain + honeypotTriggered: false`). Aucun email full logged (PII discipline, mirror R4 R20).

### i18n + a11y

- **R18.** WHEN le dictionary i18n est chargé THE SYSTEM SHALL inclure `dict.landing.beta.{heading, subheading, fieldEmail, fieldConsent, consentPrivacyLink, submit, sending, success, error}` dans `fr.json` ET `en.json` — full parity. Vitest dict-symmetry test enforce.
- **R19.** WHEN axe-core run sur `/fr#beta-signup` ET `/en#beta-signup` THE SYSTEM SHALL reporter zéro violation WCAG 2.1 AA — threshold identique à `public-support.a11y.spec.ts`.
- **R20.** WHEN le form rend THE SYSTEM SHALL exposer `<label>` pour chaque input visible, `aria-describedby` linkant consent à privacy link, `aria-live="polite"` sur success/error region, focus ring matching `Button.tsx` discipline.

## 3. Design

### 3.1 Landing section position (D1)

**Decision** : **bloc dédié post-`LandingDownloadCTA`, pré-Footer.** PAS Hero CTA, PAS modification de `LandingHero.tsx`.

Rationale :

| Option | Pour | Contre | Verdict |
|---|---|---|---|
| **(a) Hero CTA** (modifier `LandingHero.tsx`, ajouter input email à côté du CTA principal) | Maximum visibility above-the-fold ; capture intent à l'arrivée | Touche un composant existant complexe (orb animations + filtres SVG) ; densifie le Hero qui est déjà chargé (title + subtitle + 2 CTAs) ; collision potentielle avec le download CTA qui est le CTA Hero actuel ; risque a11y (focus management hero) | REJECTED |
| **(b) Post-`LandingDownloadCTA` block** (nouveau `<section>` indépendant en bas de page) | Isolation totale — aucun fichier landing existant modifié à part `page.tsx` (1 import + 1 ligne JSX) ; collision-zero avec animations hero ; respecte le flow narratif : story → showcase → features → FAQ → download → "et si t'es pas prêt à download, inscris-toi à la bêta" ; matche le placement standard SaaS "stay in the loop" en footer-ish | Moins de visibility above-the-fold ; conversion plus faible si l'utilisateur quitte avant scroll bottom | **CHOSEN** |
| **(c) Sticky banner top-of-page** | Visibility constante | Anti-pattern UX 2026, dégrade Lighthouse Best Practices, demande motion-reduce gating | REJECTED |
| **(d) Modal sur exit-intent** | Conversion technique élevée | Anti-pattern, dégrade a11y AAA, demande JS heavy | REJECTED |

**Justification finale** : on n'a PAS de baseline funnel mesurée pour arbitrer (a) vs (b) data-driven. En pre-launch + scope-tight + collision-zero avec animations Hero, **(b) est le minimum-risk maximum-isolation choice**. Si KR4 ne décolle pas semaine 1 post-launch, on a un follow-up V1.1 trivial = remonter le bloc en Hero CTA (refactor 30 min). Voir Q5.

### 3.3 BE use case (D3)

**Decision** : nouveau use case `submitBetaSignup` parallèle à `submitB2bLead`, dans le **même module `leads/`**. PAS de duplication, MAIS pas de fusion non plus.

Pourquoi ne pas étendre `submitB2bLead` avec un discriminant `kind: 'b2b' | 'beta'` ?

| Aspect | b2b | beta |
|---|---|---|
| Fields | email + name + museum + role + message (5 required) | email seul |
| Action notifier | `sendEmail()` transactionnel vers inbox interne (`B2B_INBOX_EMAIL`) | `POST /v3/contacts` Brevo (subscription liste, pas email transactionnel) |
| Port outbound | `B2bLeadNotifier.notify(payload)` | `BetaSignupNotifier.subscribe(payload)` (verbe différent — sémantique différente) |
| Success contract | Email envoyé à founder | Contact ajouté à la list 17 (ID Brevo) |
| Idempotency | Pas d'idempotence (chaque submission = nouveau mail) | Idempotent (Brevo `updateEnabled: true` + handle 400 duplicate_parameter) |

Les 2 use cases partagent **la validation pattern** (consent + email + honeypot + structured logger) MAIS ils écrivent vers 2 sous-systèmes Brevo différents avec 2 contracts différents. Fusionner forcerait à factoriser une validation helper (`validateLeadCommonFields()`) qui n'a pas encore de 3ème usage attendu. Wait-for-the-3rd-occurrence rule (R1 modal upsell mobile arrivera bientôt — c'est à ce moment qu'on extraira le helper, si justifié).

**Shape du nouveau use case** (mirror `submitB2bLead.useCase.ts:40-106`) :

```typescript
class SubmitBetaSignupUseCase {
  constructor(private readonly notifier: BetaSignupNotifier) {}

  async execute(input: SubmitBetaSignupInput): Promise<void> {
    if (!input.consent) throw badRequest('consent must be true');

    const email = input.email.trim().toLowerCase();
    if (!validateEmail(email)) throw badRequest('email must be valid');

    // Honeypot silent drop (mirror R4 R10).
    if (typeof input.website === 'string' && input.website.trim().length > 0) {
      logger.warn('beta_signup_honeypot_triggered', { requestId: input.requestId });
      return;
    }

    logger.info('beta_signup_submitted', {
      requestId: input.requestId,
```

### 3.4 Brevo integration (D4) — **CRITICAL OPEN POINT**

**Verified by inspection** : `BrevoEmailService` (`museum-backend/src/shared/email/brevo-email.service.ts`) expose **UNIQUEMENT** `sendEmail(to, subject, htmlContent)` — un wrapper raw-fetch vers `POST https://api.brevo.com/v3/smtp/email`. **AUCUNE méthode `subscribeContactToList(email, listId)` n'existe**. La port `EmailService` (`email.port.ts:2-5`) est minimale (1 méthode `sendEmail`).

R3 ne peut **PAS** réutiliser `BrevoEmailService` tel quel pour subscriber un contact à une liste — l'endpoint Brevo est différent (`/v3/contacts` vs `/v3/smtp/email`), le payload est différent, et la sémantique est différente (subscription idempotente vs envoi transactionnel one-shot).

**Decision** : créer un **nouvel adapter dédié** `BrevoBetaSignupNotifier` dans `museum-backend/src/modules/leads/adapters/secondary/notifier/brevo-beta-signup.notifier.ts`. **Pas** d'extension de `BrevoEmailService`, **pas** d'élargissement de la port `EmailService` (qui doit rester orientée transactional).

```typescript
// brevo-beta-signup.notifier.ts (NEW)
export class BrevoBetaSignupNotifier implements BetaSignupNotifier {
  constructor(
    private readonly apiKey: string,
    private readonly listId: number,
  ) {}

  async subscribe(payload: BetaSignupPayload): Promise<void> {
    const response = await fetch('https://api.brevo.com/v3/contacts', {
      method: 'POST',
      headers: {
        'api-key': this.apiKey,
        'Content-Type': 'application/json',
        Accept: 'application/json',
      },
      body: JSON.stringify({
        email: payload.email,
        listIds: [this.listId],
        updateEnabled: true,  // idempotent : add-or-update
        attributes: {
          OPT_IN: true,
          OPT_IN_AT: new Date().toISOString(),
          OPT_IN_SOURCE: 'landing_beta_waitlist',
        },
      }),
    });

    if (!response.ok) {
      const bodyText = await response.text().catch(() => '');
      // Brevo signals "already subscribed" via 400 + code: "duplicate_parameter".
      // Idempotent success per R16.
      if (response.status === 400 && bodyText.includes('duplicate_parameter')) {
        logger.info('beta_signup_already_subscribed', {
          requestId: payload.requestId,
          emailDomain: payload.email.split('@')[1],
        });
        return;
      }
      throw new Error(`Brevo contacts add failed (${response.status}): ${bodyText.slice(0, 800)}`);
    }
  }
}

export class NoopBetaSignupNotifier implements BetaSignupNotifier {
  async subscribe(payload: BetaSignupPayload): Promise<void> {
    logger.warn('beta_signup_notifier_noop', {
      requestId: payload.requestId,
      emailDomain: payload.email.split('@')[1] ?? 'unknown',
    });
  }
}
```

**Composition root** (`leads/useCase/index.ts` extension) :

```typescript
const betaSignupNotifier = (env.brevoApiKey && env.brevoBetaListId)
  ? new BrevoBetaSignupNotifier(env.brevoApiKey, env.brevoBetaListId)
  : new NoopBetaSignupNotifier();

export const submitBetaSignupUseCase = new SubmitBetaSignupUseCase(betaSignupNotifier);
```

**Brevo API contract** (verified externally, doc public Brevo `/v3/contacts` POST) :
- Endpoint : `POST https://api.brevo.com/v3/contacts`

- Success : `201 Created` avec `{ id: <contact-id> }` ; OR `204 No Content` si update via `updateEnabled: true`

## 6. Risk register

- **Risk1 (Brevo list ID misconfigured)** — `BREVO_BETA_LIST_ID` pointing vers la mauvaise liste Brevo (ex : reuse accidentel de la list ID d'un projet précédent). - **Risk5 (BE noop fallback in prod by accident)** — If `BREVO_BETA_LIST_ID` empty in prod env, all submissions silently dropped (noop notifier). User experience = "success" mais aucun email reçu.
