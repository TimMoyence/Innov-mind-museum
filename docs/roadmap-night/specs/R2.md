### 0.5 NFR — non-functional constraints

- **N1 — RFC 4180 compliance.** Quote any field containing `"` (doubled), `,`, `\n`, `\r`. Line terminator = `\r\n`. Header row first.
- **N2 — UTF-8 + optional BOM.** Body = UTF-8. Response header `Content-Type: text/csv; charset=utf-8`. BOM `﻿` prepended IF query param `?bom=1` OR `User-Agent` contains `Windows` / `Excel` — see Q5 for verdict. **Default position : always prepend BOM** (Excel-friendly, ignored by `csv` Python/Node parsers, ≤3 byte cost). Doc this in the CSV writer + matching test.
- **N3 — RFC 7807 problem+json on errors.** 4xx/5xx return JSON, not partial CSV. NO mixed-format response (e.g. a 200 with error rows in CSV is forbidden).
- **N4 — RBAC enforced server-side.** `museum_manager` MUST NOT receive rows from other museums — enforced in the WHERE clause of the repo, NOT a post-filter in the use case (defense-in-depth + saves bandwidth on large datasets). `super_admin` sees all. `admin` (B2B tenant) — see Q2 for scope (default : same as museum_manager).
- **N5 — Filename has no PII.** Pattern `<kind>-<YYYY-MM-DD>.csv`. Specifically NOT `<kind>-<userEmail>.csv` or `<kind>-<museumName>.csv`. Filename leaks via browser history + downloads folder + screenshots ; legal team flagged this on previous DSAR work.
- **N6 — Audit log durable before stream starts.** `auditService.log(...)` is `await`-ed BEFORE the first byte of CSV body is flushed. This mirrors the `me/export` pattern (`me.route.ts:83-104` — audit awaited before send). Failure rationale : if Postgres for audit hash chain is unavailable, the export MUST NOT proceed (SOC2 CC7.2 requires the audit row exists before privileged data leaves the perimeter).
- **N7 — Streaming for forward-compat.** Rows ≥ 1000 stream chunk-by-chunk via `res.write()` (`Transfer-Encoding: chunked`) to avoid heap pressure ; <1000 may buffer. Default : ALWAYS stream (one code path, no threshold). Matches the >10MB streaming logic in `me.route.ts:109-118` but starts streaming from row 0.
- **N8 — CSV injection prevention.** Any field whose first character is `=`, `+`, `-`, `@`, `\t`, `\r` is prefixed with a leading single-quote `'`. This neutralises Excel/LibreOffice formula injection (OWASP standard). Test row : a review with `comment = "=cmd|'/c calc'!A1"` → CSV cell = `"'=cmd|'/c calc'!A1"`.
- **N9 — Audit hash chain unbroken.** Audit row metadata MUST NOT include free-text user input (review comments, ticket descriptions) — only counts + filters. The hash chain (`audit-chain.ts:42`) hashes metadata JSON ; PII in metadata = retention liability beyond the audit chain's own retention (13 months per audit.service.ts:243). Metadata schema : `{ kind, rowsExported, bytesWritten, scopeMuseumId?, dateFrom?, dateTo? }`.
- **N10 — Cache-Control: no-store.** Same as `me.route.ts:68`. Exports are sensitive ; no proxy/browser cache.
- **N11 — Rate limit.** Per-user `byUserId` limit = **10 exports / hour** (mirror `me.route.ts:29-34` but looser since admins are trusted + legitimately re-pull). 11th request → 429 + `Retry-After`. NOT 1/7d like DSAR — admins are not the abuse surface DSAR worries about.
- **N12 — i18n.** Three FE buttons + states use `dict.admin.export.*` only. No hardcoded FR/EN strings in `ExportCsvButton.tsx`. CSV column headers stay English-only for parser stability (CSV headers are machine-targets ; localising them breaks downstream tooling that may consume the CSV).
- **N13 — A11y.** Button has visible label + `aria-busy={loading}` + ARIA live region announcing `dict.admin.export.ariaDownloading` start and `dict.admin.export.ariaDone` end. Disabled state during fetch. RTL-compatible (no fixed `left`/`right` ; use `start`/`end` per `feedback_rtl_start_end`).
- **N14 — Honesty UFR-013.** If RBAC scope returns 0 rows (e.g. museum_manager whose museum has no reviews), CSV = header-only row + 0 data rows. NOT 404. Audit row still emitted with `rowsExported: 0`.
- **N15 — No new prod dep.** RFC 4180 writer is ~80 LoC implemented in-house. No `csv-stringify`, `papaparse`, `fast-csv` added (see D2).
- **N16 — No DB migration.** R2 ships zero schema change. `node scripts/migration-cli.cjs generate --name=Check` must produce empty after merge.

## 1. Spec (EARS requirements)

### Backend endpoints

- **R1.** WHEN any client sends `GET /api/admin/export/sessions.csv` with a valid admin JWT (role ∈ {`super_admin`, `admin`, `museum_manager`}) THE SYSTEM SHALL respond `200 OK` with `Content-Type: text/csv; charset=utf-8`, `Content-Disposition: attachment; filename="sessions-<YYYY-MM-DD>.csv"`, `Cache-Control: no-store`, and a CSV body whose first line is the header row.
- **R2.** WHEN any client sends `GET /api/admin/export/reviews.csv` (same auth) THE SYSTEM SHALL respond identically but with `filename="reviews-<YYYY-MM-DD>.csv"` and the reviews header row.
- **R3.** WHEN any client sends `GET /api/admin/export/tickets.csv` (same auth) THE SYSTEM SHALL respond identically but with `filename="tickets-<YYYY-MM-DD>.csv"` and the tickets header row.
- **R4.** WHEN the requester's role is `visitor` OR `moderator` THE SYSTEM SHALL respond `403 Forbidden` (problem+json) before any DB read. Audit NOT emitted (failed-auth doesn't count as export — `requireRole` already emits its own access-denied path).
- **R5.** WHEN the requester is unauthenticated (no JWT) THE SYSTEM SHALL respond `401 Unauthorized` via `isAuthenticated` middleware — same shape as every other admin route.
- **R6.** WHEN the requester's role is `museum_manager` THE SYSTEM SHALL include in the result set only rows whose `museum_id` matches the requester's `user.museumId`. The scope filter is applied as a `WHERE museum_id = $scope` SQL predicate in the repository — NOT as a post-filter in the use case.
- **R7.** WHEN the requester's role is `super_admin` THE SYSTEM SHALL include all rows regardless of `museum_id` (no WHERE on tenant).
- **R8.** WHEN the requester's role is `admin` (B2B tenant) THE SYSTEM SHALL apply the same museum-scope filter as `museum_manager` (i.e. `WHERE museum_id = user.museumId`). See Q2 for confirmation.
- **R9.** WHEN the requester's role is `museum_manager` OR `admin` AND the user has no `museumId` set (anomaly) THE SYSTEM SHALL respond `403 Forbidden` with a problem+json explaining "no museum assigned". This is the safe default; it should never occur in normal data but defends against misseeded B2B accounts.
- **R10.** WHEN the requester provides query params `?from=YYYY-MM-DD` AND/OR `?to=YYYY-MM-DD` THE SYSTEM SHALL filter on `created_at` (sessions: `started_at`/`createdAt`; reviews + tickets: `createdAt`). Invalid date format → 400.
- **R11.** WHEN no `from`/`to` provided THE SYSTEM SHALL default to last 365 days (RGPD-friendly retention window — see Q4 for the configurable-vs-hard cap decision).
- **R12.** WHEN the use case starts executing THE SYSTEM SHALL audit-log via `auditService.log({ action: AUDIT_ADMIN_EXPORT_<KIND>, actorType: 'user', actorId: req.user.id, metadata: { kind, scopeMuseumId, dateFrom, dateTo, rowsExported: <computed-after>, ... }, ip, requestId })`. The audit row MUST be persisted (`await` resolved) BEFORE the first CSV byte is written to the response.
- **R13.** WHEN streaming a CSV response THE SYSTEM SHALL write the header row first, then each data row terminated by `\r\n`. The connection uses `Transfer-Encoding: chunked` (Express omits `Content-Length` when `res.write` is used).
- **R14.** WHEN a field value contains `"`, `,`, `\r`, or `\n` THE SYSTEM SHALL wrap the field in `"..."` and escape `"` as `""` (RFC 4180 § 2.6 / 2.7).
- **R15.** WHEN a field value's first character is `=`, `+`, `-`, `@`, `\t`, or `\r` THE SYSTEM SHALL prefix the field with `'` to neutralise Excel/LibreOffice formula injection (OWASP CSV-Injection).
- **R16.** WHEN N2 BOM strategy is "always prepend" THE SYSTEM SHALL begin the body with the bytes `EF BB BF` (UTF-8 BOM) before the header row.

### Anonymisation rules per role

- **R17.** WHEN exporting `sessions.csv` for any role THE SYSTEM SHALL include columns : `id, user_id, museum_id, started_at, ended_at, message_count, locale`. The `user_id` column contains the raw integer user id for `super_admin` ; for `museum_manager` and `admin` it contains a SHA-256-truncated pseudonym hex digest (16 chars) salted by env `AUDIT_PSEUDONYM_SALT` (see Q6 for salt-source confirmation).
- **R18.** WHEN exporting `reviews.csv` for any role THE SYSTEM SHALL include columns : `id, user_id_pseudonym, user_name, rating, comment, status, created_at`. Comment text is included AS-IS (with N8 injection escaping) — no PII redaction since reviews are user-facing content already published. `user_id_pseudonym` is always the salted hash (NEVER raw integer) regardless of role — review authors are end-users, raw ID would let admin pivot back to the visitor account.
- **R19.** WHEN exporting `tickets.csv` for any role THE SYSTEM SHALL include columns : `id, user_email_pseudonym, category, status, priority, subject, assigned_to, created_at, updated_at`. The `user_email` is replaced by `user_email_pseudonym` (SHA-256 truncated, salted) for ALL roles — including `super_admin`. Raw email is reachable via `/api/admin/users/<id>` for super_admin via separate audited action, NOT bundled into CSV. (See Q7 for super_admin policy.)
- **R20.** WHEN exporting any kind THE SYSTEM SHALL escape tab `\t` (HT) inside field values by wrapping in `"..."` to prevent some spreadsheet parsers treating `\t` as a column separator regardless of the comma.

### Frontend

- **R21.** WHEN any of the 4 admin pages (`analytics`, `support`, `tickets`, `reviews`) renders THE SYSTEM SHALL include a `<ExportCsvButton>` component in the header area. `kind` prop = `sessions` for analytics, `tickets` for support+tickets, `reviews` for reviews.
- **R22.** WHEN the user clicks the export button THE SYSTEM SHALL set `loading=true`, render `dict.admin.export.loading` as button label, set `aria-busy=true`, and announce `dict.admin.export.ariaDownloading` in an `aria-live="polite"` region.
- **R23.** WHEN the fetch resolves 2xx THE SYSTEM SHALL convert the response to `Blob`, create `URL.createObjectURL(blob)`, synthesise an `<a download="<filename-from-Content-Disposition>" href="…">` click, then `URL.revokeObjectURL(url)`. Loading state cleared, `aria-live` announces `dict.admin.export.ariaDone`.
- **R24.** WHEN the fetch resolves non-2xx (4xx, 5xx, 429) THE SYSTEM SHALL render `dict.admin.export.error` adjacent to the button (NOT in a modal). The Retry-After header on 429 is surfaced via the error label (e.g. "Try again in 47 minutes" — formatted via `dict.admin.export.errorRateLimit`).
- **R25.** WHEN the FE component is parsed THE SYSTEM SHALL NOT contain hardcoded FR/EN string literals (`dict.admin.export.*` only).

### 3.2 CSV library choice (D2)

**Decision** : **NO new npm dep. Implement RFC-4180 streaming writer in-house** at `museum-backend/src/shared/csv/csv-writer.ts` (~80 LoC).

Inspection : `museum-backend/package.json` (lines 126-162) contains NO csv-related dep — checked `csv-stringify`, `papaparse`, `fast-csv`, `csv-writer`. `museum-web/package.json` likewise has none.

Rationale :

| Option | Pro | Con | Verdict |
|---|---|---|---|
| **(a) `csv-stringify`** (`stream-friendly`, ~120kB unpacked, ~3 deps) | Battle-tested, supports streaming via Node streams API | New prod dep ; transitive deps need audit (Renovate, supply-chain) ; pnpm install size hit ; ADR threshold (new dep policy) | REJECTED for new-dep cost ; would be best-in-class IF deps were free |
| **(b) `papaparse`** (~50kB, no deps) | Mature, well-known | Primary use case is PARSING (admin app reads CSV) — its writer is a secondary surface ; tree-shaking imperfect | REJECTED |
| **(c) In-house writer** (~80 LoC RFC-4180 + escape) | Zero dep ; tailored to N8 injection neutralisation ; smaller surface area | We own the bugs ; we own the tests | **CHOSEN** for V1 |

The RFC-4180 spec is tiny (5 rules). Tests cover : quote-wrap-when-needed, double-quote-escape, formula-injection-neutralisation, BOM toggle, header row, multi-byte UTF-8 (emoji safety), CRLF terminator. ~50 LoC of test code. Cost-benefit < a new prod dep.

If exports grow into >10k row P95 / multi-table joins / async transforms, V1.1 reconsider `csv-stringify` via ADR.

### 3.3 Streaming vs buffer (D3)

**Decision** : **always stream**. Single code path.

Rationale :
- `me.route.ts:109-118` uses a threshold (>10MB streams, <10MB buffers). That feature was tuned for JSON serialisation cost.
- For CSV, streaming is naturally row-by-row : the writer pushes one line per row. No JSON-stringify-the-whole-thing-then-decide.
- Forward-compat : if a future museum has 50k sessions, the same code path handles it.
- Cost in code : `res.write(headerLine); for (row of rows) res.write(toCsvLine(row)); res.end()` — same code as buffered minus the intermediate Buffer. Zero perf-vs-buffer regression at small N (each `res.write` < 1ms).

The repo MUST expose `AsyncIterable<RowDTO>` (NOT `Promise<RowDTO[]>`). TypeORM `createQueryBuilder().stream()` supports this — verified via existing analytics queries that use chunked reads.

### 3.4 RBAC scoping (D4)

**Decision** : `WHERE museum_id = $scope` predicate in the repository SQL ; NOT a post-filter in the use case ; NOT a request-level guard.

Rationale (defense-in-depth) :
- A post-filter would still HIT the full dataset (DoS risk + bandwidth waste).
- A request-level guard (e.g. only check at middleware) would centralise authorisation but cannot enforce per-row scope.
- The WHERE clause IS the scope. Multiple layers (middleware + scope inference + WHERE) so a regression in one layer doesn't leak.

Computation :
```typescript
const scopeMuseumId = (() => {
  if (req.user.role === UserRole.SUPER_ADMIN) return null;          // no scope
  if (req.user.role === UserRole.MUSEUM_MANAGER || req.user.role === UserRole.ADMIN) {
    if (!req.user.museumId) throw forbidden('NO_MUSEUM_ASSIGNED');  // R9
    return req.user.museumId;
  }
  throw forbidden('EXPORT_NOT_ALLOWED');                            // visitor/moderator
})();
```

Then `repo.streamChatSessions({ scopeMuseumId, from, to })` ; repo issues `WHERE ($1::int IS NULL OR museum_id = $1)`.

**OPEN BLOCKER** : `Review` and `SupportTicket` entities have NO `museum_id` column today (verified : `review.entity.ts:13-37`, `supportTicket.entity.ts:14-45`). Museum scoping for `museum_manager` on these two kinds is therefore IMPOSSIBLE without a schema change. See Q1.

### 3.5 Audit log integration (D5)

**Decision** : reuse existing `auditService` + `audit_logs` hashed chain ; add three new action constants. NO new table.

- New constants (in `museum-backend/src/shared/audit/audit.types.ts` after line 34) :
  ```ts
  export const AUDIT_ADMIN_EXPORT_SESSIONS = 'ADMIN_EXPORT_SESSIONS';
  export const AUDIT_ADMIN_EXPORT_REVIEWS = 'ADMIN_EXPORT_REVIEWS';
  export const AUDIT_ADMIN_EXPORT_TICKETS = 'ADMIN_EXPORT_TICKETS';
  ```
- Rejected : reuse `AUDIT_DATA_EXPORT` (line 34) — that constant is reserved for user-self DSAR. Mixing user-self and admin-export rows under one action breaks the audit-chain query semantics ("show me all admin exports for this museum" becomes a metadata grep instead of `WHERE action = …`). Three explicit actions cost nothing and make audit-trail queries trivial.
- Rejected : new `admin_exports` audit table. Hash-chained `audit_logs` already covers it ; a sibling table would duplicate the integrity infrastructure for zero benefit.
- Metadata payload : `{ kind: 'sessions'|'reviews'|'tickets', scopeMuseumId: number|null, dateFrom: ISOString, dateTo: ISOString, rowsExported: number, bytesWritten: number }`. Per N9 NEVER includes row contents.

Audit-await sequencing :
```typescript
await auditService.log({ /* metadata above WITHOUT rowsExported */ ip, requestId });
// THEN stream begins. After streaming completes, OPTIONAL second audit emit
// with the final rowsExported count — see Q8 for "1 audit row or 2" decision.
```

### 3.6 Anonymisation rules (D6)

Per spec brief : "default = keep email for super_admin, anonymize for museum_manager." After inspection of existing patterns (`me.route.ts` exports raw email TO THE USER itself ; admin `/api/admin/users/:id` exposes raw email to `admin`+`moderator`) — R2 is stricter because the CSV is **portable** and **screenshot-able**, increasing leak blast radius vs an in-app JSON read.

Final policy :

| Kind | Column | super_admin | admin / museum_manager |
|---|---|---|---|
| sessions | `user_id` | raw int | pseudonym (sha256-trunc16, salted) |
| sessions | `museum_id` | raw int | raw int (their own museum) |
| reviews | `user_id_pseudonym` | pseudonym | pseudonym |
| reviews | `user_name` | raw (already public) | raw (already public) |
| reviews | `comment` | raw + N8-injection-neutralised | raw + N8 |
| tickets | `user_email_pseudonym` | pseudonym | pseudonym |
| tickets | `subject`, `description` | raw + N8 | raw + N8 |

**Why super_admin is also pseudonymised for tickets.user_email** : a screenshot of a CSV in a Slack DM is the leak vector. The super_admin can dereference the pseudonym via the audited `/api/admin/users/<id>` route if needed for a specific case ; he doesn't need 10k raw emails in a CSV. Confirms Q7.

Pseudonym helper :
```typescript
// shared/security/pseudonym.ts (NEW, or extend existing if present)
export function pseudonymise(value: string|number, salt: string): string {
  return createHash('sha256').update(`${salt}|${String(value)}`).digest('hex').slice(0, 16);
}
```

### 3.7 FE button + download trigger (D7)

**Decision** : `fetch` + `Blob` + `URL.createObjectURL` + synthesised `<a download>` click.

Rationale vs direct `<a href="/api/admin/export/sessions.csv" download>` :

| Option | Pro | Con | Verdict |
|---|---|---|---|
| **(a) Direct `<a href download>`** | Simplest, no JS, browser handles streaming natively | No loading state ; auth cookie alone may not pass through (if `samesite=strict` and rewrite weirdness) ; no error visibility ; no audit of "user clicked but never received" ; cannot show "Try again in 47 min" on 429 | REJECTED |
| **(b) `fetch` + Blob + synth click** | Loading state ; error visibility ; can read `Retry-After` ; respects `credentials: 'include'` + CSRF header on admin endpoints | Buffers in memory (browser-side) — issue if export >> 100MB which is well beyond R2 scope | **CHOSEN** |
| **(c) Stream + service worker decoder** | Optimal for huge files | Premature complexity ; R2 expected payloads <10MB | REJECTED |

Implementation skeleton :
```tsx
async function handleClick() {
  setLoading(true);
  const csrfToken = readCsrfToken();
  const res = await fetch(`/api/admin/export/${kind}.csv`, {
    credentials: 'include',
    // GET → no CSRF token required, see api.ts:147
  });
  if (!res.ok) { /* error UI */ return; }
  const blob = await res.blob();
  const url = URL.createObjectURL(blob);
  const filename = extractFilename(res.headers.get('Content-Disposition')) ?? `${kind}.csv`;
  const a = document.createElement('a');
  a.href = url; a.download = filename; document.body.appendChild(a); a.click(); a.remove();
  URL.revokeObjectURL(url);
  setLoading(false);
}
```

`api.ts` is NOT extended — the existing `apiGet` returns JSON (`res.json()`). CSV is binary-ish ; introducing an `apiGetBlob<T>` for one call site is premature abstraction. The 30-LoC inline fetch is acceptable.

### 3.9 Filename convention (D9)

Pattern : `<kind>-<YYYY-MM-DD>.csv`.

- `kind` ∈ {`sessions`, `reviews`, `tickets`} — same as endpoint suffix.
- `YYYY-MM-DD` = UTC date of the request, NOT request-time-zone (consistency across museum locales).

Example : `sessions-2026-05-14.csv`.

### Q1. `Review` and `SupportTicket` have NO `museum_id` column — **BLOCKER** for museum_manager scope.

Verified : `review.entity.ts:13-37` and `supportTicket.entity.ts:14-45` have no `museum_id` / `museumId` field. Today both tables key off `userId` only. Museum scoping for `museum_manager` on these two kinds is therefore impossible without a schema change.

Options :
- **(a) Add migration** adding `museum_id INT NULL` to `reviews` and `support_tickets`, backfilled from `users.museum_id` JOIN. Cost : 1 migration + repo touch + V1 launch risk (schema change late in cycle).
- **(b) Defer museum_manager scope on reviews+tickets to V1.1** — for R2, museum_manager export of reviews/tickets returns 403 with explicit problem+json "scope not supported for this kind yet". Only sessions supports museum_manager scope in R2.
- **(c) Reject the museum_manager role entirely from R2 export** — only super_admin can export until W3.1 RBAC completion adds the column.

**Default position** : **(b)** — ship sessions/reviews/tickets for super_admin ; for museum_manager + admin ship sessions only ; reviews + tickets return 403 with explicit error. Documented in i18n + tests. Sessions has `museum_id` already (`chatSession.entity.ts:44-45`) so scope works there. Confirm option (b) vs commission migration in R2 scope.

### Q2. `admin` role scope — same as `museum_manager` or same as `super_admin` ?

Brief said three roles : super_admin, museum_manager, moderator. `admin` (B2B tenant) is NOT mentioned. `user-role.ts:10-13` clarifies : `admin` = "B2B museum operator with full admin panel access for their tenant. NOT granted multi-tenant ops visibility."

**Default position** : `admin` scoped to `museumId` (same as `museum_manager`). The "B2B tenant operator" wording explicitly says single-tenant. Confirm.

### Q3. `moderator` role — export access ?

Spec brief said `moderator` should have "only reviews/tickets they can moderate (TBD scope)". Inspection : moderator has cross-museum read access on `/admin/reports` + `/admin/reviews` + `/admin/tickets` (`admin.route.ts:81-93,222-244,329-345,379-394`) but ZERO scope filter — moderator sees ALL museums' content. Granting moderator export = granting cross-tenant data exfiltration to a less-trusted role.

**Default position** : **moderator gets 403** on export endpoints. Moderator continues to read paginated lists in-app, but cannot bulk-pull. Tighter than the brief but consistent with the "exports are portable + screenshot-able + higher leak risk" doctrine. Confirm — if rejected, alternative is moderator-export with WHERE `status = 'pending'` (only items in their queue) but this is novel scope work outside R2.

### Q4. Date range default — last 365 days (R11) vs unbounded ?

RGPD Art. 5 §1(e) (storage limitation) — exports should not default to "all time" if retention policy is shorter than that. Current Musaium retention policy is unclear (no doc found). 365 days = generous + RGPD-friendly default.

Alternative : env-configurable cap `ADMIN_EXPORT_DEFAULT_DAYS=365`. Default position : hardcode 365 in R2 ; env-tunable in V1.1 if needed. Confirm.

### Q5. UTF-8 BOM — always prepend, or query-param toggle ?

Excel on Windows requires BOM to interpret UTF-8 (else garbled accents). LibreOffice + Python csv module + macOS Excel newer versions handle UTF-8 without BOM. BOM cost = 3 bytes ; risk of confusing CLI tools that don't strip BOM (e.g. `head -1 file.csv | grep` returns mangled match).

**Default position** : **always prepend** BOM. Excel is the dominant museum-manager tool. The 3-byte cost is negligible. Streaming parsers (Python, Node) handle BOM transparently. Confirm.

### Q6. Pseudonym salt source — env `AUDIT_PSEUDONYM_SALT` or static ?

For pseudonyms to be stable across export runs (so the same museum_manager can correlate `user_id_pseudonym=abc123` from a May export with a June export to spot returning visitors WITHOUT learning the raw user id), the salt MUST be stable.

Options :
- **(a) Env var `EXPORT_PSEUDONYM_SALT`** — set per env, stable across restarts ; rotatable for incident response.
- **(b) Database-stored secret in a `system_secrets` table** — same stability + accessible to BG jobs ; new table needed.
- **(c) Per-museum salt** — derived `hmac(globalSalt, museumId)` — stable + scopes to museum (a museum_manager can't dedupe users across his museum AND an unrelated museum).

**Default position** : **(c)** — `pseudonymise(value, hmac(env.exportPseudonymSalt, museumId ?? 'global'))`. Best privacy posture + still stable. Confirm. If unsure, fall back to (a).

### Q7. super_admin emails in tickets CSV — pseudonymise or raw ?

D6 says pseudonymise EVEN for super_admin. Brief proposed "default: keep email for super_admin". Argument for pseudonymisation : super_admin is Tim alone today, low risk ; but the CSV file lives in his downloads folder, possibly synced to iCloud, possibly screenshotted to a Slack DM. Each raw email in there = a potential leak.

**Default position** : pseudonymise for super_admin too. Tim can dereference via `/api/admin/users/<id>` (which has its own audit row). Confirm OR override to "raw for super_admin" if Tim wants single-step access to "who filed ticket 42" without two clicks.

### Q8. Audit log emission — 1 row pre-stream or 2 rows (pre + post) ?

N6 says "audit log durable before stream starts" — implies row written PRE-stream. But `rowsExported` count is only known POST-stream. Three options :

- **(a) 1 row PRE-stream** with `rowsExported: 'unknown'` ; rely on `bytesWritten` metric from access logs to estimate.
- **(b) 1 row POST-stream** after counts are known. Violates N6 (audit not durable before action).
- **(c) 2 rows** : `EXPORT_STARTED` + `EXPORT_COMPLETED`. Doubles audit volume but precise.

**Default position** : **(a)** — 1 row PRE-stream with `rowsExported: -1` sentinel ; if downstream needs exact count, the access-log line carries it (since the response stream knows its byte count). Audit-chain integrity wins over count precision. Confirm — alternative is (c) if SOC2 review prefers explicit start+end audit pairs.

### Q9. Rate limit bucket — per user or per (user, kind) ?

N11 says 10 exports/hour per user. Could be (user, kind) = 10 sessions + 10 reviews + 10 tickets = 30 per hour effective. Or strict user-bucket = 10 total regardless of kind.

**Default position** : **per (user, kind)** — admins legitimately need to re-pull one kind without burning quota for the others. Bucket name `admin:export:<kind>`. Confirm.

## 6. Risk register

- **Risk1 (PII leak via CSV file landing in Downloads folder)** — even with pseudonymisation, the file lives on the admin's disk + may sync to iCloud / OneDrive. Mitigation : pseudonymise per D6, filename has no PII per N5, `Cache-Control: no-store` per N10. Out-of-scope = client-side disk encryption (admin device hygiene).
- **Risk2 (RBAC bypass — museum_manager pulls another museum's data)** — mitigated by WHERE in SQL (D4) + multi-layer guard (middleware role + use-case scope inference + repo WHERE). - **Risk3 (large export DoS)** — if a super_admin pulls all-time data (millions of rows), the response holds a Postgres cursor open for minutes. Mitigation : N11 rate limit + streaming (D3) caps memory, NOT cursor time. V1.1 : add hard row cap `LIMIT 100000` configurable per env + return `X-Truncated: true` header.

- **Risk5 (heap pressure not asserted in CI)** — AC9 says memory is verified manually, NOT in CI. Reason : Node `process.memoryUsage()` deltas are flaky in jest workers ; a proper measurement needs an isolated runner. Accept this gap ; track if production logs show GC pressure post-launch.
- **Risk6 (audit-chain hash break if metadata schema changes)** — `audit-chain.ts:42` serialises metadata via JSON.stringify-with-sorted-keys. Adding/removing fields in `metadata` between R2 and a future release is fine (each row is hashed independently with its current metadata). Risk = developer modifies an EXISTING row's metadata after-the-fact, which breaks the chain. Mitigation : audit_logs table is append-only (already enforced).
- **Risk7 (museum_manager → reviews/tickets returns 403 confuses users)** — per Q1 resolution (b), museum_manager clicks "Export CSV" on `/admin/reviews` → 403. Mitigation : FE component checks user role + hides button on review/ticket pages for museum_manager. Backend MUST still 403 (defense-in-depth) but FE makes the hidden state explicit. Add to T2 scope.
- **Risk8 (pseudonym leak via salt rotation)** — if `EXPORT_PSEUDONYM_SALT` is rotated, the same user gets a different pseudonym pre- and post-rotation. Documented limitation ; rotate only in incident response.
- **Risk9 (Content-Disposition filename encoding)** — non-ASCII characters in `kind` are not a concern (kind is enum) but Cyrillic / Han / accent characters in browser-OS path may need RFC 5987 `filename*=UTF-8''…` syntax. Mitigation : kind enum is ASCII-only → filename is ASCII-only → standard `filename="…"` form works.

- **`user.museumId` is `number | null`** (`user.entity.ts:30-31`) — many B2B accounts may have `museumId = null` if they were seeded as `admin` role before tenant assignment. R9 + AC6 explicitly handle this (`403 NO_MUSEUM_ASSIGNED`) rather than silently treating null as super-admin scope (which would be a critical RBAC bypass).
- **The existing user-self DSAR export `/api/users/me/export` returns JSON, not CSV** (`me.route.ts:107`). R2 admin CSV is therefore not a duplicate of any existing export ; it is a complementary admin-side surface. The two paths share zero code (different data shape, different RBAC, different audit action).
- **`Content-Type: text/csv; charset=utf-8` is novel in this codebase** — no existing endpoint serves CSV (grep of `museum-backend/src` for `text/csv` returns 0 hits). R2 introduces it ; no precedent to inherit from.
