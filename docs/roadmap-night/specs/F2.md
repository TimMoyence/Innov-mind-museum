**Verified attack scenario** (free-tier user `id=42`, fresh month, count=0) :

1. User hits 4th monthly session → axios interceptor sees 402 → `PaywallProvider.open(reason)` flips `isOpen=true` → modal shows (`email=''`, `consent=false`, `state='idle'`).
2. User types `user@example.com` + ticks consent + presses submit. `httpClient.post('/api/leads/paywall-interest', ...)` resolves → `setState('success')` → green banner visible.
3. User presses close → `setIsOpen(false)`. Native modal animates dismiss. **React subtree stays mounted, all 4 state slots retained**.
4. Hours later, user hits quota again → another 402 → `open(reason)` → modal reappears.
5. User sees email **pre-filled**, consent checkbox **still ticked**, green success banner **already visible before any interaction**.

**Why it matters (R1 §0.1 coupling)** :

- **KR4 funnel signal** — `paywall_modal_shown` breadcrumb fires twice but the matching `paywall_email_captured` only fires once (user assumes submission already happened, doesn't re-submit). The KR4 conversion ratio (`docs/ROADMAP_PRODUCT.md:194`) is therefore **artificially depressed** at the exact moment we use it to gate the Stripe go/no-go decision. R1 was built to emit this signal cleanly — the bug corrupts it.
- **GDPR Art. 7** — consent must be specific, informed, unambiguous, freely given. Inheriting a tick from a previous session is **not freely given** for the second submission. R1 N6 already mandates "default unchecked" for the first open ; the same constraint MUST hold on every open. Currently violated.
- **Trust / UX** — if the second open inherits `state='error'` (previous POST failed), the user sees a red error banner reappear with no action taken — looks like a fresh failure, generates a support ticket.

## 1. Spec (EARS requirements)

- **F2.R1.** WHEN the `<QuotaUpsellModal>` `visible` prop transitions from `false` to `true` (modal opening) THE SYSTEM SHALL reset the four local state slots to their initial values : `email = ''`, `consent = false`, `website = ''`, `state = 'idle'`. This applies whether the open is the FIRST open of the session, the SECOND open after a successful submit, or the Nth open after any prior interaction.
- **F2.R2.** WHEN `<QuotaUpsellModal>` is rendered with `visible={false}` THE SYSTEM SHALL render no visible chrome (RN `<Modal visible={false}>` native behaviour preserved) — F2 does NOT change the RN Modal idiom.
- **F2.R3.** WHEN the user presses submit AND the email is valid AND consent is checked AND the POST resolves 2xx THE SYSTEM SHALL set `state='success'` AND show `t('paywall.success')`. This unchanged behaviour from R1 R27 — F2 does not touch the submit path itself, only its lifecycle on next-open.
- **F2.R4.** WHEN the user closes the modal (via close button `×` OR `onRequestClose` Android-back / iOS-swipe-down) AFTER a successful submit AND later re-opens the modal THE SYSTEM SHALL NOT display `t('paywall.success')` until the user resubmits in the new modal lifecycle. The success banner is a per-open artefact, not a session-wide one.
- **F2.R5.** WHEN the user closes the modal AFTER an error AND later re-opens the modal THE SYSTEM SHALL NOT display `t('paywall.error')` until the user resubmits AND fails again. The error banner is a per-open artefact (R28 stays satisfied within a single open ; F2 scopes that "single open" properly).
- **F2.R6.** WHEN the modal opens for the Nth time THE SYSTEM SHALL render the consent checkbox **unchecked** (visual + a11y state `accessibilityState={{ checked: false }}`). The `if (!consent) return;` guard at submit-time (line 68) MUST refuse to fire the POST until the user explicitly re-ticks — even if the user ticked it in a previous open. GDPR Art. 7 specific/informed/unambiguous/freely-given is a per-consent-act constraint, not a per-app-install one.
- **F2.R7.** WHEN the modal opens for the Nth time THE SYSTEM SHALL render the email `TextInput` with `value=''` — the user must explicitly retype (or paste) on every fresh open. Pre-filling from a previous session is forbidden.
- **F2.R8.** WHEN the user opens the modal twice in a session AND submits ONCE THE SYSTEM SHALL emit exactly 2× `paywall_modal_shown` Sentry breadcrumbs (one per open, from `PaywallProvider.open`) AND exactly 1× `paywall_email_captured` Sentry breadcrumb (one per successful submit). The funnel signal stays clean.
- **F2.R9.** WHEN the chosen D1 option is (a) `useEffect(reset, [visible])` THE effect MUST run on the `false → true` transition. Running on the `true → false` transition is acceptable as a side-effect (no-op for the next open since values are already initial) but the assertion is the `false → true` reset, not the close-time reset.
- **F2.R10.** WHEN the chosen D1 option is (b) conditional mount in `PaywallModalHost` THE SYSTEM SHALL keep `<PaywallProvider>` mounted (its `isOpen / reason / open / close` state is what drives the conditional) AND keep the `setPaywallHandler` wiring intact across modal mounts. Only the modal subtree mounts / unmounts. (Provider lifecycle is independent of modal lifecycle.)
- **F2.R11.** WHEN the existing R1 tests run (R26/R27/R28/R29 + N6 + Q7) THE SYSTEM SHALL keep them green. F2 is a strict superset of behaviour — no R1 test rewritten to accommodate the fix.
- **F2.R12.** WHEN the late `setState` race occurs (user submits, then closes before the POST resolves) THE SYSTEM SHALL NOT crash the app. Under D1 = option (a) the modal stays mounted ; the late `setState('success'|'error')` writes to the now-`visible=false` modal harmlessly ; the NEXT open's reset clears it. Under D1 = option (b)/(c) the modal subtree unmounts ; the late `setState` triggers React's "state update on unmounted component" warning but does not crash. F2 documents this trade-off in §3.1.

## 3. Design

### 3.1 Approach selection (D1) — option (a) useEffect vs (b) conditional mount vs (c) key remount

Three options weighed honestly :

| Option | Pro | Con | Verdict |
|---|---|---|---|
| **(a) `useEffect(reset, [visible])`** — inside `QuotaUpsellModal`, add `useEffect(() => { if (visible) { setEmail(''); setConsent(false); setWebsite(''); setState('idle'); } }, [visible]);`. The four state slots stay local to the modal. Reset fires on every `false → true` transition. | Smallest diff (5 lines added to `QuotaUpsellModal.tsx`, zero touch to `_layout.tsx` / `PaywallProvider.tsx`). N4 + N5 preserved automatically — the provider's `paywall_modal_shown` breadcrumb is independent. N6 race (late `setState` after close) is benign because the subtree stays mounted. TextInput focus during typing is never disturbed (component instance is stable). `useState` initialisers re-evaluated lazily — no allocator pressure. | The reset runs as a side-effect AFTER the open render, not synchronously WITH the open. One render flash where stale values could theoretically be visible (~1 frame). In practice : RN `<Modal>` animation is `slide`, takes ~300ms ; the reset effect fires before the slide finishes ; visually invisible. The test for this is the sequence assertion (post-reopen) — not the intra-frame visibility. | **PICKED** — smallest blast radius, satisfies all NFR. |
| **(b) Conditional mount** — in `PaywallModalHost` (line 97-100 of `app/_layout.tsx`), wrap : `return isOpen ? <QuotaUpsellModal visible={isOpen} reason={reason} onClose={close} /> : null;`. The modal subtree mounts on `open()` and unmounts on `close()`. State slots re-initialise from scratch on every mount. | Cleanest semantic — "state lives as long as the modal exists". No `useEffect` ceremony. | Subtree mount/unmount on every open. RN `<Modal>` itself does an internal `useEffect` teardown on unmount — there's a documented race in `react-native` where unmounting a `<Modal>` while it's mid-animation throws on iOS in dev mode (RN 0.78+, Expo 55 — Musaium is on this exact stack). Workaround = always finish the slide-out before unmounting, which means a `setTimeout(unmount, 300)`. Adds complexity. N5 (no unmount cascade) is now violated unless we re-verify Sentry breadcrumb hooks. N6 (late setState) becomes a real issue — fire-and-forget POST resolving after unmount throws the React warning. | Rejected — semantic gain not worth the iOS animation race and the late-setState complication. |
| **(c) `key={openCounter}` remount** — add an `openCounter` to `PaywallProvider` incremented on every `open()`. Pass `key={openCounter}` to `<QuotaUpsellModal key={openCounter} visible={isOpen} ... />` in `PaywallModalHost`. React remounts the component on every key change, so state slots re-initialise. | Compact ; the provider becomes the single source of truth for "which open are we on". | If `openCounter` ever bumps WHILE the user is typing (e.g. a second 402 from a background retry fires while the modal is open — unlikely but possible), React tears down the TextInput → user loses typing focus mid-keystroke. The race is rare but undebuggable when it happens. Also widens the `PaywallProvider` API for a single consumer (the modal). | Rejected — focus-loss tail is invisible in testing but corrosive in field reports. |

**Decision — option (a) "useEffect on visible transition"**, justified above.

**Why (a) over (b)** despite (b) being semantically cleaner :

1. **Diff size + blast radius** — option (a) touches one file (5 LoC), option (b) touches two (`PaywallModalHost` + `<Modal>`-on-iOS workaround), option (c) touches three (provider + host + modal).
2. **iOS RN Modal animation race** — option (b)'s unmount-mid-animation throw is a documented RN gotcha (referenced in `react-native` issues #20587 + #41098). Musaium is on Expo 55 / RN 0.83 — the issue persists. Working around it requires either `setTimeout(unmount, 300)` (fragile) or hard-coded `visible={false}` first then unmount on `onDismiss` (RN 0.71+ supports `onDismiss` on iOS only — Android needs different handling). Not worth it.
3. **N6 late-setState safety** — option (a) keeps the modal mounted ; the late `setState` after a close-during-pending-POST writes to a hidden-but-mounted tree harmlessly. Option (b) unmounts ; the late `setState` triggers React's warning AND the next open shows a fresh `state='idle'` instead of `state='success'`. The user might prefer to see "your submission worked" on reopen — but per F2.R4 spec we explicitly DON'T want that (banner is per-open). So this is moot, but the warning is still noise.
4. **TextInput focus stability** — option (a) keeps the TextInput instance ; option (c) destroys + recreates it on every open. If our keyboard auto-focus behaviour ever changes, option (a) is more forgiving.

Honesty disclosure : the reviewer recommendation in the bug ticket lists (a) as the "minimal, surgical" choice. The spec agrees. Tim makes the final call — if "do (b) anyway" is the answer, the spec re-pivots and §3.3 / §3.4 sections below adjust for the unmount cascade.

### 3.2 GDPR Art. 7 enforcement (D2) — explicit consent renewal on each open

GDPR Art. 7 :
> (1) Where processing is based on consent, the controller shall be able to demonstrate that the data subject has consented to processing of his or her personal data.
> (2) [...] the request for consent shall be presented in a manner which is clearly distinguishable from the other matters, in an intelligible and easily accessible form, using clear and plain language.

Read in conjunction with Art. 4(11) — consent must be "specific, informed, unambiguous, freely given". The Article 29 Working Party guidelines (WP259, endorsed by EDPB) clarify : consent for a SPECIFIC processing operation, given at the time of the operation. Inheriting a tick from a prior, completed operation is NOT specific or freely given for the new operation — even if the user technically ticked at some point.

**Application to F2** :

- **First open of the session** (R1 N6 mandates "default unchecked") : `consent = false`. User must explicitly tick. Specific + informed + unambiguous + freely given. OK.
- **Second open after a successful submit** : the FIRST submission's consent applied ONLY to that submission. If the user re-opens (because they hit quota again, in the same session OR a different one), and the modal pre-fills `consent = true`, the NEW submission inherits a consent record that was given for the previous operation. This is **not consent for the new operation** — it's the prior consent silently re-used. Violation.
- **F2 default (option (a))** : `useEffect(reset, [visible])` resets `consent = false` on every open. Every submission is paired with a fresh, explicit checkbox-tick. Specific + informed + unambiguous + freely given for that operation. Compliant.

**Enforcement mechanism beyond F2.R6** :

- The `if (!consent) return;` guard at submit-time (line 68) is the runtime gate.
- The visual `accessibilityState={{ checked: consent }}` (line 141) makes the state observable to a11y tools.
- The unit test `F2.AC2` (GDPR consent renewal sentinel) is the CI gate.
- The CLAUDE.md doctrine bullet (D3 below) is the human gate.

**Honesty disclosure** : GDPR Art. 7 enforcement does NOT require that we also reset the `email` field — the user CAN voluntarily reuse the same email. But our default-reset of email is justified by separate UX/funnel reasons (F2.R7 + KR4 signal). The two resets compose, but Art. 7 specifically mandates only the consent reset. We could legally pre-fill email and reset only consent ; we choose to reset both for cleanliness. Documented here so a future "let users opt back into pre-fill" V1.1 request doesn't get blocked by a misread of Art. 7.

### 3.3 Sentry breadcrumb impact (D3) — verify F2 doesn't perturb the funnel signal

The breadcrumb taxonomy from R1 N7 :

| Breadcrumb | Fired from | Cadence | F2 impact |
|---|---|---|---|
| `paywall_modal_shown` | `PaywallProvider.open()` (provider, not modal) | Once per `open()` call | None under (a) — provider is untouched. Once per 402-interceptor-trigger. |
| `paywall_cta_clicked` | `QuotaUpsellModal.onSubmit()` line 70-74 | Once per submit button press (BEFORE the POST) | None under (a) — the modal is never re-rendered into a stale "already clicked" state. |
| `paywall_email_captured` | `QuotaUpsellModal.onSubmit()` line 82-86 | Once per POST 2xx resolution | None under (a). The `state='success'` reset on reopen does NOT re-fire the breadcrumb (the breadcrumb is in the `try` block, not in the `useEffect` that paints the banner). |

**Verification** :
- F2.AC4 asserts the exact cadence (2× `paywall_modal_shown`, 1× `paywall_email_captured`, 1× `paywall_cta_clicked`) for an open → submit → close → reopen → close sequence.
- F2.N4 + F2.N5 codify the constraint at NFR level.

**Edge case under option (a) — does the useEffect itself fire a breadcrumb ?** No. The `useEffect(reset, [visible])` only calls four `setState`s, none of which emit breadcrumbs. Sentry breadcrumb-on-state-change is not configured. The breadcrumb path is purely event-driven from the user actions + provider `open()`.

**Edge case — does the late `setState` after close re-fire any breadcrumb ?** No. The `try` block's breadcrumb fires when the POST resolves ; if the user closes BEFORE the POST resolves, the breadcrumb still fires (POST is fire-and-forget from the user's perspective). The next-open's reset clears the `state` slot but the breadcrumb has already been emitted to Sentry. The funnel signal stays accurate — one submission, one breadcrumb, regardless of whether the user saw the success banner.

**No double-fire on `paywall_modal_shown`** : the provider's `open()` is called once per axios-interceptor 402 hit. F2's `useEffect` in the modal is independent. Two separate concerns.

### 3.5 V1.1 promotion (D5) — should we move state to PaywallProvider context instead of modal-local ?

**Arguments FOR moving state to provider (V1.1)** :
- Centralised "current paywall interaction" state would let other UI surfaces (e.g. a settings page) inspect "did the user already submit this month" and conditionally render.
- Cross-screen analytics : provider could emit a `paywall_session_summary` breadcrumb on logout / app-background.
- Test harness simpler — one source of truth, no `useEffect` reset ceremony.

**Arguments AGAINST (status quo for V1)** :
- The provider would grow from 4 props (`isOpen, reason, open, close`) to 8+ (`email, setEmail, consent, setConsent, submitState, setSubmitState, ...`) — bloats the context API for one consumer.
- React context is a poor fit for transient form state (re-renders all consumers on every keystroke). Mitigation = `useReducer` + memoised selectors, but that's V1.1-grade infra.
- The modal-local pattern matches React idiom (form state lives where the form lives).
- V1 launch is 2026-06-01 ; modal-local is shipped today, refactoring to provider-state without a measured signal that we need it would be premature.

**Decision** : status quo (modal-local) for V1. Revisit V1.1 if any of the following telemetry signals appears post-launch :
- ≥3 support tickets / month about "I submitted before but the modal asks me again".
- A settings-page feature wants to display "you're on the waitlist for premium — submitted 2026-04-15".
- KR4 conversion analysis shows users repeatedly re-submit the same email (telemetry on dedup at the BE).

**Captured in Q5** below — Tim confirms the V1.1 deferral.

## 6. Risk register

- **F2.Risk1 (Reset useEffect fires intra-frame stale paint)** — under option (a) the reset runs as a side-effect AFTER the open render. There's a ~1-frame window where stale values could be visible. RN's slide-animation (~300ms) masks it in practice. Mitigation : if observed in QA, switch to `useLayoutEffect` (fires before paint, RN-supported). Not done in F2 because no observation yet — premature optimisation.
- **F2.Risk2 (Late setState race)** — if user submits then closes before POST resolves, the late `setState('success'|'error')` writes to a hidden-but-mounted tree. Harmless under option (a) — the next open's reset clears it. Documented in F2.R12.
- **F2.Risk3 (Reviewer override on D1)** — if Tim picks option (b) or (c) over (a), the spec re-issues with the corresponding trade-off mitigation (iOS unmount-race workaround for (b), or `openCounter` provider widening for (c)). Captured in Q1.
- **F2.Risk4 (Breadcrumb-cadence test requires `PaywallModalHost` extraction)** — currently `PaywallModalHost` is an inline 3-line function in `_layout.tsx` (line 97-100). Not importable as-is. T1 may need to extract it to a sibling file in `features/paywall/ui/`. Extraction is small (3 lines moved + one import added in `_layout.tsx`) but ADD scope to the PR. Honesty : disclosed here. If Tim prefers to avoid scope creep, the breadcrumb-cadence test can instead mock the host inline within the test file — slightly uglier but no source-tree touch.
- **F2.Risk5 (Honeypot test brittleness)** — `display: 'none'` TextInputs are not always discoverable by `@testing-library/react-native` standard queries. F2.AC5 may need `UNSAFE_getAllByType(TextInput)` + style filter. Documented in §3.4.
- **F2.Risk6 (Test discipline drift)** — new tests must use the existing `fixtureReason()` factory (line 56-62), not inline `{ tier, currentCount, ... } as QuotaInfo`. ESLint `musaium-test-discipline` baseline cannot grow.
- **F2.Risk7 (Locale completeness)** — the existing R29 sentinel asserts `en + fr` locales ship the `paywall.*` keys. F2 doesn't add new keys, so the sentinel stays green. If a future locale (es, de) is added before F2 merges, the sentinel needs updating — orthogonal concern.
- **F2.Risk8 (GDPR consent re-tick perceived as friction)** — UX feedback could be "why do I have to re-tick consent every time, I already agreed". Counter : GDPR Art. 7 mandates per-act renewal ; the user typed nothing else either (email reset, banner reset) so re-ticking is one of multiple inputs, not the sole friction point. If UX research finds users abandon at this re-tick step, V1.1 may explore "remember consent for the user's lifetime" — but that needs separate legal review (consent withdrawal mechanics, log retention, DSAR shape). Out of F2 scope.
