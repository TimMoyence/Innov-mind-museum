## 1. Spec (EARS requirements)

### Page render + structure

- **R1.** WHEN a user navigates to `/fr/b2b` or `/en/b2b` THE SYSTEM SHALL render a Next.js server component returning a hero section, problem section, solution section, differentiators grid (×5), pricing tease section, contact form section, all reusing the marketing `Header` + `Footer` from `[locale]/layout.tsx`.
- **R2.** WHEN the page renders THE SYSTEM SHALL include `<title>` and `<meta name="description">` populated from `dict.landing.b2b.metadata.title` / `description`, with `alternates` (hreflang fr/en) and OpenGraph (`og:locale` per i18n, `og:image` = `/images/logo.png` fallback) — mirroring `support/page.tsx:10-18`.
- **R3.** WHEN the differentiators grid renders THE SYSTEM SHALL list EXACTLY 5 items in this order : (1) AI conversational contextuel multilingue, (2) Visual similarity engine (C3 image compare), (3) Multi-musées (un visiteur, plusieurs musées), (4) Voice-first (mains libres), (5) Hors-musée ET intra-musée — each with title + description from `dict.landing.b2b.differentiators[i]`.
- **R4.** WHEN the pricing tease section renders THE SYSTEM SHALL display the copy `dict.landing.b2b.pricing.tease` containing the exact i18n string `"Sur devis — pricing adapté à la taille du musée"` (FR) / `"Custom pricing tailored to museum size"` (EN). NO numeric figure rendered.
- **R5.** WHEN any TSX file under `src/app/[locale]/b2b/` is parsed THE SYSTEM SHALL contain ZERO raw French or English marketing strings — every visible string comes through `dict.landing.b2b.*` (enforced by a Vitest grep-based unit test in T1).

### Form fields + behavior

- **R6.** WHEN the contact form renders THE SYSTEM SHALL include exactly these visible labeled inputs : `email` (type=email, required), `name` (type=text, required, ≤120 chars), `museum` (type=text, required, ≤200 chars — institution name), `role` (select, required, options : `director` / `curator` / `digital` / `other`), `message` (textarea, required, 10..5000 chars), `consent` (checkbox, required, unchecked by default) + a hidden honeypot input `website` (required NOT filled — submission with non-empty value treated as spam, silently 202'd).
- **R7.** WHEN the user submits the form with all required fields valid + consent checked + honeypot empty THE SYSTEM SHALL POST to `/api/leads/b2b` (D4-preferred) OR `/api/support/contact` extended (D4-alternative) with `Content-Type: application/json` and body containing the 5 user-facing fields (no `consent`, no `website` — `consent` is enforced client-side, `website` is honeypot-only) — and SHALL show the success message `dict.landing.b2b.contact.success` once response is 2xx.
- **R8.** WHEN the response is non-2xx (network error, 5xx, 429 rate-limit, validation 400) THE SYSTEM SHALL render the error message `dict.landing.b2b.contact.error` and keep form contents intact so the user can retry — mirrors `ContactForm.tsx:36-49`.
- **R9.** WHEN the form is mid-submission THE SYSTEM SHALL show `dict.landing.b2b.contact.sending` on a disabled submit button (matches existing `ContactForm.tsx:115-117` pattern).
- **R10.** WHEN the honeypot field `website` is filled (any non-empty value) AND the form is submitted THE SYSTEM SHALL still display the success state to the user (silent accept) but the BE handler SHALL log + drop the payload without sending the Brevo email. Prevents bot enumeration via response timing.
- **R11.** WHEN the consent checkbox is unchecked THE SYSTEM SHALL block submission client-side (`required` attribute) AND BE SHALL also validate `consent === true` server-side and return 400 if absent — defense-in-depth.

### BE endpoint (if D4 = BE-proxy)

- **R12.** WHEN BE receives `POST /api/leads/b2b` with valid body THE SYSTEM SHALL apply a rate-limit middleware identical to `supportContactLimiter` (5 req / 600s / IP, `support.route.ts:23-27`) and respond 202 `{ accepted: true }` on success.
- **R13.** WHEN BE processes a valid B2B lead THE SYSTEM SHALL send a single transactional email via `BrevoEmailService.sendEmail(to=B2B_INBOX_EMAIL || SUPPORT_INBOX_EMAIL, subject=`[Musaium B2B] {museum} — {role} {name}`, html=buildB2bLeadEmail(payload))` — reusing the existing `BrevoEmailService` (no new HTTP client, no SDK).
- **R14.** WHEN BE has no `BREVO_API_KEY` configured (local dev) THE SYSTEM SHALL fall back to `NoopSupportContactNotifier`-style logger.warn behavior (`support-contact-email.notifier.ts:33-42`) — local dev still accepts submissions, just logs them.

### i18n + a11y + SEO

- **R15.** WHEN the i18n dictionary is loaded THE SYSTEM SHALL include `dict.landing.b2b.{metadata, hero, problem, solution, differentiators[], pricing, contact}` keys in BOTH `fr.json` AND `en.json` — full parity, no key missing in one locale. A Vitest dict-symmetry test enforces this.
- **R16.** WHEN axe-core runs on `/fr/b2b` AND `/en/b2b` (Playwright a11y suite) THE SYSTEM SHALL report zero WCAG 2.1 AA violations — same threshold as `public-support.a11y.spec.ts`.
- **R17.** WHEN the form renders THE SYSTEM SHALL include `<label>` (visible OR `sr-only`) for every input, `aria-describedby` linking the consent checkbox to a privacy-policy `<a href="/[locale]/privacy">` link, `aria-live="polite"` region for success/error messages, and a focus ring matching `Button.tsx` discipline (`focus-visible:outline-2`).
- **R18.** WHEN the Footer renders on any public route THE SYSTEM SHALL include a link to `/[locale]/b2b` labeled `dict.footer.links.b2b` (e.g. "Musées" FR / "For museums" EN), adjacent to the existing privacy/support/accessibility links.

### Telemetry + observability

- **R19.** WHEN the form is submitted THE SYSTEM SHALL emit a Sentry breadcrumb category `b2b.lead.submit` with sanitized fields (`museum` length, `role` enum value, NO email/name/message content) — privacy by default. No new Sentry config required ; uses `@sentry/nextjs` already in `museum-web/package.json`.
- **R20.** WHEN BE receives a B2B lead THE SYSTEM SHALL log `b2b_lead_submitted` (logger.info, requestId + museumName length + role + honeypotTriggered boolean) — no PII.

## 3. Design

### 3.1 Route + file location (D1)

**Decision** : `museum-web/src/app/[locale]/b2b/page.tsx` — server component, mirrors `[locale]/support/page.tsx:1-72`.

Rationale :
- App Router file-based routing makes `/[locale]/b2b/` automatically locale-prefixed by middleware (cf. `next.config` + `museum-web/src/app/layout.tsx`).
- `b2b/` is short, audience-recognizable, no namespace collision with `admin/`, `support/`, `privacy/`, `accessibility/`, `verify-email/`, `confirm-email-change/`, `reset-password/` (verified — full list of `[locale]/` siblings).
- Alternative names considered : `museums/`, `musees/`, `pour-musees/`, `pitch/`. Rejected : `museums/` clashes semantically with B2C "list of museums" (W3.2 future page) ; locale-specific names break the locale-agnostic `[locale]` routing convention.
- The page is reachable from Footer (R18) + optionally Header nav (left out of MVP per scope tightness, see Q4).

### 3.2 Sections layout (D2)

Linear scroll, single-column, mobile-first :

1. **Hero** — `dict.landing.b2b.hero.{title, subtitle, ctaPrimary, ctaSecondary}`. Primary CTA = `#contact` anchor scroll. Secondary CTA = `/fr` link ("Voir l'app grand public"). Background = `bg-gradient-to-b from-primary-50 to-white` (mirrors support page hero, `support/page.tsx:27`).
2. **Problem** — single paragraph framing the museum-decision-maker pain (audio guides cost, multilingual gaps, visitor data absent, fragmented per-museum apps). `dict.landing.b2b.problem.{title, body}`. Keep concise (≤80 words FR).
3. **Solution** — single paragraph + 1 inline SVG illustration (reuse `StorySection`'s SVG icons style, no new asset). `dict.landing.b2b.solution.{title, body}`.
4. **Differentiators grid** — `<ul role="list">` with 5 `<li>` cards, grid-cols-1 mobile / md:grid-cols-2 / lg:grid-cols-3-with-spanning. Each card : icon (inline SVG, `aria-hidden`) + title + description. Items per R3 above. `dict.landing.b2b.differentiators[].{title, description}` (array of 5).
5. **Pricing tease** — single centered block, `dict.landing.b2b.pricing.{title, tease, contactCta}`. The `tease` is the canonical "Sur devis…" line. Below it, a CTA button anchoring to `#contact`. NO numeric range. NO "à partir de".
6. **Contact form** — section id `contact`, max-w-2xl, mirrors `support/page.tsx:62-68`. Title via `dict.landing.b2b.contact.title`. Component = new `B2bContactForm.tsx`.
7. **Footer** — inherited from `[locale]/layout.tsx`, no per-page footer override.

NO testimonials section (would need real museum partners per memory `feedback_product_lifecycle_coherence` — pre-launch). NO "trusted by" logos (same reason). NO video. NO Calendly.

### 3.4 Form submission — BE endpoint or direct client→Brevo ? (D4)

**Decision (preferred)** : **BE proxy via new endpoint `POST /api/leads/b2b`** on `museum-backend`.

Rationale :
- **N9 / N10** : `museum-web/package.json` does NOT contain `@getbrevo/brevo` or any Brevo SDK. The `museum-backend` codebase has the canonical Brevo integration (`brevo-email.service.ts`) using raw `fetch`. Reusing it = single Brevo touchpoint, single API key custody.
- **Secrets discipline** : `BREVO_API_KEY` is `server-side only` in BE (`env.ts` line referenced earlier). Exposing it as `NEXT_PUBLIC_BREVO_API_KEY` would leak it to the client bundle — UNACCEPTABLE per honesty UFR-013 + security gates. A server-only Next.js Route Handler would also work, but R4 §0.4 N10 forbids shadowing the existing `/api/*` rewrite (which forwards to BE).
- **Rate limit** + **honeypot drop** + **logger discipline** all already wired on BE via `supportContactLimiter` + `EmailSupportContactNotifier`. Replicating server-side logic in Next.js would be a maintenance fork.
- **Defense in depth** : BE validates `consent === true` (R11), enforces zod schema, rate limit, IP capture. Direct→Brevo couldn't validate consent server-side.

**Implementation shape** :

```
museum-backend/src/modules/support/adapters/primary/http/schemas/support.schemas.ts
  + submitB2bLeadSchema (zod) — email + name(120) + museum(200) + role(enum) + message(10..5000) + consent(true literal) + website(empty-string honeypot)

museum-backend/src/modules/support/useCase/contact/submitB2bLead.useCase.ts (new)
  - validate input
  - if input.website.trim() !== '' → log warn b2b_lead_honeypot_triggered, return (no email)
  - if input.consent !== true → throw badRequest('consent required')
  - call notifier.notifyB2b({ email, name, museum, role, message, ip, requestId, userAgent })

museum-backend/src/modules/support/domain/ports/support-contact-notifier.port.ts (extend)
  + notifyB2b(payload: B2bLeadPayload): Promise<void>

museum-backend/src/modules/support/adapters/secondary/notifier/support-contact-email.notifier.ts (extend)
  + EmailSupportContactNotifier.notifyB2b → buildB2bLeadEmail template → emailService.sendEmail(b2bInbox || supportInbox, subject, html)
  + NoopSupportContactNotifier.notifyB2b → logger.warn (local dev fallback)

museum-backend/src/shared/email/templates/b2b-lead.template.ts (new)
  + buildB2bLeadEmail(payload) — HTML, mirrors buildSupportContactEmail style

museum-backend/src/modules/support/adapters/primary/http/routes/support.route.ts (extend)
  + supportRouter.post('/leads/b2b', b2bLeadLimiter, validateBody(submitB2bLeadSchema), handler)
    [or mount under a separate /api/leads router — see Q5]
```

**Alternative — direct client→Brevo via `/contactsAttribute/contact`** : REJECTED. Requires either (a) public Brevo API key (security non-starter) or (b) Edge function (not in stack). The BE proxy is the lowest-friction path.

**Mitigation if BE endpoint can't ship same PR** : the page ships with the form submit `disabled={true}` and a fallback `<a href="mailto:b2b@musaium.com">` CTA. NOT a feature flag, just a static disabled state. The BE endpoint lands in a follow-up PR same sprint. See Q2.

### 3.8 Accessibility (D8)

- Semantic landmarks : `<main>` (provided by layout), `<header>` (provided by `Header.tsx`), `<footer>` (provided by `Footer.tsx`), per-section `<section aria-labelledby="...">`. Differentiators list = `<ul role="list">`.

## 4. Open Questions

### Q1. OG image source — dedicated B2B asset, or reuse logo placeholder?
The B2C landing uses `/images/logo.png` (1024×1024) per `seo.ts:21`. A B2B-pitched OG card (with text "Musaium for museums") would convert better on LinkedIn / Twitter. **Default position : reuse logo for V1 ; design + commit a dedicated `og-b2b-{fr,en}.png` post-launch when a designer is available.** Confirm with user.

### Q2. Can the BE endpoint ship in the same PR as the FE page, or do we need a 2-PR sequence?
If BE endpoint lands first, FE has nothing to wire to. If FE lands first, submit is dead. **Default position : single-PR — green-code-agent commits FE + BE together. R4's scope is intentionally wide enough to cover both.** If concurrency cap (`max 2 features in flight`, tracker §Concurrency cap) forces splitting : FE-first with `disabled submit` + `mailto:b2b@musaium.com` fallback (cf. §3.4 mitigation). Confirm before T1.

### Q3. Reuse `SUPPORT_INBOX_EMAIL` or introduce `B2B_INBOX_EMAIL` env var?
N9 says "no new env flag" but `B2B_INBOX_EMAIL` is a config var, not a feature flag. Adding it lets the user route B2B leads to a separate inbox (e.g. `b2b@musaium.com` vs `support@musaium.com`). **Default position : add `B2B_INBOX_EMAIL` with fallback to `SUPPORT_INBOX_EMAIL` (which already defaults to `support@musaium.app` per `env.ts`).** Zero behavior change in local dev ; opt-in segregation in prod. Confirm.

### Q4. Add a Header nav link to `/[locale]/b2b`, or footer only?
Header has 3 nav items today (home / support / privacy) per `Header.tsx:50-54`. Adding a 4th ("Musées" / "For museums") increases header density on mobile. **Default position : footer only (R18), no header link.** Decision-makers reach the page via direct link (sales outreach, email signature, social) or via footer. Confirm.

### Q5. Mount the new BE route under `/api/support/contact-b2b` or `/api/leads/b2b`?
- Option A : `/api/support/contact-b2b` — keeps the contact use cases co-located, reuses `supportRouter`, minimal new route surface.
- Option B : `/api/leads/b2b` — semantically distinct ("leads" ≠ "support"), suggests a future leads router with multiple lead-source endpoints (beta signup R3, B2B R4, …).
**Default position : Option B (`/api/leads/b2b`).** Mounts a new `leadsRouter` in `app.ts` parallel to `supportRouter`. Anticipates R3 (beta signup) reusing it. But honors single-PR scope — only one endpoint shipped today. Confirm.

### Q6. RGPD consent — store the consent timestamp anywhere, or rely on Brevo's double-opt-in?
Brevo's standard transactional list signup does double-opt-in on the recipient (a confirmation email with a link). But the B2B form is NOT a list signup — it's a transactional lead. The "double opt-in" requirement applies to marketing list subscriptions under GDPR Art.7. **Default position : the form's consent checkbox + privacy-policy link satisfies RGPD informed consent for the transactional lead use case. No DB persistence of the lead in R4** (the email + logger.info trail is the audit). If the user wants leads stored in DB (CRM-light), that's a V1.1 item with its own migration. Confirm.

### Q7. Anti-spam strategy beyond honeypot — captcha tier?
Honeypot + IP rate-limit (5/600s) is reasonable for low-volume pre-launch B2B traffic. Cloudflare Turnstile or hCaptcha would harden it but cost a3rd-party dep + a11y review. **Default position : honeypot + rate-limit only in R4. Reassess if leads inbox sees >5 spam/week post-launch.** Confirm.

### Q8. Where does the i18n hard-coded check live — Vitest source-read, ESLint rule, or both?
Project has `eslint-plugin-musaium-test-discipline` per CLAUDE.md §Test Discipline. Is there an existing `no-raw-marketing-string` rule, or only `no-inline-test-entities`? **Default position : T1 ships a Vitest grep-based test (AC5) regardless. If an ESLint rule already exists or can be added cheaply, T3 polish adds it.** Verify before T1.

### Q9. Should the B2B page be accessible to crawlers (indexable)?
Indexing the B2B pitch surfaces it on Google for queries like "AI conversationnelle musée" — good for inbound. But it also exposes the "Sur devis" framing publicly. **Default position : indexable (no `noindex`). Robots default applies. Sitemap entry added (D7).** Confirm.

### Q10. Honesty crosscheck — are the 5 differentiators all DEMOABLE today?
- (1) AI conversational multilingue — YES (chat-module ships, locale routing in chat-message use case).
- (2) Visual similarity engine — YES (smoke `/api/chat/compare` proves live, `scripts/smoke-api.cjs:403-457`).
- (3) Multi-musées (un visiteur, plusieurs musées) — YES (DB model is multi-tenant on `museumId`).
- (4) Voice-first — YES (CLAUDE.md §Voice V1, always-on TTS+STT pipeline).
- (5) Hors-musée ET intra-musée — PARTIAL (LocationResolver supports both per memory `project_geolocation_pipeline`, but the "Walk V1" UX is Phase 2 NEXT per `ROADMAP_PRODUCT.md:144`). **Risk : pitching "hors-musée" copy as a present capability could mislead decision-makers.** Default position : phrase (5) as "Designed for before, during, AND after the visit" — present-tense intent, not a feature claim. T2 copy must respect this nuance. Confirm.

## 6. Risk register

- **Risk1 (copy honesty drift)** — Differentiator 5 "Hors-musée ET intra-musée" could overpromise if Walk V1 isn't shipped by museum-meeting time. Q10 + §0.5 N5. **Mitigation** : phrase as design intent, not feature claim. Reviewer (review-agent) MUST flag any "ready today" claim on Walk features.
- **Risk2 (Brevo deliverability)** — Lead emails from `no-reply@musaium.com` could land in museum-side spam filters (govt domains, museums often use `@culture.gouv.fr` etc.). **Mitigation** : nothing R4-blocking ; track open-rate post-launch via Brevo dashboard. Adjust `from`/SPF/DKIM if needed (out of scope).
- **Risk3 (form abandonment from consent friction)** — Unchecked-by-default consent + visible privacy link adds 1 extra click. Industry standard. **Mitigation** : keep the consent label short + non-legalese. Privacy policy link opens same tab (no `target="_blank"` — distracts the funnel). Acceptable.
- **Risk4 (honeypot false positives)** — Some password managers auto-fill `website` fields. **Mitigation** : use a non-obvious field name like `company_url` instead of `website`, set `autoComplete="off"` + `tabIndex={-1}`. T1 test covers honeypot behavior. If post-launch shows false-positive drop rate, swap field name (small change).
- **Risk5 (rate limit too tight)** — 5/600s/IP could lock out a museum sales team if they all submit from a NAT'd office IP at a board meeting. **Mitigation** : 5 in 10 min is generous for human submissions. If post-launch incident, raise to 10/600s.
- **Risk6 (i18n drift over time)** — One locale's `landing.b2b.*` gets a new key, the other doesn't. **Mitigation** : AC4 dict-symmetry test catches this in CI. Same pattern as existing dictionary coverage.
- **Risk7 (Brevo API rotation)** — Brevo rotates the customer key without notice. **Mitigation** : BE already falls back to noop notifier when `BREVO_API_KEY` missing (`support/useCase/index.ts:21`). B2B lead submissions logged warn-level but accepted (200 to user). Operator monitors error.log post-deploy.
- **Risk8 (SEO competition for "Musaium for museums")** — Low-traffic English keyword pre-launch, fine. **Mitigation** : none needed.
- **Risk9 (page bloat from framer-motion)** — Adding 5 differentiator animated cards could push Perf <95. **Mitigation** : reuse `viewport.once: true` pattern from `StorySection.tsx:109` so animations fire once and tear down ; no scroll listeners persist.
