## 1. Spec (EARS requirements)

### Backend — entity + migration

- **R1.** WHEN the migration `AddUserTier` runs against an empty schema THE SYSTEM SHALL create three new columns on `users` : `tier varchar(16) NOT NULL DEFAULT 'free'` with CHECK constraint `tier IN ('free','premium')`, `sessions_month_count integer NOT NULL DEFAULT 0`, and `sessions_month_start date NULL`.
- **R2.** WHEN the migration `AddUserTier` runs against a populated schema THE SYSTEM SHALL backfill all existing `users` rows with `tier='free'`, `sessions_month_count=0`, `sessions_month_start=NULL`. No row touched beyond these three columns.
- **R3.** WHEN the migration is reverted via `pnpm migration:revert` THE SYSTEM SHALL drop the three columns. Reversibility test asserts users table schema returns to pre-R1 state.
- **R4.** WHEN any process reads a `User` entity THE SYSTEM SHALL expose `tier: 'free' | 'premium'`, `sessionsMonthCount: number`, `sessionsMonthStart: Date | null` properties hydrated from the corresponding columns.

### Backend — monthly quota middleware

- **R5.** WHEN `POST /api/sessions` is received with a valid JWT AND `req.user.tier === 'premium'` THE SYSTEM SHALL pass the request to the next handler without any counter mutation or DB write.
- **R6.** WHEN `POST /api/sessions` is received with a valid JWT AND `req.user.tier === 'free'` AND the current UTC month differs from `req.user.sessionsMonthStart`'s month (OR `sessionsMonthStart === null`) THE SYSTEM SHALL atomically reset `sessions_month_count = 1` and `sessions_month_start = <first-day-of-current-UTC-month>` and pass the request to the next handler. (Reset + count-as-one happen in the same UPDATE.)
- **R7.** WHEN `POST /api/sessions` is received with a valid JWT AND `req.user.tier === 'free'` AND the current month matches `sessionsMonthStart` AND `sessions_month_count < env.freeTierMonthlySessionLimit` THE SYSTEM SHALL atomically increment `sessions_month_count` by 1 via a `RETURNING`-clause UPDATE and pass the request to the next handler.
- **R8.** WHEN `POST /api/sessions` is received with a valid JWT AND `req.user.tier === 'free'` AND `sessions_month_count >= env.freeTierMonthlySessionLimit` AND the current month matches `sessionsMonthStart` THE SYSTEM SHALL respond `402 Payment Required` with JSON body `{ code: 'QUOTA_EXCEEDED', tier: 'free', currentCount, limit, resetAt: <ISO first-day-of-next-UTC-month> }`. The handler MUST NOT execute.
- **R9.** WHEN `POST /api/sessions` is received without a JWT THE SYSTEM SHALL be rejected by `isAuthenticated` BEFORE the quota middleware (existing behavior) — quota middleware never sees anonymous requests.
- **R10.** WHEN any other chat route (`POST /api/sessions/:id/messages`, `GET /api/sessions/...`, `POST /api/sessions/:id/messages/compare`) is called THE SYSTEM SHALL NOT apply the monthly session quota middleware. The session-creation surface is the single quota gate.
- **R11.** WHEN the middleware runs the increment THE SYSTEM SHALL use a single atomic `UPDATE users SET ... WHERE id = $1 AND (sessions_month_start IS NULL OR sessions_month_start = $monthStart) RETURNING sessions_month_count` — no read-then-write race window between concurrent multi-device requests from the same user.
- **R12.** WHEN the middleware crosses the quota threshold for a user (i.e. increment fails because count is already at limit) THE SYSTEM SHALL emit `logger.info('quota_check_hit_limit', { userId, monthStart: <ISO>, currentCount, limit })` — exactly ONCE per (user, month) tuple. Subsequent 402 responses in the same month MUST be silent (no log spam).
- **R13.** WHEN env `FREE_TIER_MONTHLY_SESSION_LIMIT` is unset OR invalid (non-numeric, ≤ 0) THE SYSTEM SHALL fall back to the literal default `3`. The middleware reads `env.freeTierMonthlySessionLimit` (typed) — env parsing handles the fallback.

### Backend — admin tier override

- **R14.** WHEN `PATCH /api/admin/users/:id/tier` is called with a valid JWT AND requester role === `super_admin` AND body `{ tier: 'free' | 'premium' }` THE SYSTEM SHALL persist the new tier via `changeUserTier.useCase`, emit `AUDIT_ADMIN_USER_TIER_CHANGED` with metadata `{ from, to, actorId, targetUserId }` (awaited resolved), respond 200 with the updated `AdminUserDTO` (containing the new `tier` field).
- **R15.** WHEN `PATCH /api/admin/users/:id/tier` is called with requester role !== `super_admin` (incl. `admin`, `museum_manager`, `moderator`, `visitor`) THE SYSTEM SHALL respond `403 Forbidden` via `requireRole('super_admin')` BEFORE any DB read. Audit NOT emitted.
- **R16.** WHEN `PATCH /api/admin/users/:id/tier` is called with body `{ tier: 'enterprise' }` (or any string not in `{free,premium}`) THE SYSTEM SHALL respond `400 Bad Request` via the zod validator. Audit NOT emitted.
- **R17.** WHEN the tier flip is persisted THE SYSTEM SHALL NOT reset the user's `sessions_month_count` — flipping a free user with count=3 to premium keeps the row (`sessions_month_count=3`). Premium ignores the counter (R5). Flipping back to free retains the count. This is intentional — see Q4.

### Backend — paywall email capture

- **R18.** WHEN `POST /api/leads/paywall-interest` is received with body `{ email, consent: true, website: '' }` AND the rate-limit middleware passes (5 req / 600s / IP) THE SYSTEM SHALL invoke `submitPaywallInterest.useCase` and respond 202 `{ accepted: true }`.
- **R19.** WHEN `submitPaywallInterest.useCase` executes AND the env `BREVO_API_KEY` + `BREVO_BETA_LIST_ID` are set (per R3) THE SYSTEM SHALL call the **same `BetaSignupNotifier` port** with payload `{ email, consent: true, website: '' }` BUT extended with notifier-side metadata `OPT_IN_SOURCE: 'paywall_premium_interest'` (vs R3's `landing_beta_waitlist`). The `BrevoBetaSignupNotifier` accepts an optional `source` constructor argument OR a per-call override — see D9.
- **R20.** WHEN env Brevo creds are absent THE SYSTEM SHALL fall back to `NoopBetaSignupNotifier` and respond 202 anyway (mirror R3 R14).
- **R21.** WHEN the use case completes (success OR noop) THE SYSTEM SHALL emit `logger.info('paywall_email_captured', { requestId, emailDomain, brevoOutcome })`. Full email value MUST NOT be logged (PII discipline, mirror R3 R17).
- **R22.** WHEN body has `consent !== true` THE SYSTEM SHALL respond 400.
- **R23.** WHEN body has `website !== ''` (honeypot trip) THE SYSTEM SHALL silent-accept (202) but skip the notifier call (mirror R3 R10).

### Mobile FE

- **R24.** WHEN any HTTP response returned by `httpClient` has status 402 AND body `code === 'QUOTA_EXCEEDED'` THE SYSTEM SHALL invoke the registered paywall handler (via `setPaywallHandler`-injected callback) with the parsed `{ tier, resetAt, currentCount, limit }`. The 402 error is re-thrown after the handler returns so the calling code's `.catch()` still fires (lets the chat-session-create flow display a fallback if needed).

## 3. Design

### 3.1 User entity migration (D1) — column-vs-table for the monthly counter

**Decision** : **three columns on `users`**, NO separate `user_session_counts` table.

Three options weighed honestly :

| Option | Pro | Con | Verdict |
|---|---|---|---|
| **(a) Three columns on `users` (`tier`, `sessions_month_count`, `sessions_month_start`)** | Single row update — no JOIN ; atomic via `UPDATE … WHERE id=$1 AND sessions_month_start=$ms RETURNING …` ; no new repo ; reuses the existing `IAdminRepository` for tier update. | Pollutes `users` with stateful counter (semantically distinct from identity attributes) ; the `users` table gets one more row write per session creation. | **CHOSEN** for V1. Pre-launch users count ≪ 1000 ; row-write cost negligible vs the operational simplicity. |
| **(b) Separate `user_session_counts(user_id PK, month_start DATE, count INT)` table** | Cleanly separates identity from billing-adjacent state ; if quota tracking grows (per-feature counters, per-day, per-museum) we already have the right shape. | Needs a JOIN in the middleware ; needs a new repo + use case for the increment ; upsert semantics (counter row may not exist for month-1 users) ; adds a row per (user, month). | REJECTED for V1, candidate for V1.1 if quota model expands. |
| **(c) Redis-only counter, no DB column** | Sub-ms reads/writes ; natural TTL = end-of-month. | Cache-coherence problem : tier flip via admin DB write → Redis cache stale ; loss of counter on Redis flush ; audit invisibility (counter never on the audit chain). Fragile for billing-precursor data. | REJECTED — see N1 (no Redis V1) and Risk3 in §6. |

Trade-off honesty : option (a) is "not the cleanest schema" — `users.sessions_month_count` is a state-y attribute on an identity table. Acceptable cost vs (b)'s ~+100 LoC of repo / use case / test surface. If C6.5 funnel signals a 2nd quota dimension (e.g. per-feature) V1.1 migrates to (b).

The `tier` column is permanent regardless of (a) vs (b) — that one stays on `users` (identity-adjacent like `role`).

### 3.2 Monthly counter strategy (D2)

Increment is single atomic SQL :

```sql
UPDATE users
SET sessions_month_count =
      CASE
        WHEN sessions_month_start = $month_start THEN sessions_month_count + 1
        ELSE 1
      END,
    sessions_month_start = $month_start
WHERE id = $userId
  AND tier = 'free'
  AND (
    sessions_month_start IS NULL
    OR sessions_month_start <> $month_start
    OR sessions_month_count < $limit
  )
RETURNING sessions_month_count, sessions_month_start
```

Behavior :
- Row returned → request passes the middleware, that count is the post-increment value (= 1 on first session of month, = 2 on second, etc., up to limit).
- 0 rows returned → user is at quota, respond 402.
- Two concurrent multi-device requests from the same user race on the WHERE : first wins, second's `sessions_month_count < $limit` is false at the moment of the SQL re-eval (PostgreSQL serialises row-level locks during `UPDATE`) → second gets 0 rows → 402.
- Tier flip (admin) DOES NOT touch `sessions_month_count` or `sessions_month_start` (R17). Flipping from free→premium short-circuits the middleware entirely (R5).

`$month_start` is `'YYYY-MM-01'` in UTC (string passed as DATE param).

### 3.3 Middleware pipeline (D3) — insertion point + increment timing

**Decision** : middleware mounted ONLY on `POST /api/sessions` (the session-create handler), NOT on `POST /sessions/:id/messages` (message-create). Single quota gate, on session creation only.

Rationale :

| Option | Pro | Con | Verdict |
|---|---|---|---|
| **(a) Increment on session creation** — quota = "you can start 3 conversations per month" | Maps to the user mental model ("I started a new visit") ; one-shot decision per session ; matches the C6 wording "3 sessions/mois" | If a user opens 3 sessions then doesn't send any message, they consume the full quota. Edge case for indecisive users. | **CHOSEN** — matches the roadmap text literally + single increment point = minimal complexity. |
| **(b) Increment on first message of a new session** — quota = "you can have 3 active conversations per month" | More forgiving (UI accidents don't cost quota) | Needs to detect "is this the first message of this session?" → new column `first_message_at` on `chat_sessions` OR a count subquery → +30 LoC ; lazy increment timing is harder to reason about | REJECTED. |
| **(c) Increment on every message** | Easy to reason about per-message | Conflicts with the existing `dailyChatLimit` (which counts messages/day) — two parallel message-counting policies on same surface = confusion | REJECTED. |

The middleware sits AFTER `isAuthenticated` and BEFORE `validateBody(createSessionSchema)` in `chat-session.route.ts:29` :

```typescript
router.post(
  '/sessions',
  isAuthenticated,
  monthlySessionQuota,         // ← NEW, R1
  validateBody(createSessionSchema),
  async (req, res) => { ... }
);
```

The `dailyChatLimit` middleware on the message-POST surface (`chat-message.route.ts:191`) stays untouched — it's a per-day per-message rate limit, distinct from the per-month session quota. Two independent surfaces.

### 3.4 402 response shape (D4)

```json
{
  "code": "QUOTA_EXCEEDED",
  "tier": "free",
  "currentCount": 3,
  "limit": 3,
  "resetAt": "2026-06-01T00:00:00.000Z",
  "message": "Monthly free-tier session limit reached"
}
```

- `code: 'QUOTA_EXCEEDED'` — used by the mobile axios interceptor to decide whether to trigger the paywall (`error.response?.data?.code === 'QUOTA_EXCEEDED'`).
- `tier` — let the FE distinguish "you're free, upgrade" from a hypothetical V1.1 "you're premium but on a paused subscription".
- `resetAt` — first-day-of-NEXT-UTC-month, ISO string. Mobile renders "Resets on June 1" via `Intl.DateTimeFormat`.
- `currentCount` + `limit` — for UI substitution ("You've used 3 of 3 sessions").
- Status code 402 (Payment Required) is the semantically correct status. 429 is reserved for time-based rate-limits.

The middleware uses `res.status(402).json(body)` directly — NOT throwing an AppError — because the body shape is paywall-specific (the standard `AppError` JSON shape omits `tier` / `resetAt`).

### 3.5 Tier-change use case + audit (D5)

`changeUserTier.useCase.ts` mirrors `changeUserRole.useCase.ts:18-63` :

```typescript
export class ChangeUserTierUseCase {
  constructor(private readonly repo: IAdminRepository) {}
  async execute(input: { userId: number; newTier: UserTier; actorId: number; ip?: string; requestId?: string }): Promise<AdminUserDTO> {
    if (!Object.values(UserTier).includes(input.newTier as UserTier)) {
      throw badRequest(`Invalid tier. Must be one of: ${Object.values(UserTier).join(', ')}`);
    }
    const previous = await this.repo.findUserById(input.userId);
    if (!previous) throw notFound('User not found');
    if (previous.tier === input.newTier) {
      // No-op flip — return current DTO without audit-log spam.
      return previous;
    }
    const updated = await this.repo.changeUserTier(input.userId, input.newTier);
    if (!updated) throw notFound('User not found');
    await auditService.log({
      action: AUDIT_ADMIN_USER_TIER_CHANGED,
      actorType: 'user',
      actorId: input.actorId,
      targetType: 'user',
      targetId: String(input.userId),
      metadata: { from: previous.tier, to: input.newTier },
      ip: input.ip ?? null,
      requestId: input.requestId ?? null,
    });
    return updated;
  }
}
```

Two design points :

- **No-op skip (idempotent flip)** : flipping free → free emits no audit row + returns current DTO. Prevents double-click audit spam. Matches the spirit of `changeUserRole.useCase` (no explicit no-op guard there, BUT the constraint is "do not break the last admin" — different invariant).
- **Audit metadata** : `{ from, to }`. Does NOT include `sessionsMonthCount` or any state — state belongs to the user row, not the audit history. Per N9 of R2 doctrine (no payload in audit metadata).

### 3.7 Mobile PaywallProvider + axios interceptor (D7) — modal isolation

**Setter-injection pattern** mirroring `setUnauthorizedHandler` (`httpClient.ts:46`) / `setAuthRefreshHandler` (`httpClient.ts:57`) :

```typescript
// shared/infrastructure/httpClient.ts (NEW lines next to existing setters)
type PaywallHandler = (info: { tier: string; currentCount: number; limit: number; resetAt: string }) => void;
let paywallHandler: PaywallHandler | null = null;

export const setPaywallHandler = (fn: PaywallHandler | null): void => {
  paywallHandler = fn;
};

// In the response error interceptor (existing block ~line 214):
if (error.response?.status === 402 && error.response?.data?.code === 'QUOTA_EXCEEDED') {

    currentCount: error.response.data.currentCount,
    limit: error.response.data.limit,

```

Rationale :
- The interceptor lives in `shared/infrastructure/` → zero React import. The handler is registered via setter from `features/paywall/`. Decouples infrastructure from UI.
- The 402 error STILL propagates to the calling code's `.catch()` so existing chat-session-create flows can show their own fallback. The modal is additive, not exclusive.
- The provider wires + unwires :

The modal lives wrapped at `_layout.tsx` ABOVE chat — so the modal renders on top of any screen, including the chat screen, WITHOUT importing anything from `features/chat/`. The chat screen sees the 402 via its own mutation `.catch()` and shows its own error toast if needed ; the modal opens orthogonally.

### 3.8 Funnel telemetry (D8)

Three event sources, distinct sinks :

| Event | Where emitted | Sink | Shape |
|---|---|---|---|
| `quota_check_hit_limit` | BE middleware on FIRST 402 of (user, month) | structured logger (`logger.info`) | `{ userId, monthStart: ISO, currentCount, limit }` |
| `paywall_modal_shown` | FE `<PaywallProvider>` on `setIsOpen(true)` | Sentry breadcrumb (`category: 'paywall'`) | `{ tier, currentCount, limit, resetAt }` |
| `paywall_cta_clicked` | FE `<PaywallModal>` on submit click | Sentry breadcrumb | `{}` (empty data, the existence of the event IS the signal) |
| `paywall_email_captured` | BE `submitPaywallInterest.useCase` after notifier resolves | structured logger | `{ requestId, emailDomain, brevoOutcome: 'subscribed' \| 'duplicate' \| 'noop' }` |
| `paywall_email_captured` (FE) | FE on 202 from POST | Sentry breadcrumb | `{}` |

Why two sinks (logger vs Sentry breadcrumb) — they serve different consumers : the BE logger feeds the prod log pipeline (queryable in Loki / Sentry Logs), while Sentry breadcrumbs attach to crash reports + user-journey reconstruction. Both are needed for KR4 funnel + post-launch debugging.

No Prometheus counter in V1 (R3/R4/R2 N17 doctrine — Prom counters deferred to V1.1 if volume signal emerges). Add after we have 1 month of `quota_check_hit_limit` log data.

### 3.9 Email-capture reuse R3 vs new adapter (D9)

**Decision** : **REUSE the existing `BetaSignupNotifier` port + `BrevoBetaSignupNotifier` adapter** from R3. Distinguish by **per-call source override** (constructor variant rejected — see below).

Three options weighed :

| Option | Pro | Con | Verdict |
|---|---|---|---|
| **(a) Reuse port + adapter, hardcoded `OPT_IN_SOURCE: 'landing_beta_waitlist'`** | Zero new code, smallest diff | Brevo contact attribute is lying about origin (paywall users get tagged as landing-form users) → C6.5 funnel signal corrupted. Marketing automation in Brevo can't differentiate cohort. | REJECTED — corrupts the funnel data, the very thing R1 is supposed to emit. |
| **(b) Reuse port, new adapter `BrevoPaywallInterestNotifier` parallel to `BrevoBetaSignupNotifier`** | Cleanest separation by source ; each adapter has its own OPT_IN_SOURCE literal | ~80 LoC of near-duplicate adapter ; identical Brevo API contract ; the two adapters drift over time | REJECTED — DRY violation, no real separation of concerns. |
| **(c) Reuse port + adapter, add optional `source` parameter to the `subscribe()` method, default to current `'landing_beta_waitlist'` for R3 backward-compat** | Single adapter, single port, two call sites differentiated by argument ; preserves R3 behavior unchanged ; minimal new code (~5 LoC adapter, ~10 LoC use case) | The port signature widens slightly (one optional field) ; `NoopBetaSignupNotifier` ignores the param | **CHOSEN**. |

## 4. Open Questions

### Q1. Reset behavior when admin flips tier — does flipping premium→free reset `sessions_month_count` to 0 ?

R17 currently says NO — the counter is preserved. Rationale : flipping premium→free is a punitive admin action (B2B trial expired) ; the user shouldn't get a "free" 3 sessions just because they were premium last month. But the alternative (reset to 0 on flip) gives the demoted user a fresh start, which may be the more humane B2B flow.

**Default position** : preserve counter (R17). If a B2B pilot flow needs reset-on-demote, V1.1 adds a `resetCounter` flag to the PATCH body. Confirm.

### Q2. UTC month boundary vs user-locale month boundary ?

The middleware uses UTC (N5). A user in Asia/Tokyo on May 31 23:30 local time = May 31 14:30 UTC — still on May's quota. A user in America/Los_Angeles on June 1 06:00 local = June 1 13:00 UTC — already on June's quota. UTC-based resets are SLIGHTLY unfair to one timezone vs another (max ~12h skew).

**Default position** : UTC. Single source of truth ; matches the rest of the BE (`dailyChatLimit` uses UTC too — `daily-chat-limit.middleware.ts:21`). Per-user TZ adjustment = V1.1 (would need to persist `user.timezone` first). Confirm.

### Q3. `quota_check_hit_limit` — emit once per (user, month) or on every 402 ?

R12 says "exactly ONCE per (user, month)". Pros : log volume bounded. Cons : we don't see retry attempts (how many times did the user keep banging on session-create after the modal closed ?). Alternative : emit on every 402, but at `debug` level → still queryable in dev, dropped in prod.

**Default position** : once per (user, month) at `info`, dropped on subsequent 402s. The retry pattern is captured at the FE via Sentry breadcrumb counts (`paywall_modal_shown` fires every time). BE log stays funnel-clean. Confirm — alternative = every 402 at `info` if KR4 funnel needs cleaner cardinality.

### Q4. Should the FE show different copy depending on `reason.tier` ?

D7 passes `{ tier, currentCount, limit, resetAt }` to the provider. Currently `tier` is always `'free'` (premium users never see the modal). The copy could be unconditional. But forward-compat for V1.1 (paused subscription, trial expired) wants tier-keyed copy.

**Default position** : the i18n keys are tier-unaware in V1 (`paywall.modalBody` is a single string with `{count, limit}` substitutions). V1.1 adds tier-keyed variants. Confirm.

### Q5. Paywall email capture — auto-consent or explicit checkbox ?

The user has clicked "Join premium waitlist" — an explicit action. Could we treat that click as consent ? Legal honesty : GDPR Art. 7 requires consent to be "freely given, specific, informed, and unambiguous". A button click "Join premium" is not unambiguously consent to marketing emails (the user might think they're just expressing interest in a paywall lift, not subscribing to a newsletter).

**Default position** : **explicit checkbox** (N6). Mirror R3 doctrine. The modal has email input + consent checkbox + submit, exactly like the landing form. Two extra clicks ; legal clarity. Confirm.

### Q6. Audit row at session-creation 402 ?

The middleware emits a `logger.info('quota_check_hit_limit', ...)` once per (user, month) (R12). Should it ALSO emit an audit row (`AUDIT_QUOTA_HIT` or similar) ? Audit chain = SOC2 / RGPD trail ; quota events are not security-relevant ; logging them in `audit_logs` would inflate the chain by N users × 1 row per month.

**Default position** : NO audit row for quota hit. The `logger.info` line is enough for the funnel ; `audit_logs` stays for security + lifecycle events. Confirm.

### Q7. Mobile modal — dismissible or sticky ?

Should the user be able to close the modal without entering an email ? UX honesty : forcing a modal closed-only-via-email-submit is dark-pattern territory. Free users who don't care about premium should still be able to dismiss + use the rest of the app (browsing artworks, daily art, walks).

**Default position** : dismissible (close button visible, swipe-down to dismiss, hardware back closes). Email capture is optional. Funnel reports "modal shown vs cta clicked vs email captured" so we can measure the drop-off honestly. Confirm.

### Q8. Confirm modal for admin tier flip — required ?

R33 says yes. The button is one click + a second confirm modal click. Two clicks per flip = friction. Alternative : single-click flip with a "Undo" toast for 5 seconds.

**Default position** : confirm modal. Tier flip is billing-adjacent ; admin should mean it. Mirror the role-change UX (which also confirms). Undo-toast pattern V1.1 if B2B admin feedback says friction is too high. Confirm.

### Q9. Backfill — should `sessions_month_start` be set to first-day-of-current-month for ALL existing users at migration time, or stay NULL ?

NULL means the first session-create after deploy resets the counter to 1 and sets the date. Setting it to current-month-start at migration time means existing users start with `count=0, start=<current>`. Behavior is identical from the user perspective. Difference is purely cosmetic / schema-cleanliness.

**Default position** : NULL backfill (less migration work, NULL is a valid resting state per R6). Confirm.

### Q10. Mobile telemetry — Sentry breadcrumbs only, or also a custom event endpoint ?

Sentry breadcrumbs are attached to crashes only — they're not queryable as a stream. For KR4 funnel querying (e.g. "how many `paywall_modal_shown` last week"), we'd need a custom analytics event endpoint. R3 deferred Sentry breadcrumbs to V1.1 (R3 § OUT). R1's funnel matters more for the Stripe go/no-go decision.

**Default position** : Sentry breadcrumbs in V1. Crashes show the user journey ; aggregate funnel queries deferred. Add a Prometheus counter or analytics endpoint in V1.1 if Tim wants real-time funnel dashboards. Confirm.

## 6. Risk register

- **Risk1 (Concurrent session creation race)** — two devices submit `POST /sessions` for the same user at the same time, both pre-increment-check pass, both increment → count goes to 4 of 3. Mitigated by R11 atomic UPDATE-with-WHERE-condition (PostgreSQL row lock during UPDATE serialises concurrent UPDATEs ; the second SQL re-evaluates `sessions_month_count < $limit` and returns 0 rows). Test in `chat-session-quota.integration.test.ts` runs two parallel POSTs and asserts exactly one returns 201.
- **Risk2 (Counter drift across deploys)** — if the deploy of R1 races with users currently in session-creation, some users may see their first attempt post-deploy increment from `NULL`. R6 handles this — first hit on null start initialises to 1 (not 0+1=1, same outcome). No drift.
- **Risk3 (Cache absence vs cache-coherence)** — N1 forbids Redis. If a multi-instance BE deploys behind a LB, EACH instance reads/writes the same `users.sessions_month_count` row. PostgreSQL is the single source of truth ; no cache. Forward-compat note : if BE scales to ≥3 instances under high concurrency, the row contention on `users` becomes a hotspot. V1.1 considers a dedicated `user_session_counts` table (D1 option b) which spreads writes onto a leaf table.
- **Risk4 (Mobile modal does not open on 402)** — the axios interceptor branch is conditional on response body shape (`code === 'QUOTA_EXCEEDED'`). If BE ever changes the body shape, the modal silently fails. Mitigated by E2E integration test (AC20) + the BE 402 contract test pinning the exact `code` literal.
- **Risk5 (Email captured but user never gets a premium-launch email)** — Brevo list lives in our Brevo account ; if Stripe launch happens > 3 months from R1 ship, users may forget signing up. Mitigated by sending a "Hi, you signed up to the premium waitlist N months ago" email mid-bake. Out-of-scope for R1 ; tracked in TECH_DEBT.
- **Risk6 (Admin flipping tier in a B2B trial-expiry flow gets confused)** — current schema has no `tier_expires_at` ; a B2B "free trial of premium" needs manual flip back to free after N days. Documented limitation ; V1.1 adds expiry.
- **Risk7 (Counter not RGPD-relevant but still personal data)** — `sessions_month_count` is a usage metric per user → personal data per GDPR Art. 4(1). Existing `/api/users/me/export` DSAR path SHOULD include the new column. Mitigated by adding `tier`, `sessionsMonthCount`, `sessionsMonthStart` to the DSAR export shape — small T3 follow-up (NOT a separate spec).
- **Risk8 (BottomSheetRouter accidentally imported)** — chat-ux territory creep. AC13 sentinel + module-boundary test (T1.12) catches at CI. Continuous : pre-commit hook OPTIONAL future strengthening — V1.1.
- **Risk9 (Mobile audit log noise)** — `paywall_modal_shown` fires on every quota hit. A free user retrying 10 times in a row sends 10 breadcrumbs. Sentry breadcrumbs are capped at 100 per scope → graceful overflow. Not a bug.
- **Risk10 (Migration deployed but middleware not yet mounted)** — if the migration ships before the middleware code (deploy ordering glitch), users get the new columns but no quota enforcement. Defaults (`count=0`, `start=NULL`) mean ZERO false positives. Deploy ordering is single-blob (one PR ships migration + code together) → not a real risk.
