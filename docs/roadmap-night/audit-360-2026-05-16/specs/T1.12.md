## Design

### Approche split — recommandation

**Option A retenue : 2 fichiers nouveaux sous `features/support/` (déjà existante, "Exception accepted" features/README.md ligne 25).**

Structure proposée (s'aligne sur la convention 4-layers hexagonal-light de `features/README.md`) :
```
museum-frontend/features/support/
├── application/                       # ← NOUVEAU folder (support passe de 2 → 3 layers)
│   └── useTicketsListScreen.ts        # ← NOUVEAU : state + filter + data hooks (≈170L)
├── infrastructure/
│   └── ticketApi.ts                   # (inchangé)
└── ui/
    ├── ticketHelpers.ts               # (inchangé)
    └── TicketsListView.tsx            # ← NOUVEAU : presenter pur (≈200L, markup + StyleSheet)
```

`museum-frontend/app/(stack)/tickets.tsx` devient :
```tsx
import TicketsScreen from '@/app/(stack)/tickets';  // route entry
// ≤100L : orchestration only — call the hook, pass props to the view.
export default function TicketsScreen() {
  const state = useTicketsListScreen();
  return <TicketsListView {...state} />;
}
```

**Pourquoi Option A vs option B (`features/tickets/` nouvelle feature)** : `features/support/` existe déjà pour cette surface (ticketApi + ticketHelpers). Créer `features/tickets/` parallèle dupliquerait le scope et fragmenterait le module support. Promouvoir `support/` de "Exception 2 layers" → "Conforming 3 layers" est précisément ce que `features/README.md:25` documente comme rule "If a feature stays at ≤3 files but is **growing**, reshape into canonical folders before it crosses the threshold."

**Pourquoi NOT `useTicketsListScreen` mega-hook ?** Le hook réel proposé est un orchestrateur fin (≈170L) qui expose `{ tickets, isLoading, isRefreshing, isLoadingMore, error, statusFilter, loadTickets, loadMore, handleStatusFilter, statusLabel, dismissError }`. Si futurs T1.12 follow-ups (chat/[sessionId].tsx 504L, etc.) deviennent trop gros, splitter en sous-hooks à la `useChatSessionActions` + `useChatSessionInputHandlers`. Pour tickets 397L, **1 hook + 1 view** suffit (KISS).

### Tests à ajouter / migrer
- Tests existants (`__tests__/screens/tickets.test.tsx`, 16 cases) : **0 modif** (AC). Ils render `<TicketsScreen />` qui devient un 2-liner — le comportement testé est inchangé.
- Hook unit test : **OPTIONNEL pour T1.12** (cf. Open Question Q2). Le screen-level test couvre déjà state + filter + data. Ajouter `__tests__/features/support/useTicketsListScreen.test.ts` si on veut isoler la pagination logic (low ROI sur 397L → 100L split).
- View snapshot : **NON** — anti-pattern dans ce repo (aucun `.snap` côté `features/`, `grep -rln ".snap"` confirme 0 hits in features tree).
