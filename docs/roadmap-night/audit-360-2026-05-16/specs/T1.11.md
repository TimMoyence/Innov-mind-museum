## Design

### Constat critique
Aucune méthode actuelle ne dépasse 100L. Le file fait 611L mais ~110L sont types/interfaces, ~73L sont helpers stand-alone fin de fichier (`aggregateOutputText`, `resolveLocaleLabel`, `recordBiasMetrics`), et le reste est class body avec 7 méthodes raisonnables (max 93L). **L'AC "2 méthodes >100L" littéral est non-fondée**. Reformulation pragmatique de l'objectif vrai = "file ≤400L après split".

Cible 400L atteignable en extrayant deux blocs cohérents :
1. **V2 layers orchestration** (`runLlmJudge` 23L + `evaluateGuardrailProvider` 55L = **78L** + docstrings ~20L = ~95L removable).
2. **Output classifier** (`runArtTopicClassifier` 45L + `aggregateOutputText` 11L = ~56L removable).

Total removable du file ≈ 150L → 611 - 150 = **461L** (encore >400L) — donc une 3e extraction nécessaire.

3. **Bias metrics + locale + types** (`recordBiasMetrics` 18L + `resolveLocaleLabel` 5L + `KNOWN_LOCALES` + `GuardrailLayer` type = ~30L) → 461 - 30 = **431L** (encore au-dessus du seuil 400 sans transparence).

Pour atteindre ≤400L proprement, extraction 4 (`logBlock` 11L + `withPolicyCitation` usage move = trivial) + extraction des grosses interfaces (`GuardrailEvaluationServiceDeps` 27L) vers un fichier `guardrail-evaluation.types.ts`.

### Approche split — recommandation : Option B (sub-modules)

Pattern stateless functions — helpers reçoivent deps via args, classe service garde l'orchestration + state injecté.

#### Plan d'extraction (4 fichiers nouveaux)
- **Helper 1 — V2 layers** : `museum-backend/src/modules/chat/useCase/guardrail/eval/v2-layers.helper.ts` (~110L)
  - `runLlmJudge(text, { llmJudge, llmJudgeEnabled }): Promise<{allow}|{allow:false; reason}>`
  - `evaluateGuardrailProvider(phase, run, { guardrailProvider, observeOnly }): Promise<...>`
- **Helper 2 — Output classifier** : `museum-backend/src/modules/chat/useCase/guardrail/eval/output-classifier.helper.ts` (~75L)
  - `runArtTopicClassifier(args, { classifier, logBlock })`
  - `aggregateOutputText(text, metadata)` (move depuis stand-alone)
- **Helper 3 — Bias metrics + locale** : `museum-backend/src/modules/chat/useCase/guardrail/eval/bias-metrics.helper.ts` (~50L)
  - `recordBiasMetrics(params)`, `resolveLocaleLabel(context)`, `KNOWN_LOCALES`, `type GuardrailLayer`
- **Types extracted** : `museum-backend/src/modules/chat/useCase/guardrail/guardrail-evaluation.types.ts` (~50L)
  - `interface ArtTopicClassifierPort`, `interface InputGuardrailResult`, `interface GuardrailEvaluationServiceDeps`, `type LlmJudgeFn`, re-export `GuardrailAuditContext`.
  - **CRITIQUE** : ces types sont importés par 5 consumers ; service.ts doit ré-exporter depuis ce nouveau fichier (zero impact downstream).

Reste service.ts : constructor + 4 méthodes publiques/sous-orchestration (`evaluateInput`, `handleInputBlock`, `evaluateOutput`) + `logBlock` private + re-exports types.

Estimation finale : 611 - (95 V2 + 56 classifier + 30 bias + 50 types - 10 import lines new) = **~390L** ≤ 400 ✓.

### ADR-015 respect
- Chaque helper V2 (`runLlmJudge`, `evaluateGuardrailProvider`) reçoit ses propres deps via args (pas de service-level singleton shared) — indépendance préservée structurellement.
- Pas de fichier `v2-layers.helper.ts` "couplé" : c'est un module utilitaire qui exporte 2 fonctions pures côte-à-côte. Si demain un dev veut désactiver le judge sans toucher au provider (ou inverse), il flippe juste le flag/dep dans `chat-module.ts` — aucun import à toucher dans le helper.
- Doctrine "single source of truth content filtering" (CLAUDE.md L222) préservée : les 2 evaluators V1 (`evaluateUserInputGuardrail` + `evaluateAssistantOutputGuardrail`) restent dans `art-topic-guardrail.ts` (déjà extrait, pas touché par T1.11).
