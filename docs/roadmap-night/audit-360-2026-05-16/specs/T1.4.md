# T1.4 — Aligner ESLint majors (BE v10 vs FE/Web v9)

## Spec

**Intent (verbatim S1-types-libs.tasks.md:80-100)** : décider et appliquer un alignement ESLint major across-board (museum-backend `^10.2.0` vs museum-frontend/museum-web `^9.39.4`). `pnpm bootstrap` clean. Décision actée dans CHANGELOG ou ADR (audit propose ADR-053).

**AC (verbatim)** :
- (a) FE + Web montent à v10 : update `package.json`, run `pnpm lint` / `npm run lint` chaque app, fix breaking rule changes (ex. flat-config migration si applicable), zero net lint errors, ratchet `lintWarnings*` ne grow pas
- (b) BE descend à v9 : revert + tests
- Décision documentée dans CHANGELOG ou ADR-053. `pnpm bootstrap` (root) clean.

**Priority** : P0. **Effort spec** : 3-4h (a) ou 1h (b). **Dependencies** : none (T1.5 dépend de T1.4).

**Scope boundary** : 3 `package.json` (museum-backend, museum-frontend, museum-web) + éventuellement `eslint.config.mjs` si breaking détecté + un fichier de décision (ADR-053 OU TECH_DEBT update). Pas de refactor des règles existantes.

## Design

### Path (a) — FE + Web montent à v10 (forward-looking)

**Étapes** :
1. Bump `museum-frontend/package.json` : `eslint ^9.39.4 → ^10.4.0`, `@eslint/js ^9.39.4 → ^10.0.1`.
2. Bump `museum-web/package.json` : idem + `eslint-plugin-react-hooks ^7.1.1 → ^7.1.1` (déjà compat v10).
3. Run `pnpm bootstrap` (root) puis `pnpm lint` (FE + Web).
4. Attendre l'échec runtime sur `eslint-plugin-react` (reproduit dans TD-7).

**Risques identifiés** :
- **BLOCKER UPSTREAM** : `eslint-plugin-react@7.37.5` (latest) appelle `context.getFilename()` retiré en v10 → fail systématique sur tout fichier JSX.
- **BLOCKER UPSTREAM #2** (Web) : `eslint-plugin-jsx-a11y@6.10.2` (latest) peer `^9` ; si appelle aussi API retirée → idem.
- **BLOCKER FE** : `eslint-plugin-react-native@5.0.0` (latest, 0 release depuis 2024) peer `^9` max.

**Effort estimé sites** : N/A — bloqué upstream, pas une question de N rules à migrer mais d'attendre `jsx-eslint/eslint-plugin-react#3979` merge + release ≥ 7.38.

**Workarounds rejetés** (TECH_DEBT.md:193-195) :
- `patch-package` sur `eslint-plugin-react` : maintenance overhead, breakage à chaque release. **Non justifié J-16 launch.**
- Fork interne : idem, plus surface plugin maintenance que valeur.

**Décision documentée où** : non applicable — path (a) **impossible aujourd'hui**.

### Path (b) — BE redescend à v9 (low-risk alignment)

**Étapes** :
1. Edit `museum-backend/package.json` : `eslint ^10.2.0 → ^9.39.4`, `@eslint/js ^10.0.1 → ^9.39.4`.
2. Run `pnpm bootstrap` (root) → `pnpm install` BE.
3. Run `pnpm lint` BE (existant `--max-warnings=0`).
4. Si OK : run `pnpm lint:test-discipline` BE (le local plugin supporte `>=8 <11`, sera fine).
5. Run `pnpm test` BE (sanity) + `pnpm test` FE + `pnpm test` Web.

**Risques** :
- **Perd les améliorations v10 sur BE** (perf, nouveaux rules built-in, deprecation notices). Aucune feature v10-specific actuellement utilisée dans `museum-backend/eslint.config.mjs` (vérifié — imports = `@eslint/js`, `typescript-eslint`, plugins listés ci-dessus, tous compat v9).
- Devra re-bumper dans quelques mois quand upstream fix (estimé Q3 2026 selon vélocité PR `#3979`).
- `eslint-plugin-jsdoc@62.9.0` reste compat v9 (peer `^7..^10`) — **TECH_DEBT.md:194 affirmait "force jsdoc < 62.7"** : **C'EST FAUX** (vérifié `npm view eslint-plugin-jsdoc@62.7.0..62.9.0 peerDependencies` = tous `^9..^10`). Le claim TD-7 sur ce point est obsolète, pas de cascade jsdoc nécessaire.

**Effort minimal** : ~30 min (2 lignes edit, `pnpm install`, run lint, commit).

### Path (c) — Status quo (TD-7 doctrine actuelle)

**Étapes** :
1. Aucun edit `package.json`. Drift accepté.
2. Update `docs/TECH_DEBT.md` TD-7 avec recheck date 2026-05-16 + plan recheck 2026-06-01 et 2026-06-15.
3. Update `docs/roadmap-night/audit-360-2026-05-16/tracking.md` : T1.4 statut = "deferred to post-launch, alignment blocked upstream, TD-7 owner".

**Risques** :
- Maintenance config doublée (déjà compté par audit). Concret : la prochaine fois qu'une rule diverge entre v9 et v10, debug doublé.
- Drift sémantique potentiel sur rules built-in retravaillées (ex. `no-unused-vars` v10 modifié pour rest-siblings).

**Justification UFR-001 (product-first) + UFR-019 (no inflated estimates)** : J-16 launch, zero user-facing bénéfice à path (a) ou (b), TD-7 déjà tracké avec recheck plan.

### Recommandation finale

**Path (b) — BE descend à v9.39.4** PUIS update TD-7 en "convergé sur v9, attente upstream fix pour re-bump TOUS à v10".

**Justification chiffrée** :
- Effort réel : **~30 min** (2 lignes BE + install + lint sanity, vs. claim audit S1-tasks "1h"). FE/Web inchangés.
- Risque rollback : nul — `eslint@9.39.4` est la maintenance track upstream (≠ unsupported), BE plugins listés tous peer `^9` compatible (table ci-dessus, exhaustive).
- **Élimine** la double-maintenance config dénoncée audit S1 5.2 + TD-7 symptôme principal.
- Lance V1 2026-06-01 : 16 jours. Path (a) impossible (bloqué upstream), path (c) garde la dette ouverte sans réduction.
- UFR-001 : alignement = product-first → moins de surface debug en cas de lint flaky pre-launch.
- UFR-007 (autonomy / product-lifecycle pré-V1) : prend la décision réversible plutôt que de la différer.

**Caveat honnêteté (UFR-013)** : path (b) n'est pas "forward-looking", c'est un alignement-vers-le-bas en attendant upstream. Si user préfère path (c) (zéro touche, accepter dette TD-7 existante), c'est défendable — l'effort path (b) est faible mais non nul, et le bénéfice est interne (DX) pas user.
