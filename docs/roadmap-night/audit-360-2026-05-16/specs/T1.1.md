## Design

### Path (a) — upgrade @langfuse/tracing v5
- Étapes : (1) remove `langfuse` from package.json, add `@langfuse/tracing` + `@langfuse/otel` ; (2) refactor `langfuse.client.ts` → OTEL provider boot wiring ; (3) rewrite 7 production call-sites handle-based → active-span async ; (4) rewrite 4 test mocks ; (5) `pnpm test` BE pass ; (6) smoke `pnpm dev` startup + verify spans land on Langfuse Cloud staging ; (7) update CLAUDE.md "AI Safety" + memory `reference_otel_router_max_listeners.md` si conflit OTEL provider apparu.
- Risques : (i) provider conflict avec `sdk-node` existant ; (ii) span semantics shift → `chat_phase_complete` log contract drift (consumed par Grafana dashboards) ; (iii) silent span drop pendant bake si OTEL endpoint mal configuré.
- Tests à ajouter : couverture mock OTEL surface (`startActiveObservation`, `LangfuseSpanProcessor.export`).

### Path (b) — ADR-052 accept-EOL (DÉJÀ FAIT, juste à renommer)
Le contenu existe à `docs/adr/ADR-050-accept-langfuse-v3-eol.md` (96 lignes, dated 2026-05-13). Format suit ADR convention (Status / Date / Owner / Sunset / Trigger / Context / Decision / Consequences / Alternatives / References).

**Mitigations sécurité documentées :**
- Tilde-pin (`~3.38.20`) — patch only.
- Renovate / Dependabot abonné repo-level → CVE v3.x déclenche PR auto.
- GitHub Security Advisory subscription `langfuse-js` (v3 = thin wrapper autour `langfuse-core`, dep unique).
- Pas de PII custom property utilisé (vérifié grep `.generation(`, `usage:`).

**Sunset** : 2026-09-01 (Q3 2026, 3 mois post-launch V1).

**Rollback plan** (extrait ADR) : si CVE avant sunset, ré-ouvrir ADR ; le plan v3 → v5 dans "Alternatives considered A1" devient executable spec. Sinon, repplaner via Spec Kit (`spec.md` + `design.md` + `tasks.md`).

- **UFR-001 product-first** : chat path = fail-open via `safeTrace`, langfuse outage n'impacte pas utilisateur. Migration architecturale pré-launch = trade-off "CVE hypothétique futur" vs "regression launch-window concrète" perdant.
