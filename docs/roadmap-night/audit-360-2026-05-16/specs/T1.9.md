## Design

### Décision : helper canonical = `readEnvString`

**Recommandation** : adopter `readEnvString` comme nom canonical, l'exporter depuis
`museum-frontend/shared/lib/env.ts` (nouveau fichier), supprimer les 3 définitions locales.

Rationale :
- Nom le plus **explicite intentionnellement** ("lire une variable d'env, retourne
  string ou undefined") vs `typeofString` (décrit le mécanisme, pas l'intent) vs
  `trimOrUndefined` (suggère que le trim est l'intent — il l'est dans 1 helper sur 3 seulement).
- `trimOrUndefined` est l'helper le plus *utilisé* (7 sites) mais son nom communique
  une transformation, alors que les sites SENTRY_DSN ne veulent PAS trimmer (un DSN
  avec espaces internes serait un bug existant ; en pratique inoffensif mais l'intent
  n'est pas "trim", l'intent est "lire env-var safely").
- `readEnvString` est neutre côté transformation : on peut décider via overload ou
  param `{ trim?: boolean }` si nécessaire. **Recommandation initiale** : trim PAR
  DÉFAUT (matche le pattern majoritaire `trimOrUndefined`, défensif sans coût).

Signature retenue :
```ts
// museum-frontend/shared/lib/env.ts
export const readEnvString = (value: unknown): string | undefined => {
  if (typeof value !== 'string') return undefined;
  const trimmed = value.trim();
  return trimmed.length > 0 ? trimmed : undefined;
};
```

### Update CLAUDE.md (ligne 150 — phrase à remplacer)

**Avant** (segment) :
> Solution : helper `typeofString` au top du module — `const typeofString = (v: unknown): string | undefined => typeof v === 'string' && v.length > 0 ? v : undefined`. Predicate typeof, ESLint ne peut PAS autofix away, narrows `any` → `string` côté CI.

**Après** :
> Solution : helper canonical `readEnvString` exporté depuis `museum-frontend/shared/lib/env.ts`. Implémentation : `typeof value !== 'string' ? undefined : value.trim() || undefined`. Predicate typeof, ESLint ne peut PAS autofix away, narrows `any` → `string` côté CI. Aucun helper local dans les modules — **toujours importer** `import { readEnvString } from '@/shared/lib/env'`. Audit 2026-05-16 (T1.9) a unifié les 3 variantes historiques (`typeofString` / `trimOrUndefined` / `readEnvString`).

### ast-grep rule (Q3 — recommandée in-scope)

`tools/ast-grep-rules/use-readenv-helper.yml` :
```yaml
id: use-readenv-helper
language: tsx
severity: warning
message: |
  Direct `process.env.X` read detected. Wrap via `readEnvString` from
  `@/shared/lib/env` to bridge local/CI typing divergence (local: string|undefined ;
  CI: any → fires no-unsafe-*).
rule:
  pattern: process.env.$NAME
  not:
    inside:
      any:
        - pattern: readEnvString(process.env.$$$)
        - pattern: normalizeApiEnvironment(process.env.$$$)
files:
  - 'museum-frontend/**/*.ts'
  - 'museum-frontend/**/*.tsx'
ignores:
  - '**/*.test.ts'
  - '**/*.test.tsx'
  - '**/__tests__/**'
  - 'museum-frontend/jest.config.js'
  - 'museum-frontend/shared/lib/env.ts'  # définition du helper
```

Note : la syntaxe `not.inside` ast-grep est à valider via `ast-grep scan --rule <file>` sur
le worktree (T1.9.d). Fallback si syntaxe ne convient pas : rule purement permissive
(match `process.env.X` sans `not.inside`) + allowlist des 11 sites canoniques, plus
strict mais plus simple à maintenir.
