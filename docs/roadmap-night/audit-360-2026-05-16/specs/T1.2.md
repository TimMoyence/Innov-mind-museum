> Les 2 sites cités (JWKS response + Google token exchange response) passent par `JwksResponseSchema.safeParse()` / `GoogleTokenResponseSchema.safeParse()` au lieu de `as JwksResponse` / `as GoogleTokenResponse`. En cas d'échec : throw `ExternalApiContractError` (existant) avec event-log structuré.
> Tests unitaires ajoutés (`social-token-verifier.test.ts`, `google-token-exchange.test.ts`) couvrant : (1) response valide → OK ; (2) response shape changed → throw avec log structuré.
> `pnpm test` BE pass. `as-any-baseline.json` reste à 0.

## Design

**N/A — implémentation déjà shipped.** Section conservée pour archive doctrine.

### Pattern appliqué (déjà en place)
`safeParse` + `if (!result.success) throw new AppError({ code, statusCode, details: { issues } })`. Le `details.issues` contient le tableau Zod issue (machine-readable, surface field name + expected/received pour ops).

### Schemas en place
- `JwksResponseSchema` (colocated dans verifier) : `{ keys: array(JwksKeySchema) }` — 6 fields requis par key (`kty/kid/use/alg/n/e`).
- `GoogleTokenResponseSchema` (colocated dans exchanger) : `id_token` required min(1), reste optional (`access_token/expires_in/scope/token_type`).
- Placement colocated cohérent avec doctrine "minimal-barrel" (pas d'extraction `shared/auth/schemas/` superflue).

### Log structuré
Pas de log explicite à la frontière throw — c'est le middleware error global qui logge (`AppError.code` + `details`). Acceptable car `code` + `details.issues` sont machine-readable.
**Gap mineur** : aucun appel `logger.warn` direct dans le if-block — un log défensif "JWKS_MALFORMED at {url}" pourrait aider l'ops. Voir Q3.

### Tests
2 fichiers existent et couvrent l'AC. Aucun test net à ajouter.
