## 0. Contexte vérifié

### 0.1 Pain point — où le signal in-museum est aujourd'hui invisible

`findings.md` ligne 181 : « P17 — Pas de voice proactive géoloc — absent — LocationResolver in-museum sous-exploité — **B6** ».

L'app détecte déjà côté backend qu'un visiteur est à l'intérieur d'un musée :
- `museum-backend/src/modules/chat/useCase/location-resolver.ts:95-96` calcule
  `isInsideMuseum = nearestMuseumDistance < 200` à partir de
  `findNearbyMuseums(lat, lng, museumRepository)` et cache le résultat **20 min**
  (memory `project_geolocation_pipeline`).
- Mais ce signal n'existe QUE pendant le tour de pipeline d'un message chat —
  il n'est PAS exposé en proactif sur les écrans hors-chat.

Conséquence UX : un visiteur **dans** le Louvre regardant son écran d'accueil
voit le même home qu'un visiteur **dans son train** — aucune invitation
contextuelle à parler à l'assistant. C'est exactement le « free-form voice »
identifié dans `findings.md` §3.3 comme un gap UX museum réel non adressé par
la concurrence (Smartify, Bloomberg Connects, Google Arts & Culture).

Pattern visé (findings.md §3.2 + scope §5) : **bannière contextuelle proactive
sur le home, PAS push notification**, qui dit « Vous êtes au {museumName} —
demander à l'assistant ? » + tap = ouvre une conversation voice-first.

### 0.2 Source data — quel endpoint utiliser ?

Vérification systématique de ce qui existe déjà côté BE :

| Endpoint | Verdict B6 | Justification |
|---|---|---|
| `LocationResolver.resolve(lat, lng)` → `isInsideMuseum` | **NON exposé en HTTP** | Use-case interne, appelé seulement depuis `prepare-message.pipeline.ts` côté chat. Pas de route. |
| `GET /api/museums/directory` | Trop coarse | Retourne tous les musées actifs avec coords — le FE devrait haversiner côté client. Travail dupliqué (déjà côté BE dans `findNearbyMuseums`). |
| `GET /api/museums/search?lat&lng&radius` (`museum.route.ts:48`) | **OUI — retenu** | Retourne `{ museums: [{ id, name, distance, …, latitude, longitude }], count }` trié par `distance` asc. C'est **exactement** la donnée dont on a besoin. |
| Nouveau endpoint `GET /api/chat/proximity` | **NON V1** | Ajouterait un endpoint dédié pour `isInsideMuseum` — sur-spécifique, duplique `searchMuseums`. Règle dispatcher : minimiser changes BE. |

**Décision** : **réutiliser `GET /api/museums/search?lat=X&lng=Y&radius=1000`**
côté FE et appliquer le seuil `distance < 200m` côté FE. Aucun nouvel endpoint,
aucune migration, aucun code BE. Le seuil 200m est aligné avec
`IN_MUSEUM_THRESHOLD_M` côté BE (`location-resolver.ts:16`) pour cohérence.

Contrainte schema `searchMuseumsQuerySchema` (museum.schemas.ts:54) :
`radius` minimum 1000m, maximum 50000m. On passera donc `radius=1000` et on
filtrera client-side sur `distance < 200`. Pas de relâchement du contrat
backend (le `radius` est une borne haute, pas une borne d'intérêt).

### 0.3 Permission GPS — V1 vs V1.1

Doctrine `feedback_no_feature_flags_prelaunch` + scope dispatcher § 3 :

| Stratégie | Verdict | Justification |
|---|---|---|
| **Auto-prompt GPS au mount home** | **NON V1** | Friction sur l'écran d'accueil. Un utilisateur ouvre Musaium = on demande déjà beaucoup (locale, museumMode). Greffe la GPS-prompt sur un acte délibéré. |
| **Bouton CTA "Activer la géolocalisation" dans la bannière** | **NON V1** | Affiche une bannière sans avoir de signal géo → mensonge UX. La bannière ne doit apparaître QUE quand on a un signal in-museum confirmé. |
| **Réutiliser la permission GPS demandée par `useLocation()` sur d'autres écrans (`(tabs)/museums`)** | **OUI V1** | `useLocation` (`features/museum/application/useLocation.ts:43`) demande déjà la permission une fois — Expo Location persistence couvre les sessions subséquentes. Si l'utilisateur a déjà ouvert l'onglet `Museums`, la permission est déjà accordée et `expo-location` retournera les coords immédiatement. Si l'utilisateur n'est jamais allé sur `Museums`, la bannière n'apparaît simplement pas — pas de prompt déclenché par le home. |
| **Geofence-only post-login** (`useGeofencePreCache` pattern) | V1.1 | Pattern silent pre-cache sans prompt, mais nécessite déjà un fix GPS, donc même contrainte que ci-dessus. |

**Décision V1** : `useProactiveMuseumSuggestion` appelle `useLocation()` (qui
demande la permission au premier mount **uniquement si l'utilisateur n'a
jamais répondu**). Si `status === 'denied'`, la bannière reste null
silencieusement, pas de re-prompt, pas de message d'erreur. Si
`status === 'granted'` ET coords reçues → on fetch `searchMuseums(lat,lng)`
et on vérifie le seuil 200m.

**V1.1** : ajouter un toggle Settings « Suggérer une conversation à l'arrivée
dans un musée » qui pilote l'auto-prompt explicite (Open Q3).

### 0.4 Placement de la bannière sur le home

Findings.md §3 décision dispatcher : home top, sous le B2 resumption banner si
présent (max 2 banners stack).

Vérification du `(tabs)/home.tsx` post-B2 (baseline `325873b3`) :
```
<GlassCard style={styles.heroCard}>  // BrandMark + intro
<ConversationResumptionBanner …>     // B2 — rendu null si pas de session
<DailyArtCard … />                    // Daily art
<HomeIntentChips … />                 // 3 chips audio/camera/walk
<Pressable primaryButton>Start</…>    // Primary CTA
```

**Décision placement B6** : entre `<ConversationResumptionBanner>` et
`<DailyArtCard>`, c.-à-d. position 3 du flux home. Quand B2 + B6 sont actifs
simultanément, l'ordre est : hero → B2 (reprise session récente) → B6
(suggestion in-museum) → daily art. Justification :
1. B2 = retour-utilisateur, plus prioritaire visuellement que B6 = situation-contextuelle.
2. Hauteur cumulée B2 (~80dp) + B6 (~80dp) = 160dp, reste sous le seuil 250dp
   « scroll inévitable au-dessus du fold » même iPhone SE.
3. Si l'utilisateur dismiss B2, B6 remonte naturellement en position 2.

**Ne PAS doubler** la bannière sur le chat list ni sur l'onglet `Museums` — la
carte est un point d'entrée unique conversationnel, pas une mention répétée.

### 0.5 Critères d'affichage — décisions

| Critère | Verdict | Justification |
|---|---|---|
| Permission GPS | `useLocation().status === 'granted'` ET `latitude !== null` ET `longitude !== null` | Sans coords, pas de signal. Sans permission, pas de re-prompt. |
| Threshold in-museum | **`distance < 200m`** sur la première entrée de `museums` (la plus proche) | Aligné avec `IN_MUSEUM_THRESHOLD_M = 200` (`location-resolver.ts:16`). Garde la cohérence BE/FE. |
| Source des museums | `museumApi.searchMuseums({ lat, lng, radius: 1000 })` | Endpoint existant. `radius=1000` = bord min schema BE. Le filtre 200m s'applique client-side après. |

| Dismiss user | **4h via AsyncStorage** (vs B2 = 24h) | Proactive contextual = plus court car la situation peut changer (l'utilisateur change de salle, sort, revient). 4h = couvre une visite museum typique sans re-spam le même jour. |

| Sessions walk / audio | Indépendant — la bannière propose intent `audio` (voice-first) tap action | Cohérent avec « free-form voice proactive » du findings.md. |

## 1. Spec — format EARS

### 1.1 Hook `useProactiveMuseumSuggestion()` — détection in-museum + dismiss gate

- **R1** : `useProactiveMuseumSuggestion()` SHALL return an object `{ museum: ProactiveMuseum | null; isLoading: boolean; dismiss: () => Promise<void> }`.
- **R2** : On mount, the hook SHALL read the device location via `useLocation()`. It SHALL NOT request `expo-location` permission directly — it inherits the permission state already set by the museums tab (`(tabs)/museums.tsx:47`).
- **R3** : When `useLocation()` reports `status !== 'granted'` OR `latitude === null` OR `longitude === null`, the hook SHALL resolve `{ museum: null, isLoading: false, dismiss }` and SHALL NOT call `museumApi.searchMuseums`.
- **R4** : When the location is available, the hook SHALL call `museumApi.searchMuseums({ lat, lng, radius: 1000 })` exactly once per `(lat, lng)` pair. It SHALL NOT re-call when coords stay stable across renders.
- **R5** : The hook SHALL inspect the first entry of `museums` (the closest, BE pre-sorts by `distance` asc). When `museums.length === 0` OR `museums[0].distance >= 200`, the hook SHALL resolve `museum: null`.
- **R6** : When `museums[0].distance < 200`, the hook SHALL build a `ProactiveMuseum` from that entry and resolve `museum` accordingly.
- **R7** : Before resolving `museum`, the hook SHALL read `storage.getItem('settings.proactive_museum_banner_dismissed_until')`. If the value is a parseable ISO string `T` AND `Date.now() < T.getTime()`, the hook SHALL return `museum: null` (dismissed window still active).
- **R8** : When the storage read fails or the value is unparseable, the hook SHALL treat it as "not dismissed" and proceed with normal selection (no throw).
- **R9** : The hook SHALL expose `dismiss()` which (a) writes `settings.proactive_museum_banner_dismissed_until` with `(Date.now() + 4*60*60*1000).toISOString()`, (b) sets the local `museum` state to `null` synchronously (optimistic UI), (c) does NOT re-fetch.
- **R10** : `ProactiveMuseum` SHALL have shape `{ id: number; name: string; latitude: number; longitude: number; distanceMeters: number }`. No other field is required for V1.
- **R11** : The hook SHALL log `console.debug('[B6] proactive_museum_shown', { distance_bucket: '0-50' | '50-100' | '100-150' | '150-200' })` when `museum !== null` on first resolution. No PII (no museum id, no museum name, no exact distance, no coords) is emitted.
- **R12** : The hook SHALL NOT throw on any API error — `searchMuseums` failure resolves `museum: null` and a `console.debug('[B6] proactive_museum_fetch_failed', { reason: 'fetch-error' })` (no error stack to avoid leaking).
- **R13** : The hook SHALL constant-export `PROACTIVE_MUSEUM_BANNER_DISMISS_STORAGE_KEY = 'settings.proactive_museum_banner_dismissed_until'`, `PROACTIVE_MUSEUM_BANNER_DISMISS_DURATION_MS = 4*60*60*1000`, and `PROACTIVE_MUSEUM_IN_MUSEUM_THRESHOLD_M = 200` for testability.
- **R14** : Entries with `id <= 0` (OSM-only, no local id) SHALL be filtered out — the banner needs a `museumId` we can pass to `useStartConversation`. Use the next entry in `museums` array if any satisfies both `id > 0` AND `distance < 200`; otherwise resolve `museum: null`.

### 1.2 Composant `<ProactiveMuseumBanner>` — rendu

- **R15** : When `<ProactiveMuseumBanner museum={null}>` is rendered, the component SHALL return `null` (no DOM, no layout).
- **R16** : When `<ProactiveMuseumBanner museum={museum}>` with non-null museum is rendered, the component SHALL render a `<Pressable testID="proactive-museum-banner">` at the root with `accessibilityRole="button"`.
- **R17** : The root `accessibilityLabel` SHALL be `t('chat.proactive_museum.cta_a11y_label', { museumName: museum.name })`.

- **R20** : The component SHALL render a subtitle `<Text testID="proactive-museum-subtitle">` whose content is `t('chat.proactive_museum.subtitle')`.
- **R21** : The dismiss button SHALL be a SEPARATE `<Pressable testID="proactive-museum-dismiss">` with its OWN `accessibilityRole="button"` and its own `accessibilityLabel` = `t('chat.proactive_museum.a11y_dismiss')`. It MUST NOT be a child of the card root Pressable in a way that conflates the two tap zones — RN's nested Pressable tap handling MUST register the inner press without triggering the outer press (use `event.stopPropagation()` via inner `onPress`).

### 1.3 Wire dans `(tabs)/home.tsx`

- **R27** : The screen `(tabs)/home.tsx` SHALL pass `onStart={(museum) => void startConversation({ intent: 'audio', museumId: museum.id, museumName: museum.name, coordinates: { lat: museum.latitude, lng: museum.longitude } })}` and `onDismiss={dismiss}` from the hook to the component.
- **R28** : The banner SHALL be rendered AFTER `<ConversationResumptionBanner>` (B2) and BEFORE `<DailyArtCard>` in the screen JSX. Stack order matters: B2 first (returning user), B6 second (in-museum context), DailyArt third.
- **R29** : The home screen SHALL NOT trigger any GPS prompt of its own. Permission inheritance from the museums tab is sufficient — no `Location.requestForegroundPermissionsAsync()` call added on the home.

### 1.4 BE contract

- **R30** : **NO BE CHANGE**. `GET /api/museums/search?lat&lng&radius` is consumed as-is. OpenAPI spec unchanged. No migration. No new endpoint. No new query param.
- **R31** : The FE consumes `museumApi.searchMuseums({ lat, lng, radius: 1000 })`. The `radius=1000` value is hard-coded to satisfy the BE schema minimum (`searchMuseumsQuerySchema:radius.min(1000)`) — the actual in-museum threshold (200m) is applied client-side after the response lands.

### 1.5 Coupling FE

- **R32** : The hook SHALL NOT directly import `expo-location` — it uses the `useLocation` hook façade. This keeps all permission UX behaviour in one place (`features/museum/application/useLocation.ts`).
- **R33** : The component SHALL NOT directly call `chatApi.createSession`. Session creation lives in `useStartConversation` — the banner's `onStart` callback delegates to it. This preserves the single source of truth for createSession contract (rate limits, error mapping).

### 1.6 i18n

- **R34** : The 8 shipped locales (ar, de, en, es, fr, it, ja, zh) SHALL contain the 6 new keys listed in §0.8 under `chat.proactive_museum.*`.
- **R35** : Every `chat.proactive_museum.*` string SHALL NOT contain any Unicode emoji (regex `/[\u{1F000}-\u{1FFFF}\u{2600}-\u{27BF}]/u` returns false).
- **R36** : `chat.proactive_museum.title` per locale SHALL be ≤ 60 characters (visual budget on iPhone SE width minus icon + dismiss).

### 1.7 Out-of-scope (explicite)

- Push notification on arrival (out — distinct feature, V2). Dispatcher rule #1.
- Auto-prompt GPS V1 (V1.1 Open Q2).
- Toggle Settings "suggérer in-museum" (V1.1 Open Q3).
- Per-museum dismiss tracking (V1 = global 4h dismiss).
- Multi-museum suggestion stack (V1 = single closest only).
- Real-time room detection (BLE beacon, Wi-Fi RTT) — explicitly excluded by `findings.md` §3.3 cost/risk.
- "Vous êtes dans la salle 5" — room-level granularity needs indoor positioning infra (iBeacon refused). V1 = museum-level only.
- Refresh on tab focus (V1.1 Open Q4).
- Animation/transition (V1 = mount/unmount snap — `useReducedMotion()` N/A).
- Distance displayed verbatim ("87m") — V1 = none. Trop technique.

### 2.2 Helpers — `useProactiveMuseumSuggestion.ts` (squelette)

```ts

import { useLocation } from '@/features/museum/application/useLocation';
import { museumApi } from '@/features/museum/infrastructure/museumApi';
import { storage } from '@/shared/infrastructure/storage';

export const PROACTIVE_MUSEUM_BANNER_DISMISS_STORAGE_KEY =
  'settings.proactive_museum_banner_dismissed_until';
export const PROACTIVE_MUSEUM_BANNER_DISMISS_DURATION_MS = 4 * 60 * 60 * 1000;
export const PROACTIVE_MUSEUM_IN_MUSEUM_THRESHOLD_M = 200;
const SEARCH_RADIUS_M = 1000; // BE schema min — actual threshold is 200 client-side

export interface ProactiveMuseum {
  readonly id: number;
  readonly name: string;
  readonly latitude: number;
  readonly longitude: number;
  readonly distanceMeters: number;
}

function bucketDistance(d: number): '0-50' | '50-100' | '100-150' | '150-200' {
  if (d < 50) return '0-50';
  if (d < 100) return '50-100';
  if (d < 150) return '100-150';
  return '150-200';
}

export function useProactiveMuseumSuggestion() {
  const { latitude, longitude, status } = useLocation();
  const [museum, setMuseum] = useState<ProactiveMuseum | null>(null);
  const [isLoading, setIsLoading] = useState(true);
  const lastFetchedRef = useRef<string | null>(null);

  useEffect(() => {
    let cancelled = false;
    if (status !== 'granted' || latitude === null || longitude === null) {
      setIsLoading(false);
      setMuseum(null);
      return;
    }
    // De-dup per (lat,lng) at ~3 decimals (~111m).
    const fetchKey = `${latitude.toFixed(3)}:${longitude.toFixed(3)}`;
    if (lastFetchedRef.current === fetchKey) return;
    lastFetchedRef.current = fetchKey;

    void (async () => {
      try {
        // 1. Dismiss-until gate.
        const dismissedUntilRaw = await storage
          .getItem(PROACTIVE_MUSEUM_BANNER_DISMISS_STORAGE_KEY)
          .catch(() => null);
        if (dismissedUntilRaw) {
          const dismissedUntil = new Date(dismissedUntilRaw).getTime();
          if (!Number.isNaN(dismissedUntil) && Date.now() < dismissedUntil) {
            if (!cancelled) {
              setMuseum(null);
              setIsLoading(false);
            }
            return;
          }
        }

        // 2. Fetch.
        const response = await museumApi.searchMuseums({
          lat: latitude,
          lng: longitude,
          radius: SEARCH_RADIUS_M,
        });
        if (cancelled) return;

        // 3. Pick first entry with id > 0 AND distance < 200.
        const eligible = response.museums.find(
          (m) =>
            typeof m.id === 'number' &&
            m.id > 0 &&
            typeof m.distance === 'number' &&
            m.distance < PROACTIVE_MUSEUM_IN_MUSEUM_THRESHOLD_M &&
            typeof m.latitude === 'number' &&
            typeof m.longitude === 'number',
        );
        if (!eligible) {
          setMuseum(null);
          setIsLoading(false);
          return;
        }

        const next: ProactiveMuseum = {
          id: eligible.id as number,
          name: eligible.name,
          latitude: eligible.latitude as number,
          longitude: eligible.longitude as number,
          distanceMeters: eligible.distance as number,
        };
        setMuseum(next);
        setIsLoading(false);
        console.debug('[B6] proactive_museum_shown', {
          distance_bucket: bucketDistance(next.distanceMeters),
        });
      } catch {
        if (!cancelled) {
          setMuseum(null);
          setIsLoading(false);
          console.debug('[B6] proactive_museum_fetch_failed', { reason: 'fetch-error' });
        }
      }
    })();

    return () => {
      cancelled = true;
    };
  }, [latitude, longitude, status]);

  const dismiss = useCallback(async () => {
    setMuseum(null);
    const until = new Date(Date.now() + PROACTIVE_MUSEUM_BANNER_DISMISS_DURATION_MS).toISOString();
    await storage.setItem(PROACTIVE_MUSEUM_BANNER_DISMISS_STORAGE_KEY, until);
  }, []);

  return { museum, isLoading, dismiss };
}
```
