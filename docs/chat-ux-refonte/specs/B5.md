## 0. Contexte vérifié

### 0.1 État courant — pas de mode "sotto-voce"

`findings.md` ligne 180 : « **P16** — Pas de sotto-voce mode — absent — Museum context gap — **B5** ». Le chat dispose actuellement d'UN seul axe audio :

- `useAudioDescriptionMode` (`museum-frontend/features/settings/application/useAudioDescriptionMode.ts`) — toggle global "AI responses are automatically read aloud via TTS". Persisté sous la clé AsyncStorage `settings.audio_description_mode` via la façade `storage` (`shared/infrastructure/storage.ts`).
- `useAutoTts` (`features/chat/application/useAutoTts.ts`) — auto-play TTS sur nouveau message assistant si `enabled === true`. Hook étendu par A5 pour exposer `loading: boolean`.
- `<ChatHeader onToggleAudioDescription>` — bouton casque dans le header qui inverse un override local screen-only (`sessionAudioOverride` à `[sessionId].tsx:162-163`).

**Use case sotto-voce manquant** : dans une salle de musée silencieuse (chuchotements only), le visiteur veut couper l'audio TTS pour ne pas déranger les autres tout en gardant la conversation utile — c.-à-d. **lire** la réponse au lieu de l'écouter. Le transcript texte est déjà rendu par défaut dans les bubbles assistant (`ChatMessageBubble.tsx` + `MarkdownBubble`) ; sotto-voce n'a donc qu'à **désactiver TTS** + **persister la préférence cross-session**.

### 0.2 Anatomie actuelle du `<ChatHeader>` (A4 done)

`features/chat/ui/ChatHeader.tsx` (195 lignes) expose déjà :

- `audioDescriptionEnabled?: boolean` + `onToggleAudioDescription?: () => void` (casque headset).
- `onSummary?: () => void` + `onClose: () => void`.
- `onOpenAiDisclosure?: () => void` (badge légal EU AI Act Art.50, toujours visible).
- `collapsed?: boolean` (A4 — mini-bar mode, layout shrink mais actions restent tappables).

Les 3 boutons actions (casque audio / summary / close) sont rendus côte à côte dans `<View style={styles.headerActions}>` (`ChatHeader.tsx:84-136`). Hit target 36×36dp (`space['9']`, container rond `radius.full`). Convention HIG mini-bar acceptée (cf. A4 §0.6).

Ajouter un 4ᵉ bouton **sotto-voce** dans `<View style={styles.headerActions}>` est cohérent avec ce layout existant. R10/R19 A4 contraignent l'AI badge à rester tappable en mode collapsed ; le toggle sotto-voce hérite de la même contrainte (utile précisément en musée silencieux quand le user scrolle).

### 0.3 Décision auto-detection ambient noise = **DEFERRED V1.1**

Le tracking ligne 70 note : « Top bar toggle, audio mute, transcript live ; auto-suggest si ambient >70dB ». L'auto-suggestion via micro ambient dB est technologiquement faisable (`expo-av`/`expo-audio` accepte `metering: true` lors d'un `prepareToRecordAsync`, retourne un `audioLevel` -160..0 dBFS qu'on peut convertir en dB SPL approximatif).

**Refus V1** :
- Allume le micro en permanence → drain batterie + permission micro déjà sensible (EU AI Act + RGPD).
- Faux positif élevé (un musée bruyant n'est PAS hors-musée — l'auto-suggest sera intrusif).
- Le toggle manuel suffit pour le scénario primaire (visiteur en salle silencieuse coupe l'audio).
- Doctrine memory `feedback_no_feature_flags_prelaunch` : pas de flag — soit on ship, soit on revert. Ambient detection ajoute une mécanique non triviale (calibration dB, seuil 70dB arbitraire, debounce) qu'on ne peut pas bake en pre-launch.

**V1** : toggle manuel uniquement. Auto-suggest = **Open Q1 V1.1** (post-launch, post-B2B revenue).

### 0.5 Décision UI = **Pressable icon-button dans `<ChatHeader>` (pas Switch dans une sheet)**

Le toggle sotto-voce :

- DOIT être accessible en 1 tap depuis le chat (vide-rien aller dans Settings : le scénario primaire = utilisateur ENTRE dans une salle silencieuse, doit couper l'audio NOW).
- DOIT être visible en mode collapsed (cf. §0.2 — actions buttons restent tappables en collapsed, A4 R20).
- HÉRITE de la convention casque / summary / close (rond 36dp, Ionicons, accessibilityLabel + role button).

Icon choisi : `mic-off-outline` (sotto-voce inactif → audio normal) / `mic-off` filled (sotto-voce actif → audio coupé). Note : pas `volume-mute` car ce dernier signifie "le device est en mode silencieux", pas "je veux lire au lieu d'écouter".

Variante rejetée : `Switch` natif dans une bottom-sheet "Audio settings" → 3 taps (ouvrir sheet → toggler → fermer), pas adapté au use-case 1-tap-in-silent-room.

### 0.6 Décision a11y = **`accessibilityRole="button"` + `accessibilityState.selected`**

Convention RN : un Pressable qui agit comme bascule on/off utilise `accessibilityRole="button"` + `accessibilityState={{ selected: enabled }}` (le mode `switch` est réservé aux composants `<Switch>` natifs). VoiceOver annonce alors "Sotto-voce mode, button, on" / "Sotto-voce mode, button, off".

Label = `t('chat.sottoVoce.a11y_label')` ("Sotto-voce mode").
Hint = `t('chat.sottoVoce.a11y_hint_on')` / `_hint_off` selon l'état courant.

### 0.7 Décision effet sur TTS = **gate `useAutoTts` côté écran**

Pas de modification de `useAutoTts` (déjà extensible via `enabled: boolean`). Le screen `[sessionId].tsx` consomme :

```ts
const { enabled: audioDescEnabled } = useAudioDescriptionMode();
const { enabled: sottoVoce } = useSottoVoce();  // NEW B5
const [sessionAudioOverride, setSessionAudioOverride] = useState<boolean | null>(null);
const effectiveAudioDesc = (sessionAudioOverride ?? audioDescEnabled) && !sottoVoce;
```

**Sémantique** : sotto-voce ON dégage TOUTE source d'audio (override session OU pref globale OU défaut), sans modifier les pref sous-jacentes — quand sotto-voce repasse OFF, l'audio reprend selon les pref d'origine.

ZÉRO modification de la signature `useAutoTts({ messages, enabled })`. La gate vit dans le screen (composition).

### 0.8 Décision transcript live = **réutilisation rendu bubble existant (PAS de transcript dédié)**

Le tracking ligne 70 mentionne "transcript live". Vérification :

- Les bubbles assistant rendent déjà le texte intégral via `<MarkdownBubble>` (`features/chat/ui/MarkdownBubble.tsx`).
- En mode streaming, `<StreamingBody>` rend `text` token-par-token en cursor blink (déjà mot-par-mot perçu).
- Aucun "transcript live caché derrière l'audio" — l'audio TTS est joué EN PLUS du texte, pas À LA PLACE.

**Conclusion** : la phrase "transcript live mot-par-mot" du tracking est un mal-décrit du comportement existant. Sotto-voce = juste couper TTS, lire le texte normal. Pas besoin de composant `<LiveTranscript>` dédié.

## 1. Spec — format EARS

### 1.1 Hook `useSottoVoce`

- **R1** : The module SHALL export a hook `useSottoVoce()` from `museum-frontend/features/chat/application/useSottoVoce.ts`.
- **R2** : The hook SHALL expose the shape `{ enabled: boolean; isLoading: boolean; toggle: () => Promise<void> }`.
- **R3** : On first mount the hook SHALL initialise `enabled = false` and `isLoading = true`, then asynchronously read AsyncStorage key `'settings.sotto_voce_mode'` and set `enabled` to `true` only if the stored value equals the string `'true'`. `isLoading` SHALL flip to `false` after the read settles (resolve OR reject).
- **R4** : When `toggle()` is called the hook SHALL flip the internal `enabled` state synchronously (optimistic) AND persist the new value as the string `'true'` / `'false'` to AsyncStorage key `'settings.sotto_voce_mode'` via the existing `storage` façade (`shared/infrastructure/storage.ts`).
- **R5** : The hook SHALL NOT introduce any new dependency. It MUST reuse the `storage` façade.
- **R6** : When the storage read fails (rejection), the hook SHALL NOT throw — it SHALL fall back to `enabled = false` and set `isLoading = false`. The user can still toggle ; the write attempt is best-effort.
- **R7** : The hook SHALL guard against state updates after unmount (a `cancelled` flag in the initial-load effect, mirror of `useAudioDescriptionMode`).
- **R8** : The storage key constant `SOTTO_VOCE_STORAGE_KEY = 'settings.sotto_voce_mode'` SHALL be exported from the module for testability.

### 1.2 Composant `<SottoVoceToggle>`

- **R9** : The module SHALL export a React component `SottoVoceToggle` from `museum-frontend/features/chat/ui/SottoVoceToggle.tsx`.
- **R10** : The component SHALL accept the props `{ enabled: boolean; onToggle: () => void }`. NO other prop is consumed. The component is purely presentational (the hook lives at the screen level, the state is propagated through `<ChatHeader>`).
- **R11** : The component SHALL render a `<Pressable>` with `testID="sotto-voce-toggle"`.
- **R12** : The component SHALL render an `Ionicons` icon : `mic-off` (filled) when `enabled === true`, `mic-off-outline` when `enabled === false`. Size 20 (match other ChatHeader action buttons).
- **R13** : The component SHALL set `accessibilityRole="button"` and `accessibilityState={{ selected: enabled }}`. (Convention RN — `selected` for Pressable used as on/off bascule. `role="switch"` is reserved for `<Switch>`.)
- **R14** : The `accessibilityLabel` SHALL come from i18n key `'chat.sottoVoce.a11y_label'`. The `accessibilityHint` SHALL come from i18n key `'chat.sottoVoce.a11y_hint_on'` when `enabled === true`, otherwise `'chat.sottoVoce.a11y_hint_off'`.
- **R15** : Hit target SHALL be ≥ 36 × 36 dp (`space['9']` rounded container, parity with the other action buttons in `<ChatHeader>`).
- **R16** : The component SHALL NOT introduce any animation (state flip is instantaneous). `useReducedMotion()` is therefore not consulted — there is no motion to reduce.
- **R17** : The component SHALL emit `console.debug('[chat.sottoVoce]', { next: !enabled })` from inside `onToggle` (telemetry — same pattern as A3-A6). NO Sentry breadcrumb in V1 (Open Q3).

### 2.2 Hook — `useSottoVoce.ts`

```ts
import { useCallback, useEffect, useState } from 'react';
import { storage } from '@/shared/infrastructure/storage';

export const SOTTO_VOCE_STORAGE_KEY = 'settings.sotto_voce_mode';

/**
 * Manages the sotto-voce mode toggle (silent-room: text-only, no TTS) with
 * AsyncStorage persistence cross-session.
 *
 * - `enabled === true` → screen MUST gate `useAutoTts.enabled` to `false`.
 * - Persistence : AsyncStorage key `settings.sotto_voce_mode` via shared
 *   `storage` façade (zero new dep).
 * - Read failure is non-fatal (defaults to `false`).
 */
export function useSottoVoce() {
  const [enabled, setEnabled] = useState(false);
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    let cancelled = false;
    void storage
      .getItem(SOTTO_VOCE_STORAGE_KEY)
      .then((value) => {
        if (!cancelled) {
          setEnabled(value === 'true');
          setIsLoading(false);
        }
      })
      .catch(() => {
        if (!cancelled) setIsLoading(false);
      });
    return () => {
      cancelled = true;
    };
  }, []);

  const toggle = useCallback(async () => {
    const next = !enabled;
    setEnabled(next);
    await storage.setItem(SOTTO_VOCE_STORAGE_KEY, String(next));
  }, [enabled]);

  return { enabled, isLoading, toggle };
}
```

## 5. Open questions

- **Q1 (Auto-detection ambient noise via micro)** : Tracking ligne 70 mentionne "auto-suggest si ambient >70dB". Spec V1 = **deferred** (cf. §0.3). Implémentation potentielle : `expo-audio` `prepareToRecordAsync({ metering: true })` + lecture périodique `audioLevel` (dBFS) + conversion approximative dB SPL via offset device-specific + debounce 5s + banner suggestion ("Activate sotto-voce ?"). - **Q2 (`AccessibilityInfo.announceForAccessibility()` au toggle)** : Spec V1 livre les i18n keys `announcement_on/off` mais NE les appelle PAS. Pour les utilisateurs VoiceOver, le `selected` state change déjà annoncé par VO. L'announcement explicite ajoute redondance mais peut être plus clair pour low-vision users. **Proposition** : V1.1 post-feedback a11y user.
- **Q3 (Sentry breadcrumb)** : Spec V1 = `console.debug` uniquement (cohérent A3-A6). Ajout breadcrumb Sentry = signal extra pour debugging support (e.g., "user toggled sotto-voce 3× / minute → UI confusion ?"). **Proposition** : V1.1 si support remonte des cas. Méta-question : sentry events sont déjà très bavards.
- **Q4 (Haptic feedback au toggle)** : `Haptics.selectionAsync()` (expo-haptics, déjà installé) — feedback tactile au toggle. Cohérent avec les autres boutons header ? Vérification : grep `Haptics` dans `<ChatHeader>` → 0 résultats.
