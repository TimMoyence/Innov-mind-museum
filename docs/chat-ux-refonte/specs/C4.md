## 1. Spec — format EARS

### 1.1 Ouverture / commande

- **R1** : When a call-site invokes `useBottomSheetRouter().open(route, params)`, the system SHALL mount the corresponding sheet component above the chat surface and trigger its entrance animation.
- **R2** : When a call-site invokes `open(route, params)` while another sheet is already open, the system SHALL replace the active sheet with the new one (last-write-wins, single-slot policy) unless the active route is marked `blocking: true` (cf. R6).
- **R3** : When the `BottomSheetRouter` mounts, the system SHALL NOT render any sheet until `open()` is called (no pre-render of inactive routes).

### 1.2 Empilement / collision

- **R4** : While a sheet of route `R_a` is open and a call-site invokes `open(R_b)`, the system SHALL log a `bottomSheetRouter.replace` debug event in dev (Sentry breadcrumb in prod) and proceed to replace.
- **R5** : While a sheet is opening (anim in-flight), and a call-site invokes `close()`, the system SHALL cancel the entrance and play the dismiss animation from the current position (no jank, no double-anim).
- **R6** : While the active route is `consent` or `daily-limit`, the system SHALL refuse `open()` calls for non-blocking routes (`context-menu`, `summary`, `browser`) — these are gated behind the blocking modal (consent must be accepted, daily-limit dismissed). Voice-intro is also blocking and SHALL win over consent if both are pending (consent is shown first chronologically per current flow; see Open Q1).

### 1.3 Fermeture

- **R7** : When the user taps the backdrop (semi-transparent overlay) on a non-blocking sheet, the system SHALL close the sheet and invoke the route's `onClose` callback.
- **R8** : When the user swipes the sheet down past the dismiss threshold (50% of the sheet height by default, or `enableSwipeDown=false` for blocking routes), the system SHALL close the sheet and invoke `onClose`.
- **R9** : When the user presses the Android hardware back button while a sheet is open, the system SHALL close the active sheet AND consume the event (return `true` from `BackHandler.addEventListener('hardwareBackPress', ...)`) so the Expo Router stack does NOT pop the screen.
- **R10** : When a call-site invokes `useBottomSheetRouter().close()`, the system SHALL dismiss the active sheet and invoke its `onClose` callback.
- **R11** : When the user attempts backdrop tap or swipe-down on a `blocking: true` route (consent, daily-limit, voice-intro), the system SHALL ignore the gesture (no close, no callback). Only the route's primary CTA (Accept / Dismiss / Acknowledge) closes a blocking sheet.

### 1.4 Transition entre sheets

- **R12** : When a sheet closes and a queued `open()` exists from the same render cycle (programmatic chain e.g. `consent` accepted → optional `voice-intro`), the system SHALL play the close animation first, then mount the next sheet and play its entrance animation (no overlap, no flicker).
- **R13** : While `useReducedMotion()` returns `true`, the system SHALL skip enter/exit animations and snap the sheet to its open/closed position instantaneously (still respecting R12 sequencing — sequential snap, not concurrent).

### 1.5 Accessibilité

- **R14** : When a sheet mounts, the system SHALL set `accessibilityViewIsModal={true}` on the sheet container so VoiceOver/TalkBack ignore content behind it.
- **R15** : When a sheet mounts, the system SHALL move screen-reader focus to the sheet's first focusable element (or to the sheet's `accessibilityLabel` header if no focusable child).
- **R16** : When a sheet closes, the system SHALL return screen-reader focus to the element that invoked `open()` (focus restore), best-effort via `AccessibilityInfo.setAccessibilityFocus`.
- **R17** : While a sheet is open, the system SHALL announce the sheet's title via `AccessibilityInfo.announceForAccessibility(<route.a11yAnnounce>)` on mount.
- **R18** : When a sheet is rendered, the system SHALL set `accessibilityRole="dialog"` (RN ≥0.72 supports it) on the sheet container.

### 1.6 Reduced motion

- **R19** : See R13. Applies to: enter slide, exit slide, swipe-down spring, sequential transition (R12).

### 1.7 Bilingual + theming

- **R20** : The system SHALL NOT introduce any user-facing string in `BottomSheetRouter` itself — all copy lives in the route's content component, already i18n'd via `useTranslation()`.
- **R21** : The system SHALL respect the existing `ThemeContext` (light/dark) — backdrop opacity, handle color, sheet background sourced from `theme.modalOverlay`, `theme.cardBackground`, `theme.cardBorder`. No new tokens.

## 2. Design

### 2.1 Décision archi : state machine maison (pas `@gorhom/bottom-sheet`)

**Verdict** : **state machine maison** + RN `<Modal>` natif + `Animated` (Reanimated 4) + `react-native-gesture-handler` pour swipe-down.

**Pourquoi pas `@gorhom/bottom-sheet`** :

| Critère | `@gorhom/bottom-sheet` v5 | State machine maison |
|---|---|---|
| Bundle weight | +~50 KB (deps `@gorhom/portal` + types) | 0 KB (RN built-in) |
| Setup root | **Requires `GestureHandlerRootView` at app root** — `museum-frontend/app/_layout.tsx` ne le wrap PAS aujourd'hui. Risque régression iOS gesture handlers globaux | Pas de root setup. Modal natif déjà rodé |
| Reanimated 4 compat | v5 compatible Reanimated 3+ ; 4.2.1 supportée mais nécessite peer alignment | Aucune dep |
| API surface | `BottomSheetModal` + `BottomSheetModalProvider` + `useBottomSheetModal` + portal — découplé du sheet content, mais plus de couches | Provider + hook + switch sur `activeRoute` — 1 fichier |
| 3 modals SONT déjà full-screen (consent, voice-intro, in-app-browser) | Bottom-sheet pousse vers la convergence visuelle "sheet" — pas adapté pour browser WebView | On garde la flexibilité : route déclare son `presentation: 'sheet' \| 'fullscreen'`, le router applique le shape |
| EU AI Act blocking modal | Faisable mais nécessite `enableContentPanningGesture={false}` + `enablePanDownToClose={false}` + custom backdrop component | Route-level `blocking: true` flag, géré par le router |
| Tests existants | Nécessitent mocks `@gorhom/bottom-sheet/__mocks__/` (officiels existent) | Tests existants `__tests__/components/*.test.tsx` continuent de rendre les content components directement (sans le router), pas de mocks ajoutés |
| Maintenance | Dep tierce additionnelle dans un repo qui a déjà `expo-image-manipulator` deprecated (C5) — minimiser surface | Code 100% nous, ~150 LOC |
| Risque migration future | Si Musaium veut passer à `@gorhom/bottom-sheet` plus tard, la state machine maison expose la même API publique (`useBottomSheetRouter().open(...)`) — swap drop-in | N/A |

**Trade-off accepté** : on ne gagne pas la "snap-to-snap-points" naturelle de `@gorhom/bottom-sheet` (pas de demi-position pour le sheet). Aucune des 6 surfaces actuelles n'en a besoin (consent / voice-intro / browser = full ; context-menu = single position bottom ; summary / daily-limit = centered card). Si une feature future (A1 unified composer media sheet ?) demande des snap points, on re-évaluera la migration vers `@gorhom/bottom-sheet`. Surfacé en Open Q3.

### 2.3 API publique (signatures TypeScript)

```ts
// routes.ts
export type BottomSheetRouteId =
  | 'browser'
  | 'context-menu'
  | 'consent'
  | 'summary'
  | 'daily-limit'
  | 'voice-intro'
  | 'ai-disclosure';

export interface BottomSheetRouteParams {
  browser: { url: string };
  'context-menu': { message: ChatUiMessage };
  consent: Record<string, never>;
  summary: { summary: VisitSummary };
  'daily-limit': Record<string, never>;
  'voice-intro': { locale: string };
  'ai-disclosure': Record<string, never>;
}

export interface BottomSheetRouteDefinition<K extends BottomSheetRouteId> {
  readonly id: K;
  /**
   * `sheet` = slide-up from bottom, backdrop, swipe-down to dismiss.
   * `fullscreen` = slide-up taking full screen (WebView, consent, voice-intro).
   * `card` = centered card (summary, daily-limit).
   */
  readonly presentation: 'sheet' | 'fullscreen' | 'card';
  /**
   * When `true`: backdrop tap, swipe-down, and Android back DO NOT close.
   * Only the primary CTA inside the content can close.
   */
  readonly blocking: boolean;
  /** i18n key for `AccessibilityInfo.announceForAccessibility` on mount. */
  readonly a11yAnnounceKey: string;
  /** Function rendering the route content. Receives params + close handle. */
  readonly Content: React.FC<BottomSheetRouteParams[K] & { close: () => void }>;
}

// useBottomSheetRouter.ts
export interface BottomSheetRouter {
  readonly activeRoute: BottomSheetRouteId | null;
  open<K extends BottomSheetRouteId>(route: K, params: BottomSheetRouteParams[K]): void;
  close(): void;
}

export function useBottomSheetRouter(): BottomSheetRouter;

// BottomSheetRouter.tsx
export const BottomSheetRouter: React.FC<{
  /** Optional callbacks bridged from screen state (e.g. `recheckConsent` after privacy nav). */
  onRouteClose?: (route: BottomSheetRouteId) => void;
}>;
```

```ts
export type BottomSheetState =

```

### 2.4 Migration plan (ordre de bascule)

**Décision : big bang** — toute la suite des 7 surfaces migre dans le même chantier C4, dans l'ordre des sous-tâches T1.x → T7.x. Raisons :

1. `ChatSessionModals.tsx` est un seul composant — le couper en deux pour garder 3 modals legacy + 3 routés = 2× la complexité et un état hybride dans `chat/[sessionId].tsx`.
2. La doctrine `feedback_no_feature_flags_prelaunch` interdit un flag `BOTTOM_SHEET_ROUTER_ENABLED`.
3. Le screen `chat/[sessionId].tsx` est testé end-to-end par `chat-session-deep.test.tsx` — si on casse l'intégration, le test rouge le signale dès la première itération.

**Ordre interne des tâches** : foundations (T1) → un sheet "simple non-blocking" pour valider (T2, context-menu) → blocking critique (T3, consent + voice-intro) → reste (T4-T7) → cleanup (T8). Si une tâche échoue, on rollback ce commit-là, pas tout le chantier.

### 2.5 A11y détaillée

- `accessibilityRole="dialog"` sur le container du content (R18).
- `accessibilityViewIsModal={true}` sur le content (R14) — déjà fait sur `MuseumSheet.tsx`, pattern repris.
- Sur mount du sheet :
  - `AccessibilityInfo.announceForAccessibility(t(route.a11yAnnounceKey))` (R17).
  - Capture la `currentlyFocusedTag` via `findNodeHandle` avant mount pour focus restore.
- Sur unmount :
  - `AccessibilityInfo.setAccessibilityFocus(savedTag)` (R16) — best-effort, on log warn si échec mais on ne crash pas.
- Focus initial dans le sheet : laisser RN gérer naturellement le tab order. Si un sheet n'a pas de focusable visible, le screen reader lit le `accessibilityLabel` du container. Suffisant pour V1.

### 2.6 Performance

- `BottomSheetRouter` est rendu **une seule fois** au bottom du tree (`<LiquidScreen>` end). Quand `activeRoute === null`, le router rend `null` — pas de modal mounté, pas de re-render parent.
- L'état machine vit dans un `Context` dédié — providers consommateurs (les call-sites qui appellent `open()`) ne re-renderent QUE quand le hook leur retourne un nouvel objet. On stabilise via `useMemo` sur `{ open, close, activeRoute }`.
- Animations via `useNativeDriver: true` (transform + opacity uniquement, pattern déjà standard dans `StreamingBody.tsx`). Reanimated 4 disponible mais Animated suffit ici — pas besoin d'introduire Reanimated worklets pour 7 sheets simples.
- Backdrop opacity animée en parallèle de la transform — opacity-only ≤16ms / frame.

### 2.7 Compatibilité dark mode

Tokens existants : `theme.modalOverlay`, `theme.cardBackground`, `theme.cardBorder`, `theme.separator`, `theme.textPrimary`, etc. Pas de nouveaux tokens. La `MessageContextMenu.tsx` actuelle utilise déjà `theme.glassBackground` pour dark — on garde la même logique dans le sheet content.

## 7. Open questions

- **Q1** : Ordre des blocking modals si plusieurs sont pending simultanément. Avec single-slot policy (R2), le router doit choisir. Proposition : `consent` > `daily-limit` > `voice-intro` > `ai-disclosure` > non-blocking.
