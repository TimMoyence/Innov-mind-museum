## 0. Contexte vérifié

### 0.1 Trouvaille déclenchante (findings.md P5)

`docs/chat-ux-refonte/findings.md:169` — **« Cache LLM bypass total dès image présente — `chat-message.service.ts:319` — LLM cost + latence répétée — C3 »**.

Le bypass est implémenté à deux sites :

- `museum-backend/src/modules/chat/useCase/message/chat-message.service.ts:330` (lookup) :
  ```ts
  if (!llmCache || ctx.input.image || ctx.orchestratorInput.image) return null;
  ```

- `museum-backend/src/modules/chat/useCase/message/chat-message.service.ts:349` (store) — même garde miroir.

Conséquence : un visiteur qui scanne **deux fois la même œuvre** (e.g. devant *La Liseuse*, hors de portée Wi-Fi flake, ré-essai 30s plus tard) repaye **100 % du LLM call** (~2-4s + ~3-8 ¢ selon prompt), alors que la réponse précédente est identique à un epsilon stable près (mêmes locale, mêmes prefs, même œuvre détectée).

### 0.2 Cache LLM existant (ADR-036, post-PR-B 2026-05-08)

**Source de vérité unique** : `museum-backend/src/modules/chat/useCase/llm/llm-cache.service.ts`. Single-layer L1, exposed à travers le port `LlmCacheService` (`museum-backend/src/modules/chat/useCase/llm/llm-cache.types.ts`).

- **Key shape actuelle** : `llm:v1:{contextClass}:{museumId|none}:{userId|anon}:{sha256OfCanonicalInput}` (32 chars hex).
- **Canonical input fields** : `{model, systemSection, locale, museumName, userPreferencesHash, prompt}` sorted-keys JSON.
- **Context class** : `personalized` si `userPreferencesHash` set, `museum-mode` si `museumContext.museumId` set, sinon `generic`.
- **TTL** : 7 j (generic) / 1 j (museum-mode) / 1 h (personalized).
- **Bypass** owned by caller (`chat-message.service.ts`) — le port lui-même calcule toujours la clé.
- **Kill-switch** : `LLM_CACHE_ENABLED=false` désactive lookup + store dans le caller.
- **Fail-open** : exceptions cache → log warn `llm_cache_lookup_failed` / `llm_cache_store_failed`, request continue.

L'ADR-036 dit explicitement (lignes 75-76) : **« Semantic similarity matching: deferred to G Phase 2. This ADR retains the deferral. The current cache is exact-match only on the canonical input hash. »**. C3 **n'introduit PAS** de matching sémantique — il introduit une **clé déterministe additionnelle** pour le cas image-présente.

### 0.3 Visual signature aujourd'hui

`museum-backend/src/modules/chat/useCase/visual-similarity/compare.use-case.ts:171` calcule déjà :
```ts
const queryEmbeddingHash = createHash('sha256').update(input.buffer).digest('hex');
```

`museum-backend/src/modules/chat/useCase/visual-similarity/similarity.service.ts:215` calcule le même `sha256(buffer)` côté kNN cache (encoder-output cache). Pattern stable, hex 64 chars.

**Mais** : ni l'un ni l'autre ne **publie** le hash au-delà de son audit log (`compare_request` Pino event) ou de son cache interne. Le pipeline `chat-message.service.ts` ne **reçoit pas** ce hash — il consomme `ProcessedImage` (`museum-backend/src/modules/chat/useCase/image/image-processing.service.ts`) qui ne contient que `{imageRef, orchestratorImage}`.

**Décision C3** : étendre `ProcessedImage` avec un champ `imageContentHash: string` (32 hex chars, SHA-256 du buffer post-EXIF-strip, slice(0, 32) pour aligner avec le hash use-case-level `sha256OfCanonicalInput`). Calculé une seule fois dans `image-processing.service.ts.processImage()` à l'endroit où on a déjà le buffer post-strip. **Nom interne `imageContentHash`** (pas `siglip…` — voir Q4) pour ne pas suggérer un embedding alors que c'est un hash de bytes : la dérivation reste **content-hash**, pas vector-hash. La tracking évoque `artworkSigLIPHash` pour des raisons historiques (initial naming) ; le présent spec et les tests utilisent `imageContentHash` (plus honnête, UFR-013).

### 0.4 Prefs hash existant

`museum-backend/src/modules/chat/useCase/message/chat-message.service.ts:471` calcule déjà :
```ts
userPreferencesHash: prep.userMemoryBlock ? hashString16(prep.userMemoryBlock) : undefined
```

Le `userMemoryBlock` (`prepare-message.pipeline.ts`) est le snapshot stable des prefs utilisateur (expertise, audioDescEnabled, langue préférée, sotto-voce, etc.) consolidées en une string déterministe. C3 **réutilise** ce hash — pas de nouveau prefs hash, pas de duplication. C'est le `prefsHash` du tracking.

### 0.5 Bypass actuel et conditions

`chat-message.service.ts:330` ne bypasse PAS uniquement sur image — il bypasse aussi (implicitement) quand `buildLlmCacheInput` retourne `null` (cas `sanitizedText` vide, image-only). C3 ne change PAS le comportement quand prompt vide : **C3 active le cache pour image présente + prompt présent**. Image-only (no text) reste hors cache car non-déterministe (le LLM voit l'image et improvise — pas de canonical à hasher).

Les autres bypass implicites (PII flag, history>0, oversize text…) sont **owned par L2 supprimé en PR-B** — ils ne sont pas réintroduits en L1, donc pas réintroduits en C3 (R12 ci-dessous).

## 1. Spec — format EARS

### 1.1 Image content hash exposure

- **R1** : When `ImageProcessingService.processImage(image, sessionId, ownerId)` returns successfully, the system SHALL include a field `imageContentHash: string` in the resulting `ProcessedImage` envelope. The hash MUST equal `sha256(strippedBuffer).digest('hex').slice(0, 32)` where `strippedBuffer` is the post-EXIF-strip image buffer that already exists on the success path.
- **R2** : When the input `image.source === 'url'`, the system SHALL skip hash computation (no buffer to hash) and SHALL NOT include `imageContentHash` in the returned `ProcessedImage` (or include it with value `undefined`). C3 cache key derivation MUST treat this case as "no visual signature available" and SHALL fall back to the current bypass (no cache).
- **R3** : When the same buffer is submitted twice in two separate requests (identical bytes), the system SHALL produce **identical** `imageContentHash` values. Stable across processes, identical to bytewise SHA-256.
- **R4** : When two different buffers are submitted, the system SHALL produce different `imageContentHash` values with cryptographic collision resistance (SHA-256 32-char prefix collision probability ≈ 2⁻¹²⁸ — negligible).
- **R5** : When the EXIF-strip step changes the buffer bytes (e.g. metadata removed), the system SHALL hash the **post-strip** buffer (not the user-submitted bytes), so two uploads of the same photo with different EXIF (e.g. different cameras' timezone metadata) produce the SAME hash and hit the same cache entry.

### 1.2 Cache key extension

- **R6** : When `chat-message.service.ts.buildLlmCacheInput(prep, sanitizedText)` is called AND `prep.processedImage?.imageContentHash` is set AND `sanitizedText` is non-empty, the system SHALL include `imageContentHash` in the canonical input that feeds `sha256OfCanonicalInput`. The new canonical field is named `imageContentHash` and is `null` when absent.
- **R7** : When `imageContentHash` is added to the canonical input, the system SHALL NOT change the key prefix or version (`llm:v1:` remains). Two entries with different `imageContentHash` produce different keys (different sha256 suffix). Two entries with identical `imageContentHash` AND identical other canonical fields produce the SAME key.
- **R8** : When `LlmCacheKeyInput.imageContentHash` is absent (no image OR url-source image — R2), the system SHALL behave **identically to today** : key derivation does NOT include any visual signature, no regression on text-only paths.
- **R9** : When the cache key includes `imageContentHash`, the system SHALL classify the entry into a `LlmContextClass` using the existing `classify()` rules (`userPreferencesHash` → personalized, `museumContext.museumId` → museum-mode, else generic). Image presence does NOT introduce a new class.
- **R10** : The system SHALL extend `LlmCacheKeyInput` (`museum-backend/src/modules/chat/useCase/llm/llm-cache.types.ts`) with an OPTIONAL field `imageContentHash?: string`. The field is included in `sha256OfCanonicalInput` when present (R6).

### 1.3 Bypass lift

- **R11** : When `chat-message.service.ts.tryLlmCacheLookup(ctx)` runs AND `ctx.input.image` is present AND `ctx.prep.processedImage?.imageContentHash` is set AND `buildLlmCacheInput(...)` returns a non-null `cacheInput` (i.e. text present), the system SHALL perform the cache lookup (NOT bypass). The current bypass `if (!llmCache || ctx.input.image || ctx.orchestratorInput.image) return null;` MUST be modified so that image presence alone is NO LONGER a bypass condition when the visual hash is available.
- **R12** : When `chat-message.service.ts.tryLlmCacheLookup(ctx)` runs AND `ctx.input.image` is present AND `ctx.prep.processedImage?.imageContentHash` is **undefined** (url-source per R2, or processing failed), the system SHALL preserve the current bypass behavior (return null, no lookup, no store). Same for `tryLlmCacheStore`.
- **R13** : When `chat-message.service.ts.tryLlmCacheStore(ctx, aiResult)` runs, the system SHALL apply the **same** activation condition as `tryLlmCacheLookup` (R11+R12) — if lookup ran, store runs ; if lookup bypassed, store bypasses. Symmetry is invariant (preserves miss→store→hit replay).
- **R14** : When `LLM_CACHE_ENABLED=false` env flag is set, the system SHALL preserve the existing kill-switch behavior — no lookup, no store, regardless of image presence. C3 does NOT introduce a separate kill-switch.

### 1.4 TTL

- **R15** : When `LlmCacheServiceImpl.store(input, value)` runs with an `input.imageContentHash` set AND `input.museumContext.museumId` set, the system SHALL apply the existing `museum-mode` TTL constant (`TTL_MUSEUM_MODE_S = 24 * 60 * 60 = 86400` seconds = 24 h). This aligns with the tracking note "TTL 24h" and is **already** the museum-mode TTL — no new constant introduced.
- **R16** : When `LlmCacheServiceImpl.store(input, value)` runs with an `input.imageContentHash` set AND `input.userPreferencesHash` set, the system SHALL apply the existing `personalized` TTL (`TTL_PERSONALIZED_S = 3600` seconds = 1 h). Personalized class wins over museum-mode in `classify()`.
- **R17** : When `LlmCacheServiceImpl.store(input, value)` runs with an `input.imageContentHash` set AND no museum/prefs context (e.g. anon user no museum), the system SHALL apply the existing `generic` TTL (7 days). This case is rare in production (most image scans are in-museum) but possible and must not crash.
- **R18** : The system SHALL NOT introduce a new TTL constant for image-cache entries. Tuning is gated by ADR-036 R11 protocol (≥7-day bake + Grafana screenshot + ADR amendment) — see §6 NFR and §X Open Questions.

### 1.5 Invalidation

- **R19** : When `LlmCacheServiceImpl.invalidateMuseum(museumId)` is called (admin museum update hook, existing path), the system SHALL drop all entries scoped to that museum, including image-bearing entries (because `museumId` is in the key prefix BEFORE the sha256 suffix — already covers the new entries with no code change).
- **R20** : The system SHALL NOT introduce a per-image invalidation API (no `invalidateImage(imageContentHash)`). Bake observation will reveal if needed (Open Q3).

### 1.6 Privacy / security

- **R21** : When `imageContentHash` is logged (existing `llm_cache_hit` / `llm_cache_miss` log events in `chat-message.service.ts:335,353`), the system SHALL log the hash hex prefix only (already truncated to 32 chars in R1) and SHALL NOT log raw image bytes, base64, or storage refs. Truncated hex is not PII (one-way SHA-256).
- **R22** : When `imageContentHash` is included in the cache key, the system SHALL NOT include `userId` in the visual-signature derivation (the existing key shape already places userId in a separate segment — `{userId|anon}` before the hash — so two users scanning the same artwork still get different keys today; C3 preserves this). For `museum-mode` class without prefs, `userId='anon'` collapses cross-user sharing (acceptable: same artwork + same museum + same locale + no prefs ≈ same answer — UFR-013 honesty: no user-specific content leaks because no user-specific input is in the canonical).
- **R23** : When `imageContentHash` is computed, the system SHALL NOT persist the raw buffer beyond the existing `imageStorage.save()` lifecycle. C3 introduces NO new storage of image bytes.

### 1.7 Observability

- **R24** : When a cache hit occurs on an image-bearing request (R11 path), the system SHALL log `llm_cache_hit` with the existing structured fields `{contextClass, userId, sessionId, requestId}` AND a new field `hasImage: true`. Cache miss logs SHALL also include `hasImage: true` for symmetry.
- **R25** : The existing Prom counters `llm_cache_hits_total{context_class}` and `llm_cache_misses_total{context_class}` SHALL be incremented as today. C3 does NOT add a per-image label (would explode cardinality — single sha256 prefix is high-entropy). Observability of image-cache effectiveness comes from log-based dashboarding (Loki query `hasImage=true` filter), NOT from Prom labels.
- **R26** : When a cache hit returns a previously-stored `OrchestratorOutput`, the system SHALL forward it to `commitResponse` unchanged. No re-running of guardrails on the cached payload (the cached payload already cleared output guardrail at store time).

## 2. Design

### 2.1 Cache key shape — exacte

**Current** (ADR-036, post-PR-B authoritative — `LlmCacheServiceImpl.buildKey`) :
```
llm:v1:{contextClass}:{museumId|none}:{userId|anon}:{sha256(canonical)32hex}
```
where `canonical = JSON.stringify({model, systemSection, locale, museumName, userPreferencesHash, prompt}, sortedKeys)`.

**C3 extension** : `canonical` gains an OPTIONAL field `imageContentHash`. Sorted-keys order recomputed (alphabetical), so existing keys (without `imageContentHash`) are NOT invalidated — the JSON serialization simply gains a new key when present. Existing entries' keys remain hashable to the same value because `JSON.stringify({…, imageContentHash: null}, sortedKeys)` ≠ `JSON.stringify({…}, sortedKeys)` (the `null` value DOES appear in the serialized form).

**Key collision risk** : 32 hex chars = 128 bits of entropy. Collision probability across 10⁹ entries ≈ 10⁻²⁰. Negligible.

**Decision** : `imageContentHash` is **always present** in the canonical when the `LlmCacheKeyInput.imageContentHash` field is set (R10). When absent, it is **omitted** from the canonical (NOT serialized as `null`), so legacy text-only keys remain bit-identical to today's keys. This is critical for the bake/migration path: PR ships without invalidating the existing 7-day generic cache or 24h museum-mode cache. Implementation pattern :

```ts
const canonical: Record<string, unknown> = {
  model, systemSection, locale,
  museumName: museumContext?.museumName ?? null,
  userPreferencesHash: userPreferencesHash ?? null,
  prompt,
};
if (imageContentHash !== undefined) {
  canonical.imageContentHash = imageContentHash;
}
```

### 2.2 Comment obtenir `imageContentHash` (data flow)

```
PostMessageInput.image (upload base64 or legacy)
   │
   ▼  (chat-message.service.postMessage → pipeline.prepare → imageProcessor.processImage)
ImageProcessingService.processImage
   │  (post-EXIF-strip buffer is already computed for size assertion + storage save)
   │  NEW: imageContentHash = sha256(strippedBuffer).slice(0, 32)
   ▼
ProcessedImage { imageRef, orchestratorImage, imageContentHash?: string }
   │  (returned by pipeline.prepare into PrepareReady)
   ▼
PrepareReady.processedImage  (new field — pipeline result extension)
   │
   ▼  (chat-message.service.buildLlmCacheInput consumes prep.processedImage)
LlmCacheKeyInput { …, imageContentHash: prep.processedImage?.imageContentHash }
   │
   ▼
LlmCacheServiceImpl.lookup / .store
```

**Why hash post-strip, not pre-strip** : per R5, two photos with different EXIF (timezone, camera serial) but identical pixels should hit the same cache. EXIF strip is deterministic (same input → same output), so post-strip bytes are a stable visual signature. This matches the pattern already used by `compare.use-case.ts:171` (which hashes the buffer AFTER `imageProcessor.process` runs).

**Reuse of existing buffer** : `stripExif()` returns `{base64, mimeType, sizeBytes}`. We need the **buffer** to hash. Modify `stripExif()` private signature to also return the buffer (no extra allocation — the buffer is already constructed internally at `image-processing.service.ts:163` and at the no-processor branch line 160-161). Hash is computed at the return site.

### 2.3 `ProcessedImage` shape extension

**Current** (`museum-backend/src/modules/chat/domain/processed-image.ts` or inline in `image-processing.service.ts`) :
```ts
export interface ProcessedImage {
  imageRef: string;
  orchestratorImage: PostMessageInput['image'];
}
```

**Extended** :
```ts
export interface ProcessedImage {
  imageRef: string;
  orchestratorImage: PostMessageInput['image'];
  /**
   * C3 — Visual content signature (SHA-256 of the post-EXIF-strip buffer,
   * 32-char hex prefix). Present for `upload` and legacy-base64 sources ;
   * absent for `url` source (no buffer to hash). Feeds the LLM cache key
   * derivation in chat-message.service.buildLlmCacheInput.
   */
  imageContentHash?: string;
}
```

`PrepareReady` (`prepare-message.pipeline.ts`) gains a corresponding optional field `processedImage?: ProcessedImage` if it doesn't already carry the full envelope. Inspect: `prep.processedImage` may already exist via `processInputImage` step. Verify in green phase ; if absent, plumb through.

### 2.4 `LlmCacheKeyInput` extension

```ts
// llm-cache.types.ts
export interface LlmCacheKeyInput {
  // …existing fields…
  /**
   * C3 — SHA-256 hex prefix (32 chars) of the post-EXIF-strip image buffer.
   * Present iff the request carries an upload-source or legacy-base64 image.
   * When present, the cache key derivation includes it in the canonical input
   * hash, lifting the today's image-bypass and enabling repeat-scan hits.
   * When absent (text-only or url-source), key derivation is byte-identical
   * to today (R8 — no regression on legacy entries).
   */
  imageContentHash?: string;
}
```

### 2.5 `buildLlmCacheInput` patch

```ts
private buildLlmCacheInput(prep: PrepareReady, sanitizedText: string): LlmCacheKeyInput | null {
  if (!sanitizedText) return null;
  return {
    model: env.llm.model,
    userId: prep.ownerId ?? 'anon',
    systemSection: 'chat-default',
    locale: prep.requestedLocale ?? 'en',
    museumContext: { museumId: prep.session.museumId ?? null, museumName: prep.session.museumName ?? null },
    userPreferencesHash: prep.userMemoryBlock ? hashString16(prep.userMemoryBlock) : undefined,
    prompt: sanitizedText,
    // C3 — add visual signature when available (upload / legacy-base64 paths)
    ...(prep.processedImage?.imageContentHash !== undefined
      ? { imageContentHash: prep.processedImage.imageContentHash }
      : {}),
  };
}
```

### 2.6 Bypass patch

```ts
// Before (line 330) — bypass on any image
if (!llmCache || ctx.input.image || ctx.orchestratorInput.image) return null;

// After — bypass only when visual signature unavailable
if (!llmCache) return null;
const hasImage = Boolean(ctx.input.image || ctx.orchestratorInput.image);
const hasVisualSignature = Boolean(ctx.prep.processedImage?.imageContentHash);
if (hasImage && !hasVisualSignature) return null;  // url-source or processing failed
// fall through to lookup (with cache input that includes imageContentHash when set)
```

Mirror in `tryLlmCacheStore` (R13 symmetry).

### 2.8 ADR-036 alignment

| ADR-036 lock | C3 stance |
|---|---|
| Single-layer L1 only, no L2 adapter decorator | **PRESERVED**. Zero adapter-level change. |
| Key prefix `llm:v1:` | **PRESERVED**. No version bump. |

## 4. Acceptance criteria (machine-verifiable)

- **AC1** : `ImageProcessingService.processImage` with upload-source image returns `imageContentHash` matching `sha256(strippedBuffer).slice(0, 32)`. Asserted via spec test with known buffer fixture.
- **AC2** : `ImageProcessingService.processImage` with url-source image returns NO `imageContentHash` (field `undefined`).
- **AC3** : Same buffer submitted twice → identical `imageContentHash`. Asserted via two `.processImage()` calls with same upload payload.
- **AC4** : Different buffers (1 byte diff) → different `imageContentHash`.
- **AC5** : `LlmCacheServiceImpl.lookup({…, imageContentHash: 'abc…'})` and `.lookup({…})` (without imageContentHash) on otherwise-identical inputs produce DIFFERENT cache keys (asserted via `cache.get.mock.calls[i][0]`).
- **AC6** : `LlmCacheServiceImpl.lookup({…})` (text-only, no image) at baseline AND after C3 produces the SAME cache key (R8 non-regression — existing `llm-cache.service.test.ts` `lookup returns hit=true and value on cache hit` continues to pass byte-identically).

## 6. NFR

### Performance

- SHA-256 compute on a 2 MB buffer: ~5-10 ms on Node 22 native crypto. Single-pass on the post-strip buffer (already in memory). **Latency impact: <10 ms added to image processing**. Acceptable vs. the 1-3s saved on a cache hit (200-500x).
- Cache hit on image-bearing request: ~5-15 ms (Redis GET + JSON parse + commit response). vs. ~2-4s LLM call (cold path). **Speedup: ~200-400x** on hit.
- Hit-rate expectation post-bake: ≥3% on museum-mode class (rough estimate based on repeat-scan UX in B2 conversation resumption window). Validated by ≥7-day bake per ADR-036 R11.

### Security

- SHA-256 of image buffer is **one-way** : no privacy leak from logging the 32-char hex prefix.
- No new storage of raw image bytes.
- `userId` already segmented in the key prefix BEFORE the canonical hash — no cross-user leak (R22).
- Museum tenant isolation preserved via `museumId` key prefix (R19, R22).
- Kill-switch `LLM_CACHE_ENABLED=false` continues to disable the new path (R14).

### Bake (ADR-036 R11)

- **≥7 consecutive days** of measured hit-rate per `context_class` post-deploy required BEFORE any TTL tuning PR.
- Hit-rate observation via existing Prom counters `llm_cache_hits_total{context_class}` / `llm_cache_misses_total{context_class}` + new Loki filter `hasImage=true` (R24).
- If museum-mode hit-rate <10% post-bake, the value proposition of C3 is weak ; revisit Open Q3 (per-image invalidation) or accept the floor.

### Honesty (UFR-013)

- Logs distinguish text-only cache hit (`hasImage: false`) from image-bearing cache hit (`hasImage: true`). No conflation.
- The `imageContentHash` is a **content hash**, NOT a SigLIP embedding hash. The spec, code comments, and field name reflect this honesty (despite the tracking note using `artworkSigLIPHash` historically).
- Two visually-similar but byte-different images of the same artwork produce DIFFERENT hashes and DIFFERENT cache entries. Spec calls this out (Out-of-scope §5 + Open Q2). No false claim of "semantic" matching.

### Test discipline

- New test file MUST use shared factories where applicable. For `LlmCacheKeyInput` and `ProcessedImage`, inline builders are acceptable (small leaf types, no shared factory exists today — see existing `llm-cache.service.test.ts` pattern).

## X. Open Questions

### Q1 — ADR-036 amendment required for this PR?

**Stake**: ADR-036 Cache contract table (post-PR-B authoritative) describes the canonical input fields as `{model, systemSection, locale, museumName, userPreferencesHash, prompt}`. C3 adds `imageContentHash` to that set. Is this a **strategy change** requiring an ADR amendment, or a **key-derivation extension** that does not?

**Proposed answer (R10/R11 §2.8)**: NOT required for the C3 ship PR. The key prefix (`llm:v1:`) and TTL constants are unchanged ; the change is additive (legacy keys remain bit-identical). ADR amendment IS required if and when TTL is data-tuned post-bake (per ADR-036 R11+R13).

**However**, the dispatcher may decide to land a **documentation-only ADR-036 amendment** alongside C3 to record the canonical-input expansion, for traceability. Decision deferred to dispatcher.

### Q2 — Semantic matching for "visually close" images?

**Stake**: Two photos of the same artwork from slightly different angles (or with different EXIF jitter that survives strip — none today, but theoretical) produce different `imageContentHash` values and different cache entries. The user experience of a repeat scan is therefore only sped up when the photo is **byte-identical** to a prior one (rare — re-tap "send" on the same captured frame).

**Decision**: V1 ships content-hash only. If post-bake telemetry shows museum-mode hit-rate <3% specifically on image-bearing requests, escalate to a **separate ADR proposal** for SigLIP-embedding-based key derivation (deferred per ADR-036 "G Phase 2"). Estimated cost: encoder warm path + cosine threshold + cache key shape change (vector-quantization-or-LSH-style). Major design — out of C3 scope.

### Q3 — Per-image invalidation API?

**Stake**: An admin updating an artwork (room, label, attribution) currently invalidates ALL museum-scoped entries via `invalidateMuseum(museumId)`. With image-cache active, this drops repeat-scan hits for unrelated artworks in the same museum. Acceptable for V1 (museum updates are rare). If museum-update frequency rises post-launch, consider `invalidateByImageContentHash(hash)` API.

**Decision**: V1 ships without per-image invalidation. Re-evaluate post-bake.

### Q4 — Field naming: `artworkSigLIPHash` (tracking) vs `imageContentHash` (spec)?

**Stake**: The tracking line 72 names the field `artworkSigLIPHash`, suggesting a SigLIP embedding hash. The actual implementation is a SHA-256 of post-EXIF-strip bytes — NOT a SigLIP vector hash.

**Decision**: Use `imageContentHash` in code and types (UFR-013 honesty). Update the tracking line 72 to use the same name when the feature flips to `done`. The two are equivalent for the V1 cache purpose (both are stable per-image signatures), but the naming should reflect what the code actually does.

### Q5 — Should we pre-compute `imageContentHash` BEFORE EXIF strip (to also catch pre-strip duplicates)?

**Stake**: If a user takes two photos of the same artwork seconds apart, the post-strip buffers are identical (deterministic strip on identical pixels). If two users take photos from the same vantage point at different times, EXIF differs but pixels are identical → post-strip buffers are identical. Pre-strip hash would diverge. Post-strip is the right choice (R5).

**Decision**: Post-strip (locked in R5).

### Q6 — Buffer-vs-pixel-array hash?

**Stake**: The SHA-256 of the post-strip buffer hashes the **compressed** bytes (JPEG/PNG/WebP). If `imageProcessor.stripExif` transcodes (e.g. HEIC → JPEG), the buffer is JPEG bytes. Two scans of the same source HEIC photo from the same device produce identical post-transcode JPEG bytes (deterministic transcode if the same encoder is used) — assumption to verify in green phase. If non-deterministic transcode, the hash diverges and the cache hit-rate is lower than expected.

**Decision**: Verify deterministic transcode in green phase via integration test. If non-deterministic, fall back to **pixel-array hash** (decode + canonical pixel buffer + sha256) — costlier (~50-100 ms on 2 MP image) but stable. Documented as a known follow-up if observed.
