## 0. Contexte vérifié

### 0.1 État courant — chrome composer (le mal à soigner)

Aujourd'hui, le bas du `ChatSessionScreen` empile **trois bandes UI persistantes** :

```
<MediaAttachmentPanel />        # toujours visible : 3 boutons "Gallery / Lens / Audio" + audio preview card
<WalkSuggestionChips />          # mode balade uniquement (déjà gated)
<ChatInput />                    # TextInput + send button + image thumb
```

`MediaAttachmentPanel` (`museum-frontend/features/chat/ui/MediaAttachmentPanel.tsx`, 161 lignes) rend une row de 3 `Pressable` (Ionicons + label) + une `<GlassCard>` "audio preview" quand `recordedAudioUri` est non-null. Cette row est **toujours rendue** (pas de condition mount), mange ~56dp de hauteur même quand l'utilisateur n'a rien à attacher.

Findings.md ligne 173 (P9) acte le verdict : « `MediaAttachmentPanel` toujours visible (mange écran) — Chrome > œuvre — **A1** ». Référence pattern 2026 (findings.md §3.1) :

> 1. **Unified voice+text surface (ChatGPT nov 2025)** — fini le blue-orb plein écran qui isole. Voix vit *dans* le thread, contexte visuel toujours visible.

L'objectif A1 est de remplacer cette bande par un **composer minimaliste 1-ligne** où :

- Le texte est l'input principal (réutilise le `<ChatInput>` existant — pas réécrit).
- Un bouton **mic** vit dans la même barre (tap → toggle voice mode, gated EU AI Act Art.50 inchangé).
- Un bouton **+ (plus)** dans la même barre ouvre une **slide-up bottom sheet** (`attachment-picker`) qui héberge les actions secondaires : camera, gallery, et — quand audio recordé — la card preview / play / clear / record-stop.

### 0.2 Infrastructure C4 BottomSheetRouter — DOIT être réutilisée

C4 a livré (`67a49a28`) un `BottomSheetRouter` complet (state machine maison + RN `<Modal>` + `PanResponder` swipe + a11y focus restore). Il sert déjà 7 routes (`browser`, `context-menu`, `consent`, `summary`, `daily-limit`, `voice-intro`, `ai-disclosure`).

**Décision contraignante (doctrine reuse)** : A1 ajoute **une 8ème route** `attachment-picker` dans `museum-frontend/features/chat/ui/bottom-sheet-router/routes.ts`. Présentation `sheet` (slide-up, swipe-down to dismiss), `blocking: false`. Pas de nouveau lib (zéro nouvelle dep), pas de nouveau registry. Cohérent avec la spec C4 §7 Q6 qui anticipait explicitement « si une feature future (A1 unified composer media sheet) demande des snap points, on re-évaluera ». A1 confirme : **pas de snap points multi-position nécessaires** — single position bottom sheet suffit pour 3 actions (camera / gallery / inline audio preview).

Le routes registry actuel (`routes.ts:24-31`) :

```ts
export type BottomSheetRouteId =
  | 'browser'
  | 'context-menu'
  | 'consent'
  | 'summary'
  | 'daily-limit'
  | 'voice-intro'
  | 'ai-disclosure';
```

Devient :

```ts
export type BottomSheetRouteId =
  | 'browser'
  | 'context-menu'
  | 'consent'
  | 'summary'
  | 'daily-limit'
  | 'voice-intro'
  | 'ai-disclosure'
  | 'attachment-picker';
```

Avec entry correspondant dans `ROUTES` (cf. §2.2).

### 0.3 Mic button — réutilise EU AI Act gate existant

Le mic n'est pas une feature nouvelle dans A1. Le hook `useAudioRecorder` (existant, `application/useAudioRecorder.ts`) expose `isRecording`, `recordedAudioUri`, `toggleRecording`. Le screen wrappe déjà `toggleRecording` par la disclosure gate `useVoiceDisclosure` (`[sessionId].tsx:138-152`) :

```ts
const toggleRecording = useCallback(async () => {
  if (!voiceDisclosureAcknowledged) {
    setPendingVoiceAction(true); // déclenche le sheet voice-intro
    return;
  }
  await rawToggleRecording();
}, [voiceDisclosureAcknowledged, rawToggleRecording]);
```

A1 NE TOUCHE PAS à cette logique. Le bouton mic du composer appelle `toggleRecording` (la version wrappée) — identique au comportement actuel quand l'utilisateur tape le bouton `chat.audio` de `MediaAttachmentPanel`.

### 0.4 Mic behavior — décision Tier 1 (tap-to-toggle) ; long-press push-to-talk = Open Q1

L'audit demande de proposer 2 options pour le mic :

| Option | Effort | UX |
|---|---|---|
| **Tap court → toggle** (start/stop recording) | Aucun changement `useAudioRecorder` | Identique au pattern actuel (`chat.audio` button). Familier. |
| **Long-press → push-to-talk** (record while held, send on release) | Refactor `useAudioRecorder` pour exposer `start()` / `stop()` séparés + intégration `Gesture.LongPress()` ou `onPressIn/onPressOut` | Pattern voice-first 2026 (WhatsApp / Telegram / iMessage). Une seule action pour une seule pensée vocale. |

**Décision V1** : **tap-to-toggle** (option 1) — zéro refactor `useAudioRecorder`, gate EU AI Act préservée, comportement identique au bouton actuel. Le push-to-talk est tracé Open Q1 (post-launch).

Cette décision respecte la doctrine `feedback_quality_doctrine` (product-first, pas minimal-fix) en gardant le scope A1 sur le **layout** (chrome → composer minimaliste) et pas sur la mécanique voice elle-même (qui marche).

### 0.5 Comportement quand audio recordé

Aujourd'hui, quand `recordedAudioUri != null`, la `<GlassCard>` "audio preview" apparaît au-dessus de la row d'attach buttons dans `<MediaAttachmentPanel>` — toujours visible.

A1 migre cette card **dans la sheet `attachment-picker`**. Quand l'utilisateur a recordé un audio mais n'a pas encore envoyé :

- Dans le composer minimaliste : une **mini-pill audio** (~32dp) apparaît à côté du bouton mic dans la barre. Tap → ouvre l'attachment-picker pour preview/play/clear. Cohérent avec ChatInput.image-thumb (déjà inline).
- Alternative considérée mais rejetée : forcer l'ouverture du sheet à `recordedAudioUri` change. Trop intrusif. La mini-pill suffit comme signal visuel.

Le `<Composer>` est UN nouveau composant qui **absorbe `<ChatInput>` existant** comme building block interne. Le `<ChatInput>` actuel reste utilisable seul (non-régression) ; le `<Composer>` l'enveloppe avec :

- Un bouton `+` à gauche (`Ionicons name="add-circle-outline"`) qui appelle `bottomSheetRouter.open('attachment-picker', {...})`.
- Un bouton mic à droite (avant le send) (`Ionicons name="mic-outline"` ou `stop-circle-outline` si `isRecording`).
- Une mini-pill audio si `recordedAudioUri != null`, tappable, ouvre aussi l'attachment-picker.

Le bouton **send** existant du `<ChatInput>` reste inchangé.

## 1. Spec — format EARS

### 1.1 Composant `<Composer>` — barre minimaliste

- **R1** : When `<Composer>` is rendered, the system SHALL render exactly one row containing (left-to-right LTR / right-to-left RTL respected) : a `+` button, the existing `<ChatInput>` building block (text input + send button), an optional audio mini-pill (when `recordedAudioUri != null`), and a mic button.
- **R2** : When `<Composer>` is rendered without `recordedAudioUri`, the audio mini-pill SHALL NOT be present in the layout (no DOM, no layout space taken).
- **R3** : When `<Composer>` is rendered with `recordedAudioUri` non-null, an audio mini-pill SHALL appear in the row with `accessibilityRole="button"`, `accessibilityLabel = t('chat.composer.a11y.audio_pill')`, and tapping it SHALL invoke `onOpenAttachments()` (same handler as the `+` button — surfaces the picker).
- **R4** : The `+` button SHALL be a `Pressable` with `accessibilityRole="button"`, `accessibilityLabel = t('chat.composer.a11y.open_attachments')`, `accessibilityHint = t('chat.composer.a11y.open_attachments_hint')`, `testID="composer-attach-button"`. On press, the system SHALL invoke `onOpenAttachments()`.
- **R5** : The mic button SHALL be a `Pressable` with `accessibilityRole="button"`, `testID="composer-mic-button"`. The `accessibilityLabel` SHALL be `t('chat.composer.a11y.mic')` when `isRecording === false` and `t('chat.composer.a11y.mic_recording')` when `isRecording === true`. The icon SHALL be `Ionicons name="mic-outline"` when not recording, `Ionicons name="stop-circle-outline"` when recording. On press, the system SHALL invoke `toggleRecording()`.
- **R6** : The `<Composer>` SHALL NEVER use Unicode emoji (doctrine `feedback_no_unicode_emoji`). Ionicons + tokens only.
- **R7** : The `<Composer>` SHALL forward `text`, `onChangeText`, `onSend`, `isSending`, `disabled`, `imageUri`, `onClearImage` to the embedded `<ChatInput>` unchanged. The contract surface of `<ChatInput>` is NOT broken.

### 1.2 Route `attachment-picker` — registry C4

- **R8** : The system SHALL extend `BottomSheetRouteId` union with the value `'attachment-picker'`.
- **R9** : The system SHALL register `'attachment-picker'` in `ROUTES` with `presentation: 'sheet'`, `blocking: false`, `a11yAnnounceKey: 'a11y.attachmentPicker.opened'`, `Content: AttachmentPickerSheetContent`.
- **R10** : The route SHALL define typed params via `BottomSheetRouteParams['attachment-picker']` with shape `{ recordedAudioUri: string | null; isPlayingAudio: boolean; isRecording: boolean; onPickImage: () => void; onTakePicture: () => void; toggleRecording: () => Promise<void> | void; playRecordedAudio: () => Promise<void> | void; clearMedia: () => void; }`.

### 1.3 Composant `<AttachmentPickerSheetContent>` — sheet content

- **R11** : The system SHALL ship a new component `<AttachmentPickerSheetContent>` at `museum-frontend/features/chat/ui/AttachmentPickerSheetContent.tsx` accepting the params of R10 plus the C4-standard `close: () => void` callback.
- **R12** : When `<AttachmentPickerSheetContent>` is rendered, the system SHALL render a `Text` header with `t('chat.attachmentPicker.title')`.
- **R13** : When `<AttachmentPickerSheetContent>` is rendered, the system SHALL render exactly two primary action `Pressable` buttons : "camera" (`testID="attachment-picker-camera"`, `accessibilityLabel = t('chat.attachmentPicker.camera')`, Ionicons `camera-outline`) and "gallery" (`testID="attachment-picker-gallery"`, `accessibilityLabel = t('chat.attachmentPicker.gallery')`, Ionicons `images-outline`).
- **R14** : When the user presses the camera button, the system SHALL invoke `onTakePicture()` then `close()`.
- **R15** : When the user presses the gallery button, the system SHALL invoke `onPickImage()` then `close()`.
- **R16** : When `recordedAudioUri === null` AND `isRecording === false`, the system SHALL render a tertiary action `Pressable` "record audio" with `testID="attachment-picker-record"`, `accessibilityLabel = t('chat.attachmentPicker.record_audio')`, Ionicons `mic-outline`. On press, the system SHALL invoke `toggleRecording()` (NOT close — the sheet remains open while user records to give visual feedback).
- **R17** : When `isRecording === true`, the tertiary action SHALL become "stop recording" with `accessibilityLabel = t('chat.attachmentPicker.stop_audio')`, Ionicons `stop-circle-outline`. On press, the system SHALL invoke `toggleRecording()` (and the sheet remains open until the user explicitly dismisses or interacts further).
- **R18** : When `recordedAudioUri !== null`, the system SHALL render the audio preview block (header `t('chat.voice_ready')` + play button + clear button) using the SAME visual contract as the legacy `<MediaAttachmentPanel>` `GlassCard` audio block (preserves user-facing copy `chat.voice_ready` / `chat.play` / `chat.playing` / `chat.clear`).
- **R19** : The play button SHALL invoke `playRecordedAudio()` (NOT close). The clear button SHALL invoke `clearMedia()` then `close()`.
- **R20** : The `<AttachmentPickerSheetContent>` SHALL NEVER use Unicode emoji. Ionicons + tokens only.

### 1.4 Screen wiring (`chat/[sessionId].tsx`)

- **R21** : The chat screen SHALL render `<Composer>` in place of `<MediaAttachmentPanel>` + `<ChatInput>`. The legacy `<MediaAttachmentPanel>` import + render call SHALL be removed.
- **R22** : The chat screen SHALL define an `onOpenAttachments` callback that calls `bottomSheetRouter.open('attachment-picker', { recordedAudioUri, isPlayingAudio, isRecording, onPickImage, onTakePicture, toggleRecording, playRecordedAudio, clearMedia, })`.
- **R23** : The chat screen SHALL pass to `<Composer>` the props : `text`, `onChangeText`, `onSend`, `isSending`, `imageUri`, `onClearImage`, `recordedAudioUri`, `isRecording`, `toggleRecording`, `onOpenAttachments`. (Note : `clearMedia` lives on the picker sheet, not the composer.)
- **R24** : The legacy file `museum-frontend/features/chat/ui/MediaAttachmentPanel.tsx` SHALL be deleted in the same commit as the green code lands (doctrine `feedback_bury_dead_code`). Its test file `__tests__/components/MediaAttachmentPanel.test.tsx` SHALL also be deleted (no zombie tests).

### 1.5 A11y

- **R25** : On mount of the `attachment-picker` sheet, the system SHALL announce `t('a11y.attachmentPicker.opened')` via `AccessibilityInfo.announceForAccessibility` (handled by `BottomSheetRouter` per C4 R17 — no new code needed, just the i18n key).
- **R26** : The `<Composer>` row SHALL satisfy WCAG 2.5.5 hit target minimum 44dp on each interactive element (`+`, mic, send). The audio mini-pill SHALL be ≥ 32dp (acceptable per iOS HIG toolbar convention, cohérent A2 §2.9 trade-off).
- **R27** : When `<Composer>` renders the mic button, the `accessibilityState` SHALL include `{ busy: isRecording }` so screen readers announce active recording.

### 1.6 i18n

- **R28** : The 8 shipped locales (ar, de, en, es, fr, it, ja, zh) SHALL contain the 12 new keys listed in §0.8 (`chat.composer.*` × 5 + `chat.attachmentPicker.*` × 6 + `a11y.attachmentPicker.opened` × 1).
- **R29** : Every new `chat.composer.*` / `chat.attachmentPicker.*` / `a11y.attachmentPicker.*` string SHALL NOT contain any Unicode emoji (regex `/[\u{1F000}-\u{1FFFF}\u{2600}-\u{27BF}]/u` returns false).
- **R30** : `chat.attachmentPicker.title` SHALL be ≤ 30 characters per locale (sheet header width budget on iPhone SE).

### 1.7 Reduced motion

- **R31** : The slide-up animation of the `attachment-picker` sheet SHALL be inherited from `BottomSheetContainer` (C4 R13). When `useReducedMotion()` returns `true`, the sheet SHALL snap open/closed instantaneously. **A1 introduces zero new animation logic.**

### 1.8 No feature flag

- **R32** : The system SHALL NOT introduce any `*_ENABLED` flag (doctrine `feedback_no_feature_flags_prelaunch`). The composer ships or it reverts.

### 1.9 Backend — pas de changement

- **R33** : `museum-backend/` SHALL NOT be modified by A1. A1 is purely a frontend layout refactor consuming existing hooks and existing send pipeline.

### 1.10 Out-of-scope (explicite)

- Long-press push-to-talk mic interaction (Open Q1, deferred V1.1).
- Multi-image gallery picker (memory `project_c2_ai_side_only` interdit explicitement multi-image V1).
- Auto-open sheet sur `recordedAudioUri` change (rejected §0.5 — trop intrusif).
- New send strategy or pipeline change (R33 — purement UI).
- Animation custom au-delà du slide-up hérité C4 (no parallax, no spring, no shared-element).
- Camera live commentary in-sheet (V2, findings.md §3.1 anti-pattern hors scope).
- Replace existing `<ChatInput>` internals (R7 — non-regression).
- Wrap `<Composer>` in a `GlassCard` distinctly from the existing `<ChatInput>` look. The existing `<ChatInput>` already uses `<GlassCard>` ; the row outside (mic + plus buttons) lives in a `<View>` with same `marginTop: semantic.screen.gapSmall`. No new container token introduced.

### 2.3 Composant — `<Composer>`

```tsx
interface ComposerProps {
  readonly text: string;
  readonly onChangeText: (text: string) => void;
  readonly onSend: () => void;
  readonly isSending: boolean;
  readonly disabled?: boolean;
  readonly imageUri?: string | null;
  readonly onClearImage?: () => void;
  readonly recordedAudioUri: string | null;
  readonly isRecording: boolean;
  readonly toggleRecording: () => Promise<void> | void;
  readonly onOpenAttachments: () => void;
}

export const Composer = React.memo(function Composer(props: ComposerProps) {
  const { theme } = useTheme();
  const { t } = useTranslation();

  const hasAudio = props.recordedAudioUri !== null;

  return (
    <View style={styles.row}>
      <Pressable
        onPress={props.onOpenAttachments}
        testID="composer-attach-button"
        accessibilityRole="button"
        accessibilityLabel={t('chat.composer.a11y.open_attachments')}
        accessibilityHint={t('chat.composer.a11y.open_attachments_hint')}
        style={[styles.iconButton, { backgroundColor: theme.surface, borderColor: theme.cardBorder }]}
      >
        <Ionicons name="add-circle-outline" size={22} color={theme.textPrimary} />
      </Pressable>

      <View style={styles.inputWrap}>
        <ChatInput
          value={props.text}
          onChangeText={props.onChangeText}
          onSend={props.onSend}
          isSending={props.isSending}
          disabled={props.disabled}
          imageUri={props.imageUri}
          onClearImage={props.onClearImage}
        />
      </View>

      {hasAudio ? (
        <Pressable
```
