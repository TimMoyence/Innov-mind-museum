## 0. Contexte vérifié

### 0.1 Pain point — pattern Cooper Hewitt manqué

`findings.md` ligne 178 : « P14 — Pas de carnet visite post-visite — absent — Différenciateur Cooper Hewitt manqué — **B1** ».
`findings.md` §3.3 pattern #2 : « Cooper Hewitt trace post-visite : 30% retour post-visite consulter sa collection web → différenciateur fort sous-exploité en mobile » (caveat §7 : case study Local Projects, non peer-reviewed — directional).

Aujourd'hui, après une visite Musaium :
- Les sessions chat existent (table `chat_sessions`), accessibles via l'onglet `(tabs)/conversations` qui les liste **à plat** triées par `updatedAt DESC`.
- Le composant `<VisitSummarySheetContent>` (livré par C4) montre un résumé **par session, dans la session ouverte** — bouton « Résumé de visite » du menu chat. C'est de l'in-session, pas du post-visite.
- Aucun écran ne propose une vue **regroupée par musée / date** des œuvres scannées, des transcripts et des photos. L'utilisateur n'a pas de « carnet » à rouvrir le soir.

B1 ajoute ce point d'entrée post-visite, en MVP — scope strictement borné pour pre-launch V1 (decisions §0.7).

### 0.4 Architecture du carnet — pattern d'agrégation

**Vue liste « Mon carnet »** = sessions **groupées par musée puis par date** (descendant). Le grouping est **client-side**, opéré par une fonction pure `groupSessionsByMuseumAndDate(sessions, locale)` produisant :

```
VisitCarnetGroup[] = [
  {
    museumKey: string,           // `museumId:${n}` si museumId défini, sinon `museumName:${name}`, sinon `unknown`
    museumLabel: string,         // museumName ?? i18n('carnet.unknownMuseum')
    sessions: VisitCarnetCard[], // tri updatedAt DESC dans le groupe
  },
  ...
]                                // tri groupes : par updatedAt MAX du groupe DESC
```

**`VisitCarnetCard`** (view-model par session) :
```
{
  id: string,                    // sessionId
  title: string,                 // session.title ?? lastArtworkTitle ?? preview?.text ?? i18n('carnet.untitledSession')
  museumLabel: string,
  dateLabel: string,             // "21 Apr 2026", locale BCP-47
  messageCount: number,
  lastArtworkTitle: string | null,
  hasArtworks: boolean,          // lastArtworkTitle non-null OR preview suggests image
}
```

Pas de fetch détail à la liste — uniquement l'item-shape déjà exposée par `/api/chat/sessions`. Pas de **count d'œuvres** côté carte (pas dispo à la liste — un seul `lastArtworkTitle`). Voir Open Q1.

**Vue détail « Visite »** = écran `app/(stack)/carnet/[sessionId].tsx` qui :
1. Fetch `chatApi.getSession(sessionId)` au mount.
2. Construit le carnet via une fonction pure dédiée `buildVisitCarnetDetail(session, messages, locale)` (réutilise `buildVisitSummary()` en interne, ajoute le transcript).
3. Rend trois sections verticales scrollables (`<ScrollView>`) :
   - **Header** : museumName + date + durée + nombre messages.
   - **Œuvres scannées** : list des `summary.artworks[]` (titre + artist? + room? + thumbnail si `imageUrl`).
   - **Transcript** : lecture-only (user + assistant interleavés, ordre chronologique, pas d'input, pas de TTS). Aucune action (pas de feedback, pas de report — c'est le screen `chat/[sessionId].tsx` qui les porte, voir §0.6 décision sur l'overlap).

### 0.5 Placement nav — décision

Le prompt du dispatcher propose deux options : (a) nouveau onglet `/(tabs)/carnet.tsx`, (b) link depuis home.

**Décision V1 — link depuis home, PAS nouveau onglet** :

| Critère | Pondération | Verdict |
|---|---|---|
| Cohérence avec tabs existants | `(tabs)/conversations` liste déjà sessions à plat → ajouter `carnet` dupliquerait visuellement | (b) |
| Saturation tab bar (`_layout.tsx`) | 3 onglets visibles (conversations, museums, home) — ajouter 4ème = OK techniquement mais charge UI | neutre |
| Découverte post-visite | Pattern Cooper Hewitt : « le visiteur retrouve son carnet le soir » → home (port d'entrée principal après app-launch) plus naturel | (b) |
| Friction sortie de visite | Si tab, friction tap. Si link home, plus visible mais perdu sous d'autres CTA | (a) si visibilité critique |
| Doctrine `feedback_bury_dead_code` + KISS V1 | 1 lien testé > 1 onglet à concevoir+styler+tester | (b) |

→ **V1 = bouton link discret sur le home screen** (`(tabs)/home.tsx`), texte « Mon carnet de visite » + Ionicons `journal-outline`, placé entre `<DailyArtCard>` et le primary CTA — assez visible sans dominer.
→ **V1.1 deferred** : promotion en tab dédié si analytics retour-visite ≥ 15 % usage.

L'écran liste lui-même est un **stack screen** sous `(stack)/` : `/(stack)/carnet.tsx`. Le détail est `/(stack)/carnet/[sessionId].tsx`. Justification — un screen « carnet » a une back-button native + n'est pas dans la bottom-tab → toujours scoped à un parcours « j'ai voulu rouvrir mon carnet ». Cohérent avec `/(stack)/chat/[sessionId].tsx` qui suit le même pattern.

### 0.6 Overlap avec écran chat — décision

Le `chat/[sessionId].tsx` existant permet déjà de **continuer une conversation** sur une session passée — il a TTS, composer, menu actions. Le carnet, lui, est **read-only** : on regarde, on ne reprend pas.

**Décision** : depuis le détail carnet, **un bouton « Continuer cette conversation »** ouvre le chat sur la session (`router.push('/(stack)/chat/${id}')`). Pas de reprise auto, pas de scroll synchronisé. Garantit la séparation : carnet = consultation post-visite ; chat = conversation active.

## 1. Spec — EARS (Easy Approach to Requirements Syntax)

### 1.1 Hook `useVisitCarnet` — data layer liste

> R1 — WHILE the user is on the carnet list screen, the system SHALL fetch the most recent chat sessions via `chatApi.listSessions({ limit: 50 })` exactly once at mount.

> R2 — WHEN the API returns successfully, the system SHALL filter OUT sessions with `messageCount === 0` (vide, never used).

> R3 — WHEN the eligible sessions list is produced, the system SHALL group them by museum identity, using the precedence rule : `museumId` if defined (key `museumId:${n}`), else `museumName` case-insensitive trimmed if non-empty (key `museumName:${slug}`), else key `unknown`.

> R4 — WHEN groups are produced, the system SHALL sort sessions within each group by `updatedAt` DESC, and sort groups themselves by max(`updatedAt`) of their sessions DESC.

> R5 — WHEN building each session card, the system SHALL derive `title` with precedence : `session.title` → `session.lastArtworkTitle` → `session.preview?.text` (truncated to 90 chars) → i18n key `carnet.untitledSession`.

> R6 — WHEN building each session card, the system SHALL produce `dateLabel` formatted via `Date.toLocaleString(locale, { day: '2-digit', month: 'short', year: 'numeric' })` (no third-party date library introduced — NFR2).

> R7 — IF the API call throws OR returns a non-conformant payload (type-guard `isListSessionsResponseDTO` false), THEN the hook SHALL expose `error: string` and `groups: []` ; the screen SHALL render the standard `<ErrorState>` component.

> R8 — WHILE the fetch is in flight, the hook SHALL expose `isLoading: true`, `groups: []`, `error: null`.

> R9 — WHEN the hook is unmounted before fetch resolves, the system SHALL ignore the late response (no setState on unmounted — pattern `useResumableSession.ts`).

> R10 — The hook SHALL expose a stable shape `{ isLoading: boolean ; error: string | null ; groups: VisitCarnetGroup[] ; refresh: () => Promise<void> }`.

### 1.2 Screen `app/(stack)/carnet.tsx` — liste « Mon carnet »

> R11 — WHEN the user opens the carnet screen, the screen SHALL display sessions grouped by museum, each group rendered as a section with `museumLabel` as header text (`accessibilityRole="header"`).

> R12 — WHILE `groups.length === 0` AND `isLoading === false` AND `error === null`, the screen SHALL render `<EmptyState variant="conversations">` with i18n keys `carnet.empty.title` / `carnet.empty.description` and a primary action « Start a conversation » that calls `useStartConversation().startConversation()`.

> R13 — WHEN the user taps a session card, the screen SHALL navigate via `router.push('/(stack)/carnet/${sessionId}')` (NOT to `/(stack)/chat/...`).

> R14 — WHILE `isLoading === true`, the screen SHALL render ≥1 `<SkeletonConversationCard>` placeholders (reuse existing component, no new skeleton).

> R15 — WHEN `error !== null`, the screen SHALL render `<ErrorState>` with the verbatim error message and a `onDismiss` that clears the error.

> R16 — The screen SHALL provide pull-to-refresh that calls `refresh()` from the hook (standard `RefreshControl`).

> R17 — The screen header SHALL render `<BrandMark variant="header" />` + a title via i18n `carnet.title` (« My visit notebook » / « Mon carnet de visite »).

### 1.3 Component `<CarnetSessionCard>` — item de liste

> R18 — `<CarnetSessionCard>` SHALL accept props `{ card: VisitCarnetCard ; onPress: (id: string) => void }` and be `React.memo`-wrapped with shallow-compare on `card.id` + `card.dateLabel` + `card.title`.

> R19 — `<CarnetSessionCard>` SHALL render a single `<Pressable>` row with `accessibilityRole="button"` and `accessibilityLabel` = i18n key `a11y.carnet.session_card` interpolated with `{ title, museum, date }`.

> R20 — `<CarnetSessionCard>` SHALL render the `card.title` (single line, truncated), the `card.dateLabel` as secondary line, and a chevron `chevron-forward` Ionicon as trailing element.

> R21 — `<CarnetSessionCard>` SHALL apply existing design tokens (`semantic.card.padding`, `theme.cardBackground`, `theme.separator`) and NEVER introduce raw hex colors or magic spacing (eslint `react-native/no-color-literals` + design-token discipline).

> R22 — `<CarnetSessionCard>` SHALL respect `useReducedMotion()` if any tap animation is added (V1 : no animation, native press feedback only — `android_ripple` + `pressed.opacity 0.7`).

### 1.4 Screen `app/(stack)/carnet/[sessionId].tsx` — détail visite

> R23 — WHEN the screen mounts with a `sessionId` route param, the system SHALL fetch `chatApi.getSession(sessionId)` exactly once.

> R24 — WHEN the response is received, the system SHALL build a `VisitCarnetDetail` via the pure function `buildVisitCarnetDetail(session, messages, locale)` defined in `features/chat/application/chatSessionLogic.pure.ts` (or sibling file) — that function internally reuses `buildVisitSummary()` for the artworks list and augments with `transcript: ChatUiMessage[]`.

> R25 — The screen SHALL render three sections vertically in a `<ScrollView>` :
>   - Header card : museumName + date + duration minutes + messageCount.
>   - « Œuvres scannées » section : list of `summary.artworks[]` with thumbnail (image or Ionicons placeholder) + title + artist? + room?.
>   - « Transcript » section : list of `transcript[]` rendered as alternating user/assistant rows (compact, read-only).

> R26 — WHEN the « Continuer cette conversation » CTA at the bottom of the screen is pressed, the system SHALL `router.push('/(stack)/chat/${sessionId}')`.

> R27 — IF the API throws or the payload is non-conformant, THEN the screen SHALL render `<ErrorState>` and a « Retour au carnet » button that calls `router.back()`.

> R28 — WHILE the fetch is in flight, the screen SHALL render skeletons (reuse `<SkeletonChatBubble>` × 3 stub for transcript section).

> R29 — The transcript section SHALL be read-only : NO composer, NO mic, NO action menu, NO feedback buttons, NO TTS auto-play.

### 1.5 Function `buildVisitCarnetDetail` (pure)

> R30 — `buildVisitCarnetDetail(session, messages, locale)` SHALL be a pure function (no side effect, no random, no Date.now beyond `formatSessionTime`-compatible reading of the inputs).

> R31 — `buildVisitCarnetDetail` SHALL return `{ summary: VisitSummary ; transcript: ChatUiMessage[] ; dateLabel: string ; durationLabel: string }`. `summary` = result of `buildVisitSummary(messages, session.title)`. `transcript` = `messages` mapped via `mapApiMessageToUiMessage()` and `sortByTime()` (chronological).

> R32 — `buildVisitCarnetDetail` SHALL produce `dateLabel` via the same precedence as R6 (using `session.updatedAt` ISO).

> R33 — `buildVisitCarnetDetail` SHALL produce `durationLabel` from `summary.duration.minutes` using an i18n-pluralizable key — V1 returns the integer minutes (the screen runs i18n interpolation). If `minutes === 0`, returns `0` (the screen renders « < 1 min »).

### 1.6 Telemetry

> R34 — WHEN the carnet list screen mounts and a non-empty `groups` set is rendered, the system SHALL emit a `carnet_list_viewed_total` counter increment (in-memory via `phase-telemetry.ts` existing pattern, no Prometheus FE pipeline — V1).

> R35 — WHEN the carnet detail screen mounts and a session is loaded successfully, the system SHALL emit a `carnet_detail_viewed_total` counter increment.

> R36 — WHEN the « Continuer cette conversation » CTA is pressed, the system SHALL emit a `carnet_continue_pressed_total` counter increment BEFORE invoking the router.

### 1.7 Non-functional requirements

> NFR1 — V1 SHALL ship **ZERO backend changes** — only consume existing endpoints `GET /api/chat/sessions` and `GET /api/chat/sessions/{id}`.

> NFR2 — V1 SHALL introduce **ZERO new third-party dependencies** (no react-native-pdf, no date-fns, no map library).

> NFR3 — V1 SHALL emit **ZERO Unicode emojis** in any UI string (per `feedback_no_unicode_emoji`). Use Ionicons + design-token icons only.

> NFR4 — V1 SHALL NOT introduce any `*_ENABLED` feature flag (per `feedback_no_feature_flags_prelaunch`).

> NFR5 — V1 SHALL use the storage façade `@/shared/infrastructure/storage` if any persistence is needed (V1 : no persistence — V1.1+ : last-viewed carnet timestamp).

> NFR6 — All i18n keys SHALL exist in **all 8 supported locales** (`fr`, `en`, `de`, `es`, `it`, `ja`, `zh`, `ar`) before merge — same discipline as B2/B4/B5/B6.

> NFR7 — A11y : every interactive element SHALL have `accessibilityRole` + `accessibilityLabel`. Group headers SHALL have `accessibilityRole="header"`.

> NFR8 — Test coverage : ≥ 4 unit test files (hook, card component, list screen, detail screen) + ≥ 1 pure-function test file for `buildVisitCarnetDetail`.

> NFR9 — `pnpm lint` (typecheck `tsc --noEmit`) PASS scoped on touched files.

> NFR10 — `npm test` PASS for all new + touched test files at green time.

## 4. Acceptance criteria

### AC1 — Liste : groupes par musée et tri

GIVEN 4 sessions avec museumId/museumName mixtes :
- s1 `{ museumId: 12, museumName: 'Louvre', updatedAt: 2026-04-21 }`
- s2 `{ museumId: 12, museumName: 'Louvre', updatedAt: 2026-04-20 }`
- s3 `{ museumId: 14, museumName: 'Orsay', updatedAt: 2026-04-22 }`
- s4 `{ museumId: null, museumName: null, updatedAt: 2026-04-19 }`
WHEN `groupSessionsByMuseumAndDate(sessions, 'fr-FR')` is called,
THEN it returns 3 groups in order : `[Orsay (s3), Louvre (s1, s2), Unknown (s4)]`.

### AC2 — Liste : exclusion des sessions vides

GIVEN une session avec `messageCount === 0`,
WHEN `useVisitCarnet()` produit `groups`,
THEN cette session est ABSENTE de tous les groupes.

### AC3 — Liste : title précédence

GIVEN une session sans `title` mais avec `lastArtworkTitle === 'La Joconde'`,
WHEN la carte est buildée,
THEN `card.title === 'La Joconde'`.

GIVEN une session sans `title`, sans `lastArtworkTitle`, avec `preview.text === 'A long preview message...'`,
WHEN la carte est buildée,
THEN `card.title === 'A long preview message...'` (truncated to ≤ 90 chars).

GIVEN une session sans `title`, ni `lastArtworkTitle`, ni `preview`,
WHEN la carte est buildée,
THEN `card.title === t('carnet.untitledSession')`.

### AC4 — Liste : loading state

GIVEN un fetch en cours,
WHEN le screen `carnet.tsx` est rendu,
THEN il affiche ≥ 1 `<SkeletonConversationCard>`.

### AC5 — Liste : empty state

GIVEN aucune session éligible (toutes vides ou aucune session),
WHEN le screen `carnet.tsx` est rendu,
THEN il affiche `<EmptyState variant="conversations">` avec un CTA « Start a conversation ».

### AC6 — Liste : error state

GIVEN un fetch qui throw,
WHEN le screen `carnet.tsx` est rendu,
THEN il affiche `<ErrorState>` (variant inline) ; le hook expose `groups: []`.

### AC7 — Liste : navigation au tap carte

GIVEN une carte session id=`'sess-9'`,
WHEN l'utilisateur tap dessus,
THEN `router.push('/(stack)/carnet/sess-9')` est appelé.

### AC8 — Détail : fetch & build

GIVEN un sessionId `'sess-7'`,
WHEN la screen `carnet/[sessionId].tsx` mount,
THEN `chatApi.getSession('sess-7')` est appelé exactement 1 fois ; au resolve, `buildVisitCarnetDetail(...)` est invoquée ; les 3 sections (header / artworks / transcript) sont rendues.

### AC9 — Détail : œuvres dédupliquées

GIVEN un transcript avec 3 messages assistant pointant 2 fois `La Joconde` et 1 fois `Vermeer Liseuse`,
WHEN `buildVisitCarnetDetail` est calculé,
THEN `summary.artworks.length === 2` (dedupe via `seenTitles` Set, comme `buildVisitSummary` existant — invariant déjà testé `build-visit-summary.test.ts`).

### AC10 — Détail : transcript read-only

WHEN la screen détail est rendue,
THEN aucun composer / aucun champ de saisie / aucun bouton micro / aucun bouton feedback / aucun bouton report N'EST visible dans le DOM (queryByTestId all return null).

### AC11 — Détail : CTA « Continuer cette conversation »

WHEN l'utilisateur tap « Continuer cette conversation »,
THEN `router.push('/(stack)/chat/${sessionId}')` est appelé.

### AC12 — Détail : error state

GIVEN un fetch qui throw,

## 6. Honnêteté UFR-013 — caveats sur la spec

1. **Cooper Hewitt « 30 % retour post-visite »** : case study Local Projects, pas une étude indépendante (cf. findings.md §7). Le pattern différenciateur reste pertinent, mais le ROI précis est indicatif.
2. **`messageCount === 0` filter** : on assume ce que `listSessions` retourne pour les sessions complètement vides. À vérifier au green-time qu'on ne reçoit pas `messageCount: undefined` pour certaines anciennes sessions — sinon ajuster le filtre (`> 0` strict).
3. **`buildVisitSummary` réutilisation** : on suppose que la fonction existante est correcte (couverte par `__tests__/features/chat/build-visit-summary.test.ts`). Si le green-agent trouve un bug, le scope B1 ne le corrige pas — ouvrir un ticket séparé.
4. **`<MarkdownBubble>` vs `<ChatMessageBubble>` pour transcript** : décision à finaliser au green-time. Le risque est de réintroduire des comportements interactifs (long-press menu, feedback) qu'on veut désactiver. Le green-agent doit privilégier `<MarkdownBubble>` pur ou créer un `<TranscriptBubble>` minimaliste si nécessaire. Mention §1.4 R29 explicite.
5. **Telemetry FE-only** : V1 ne push pas vers Prometheus. Si user demande la métrique post-launch, V1.1 ajoute un push via le pipeline existant (compteurs Sentry transactions ou pipeline analytics propre).
6. **Locale `fr-FR` vs `fr` BCP-47** : on utilise `useRuntimeSettingsStore().defaultLocale` qui peut être `'fr'` ou `'fr-FR'`. `Date.toLocaleString` tolère les deux. Pas de hard-fail.
7. **Empty state CTA réutilise `useStartConversation`** — la session démarrée n'est pas attachée à un musée (intent=default). Si V1.1 le carnet doit pouvoir lancer depuis un musée précis, ouvrir Open Q3.
