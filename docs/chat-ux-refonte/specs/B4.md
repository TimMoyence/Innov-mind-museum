## 0. Contexte vérifié

### 0.1 Pourquoi B4 — le pain point exact (findings.md P15)

> "Pas de fallback QR cartel — absent — Low-light failure path — **B4**" (findings.md ligne 179)

Aujourd'hui, le seul chemin pour "demander à l'IA quelque chose à propos de l'œuvre que je regarde" est la **photo de l'œuvre** via camera/gallery (`<AttachmentPickerSheetContent>` post-A1). Ce chemin échoue dans deux contextes museum classiques :

1. **Low-light / éclairage spot ponctuel** : œuvre dans une vitrine sombre, capteur du téléphone bruite, image floue, l'enrichment image-side (KE pipeline) sort un detectedArtwork null, l'utilisateur ne comprend pas pourquoi son scan ne marche pas.
2. **Vitrines réfléchissantes / verres anti-vol** : reflets, halos UV, l'œuvre est partiellement obscurcie.

Dans les deux cas, le **cartel** (label informatif posé à côté de l'œuvre) reste lisible, et porte typiquement un **code QR** ou un **numéro d'inventaire** (catalog number, Inv. n°, ID). C'est exactement le pattern **Bloomberg Connects** (museum apps, lookup number) — fallback robuste low-light qui complète la photo.

L'objectif B4 est de donner à l'utilisateur un chemin secondaire : **scanner le QR du cartel** (ou taper le numéro manuellement, V1.1+) plutôt que photographier l'œuvre. Le code est ensuite remonté au pipeline de chat existant sous forme texte — l'orchestrateur LLM résout le contexte (musée courant déjà connu via `museumName` + `locationString` de la session, cf. `useChatSession.ts:127-129`).

### 0.2 Pattern de référence — Bloomberg Connects

Bloomberg Connects (findings.md §3.3 ligne 146) : "lookup number cartel = QR/code manuel, robuste fallback quand scan visuel échoue (low-light, vitrines réfléchissantes)". C'est le seul des museum apps benchmarkés (Smartify, Cooper Hewitt, Google Arts, Bloomberg, MoMA, Rijksmuseum, Louvre) qui implémente ce pattern de façon visible — pour Musaium c'est différenciant face à la concurrence et complète l'attaque "voice + camera + cartel" (les 3 entrées physiques d'un visiteur).

### 0.3 État courant — entry point décidé : 4ème option `AttachmentPickerSheetContent`

Post-A1 (commit `d4a94f73`), `<AttachmentPickerSheetContent>` (sheet content monté par le `BottomSheetRouter` C4) propose 3 actions primaires en row :

```
[ camera ]  [ gallery ]  [ record audio / stop ]
```

**Décision contraignante (cohérence A1 + KISS)** : B4 ajoute une **4ème action** dans cette même `actionsRow` :

```
[ camera ]  [ gallery ]  [ record audio / stop ]  [ scan cartel ]
```

Alternative considérée et **rejetée** : bouton dédié `Scan QR` directement sur le `<Composer>` (au lieu du picker sheet). Refus motivé :
- Erode la barre composer minimaliste (3 boutons + ChatInput) que A1 a justement épurée.
- Sépare le cartel des autres "moyens d'attacher du contenu" — incohérent UX.
- Le cartel scan est un **fallback** (low-light), pas l'action primaire d'un utilisateur typique → enfouir derrière le `+` est volontaire (réactif, pas proactif, doctrine `project_hybrid_product_philosophy`).

### 0.4 État courant — librairie scanner décidée : `expo-camera`

Vérifié `museum-frontend/package.json` ligne ~71 : `"expo-camera": "~55.0.13"` est **déjà installé** (utilisé par `expo-image-picker` pour la capture photo + permission CAMERA déjà déclarée dans `app.config.ts:267-274`).

Vérification (Open Q3, à confirmer en green-phase) : `museum-frontend/node_modules/expo-camera/build/next/CameraView.d.ts` expose `barcodeScannerSettings?: BarcodeSettings` + `onBarcodeScanned?: (event: BarcodeScanningResult) => void`. Si l'export n'est pas présent dans la version installée, fallback strict sur `@expo/vector-icons` placeholder + Open Question pour bump `expo-camera` (la 55.0.13 documente le support officiel selon le changelog Expo SDK 55).

### 0.5 État courant — pas de BE artwork lookup

Grep `museum-backend/src` sur `cartel|catalogNumber|inventoryNumber|byCatalog|byNumber` retourne **zéro résultat**. Le seul "catalogue d'œuvres" côté BE est `museum-backend/src/modules/daily-art/adapters/secondary/catalog/artworks.data.ts` — liste statique de 7-10 œuvres pour le daily-art rotation, sans champ `cartelCode` ni `inventoryNumber`. Aucun endpoint `/api/artwork/lookup` n'existe.

**Décision V1** : **PAS de nouveau endpoint BE**. Le code scanné est traité comme une **chaîne de contexte** ajoutée au prompt du chat. Le pipeline LLM existant (`chat.service.ts → langchain.orchestrator.ts`) reçoit `text = "Œuvre identifiée par le cartel: <code>"` (i18n template) et résout via :
- Contexte session : `museumName` est déjà passé au prompt (`museum-mode` cache class).
- Enrichment : `knowledge-extraction` module peut consommer le code comme query Wikidata si la chaîne lookup s'oriente vers un identifiant interprétable.
- Fallback "je ne reconnais pas ce cartel" si rien ne matche — déjà géré par les guardrails.

**Pourquoi cette décision** :
1. **YAGNI** — aucun musée B2B n'est encore onboardé (memory `project_no_staging_v1`, launch V1 2026-06-01). On n'a pas de catalogue museum-side à interroger.
2. **Réversible** — si V1.1 livre un endpoint BE catalog lookup, le call-site frontend peut basculer sans changer la route ou le composant scanner.
3. **Cohérent avec project_hybrid_product_philosophy** — l'LLM est la source de vérité pour la conversation, on n'introduit pas une mécanique RPC dédiée pour 1 use-case low-light.
4. **Cohérent UFR-013** — on ne fabrique pas un endpoint qui ne renvoie rien d'utile faute de données.

L'introduction d'un endpoint `/api/artwork/lookup?cartel=<code>&museumId=<id>` est tracée Open Q1 V1.1+ (post-B2B onboarding).

### 0.6 État courant — pipeline d'envoi text-only existant

`useChatSession.sendMessage({ text: "..." })` est l'API existante. Aucun changement BE n'est requis : le code scanné est concaténé dans un template i18n localisé (e.g. `chat.cartel.lookup_template = "Tell me about the artwork with cartel code: {{code}}"`), puis envoyé via `sendMessage({ text: template })` — pipeline standard `pickSendStrategy = 'cache' | 'streaming'` selon `museumName`.

Le code scanné est **sanitisé côté FE** avant template (R12) : seul un sous-ensemble alphanumérique + tirets / underscores accepté, max 64 chars, normalisation Unicode. Cohérent avec `sanitizePromptInput()` BE (CLAUDE.md §AI Safety), mais doublé côté FE pour ne JAMAIS injecter raw user-controlled barcode payload dans le template.

### 0.7 État courant — décisions de flow

1. **Scan immédiat-send vs prefill-review** ?
   - **Décision** : **scan immédiat-send** (Bloomberg Connects pattern : fewer taps, le visiteur veut une réponse rapide). Le scanner s'ouvre fullscreen, scanne, ferme, et `sendMessage` est appelé direct avec le template i18n. L'utilisateur voit sa question dans le thread comme une bulle user normale.
   - Alternative rejetée : prefiller `composer.text = "Cartel: <code>"` pour review utilisateur. Plus de friction, peu de valeur (l'utilisateur a déjà visé le QR, il sait ce qu'il fait).
2. **Multi-scan dans la même session ouverte** ?
   - **Décision** : **non**. Une seule détection par ouverture du scanner. `onBarcodeScanned` est gated par un `scannedRef` (idempotence), la sheet ferme après le 1er hit, l'utilisateur ré-ouvre s'il veut scanner un autre cartel.
3. **Manual entry fallback** (taper le numéro) ?
   - **Décision V1** : **non**. Trop UI à designer (clavier numérique modal, validation format), peu de fréquence (QR codes ubiquitous post-COVID). V1.1+ Open Q4 (si feedback B2B le demande).
4. **Format codes acceptés** ?
   - **Décision V1** : **QR seulement** (`barcodeTypes: ['qr']`). KISS, format dominant des cartels modernes (smartify, bloomberg). Code128 / EAN-13 / Aztec V1.1+ Open Q2.
5. **Permission camera refusée** ?
   - **Décision** : screen explicite "Camera access required" + bouton "Open settings" (`Linking.openSettings()`). Cohérent UX des autres permission gates (locationConsent, etc.). Pas de fallback dégradé.

## 1. Spec — format EARS

### 1.1 Entry point — bouton "scan cartel" dans `<AttachmentPickerSheetContent>`

- **R1** : When `<AttachmentPickerSheetContent>` is rendered, the system SHALL render a 4th action `Pressable` in the actions row with `testID="attachment-picker-scan-cartel"`, `accessibilityRole="button"`, `accessibilityLabel = t('chat.attachmentPicker.scan_cartel')`, Ionicons `qr-code-outline`.

- **R3** : The "scan cartel" button SHALL be rendered after the "record audio" button in left-to-right order (LTR locales) and mirrored under RTL.
- **R4** : The `<AttachmentPickerSheetContent>` component SHALL accept a new prop `onOpenScanner: () => void` (required, no default — the screen wires it).

### 1.2 Route `cartel-scanner` — C4 registry extension

- **R5** : The system SHALL extend `BottomSheetRouteId` union with `'cartel-scanner'`.
- **R6** : The system SHALL register `'cartel-scanner'` in `ROUTES` with `presentation: 'fullscreen'`, `blocking: false`, `a11yAnnounceKey: 'a11y.cartelScanner.opened'`, `Content: CartelScannerSheetContent`.
- **R7** : The route SHALL define typed params via `BottomSheetRouteParams['cartel-scanner']` with shape `{ onScanned: (code: string) => void; }`.

### 1.3 Component `<CartelScannerSheetContent>` — scanner surface

- **R8** : The system SHALL ship `museum-frontend/features/chat/ui/CartelScannerSheetContent.tsx` exporting a React component accepting the params of R7 plus the standard `close: () => void` callback.
- **R9** : When `<CartelScannerSheetContent>` is mounted, the system SHALL request camera permission via `useCameraPermissions()` from `expo-camera`. While the permission status is `null` (not yet decided), the component SHALL render a permission-pending placeholder (centered text `t('chat.cartelScanner.permission_title')` + activity indicator). No camera preview shown yet.
- **R10** : When the camera permission status is `granted`, the system SHALL render `<CameraView>` from `expo-camera` with `facing='back'`, `barcodeScannerSettings={{ barcodeTypes: ['qr'] }}`, occupying the available fullscreen area below the safe-area top inset, with an instructions text overlay (`t('chat.cartelScanner.instructions')`) and a cancel button (`testID="cartel-scanner-cancel"`) anchored to the top-right of the screen.
- **R11** : When the camera permission status is `denied`, the system SHALL render a denied-state view : title `t('chat.cartelScanner.permission_title')`, body `t('chat.cartelScanner.permission_body')`, primary button `t('chat.cartelScanner.permission_open_settings')` (`testID="cartel-scanner-open-settings"`) wired to `Linking.openSettings()`, secondary cancel button (`testID="cartel-scanner-cancel"`) wired to `close()`.
- **R12** : When `<CameraView.onBarcodeScanned>` fires, the system SHALL :
  - sanitize the raw `data` field (Unicode normalize NFKC, strip zero-width chars, slice 0..64, reject if empty after sanitization),
  - on a valid sanitized code : invoke `onScanned(sanitizedCode)` exactly once (idempotence via a `useRef<boolean>` guard — subsequent `onBarcodeScanned` events are ignored after the first hit),
  - then invoke `close()`.
- **R13** : When the user presses the cancel button (`testID="cartel-scanner-cancel"`), the system SHALL invoke `close()` without calling `onScanned`.
- **R14** : The `<CartelScannerSheetContent>` SHALL announce `t('a11y.cartelScanner.scan_success')` via `AccessibilityInfo.announceForAccessibility` on a successful sanitized scan (R12), before `close()`.
- **R15** : The `<CartelScannerSheetContent>` SHALL NEVER use Unicode emoji (doctrine `feedback_no_unicode_emoji`). Ionicons + tokens only.

### 1.4 Screen wiring — `chat/[sessionId].tsx`

- **R16** : The chat screen SHALL define an `onOpenCartelScanner: () => void` callback (memoized via `useCallback`) that calls `bottomSheetRouter.open('cartel-scanner', { onScanned: handleCartelScanned })`.
- **R17** : The chat screen SHALL define a `handleCartelScanned: (code: string) => void` callback (memoized via `useCallback`) that calls `sendMessage({ text: t('chat.cartelScanner.lookup_template', { code }) })`. The callback SHALL NOT directly write to `<Composer>` state — the bubble appears via the standard `messages` reducer.
- **R18** : The chat screen SHALL pass `onOpenScanner: onOpenCartelScanner` to the existing `bottomSheetRouter.open('attachment-picker', {...})` params object (R22 of A1 spec extended).

### 1.5 Sanitization + safety

- **R19** : The `sanitizeCartelCode(raw: string): string | null` helper SHALL apply, in order : Unicode normalize NFKC, strip control + zero-width chars (`​-‏`, `﻿`), restrict to alphanumeric + `-` + `_` + `.` (regex `/[^A-Za-z0-9._-]/g` → empty), trim, slice 0..64. Returns `null` if the result is empty.
- **R20** : The `chat.cartelScanner.lookup_template` i18n key SHALL contain the `{{code}}` placeholder exactly once. The template SHALL NOT contain unescaped HTML, markdown, or LLM instruction markers (e.g. `[END OF SYSTEM INSTRUCTIONS]`, `<|im_end|>`).
- **R21** : The system SHALL NOT pass the raw barcode payload to `console.log` / `console.debug` at any level above `debug`. Sentry breadcrumb (Open Q7) deferred V1.1+.
- **R22** : The system SHALL NOT send the cartel code via any other transport than `sendMessage` (no direct API call, no analytics event with the value).

```ts
const CONTROL_AND_ZW_RE = /[ -​-‏﻿]/g;
const FORBIDDEN_RE = /[^A-Za-z0-9._-]/g;

export function sanitizeCartelCode(raw: string): string | null {
  if (typeof raw !== 'string') return null;
  const normalized = raw.normalize('NFKC');
  const stripped = normalized.replace(CONTROL_AND_ZW_RE, '');
  const filtered = stripped.replace(FORBIDDEN_RE, '');
  const trimmed = filtered.trim();
  if (trimmed.length === 0) return null;
  return trimmed.slice(0, SANITIZE_MAX_LEN);
}
```

```tsx
  const { t } = useTranslation();
```

## 5. Open questions

- **Q1 (BE artwork lookup endpoint)** : Aujourd'hui le cartel code est traité comme du texte. Une fois Musaium B2B onboardé (cf. memory `project_no_staging_v1` post-launch), un endpoint dédié `/api/artwork/lookup?cartel=<code>&museumId=<id>` retournerait un objet `Artwork` enrichi (image, year, description, room, etc.) que le FE pourrait pre-fill dans `<ArtworkHeroCard>` (A2) avant même que le LLM réponde. **Proposition V1** : NON — pas de données catalog pour interroger. V1.1+ post-B2B avec catalog ingestion KE pipeline étendue.
- **Q2 (Formats codes acceptés)** : V1 = QR seulement. Les museums vintage utilisent encore Code128 / EAN-13 sur cartels imprimés. `barcodeScannerSettings.barcodeTypes` supporte `['qr', 'code128', 'ean13', 'aztec', 'datamatrix']`. **Proposition V1** : QR-only (KISS). Multi-format V1.1+ après B2B feedback.
- **Q3 (Vérification API `expo-camera ~55.0.13`)** : La spec assume `CameraView` + `useCameraPermissions` + `onBarcodeScanned` exposés à cette version. Si la version installée n'expose pas ces symboles (Expo SDK 55 changelog ambigu), il faudra soit bump minor d'`expo-camera`, soit annoter Open Q dans le rapport green. À vérifier au début du T2.1.
- **Q4 (Manual numeric entry fallback)** : Si la caméra refuse / l'utilisateur préfère taper le numéro d'inventaire au lieu du QR. Nécessite un `<TextInput>` + validation pattern + clavier numérique. **Proposition V1** : non — peu de fréquence (QR ubiquitaire post-COVID), trop UI à designer. V1.1+ si B2B feedback le demande.
- **Q5 (Traductions 6 locales non-en/fr)** : Le red-test ne fixe que sentinelles en/fr. Les 6 autres locales reçoivent placeholders ≤ 50 chars (valeurs EN copiées) ; produit valide post-green (cohérent A1/A2/A6/B3/B5 pattern).
- **Q6 (Multi-scan continu)** : V1 = 1 scan / 1 ouverture du scanner (`scannedRef` gating). Pour scanner plusieurs cartels rapidement (e.g. visite de groupe, comparaison 2 œuvres), il faudrait un mode "rapid scan" qui ne ferme pas après chaque hit. **Proposition V1** : non — friction visible (haptic + a11y announce + sheet stays open) à designer. V1.1+ Open Question.
- **Q7 (Telemetry Sentry breadcrumb)** : R21 interdit de logger le raw payload, mais un breadcrumb `cartel_scanned { length: code.length }` (sans le code lui-même) serait utile pour mesurer l'adoption. **Proposition V1** : `console.debug('[B4] cartel_scanned', { length })` only, cohérent A2/A6/B3/B5. Sentry breadcrumb V1.1+.
