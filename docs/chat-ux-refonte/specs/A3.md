## 0. Contexte vérifié

### 0.1 État courant — bulles indistinguées par texture (findings.md P18)

Aujourd'hui, `museum-frontend/features/chat/ui/ChatMessageBubble.tsx` rend **deux variantes du même container** pour user et assistant — un `<Pressable>` (assistant, pour long-press → report) et un `<View>` (user), tous deux portés par le même style `styles.bubble` (radius + padding + borderWidth), différenciés uniquement par les tokens **couleur** `theme.userBubble` / `theme.assistantBubble` (lignes 165-196 du fichier) :

```ts
// assistant : Pressable
backgroundColor: theme.assistantBubble,     // 'rgba(255, 255, 255, 0.72)' light / 'rgba(30, 41, 59, 0.72)' dark
borderColor: theme.assistantBubbleBorder,   // 'rgba(148, 163, 184, 0.22)' light / '...0.18' dark
alignSelf: 'flex-start'

// user : View
backgroundColor: theme.userBubble,          // 'rgba(30, 64, 175, 0.88)' light / '...0.92' dark
borderColor: theme.userBubbleBorder,        // 'rgba(191, 219, 254, 0.6)' light / '...0.4' dark
alignSelf: 'flex-end'
```

Vérifié dans `museum-frontend/shared/ui/tokens.functional.ts:32-35` et `:56-59`. Les tokens utilisateur sont **déjà ~mat** (alpha 0.88/0.92 — presque opaque) mais le rendu se fait via un `View` ordinaire : **pas de BlurView**, donc même la couleur translucide ne capture rien du fond visuellement → la perception est uniforme. Le pain point P18 (findings.md ligne 182) acte : « Bubbles user/assistant indistinguées par texture — tokens uniformes — Glassmorphism 2026 trend ».

L'objectif A3 est de matérialiser **deux textures distinctes** :

- **User bubble = mat solide** : `View` natif avec `backgroundColor` opaque (pas de BlurView, pas d'alpha). Réutilise `theme.userBubble` mais via une variante **opaque** (alpha 1.0) dérivée par calcul d'opacité — pas un nouveau hex.
- **Assistant bubble = glass** : container `BlurView` (intensité modérée, `tint` adaptatif dark/light) avec `theme.glassBackground` + `theme.glassBorder` en overlay. Réutilise les tokens DS existants `glassBackground` / `glassBorder` / `blurTint` consommés déjà par `<GlassCard>` (`shared/ui/GlassCard.tsx:18-32`) et `<FloatingContextMenu>` (`shared/ui/FloatingContextMenu.tsx:50-94`).

### 0.4 Accessibilité — iOS `reduceTransparency`

Le glassmorphism (assistant bubble) est **décoratif** : il rend la bulle assistant visuellement distincte (texture frosted) sans changer le contenu lu par VoiceOver. iOS et Android offrent une option d'accessibilité « **Reduce Transparency** » (iOS Settings → Accessibility → Display & Text Size → Reduce Transparency ; Android pas d'API directe — par défaut `false`). Quand activé, l'utilisateur signale qu'il ne tolère pas le blur (lisibilité dégradée, latence).

**Décision A3** : créer un hook `useReducedTransparency()` (sibling de `useReducedMotion`) qui :

- Consomme `AccessibilityInfo.isReduceTransparencyEnabled()` + le listener `reduceTransparencyChanged` (RN ≥ 0.73, disponible dans Expo SDK 55). Sur Android, l'API retourne `false` silencieusement par défaut (cohérent comportement `useReducedMotion`).
- Retourne `true` si l'utilisateur a activé le réglage.
- Quand `true`, **`<ChatMessageBubble>` assistant SHALL fallback en mat solide** (View opaque avec `theme.assistantBubble` consommé en `backgroundColor` mais opaque, identique au pattern user) — **PAS de BlurView**.

Pattern WCAG : Reduce Transparency = lisibilité fonctionnelle. Cohérent avec la décision déjà appliquée pour `useReducedMotion` (motion décoratif désactivé). NFR2 (a11y) couvert.

## 1. Spec — format EARS

### 1.1 Hook `useReducedTransparency()` — fallback a11y

- **R1** : When `useReducedTransparency()` is invoked on mount, the hook SHALL call `AccessibilityInfo.isReduceTransparencyEnabled()` once and update its internal state to the resolved boolean.
- **R2** : When the OS-level "Reduce Transparency" setting changes during the component lifecycle, the hook SHALL update its returned value via the `AccessibilityInfo.addEventListener('reduceTransparencyChanged', ...)` subscription.
- **R3** : When `AccessibilityInfo.isReduceTransparencyEnabled()` rejects (older Android — API absent), the hook SHALL default to `false` and SHALL NOT throw.
- **R4** : The hook SHALL remove its listener on unmount (no leak).

### 1.2 Util `forceOpaque(rgbaString)` — pure derivation, NO new color

- **R5** : When `forceOpaque(rgba: string)` is invoked with an `rgba(R, G, B, A)` formatted string, the function SHALL return `rgba(R, G, B, 1)`. The RGB values SHALL be preserved verbatim (no rounding, no hex conversion).
- **R6** : When `forceOpaque(rgb: string)` is invoked with an already-opaque `rgb(R, G, B)` formatted string, the function SHALL return the input string unchanged.
- **R7** : When `forceOpaque(hex: string)` is invoked with a hex string (e.g. `'#1e40af'`), the function SHALL return the input string unchanged (hex is opaque by definition).
- **R8** : The function SHALL be pure (no state, no I/O, deterministic).

### 1.3 `<ChatMessageBubble>` — user bubble (mat solide)

- **R9** : When `<ChatMessageBubble message={{role:'user', ...}} />` is rendered, the system SHALL render a `<View>` container (NOT a `<BlurView>`) at the root of the bubble.
- **R10** : The user bubble container SHALL expose `testID="chat-bubble-user"`.
- **R11** : The user bubble container `backgroundColor` SHALL be `forceOpaque(theme.userBubble)` (alpha=1) — mat solid, no transparency on the bubble.
- **R12** : The user bubble container SHALL preserve `alignSelf: 'flex-end'`, `borderColor: theme.userBubbleBorder`, `borderWidth: semantic.input.borderWidth`, `borderRadius: semantic.chat.bubbleRadius`, `padding: semantic.chat.bubblePadding`, `maxWidth: '85%'`.

### 1.4 `<ChatMessageBubble>` — assistant bubble (glass)

- **R13** : When `<ChatMessageBubble message={{role:'assistant', ...}} />` is rendered AND `useReducedTransparency() === false`, the system SHALL render a `<BlurView>` (from `expo-blur`) as the inner container of the bubble.
- **R14** : The assistant `<BlurView>` SHALL expose `testID="chat-bubble-assistant"`.
- **R15** : The assistant `<BlurView>` SHALL have `intensity={42}` (constant exported `ASSISTANT_BUBBLE_BLUR_INTENSITY` from `ChatMessageBubble.tsx`).
- **R16** : The assistant `<BlurView>` SHALL have `tint={theme.blurTint}` (i.e. `'light'` in light mode, `'dark'` in dark mode).
- **R17** : The assistant `<BlurView>` SHALL have `backgroundColor: theme.assistantBubble` (semi-transparent overlay tint, NOT forced opaque — the alpha allows the blur to show through).
- **R18** : The assistant `<BlurView>` SHALL have `borderColor: theme.assistantBubbleBorder`, `borderWidth: semantic.input.borderWidth`, `borderRadius: semantic.chat.bubbleRadius`, `padding: semantic.chat.bubblePadding`, `maxWidth: '85%'`, `alignSelf: 'flex-start'`, `overflow: 'hidden'` (clip blur to radius).

### 1.5 Reduced transparency fallback (assistant only)

- **R19** : When `<ChatMessageBubble message={{role:'assistant', ...}} />` is rendered AND `useReducedTransparency() === true`, the system SHALL render a `<View>` container (NOT a `<BlurView>`) at the root of the bubble.
- **R20** : The fallback assistant `<View>` SHALL still expose `testID="chat-bubble-assistant"` (so consumers can assert presence regardless of the texture choice).
- **R21** : The fallback assistant `<View>` SHALL have `backgroundColor: forceOpaque(theme.assistantBubble)` (alpha=1, mat opaque), preserving all other style properties of R18 except `overflow` (no clip needed without blur).
- **R22** : The fallback assistant `<View>` SHALL behave identically to the non-fallback bubble for accessibility (same `accessibilityRole="text"`, same `accessibilityLabel`, same long-press → report).

### 2.2 Util — `forceOpaque(color)`

```ts
// museum-frontend/shared/ui/colorUtils.ts

/**
 * Returns the same color with alpha forced to 1 (fully opaque).
 *
 * - `rgba(R, G, B, A)` → `rgba(R, G, B, 1)` (RGB preserved)
 * - `rgb(R, G, B)`     → unchanged (already opaque)
 * - `#hex`             → unchanged (already opaque)
 * - Any unrecognized format → unchanged (defensive — caller's responsibility)
 *
 * Used to express the A3 "user bubble = mat solide" rule WITHOUT introducing
 * a new design token — derived deterministically from the existing
 * `theme.userBubble` / `theme.assistantBubble` rgba values.
 */
const RGBA_RE = /^rgba\(\s*(\d+)\s*,\s*(\d+)\s*,\s*(\d+)\s*,\s*[0-9.]+\s*\)$/;

export function forceOpaque(color: string): string {
  const match = RGBA_RE.exec(color);
  if (!match) return color;
  const [, r, g, b] = match;
  return `rgba(${r}, ${g}, ${b}, 1)`;
}
```

Justification du regex strict : on couvre uniquement le format produit par `tokens.functional.ts` (rgba) sans toucher aux formats que `forceOpaque` ne devrait pas modifier (hex / rgb / inconnu). Cohérent doctrine UFR-013 — explicite > implicite.

### 2.3 Hook — `useReducedTransparency`

```ts
// museum-frontend/shared/ui/hooks/useReducedTransparency.ts

import { useEffect, useState } from 'react';
import { AccessibilityInfo } from 'react-native';

/**
 * Returns `true` when the OS-level "Reduce Transparency" accessibility setting
 * is enabled (iOS: Settings → Accessibility → Display & Text Size → Reduce
 * Transparency). On Android, the API is absent → defaults to `false`.
 *
 * Consumers should fall back to opaque solid surfaces (no BlurView, no alpha)
 * when this returns true — translucent surfaces reduce text contrast and
 * cause visual fatigue for the targeted users.
 *
 * Sibling of `useReducedMotion` — same lifecycle pattern (mount fetch +
 * change listener + unmount cleanup + reject → false).
 *
 * WCAG 2.1 AA — informed by Apple HIG "Reduce Transparency" guidance.
 */
export function useReducedTransparency(): boolean {
  const [reduced, setReduced] = useState(false);

  useEffect(() => {
    let mounted = true;

    AccessibilityInfo.isReduceTransparencyEnabled()
      .then((value) => {
        if (mounted) setReduced(value);
      })
      .catch(() => {
        // Platform returns nothing on older Android — default to false.
      });

    const subscription = AccessibilityInfo.addEventListener(
      'reduceTransparencyChanged',
      (value) => {
        if (mounted) setReduced(value);
      },
    );

    return () => {
      mounted = false;
      subscription.remove();
    };
  }, []);

  return reduced;
}
```

```tsx
          styles.bubbleAssistantOpaque,
          {
            backgroundColor: forceOpaque(theme.assistantBubble),
            borderColor: theme.assistantBubbleBorder,
          },
        ]}
      >
        {bubbleContent}
      </View>
    ) : (
      <BlurView
        testID="chat-bubble-assistant"
        intensity={ASSISTANT_BUBBLE_BLUR_INTENSITY}
        tint={theme.blurTint}
        style={[
          styles.bubble,
          styles.bubbleAssistantBlur,
          {
            backgroundColor: theme.assistantBubble,
            borderColor: theme.assistantBubbleBorder,
          },
        ]}
      >
        {bubbleContent}
      </BlurView>
    )}
  </Pressable>
) : (
  <View
    testID="chat-bubble-user"
    style={[
      styles.bubble,
      styles.bubbleUser,
      {
        backgroundColor: forceOpaque(theme.userBubble),
        borderColor: theme.userBubbleBorder,
      },
    ]}
    accessibilityRole="text"
    accessibilityLabel={t('a11y.chat.user_message')}
  >
    {bubbleContent}
  </View>
)}
```

### 2.5 Tokens DS — table d'usage

| Usage | Token | Note |
|---|---|---|
| User bg | `forceOpaque(theme.userBubble)` | Token existant + alpha forced 1 |
| User border | `theme.userBubbleBorder` | Existant |
| Assistant bg (glass) | `theme.assistantBubble` | Existant (alpha 0.72) — overlay tint sur blur |
| Assistant bg (fallback opaque) | `forceOpaque(theme.assistantBubble)` | Existant + alpha forced 1 |
| Assistant border | `theme.assistantBubbleBorder` | Existant |
| Blur tint | `theme.blurTint` | Existant (`'light' | 'dark'`) |

### 2.6 Performance

- 1 `BlurView` par message assistant (n ≤ ~50 typique). React.memo `ChatMessageBubble` empêche re-render hors changement de `text` / `isStreaming` / `ttsPlaying` etc. (comparator existant `:243-262` inchangé).
- `expo-blur` natif iOS = `UIVisualEffectView` (~5-15ms init iOS A15, ~10-25ms render Android). Pas de re-blur sur scroll grâce à `React.memo`.
- `forceOpaque` = regex one-shot, négligeable (< 1µs par appel).
- `useReducedTransparency` = mount-only + listener event-driven. Pas de polling.

### 2.7 A11y

- `accessibilityRole="text"` préservé sur user et assistant.
- `accessibilityLabel` préservé.
- `accessibilityHint` (long-press) préservé.
- Contraste texte : user blanc sur bleu `rgba(30, 64, 175, 1)` = WCAG AA ✓ (ratio > 4.5). Assistant : texte `theme.textPrimary` (dark slate light mode / white dark mode) sur `theme.assistantBubble` (white-72% light / slate-72% dark) = WCAG AA ✓ déjà validé en baseline. Le blur ne dégrade pas le contraste (le tint blanc-72% reste l'élément dominant pour le texte).
- iOS Reduce Transparency : fallback opaque (R19-R22). Cohérent Apple HIG.
- VoiceOver : ne distingue pas mat/glass — toujours lit le contenu de la bulle. Pas de label change.

### 2.8 Dark mode

`theme.blurTint = 'dark'` en dark theme → BlurView sombre. `theme.assistantBubble = rgba(30, 41, 59, 0.72)` (slate-800 / 0.72) → overlay tint sombre cohérent "dark glassmorphism trend" (findings.md §3.2). Aucune nouvelle couleur.

### 2.9 Caveats / pièges

- **`overflow: 'hidden'`** est REQUIS sur le BlurView pour que le blur soit clipé au `borderRadius`. Sans, le blur déborde et casse la silhouette de la bulle.
- **Jest mock `expo-blur`** retourne un `<View>` qui spread les props — donc les tests qui matchent par `getByText` continuent de fonctionner (l'enfant texte est rendu). Pour asserter sur `intensity` / `tint`, on récupère l'élément via `getByTestId('chat-bubble-assistant')` et on lit `props.intensity` / `props.tint`.
- **Pas de double React.memo** : la `BlurView` n'est PAS memoized séparément — elle vit comme enfant direct du composant déjà memoized.
- **Streaming** : la BlurView reste active pendant le streaming. Pas de toggle à mi-chemin (qui flickerait). Le comparator React.memo existant skip re-render si seul le texte change.
