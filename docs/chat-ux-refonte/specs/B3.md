## 0. Contexte vérifié

### 0.1 Le gap — findings P12

### 0.3 Source du follow-up — décision BE-led (LLM-generated)

**Décision retenue** : **LLM-generated côté BE**, NON heuristique FE.

Rationale (doctrine + ergonomie) :
- La doctrine tracking « JAMAIS 3 boutons — référence un fact précis ou rien » exige une **vraie référence sémantique** à un fact concret dans la dernière réponse. Une heuristique FE par regex (e.g. extraction de NER first-NP, "Tell me more about X") produit des génériques qui se rapprochent du anti-pattern 3-choice chatbot — sans atteindre la précision Be My AI.
- Le LLM a déjà le contexte complet (la réponse + l'image + l'historique + le visit context) pour rédiger UNE question naturelle qui prolonge la conversation. C'est exactement le rôle du structured-output qu'on utilise pour `followUpQuestions[]` aujourd'hui — l'enrichissement de prompt est minime, le coût/latence quasi-nul (un champ de plus dans le schéma).
- Une heuristique FE forcerait des deltas comportementaux entre locales (la prosody anglaise vs française d'un "Tell me more" ≠ "Dis-m'en plus" se prête mal au regex pur).
- Le BE peut **refuser explicitement** de proposer un follow-up quand le contexte ne s'y prête pas (refus guardrail, message court non-factuel, daily-art récap déjà clos) — cette décision sémantique appartient au LLM, pas au FE.

**Caveat scope BE** : c'est un changement BE — pas un scope creep caché. C'est documenté en §2.1 (changes BE list explicite) + R1-R8 (BE EARS) + Open Q1 (deferred FE-heuristic fallback si BE dévie en bake).

### 0.4 Forme du composant — chip singulier, jamais 3 boutons

| Décision | Détail |
|---|---|
| Cardinalité | UN seul chip (`suggestedFollowUp` est un **string** singulier, pas un array — la grammaire BE/FE refuse plusieurs) |
| Placement | Sous la bubble assistant, dans `<MessageActions>`, AU-DESSUS de `<FollowUpButtons>` legacy (le legacy disparaît V1.1 quand le BE arrête d'émettre `followUpQuestions`) |
| Longueur visible | ≤ 80 caractères (R5) — au-delà, le BE clip + ellipse, le FE garde 80 chars max via `String.prototype.slice` defensive |
| Aspect | Single Pressable, Ionicons `arrow-forward-circle-outline` (PAS `send-outline` — différencie visuellement du legacy vertical-list), texte 1 ligne `numberOfLines={1}` ellipse |
| Tap | Envoie le `suggestedFollowUp` comme nouveau message user via `onFollowUpPress(text)` existant — réutilise zéro nouveau handler |
| Disparition | Disparaît automatiquement dès qu'un message user/assistant nouveau arrive (la `MessageActions` n'est rendue que pour `lastAssistantMessage` — déjà existant dans `ChatMessageList.tsx:197`) |
| Dismiss explicite | NON — pas de croix close, pas de swipe. Disparition implicite via "next message". KISS doctrine. |
| Disabled state | Pendant un `isSending=true`, le chip est rendu avec opacity 0.5 et `onPress` no-op (cohérent FollowUpButtons existant) |

### 0.5 Décisions BE — prompt + schema + parser

**Prompt enrichment** (`llm-sections.ts:249`) : remplacer la ligne actuelle par UN bloc d'instruction explicite :

> In `suggestedFollowUp`, propose ONE single short follow-up question (≤80 chars) the visitor might want to ask next, ANCHORED TO A SPECIFIC FACT mentioned in your answer. Never suggest a generic prompt like "Tell me more". If the answer does not contain a concrete fact (refusal, short clarification, daily-art recap), set `suggestedFollowUp` to null.

**Schema** (`main-assistant-output.schema.ts:89-94`) : remplacer `followUpQuestions: z.array(z.string()).nullable()` par `suggestedFollowUp: z.string().nullable()` (garde le champ existant pour 1 cycle backward-compat — voir §1.6).

**Parser** (`assistant-response.ts:72-81`) : ajouter `toSuggestedFollowUp(value): string | undefined` qui (a) accepte `string` non-vide ≤ 80 chars, (b) sinon retourne `undefined`. Le parser `toFollowUpQuestions` legacy reste, mais NFR8 garantit que le BE n'émet plus simultanément les deux — au choix du prompt, soit `suggestedFollowUp` (B3 path), soit `followUpQuestions` (legacy path, deprecated 1 cycle).

**Metadata** (`chat.types.ts:293`) : ajouter `suggestedFollowUp?: string` à `ChatAssistantMetadata`. Garder `followUpQuestions?: string[]` pour 1 cycle (NFR8). Documenter `@deprecated` JSDoc tag sur `followUpQuestions`.

**HTTP contract** (`chat.contracts.ts:111`) : ajouter `suggestedFollowUp?: string` au payload assistant message.

## 1. Spec — format EARS

### 1.1 Backend — schema + prompt + parser

- **R1** : The `mainAssistantOutputSchema` in `museum-backend/src/modules/chat/useCase/llm/llm-sections/main-assistant-output.schema.ts` SHALL add a field `suggestedFollowUp: z.string().max(80).nullable()` with a `.describe()` instructing the LLM to emit ONE follow-up question anchored to a fact in the answer, or `null` when no fact is concrete enough to anchor.
- **R2** : The prompt builder in `museum-backend/src/modules/chat/useCase/llm/llm-sections.ts` SHALL replace the current `followUpQuestions` instruction with an instruction that (a) imposes exactly ONE follow-up, (b) requires a factual anchor inside the answer, (c) instructs `null` when the answer is a refusal / clarification / has no factual anchor.
- **R3** : The schema field SHALL be a `string` singular type, NEVER `array(string)`. The compile-time and runtime test contracts in `museum-backend/tests/unit/chat/follow-up-suggestion.test.ts` SHALL fail if a future contributor changes `suggestedFollowUp` to an array.
- **R4** : `museum-backend/src/modules/chat/useCase/orchestration/assistant-response.ts` SHALL expose a pure function `toSuggestedFollowUp(value: unknown): string | undefined` that returns the trimmed string if `typeof value === 'string'` AND `0 < trimmed.length ≤ 80`, and `undefined` otherwise.
- **R5** : `toSuggestedFollowUp` SHALL truncate strings strictly longer than 80 characters to `undefined` (NOT silently slice to 80) — this is the strict-validation path that catches LLM prompt drift early. The schema `.max(80)` is a separate defence layer.
- **R6** : `extractMetadata` in `assistant-response.ts` SHALL call `toSuggestedFollowUp(parsed.suggestedFollowUp)` and assign the result to `metadata.suggestedFollowUp`.
- **R7** : `ChatAssistantMetadata` in `museum-backend/src/modules/chat/domain/chat.types.ts` SHALL declare `suggestedFollowUp?: string` AND keep `followUpQuestions?: string[]` for ONE release cycle (NFR8). The new field's JSDoc SHALL state « single, factual-anchor follow-up — B3 ».
- **R8** : `ChatMessageService.postMessage` (`message-commit.ts` consumer) SHALL persist the value verbatim on success — no truncation, no fallback, no synthetic default.

### 1.2 Backend — HTTP contract

- **R9** : The OpenAPI-mirrored type in `museum-backend/src/modules/chat/adapters/primary/http/chat.contracts.ts` SHALL declare `suggestedFollowUp?: string` on the assistant message payload, alongside the existing `followUpQuestions?: string[]` (deprecated 1 cycle).
- **R10** : The HTTP serialiser SHALL emit `suggestedFollowUp` ONLY when `metadata.suggestedFollowUp` is a non-empty string ≤ 80 chars. NEVER `null` on the wire (`undefined` => field omitted, per existing optional convention).

### 1.3 Frontend — type contract

- **R11** : `museum-frontend/features/chat/application/chatSessionLogic.pure.ts` `ChatUiMessageMetadata` SHALL add `suggestedFollowUp?: string`. Keep `followUpQuestions?: string[]` 1 cycle.
- **R12** : The OpenAPI-generated `openapi.ts` types SHALL eventually expose the field after `npm run generate:openapi-types` — out-of-scope to assert in red tests (regenerated artefact).

### 1.4 Frontend — composant atomique `<AskMoreChip>`

- **R13** : The `<AskMoreChip>` SHALL be a `<Pressable>` with rounded shape (`borderRadius: semantic.button.radius`), border 1px (`borderColor: theme.primaryBorderSubtle`), padding `paddingHorizontal: space['3']` / `paddingVertical: space['2']`, background `theme.primaryTint`.
- **R14** : The `<AskMoreChip>` SHALL render an `Ionicons` icon `arrow-forward-circle-outline` (size 16) to the LEFT of the label text. NEVER `send-outline` (visually differentiates from the legacy `<FollowUpButtons>` vertical list). NEVER Unicode emoji (doctrine `feedback_no_unicode_emoji`).
- **R15** : The `<AskMoreChip>` SHALL render the follow-up text using `<Text numberOfLines={1} ellipsizeMode="tail">` capped visually by 80 chars effective via `slice(0, 80)` on the prop.
- **R16** : The `<AskMoreChip>` SHALL accept a single string prop `text` (non-empty assertion at render time — returns `null` if `text.trim().length === 0`) and a callback `onPress: (text: string) => void` invoked with the verbatim text on tap. NEVER pass an array, never accept multiple suggestions.
- **R17** : The `<AskMoreChip>` SHALL set `accessibilityRole="button"`, `accessibilityLabel` to `t('chat.askMore.a11y_label', { text })` (resolved interpolated), and `accessibilityHint` to `t('chat.askMore.a11y_hint')`.
- **R18** : When the `disabled` prop is `true`, the `<AskMoreChip>` SHALL render with `opacity: 0.5` and `onPress` SHALL be a no-op (mirror `FollowUpButtons` legacy contract).
- **R19** : The `<AskMoreChip>` SHALL NOT render any close button, swipe handler, or dismiss UI — disappearance is driven by the parent `<MessageActions>` only mounting it on `lastAssistantMessage` (existing logic, ChatMessageList.tsx:197).
- **R20** : The `<AskMoreChip>` SHALL NOT animate on appear — `useReducedMotion()` doctrine respected by absence (no Animated.* / Reanimated worklet introduced). WCAG 2.3.3.

### 1.5 Frontend — intégration `<MessageActions>`

- **R21** : `<MessageActions>` (`museum-frontend/features/chat/ui/MessageActions.tsx`) SHALL render `<AskMoreChip>` ABOVE `<FollowUpButtons>` when `metadata.suggestedFollowUp` is a non-empty string. The chip SHALL use the same `onFollowUpPress` callback prop as the legacy buttons — zero new handler wiring.
- **R22** : When `metadata.suggestedFollowUp` is `undefined` OR an empty string OR longer than 80 chars (defence in depth), `<AskMoreChip>` SHALL NOT render (return null).
- **R23** : `<MessageActions>` SHALL NEVER render more than ONE `<AskMoreChip>` per message — the prop is singular by type contract (R11) and a TypeScript-level test asserts this (no `.map`-style fanout possible).
- **R24** : When `<AskMoreChip>` AND `<FollowUpButtons>` are both rendered (transient cohabitation V1, BE emits both during deprecation cycle), the chip SHALL appear FIRST in source order (above the buttons). V1.1 task removes `<FollowUpButtons>` when BE stops emitting `followUpQuestions`.

### 2.3 BE — parser helper

```ts
// assistant-response.ts (new helper)
const MAX_SUGGESTED_FOLLOWUP_CHARS = 80;

const toSuggestedFollowUp = (value: unknown): string | undefined => {
  if (typeof value !== 'string') return undefined;
  const trimmed = value.trim();
  if (trimmed.length === 0) return undefined;
  if (trimmed.length > MAX_SUGGESTED_FOLLOWUP_CHARS) return undefined; // R5 strict drop
  return trimmed;
};
```

### 2.4 FE — composant `<AskMoreChip>`

```tsx
interface AskMoreChipProps {
  readonly text: string;
  readonly onPress: (text: string) => void;
  readonly disabled?: boolean;
}

const MAX_CHARS = 80;

export const AskMoreChip = React.memo(({ text, onPress, disabled = false }: AskMoreChipProps) => {
  const { theme } = useTheme();
  const { t } = useTranslation();

  const trimmed = text.trim();
  if (trimmed.length === 0) return null;
  const safeText = trimmed.length > MAX_CHARS ? trimmed.slice(0, MAX_CHARS) : trimmed;

  return (
    <Pressable
      onPress={disabled ? undefined : () => onPress(safeText)}
      disabled={disabled}
      style={[
        styles.chip,
        { borderColor: theme.primaryBorderSubtle, backgroundColor: theme.primaryTint },
        disabled && styles.disabled,
      ]}
      accessibilityRole="button"
      accessibilityLabel={t('chat.askMore.a11y_label', { text: safeText })}
      accessibilityHint={t('chat.askMore.a11y_hint')}
      hitSlop={6}
    >
      <Ionicons name="arrow-forward-circle-outline" size={16} color={theme.primary} />
      <Text numberOfLines={1} ellipsizeMode="tail" style={[styles.label, { color: theme.primary }]}>
        {safeText}
      </Text>
    </Pressable>
  );
});
```

## 6. NFR

- **NFR1 (Perf)** : `<AskMoreChip>` = pure render, React.memo, zéro useEffect/timer. `slice(0, 80)` est O(1). BE `toSuggestedFollowUp` = pure, O(n) sur string length, n ≤ 80. Latence impact LLM ≈ 0 (1 field de plus dans le schema structured output).
- **NFR2 (A11y)** : button role + label interpolé + hint constant. Hit target ≥ 44×44 via padding + hitSlop. WCAG 4.1.3 + 2.5.5.
- **NFR3 (i18n coverage)** : 2 clés × 8 locales = 16 strings ajoutées. Token `{{text}}` présent dans `a11y_label` chaque locale. Pas d'emoji.
- **NFR4 (Security)** : aucun payload telemetry (V1, Q3 deferred). Pas de fuite PII : le texte du follow-up est généré par le LLM, jamais le user-input ; pas de XSS surface (texte rendu via `<Text>` RN, pas WebView).
- **NFR5 (Backward-compat)** : `metadata.suggestedFollowUp === undefined` sur les messages legacy persisted → `<AskMoreChip>` ne rend pas (R22). `followUpQuestions: string[]` reste lu par `<FollowUpButtons>` legacy (NFR8 1-cycle).
- **NFR6 (Schema stability — singularity enforced)** : `suggestedFollowUp: string` (not `string[]`) is a domain invariant. Compile-time + Zod-introspection tests block any future array drift (R3 + AC2).
- **NFR7 (No new dep)** : `package.json` BE + FE inchangé. Ionicons + tokens DS uniquement.
- **NFR8 (1-cycle backward-compat)** : `followUpQuestions` (BE + FE + DB) reste pour 1 cycle (V1 → V1.1). Cleanup PR dédiée post-bake.
- **NFR9 (Doctrine UFR-013)** : la décision BE-led (LLM-generated + factual anchor) traduit la doctrine UFR-013 — pas de pretense visuel ("Tell me more" générique) qui ferait croire à un follow-up intelligent.
- **NFR10 (No emoji)** : Ionicons + tokens DS uniquement.
- **NFR11 (No feature flag)** : pas de `*_ENABLED`. Live or revert (doctrine `feedback_no_feature_flags_prelaunch`).
- **NFR12 (No solo-dev calendar)** : pas d'estimation calendrier dans la spec — frame trade-offs as risk/dependency/scope (doctrine `feedback_no_solo_dev_estimates`).
- **NFR13 (1 chip jamais 3)** : doctrine `project_hybrid_product_philosophy` enforce. Schema `string` (singular), composant accepte 1 string, jamais d'API multi-suggestion. Compile-time + runtime + lexical assertions.

## 7. Verdict

**READY** — B3 introduit un nouveau champ `suggestedFollowUp: string` (singular) côté BE (schema Zod + prompt + parser + types + HTTP contract) et un composant FE `<AskMoreChip>` rendu sous la dernière bubble assistant. La doctrine "JAMAIS 3 boutons — référence un fact précis ou rien" est encodée à 4 niveaux de défense : (1) prompt LLM imposant 1 follow-up factuel ou null, (2) Zod schema `string.max(80).nullable()`, (3) parser `toSuggestedFollowUp` qui drop arrays/longs strings, (4) composant FE qui n'accepte qu'un `text: string` prop. Le scope BE est explicite (changes documentés §0.5 + §2.1), pas un creep caché. Cohabitation 1 cycle avec `followUpQuestions: string[]` legacy (NFR8). Tests rouges couvrent : (a) BE schema + parser + extractMetadata + prompt lexical assertion, (b) FE composant atomique (render, tap, a11y, disabled, slice 80, empty no-op), (c) FE intégration `<MessageActions>` (render conditional, ordre cohabitation, wiring `onFollowUpPress`). Open Q1-Q5 sont des arbitrages secondaires, non bloquants pour démarrer T1.1 (BE schema) ou T3.2 (FE component). Q4 (legacy cleanup) doit être tranchée AVANT V1.1.
