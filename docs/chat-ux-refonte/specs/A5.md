| Question | Verdict |
|---|---|

## 1. Spec — format EARS

### 1.1 Backend — exposition de la phase terminale

- **R1** : When `ChatMessageService.postMessage()` returns successfully, the system SHALL set `result.metadata.phase = 'done'` so that any consumer can verify pipeline completion via the response payload.
- **R2** : When `ChatMessageService.postMessage()` is invoked with an `image` field, the system SHALL ensure the pipeline traversed an `analyzing-image` phase (observed via Langfuse span) before the LLM call. Phase tracking is internal-only — only the terminal `done` (or last-attempted phase on failure) is surfaced in `metadata.phase`.
- **R3** : When `ChatMessageService.postMessage()` is invoked WITHOUT an `image` field, the system SHALL skip the `analyzing-image` phase entirely (no Langfuse span emitted for it).
- **R4** : While the orchestrator is computing the LLM response (after enrichment, before commit), the system SHALL classify this as the `composing` phase (Langfuse span name: `chat.phase.composing`).
- **R5** : When TTS synthesis is invoked (`chat-media.route.ts` path / audio response path), the system SHALL classify the synthesis window as the `synthesizing-voice` phase (Langfuse span name: `chat.phase.synthesizing-voice`).
- **R6** : When enrichment fan-out runs (KB + web search + image enrichment in `enrichAndResolveLocation`), the system SHALL classify this window as the `searching-collection` phase.
- **R7** : The type `ChatPipelinePhase` defined in `museum-backend/src/modules/chat/domain/chat.types.ts` SHALL be the EXACT union `'analyzing-image' | 'searching-collection' | 'composing' | 'synthesizing-voice' | 'done'`. Any change to this union MUST trigger a coordinated update in `museum-frontend/features/chat/application/phases.ts` (i18n keys are derived from this union).
- **R8** : The `metadata.phase` field SHALL be an optional `ChatPipelinePhase` (backward-compat NFR8 — legacy persisted assistant messages have no `phase`, FE MUST tolerate absence).
- **R9** : While Langfuse SDK is unavailable / disabled / throwing, the system SHALL fail-open : pipeline keeps running, no `phase` Langfuse span is emitted, `metadata.phase = 'done'` is still set on success. Cf. pattern `safeTrace` already used in `chat-phase-timer.ts`.

### 1.2 Frontend — surface visuelle

- **R10** : When the chat session enters `isSending = true` (user message just dispatched, awaiting response), the system SHALL render a `<StatusIndicator>` component in place of (or replacing) the legacy `<TypingIndicator>` 3-dots row.
- **R11** : The `<StatusIndicator>` SHALL display ONE of the five phase strings drawn from `chat.status.<phase-key>` i18n keys, in the user's active locale.
- **R12** : Every i18n string under `chat.status.*` SHALL be ≤ 35 characters (incl. terminal ellipsis `…`) in every shipped locale (ar, de, en, es, fr, it, ja, zh).
- **R13** : When the user message includes an image (`hasImage = true` in the send context), the initial displayed phase SHALL be `analyzing-image` ("Looking at the artwork…" in en).
- **R14** : When the user message is text-only, the initial displayed phase SHALL be `searching-collection` ("Searching the collection…" in en).
- **R15** : While the assistant response is still pending after a configurable threshold (`PHASE_TICK_MS = 1200`), the system SHALL advance the displayed phase one step along the canonical sequence: `analyzing-image → searching-collection → composing` (text path skips the first). The displayed phase SHALL NOT advance past `composing` while waiting — it loops/stays on `composing` indefinitely until response arrives.
- **R16** : When the response arrives AND `metadata.suggestedTts === true` (or any future signal that TTS is being prepared client-side via `useAutoTts`), the displayed phase SHALL transition to `synthesizing-voice` until TTS audio is ready or playback starts.
- **R17** : When the response arrives AND no TTS phase is active, the system SHALL unmount the `<StatusIndicator>` (no terminal "Done" string — silence is the success signal).
- **R18** : When `useReducedMotion()` returns `true`, the system SHALL NOT animate the appearance/disappearance of `<StatusIndicator>` — instant mount/unmount, no fade.
- **R19** : The `<StatusIndicator>` container SHALL set `accessibilityLiveRegion="polite"` and `accessibilityRole="text"` so screen readers announce each phase change without interrupting other speech (WCAG 4.1.3).
- **R20** : The `<StatusIndicator>` SHALL set `accessibilityLabel` to the current phase string (e.g. "Searching the collection…") so VoiceOver/TalkBack read the phase rather than the generic decorative content.
- **R21** : The `<StatusIndicator>` SHALL NOT introduce any Unicode emoji in any locale string (doctrine `feedback_no_unicode_emoji` — Ionicons + PNG require only). Terminal ellipsis is the Unicode character `…` (`…`), which is punctuation, not an emoji.

### 1.3 Coupling FE↔BE

- **R22** : When the response payload carries `metadata.phase`, the FE SHALL log it (dev only — `console.debug`, or Sentry breadcrumb in prod) but SHALL NOT use it to drive the displayed phase. The displayed phase is a client-side simulation (per §0.5).
- **R23** : When the response payload omits `metadata.phase` (legacy messages, refusal paths), the FE SHALL NOT throw — the StatusIndicator already unmounts at response time (R17), `metadata.phase` is consumed for telemetry only.

### 1.4 Out-of-scope (explicite)

- ❌ Real-time push of phases via SSE/WS — see §0.5.
- ❌ Per-section LLM phase exposure (we surface ONE coarse `phase`, not the 6-step pipeline). Diagnostics already covers per-section telemetry.
- ❌ Animation morphing between phase strings.
- ❌ User-facing "Done" string — silence is success.
- ❌ Configurable phase strings per museum / per locale beyond i18n.
- ❌ Persistence of phase history per message (Langfuse covers it).

### 2.2 Choix architectural : phase client-side simulée

**Verdict** : state machine pure côté FE alimentée par signaux locaux, BE expose juste la phase terminale `done` en métadonnée pour audit.

**Pourquoi pas live BE→FE** :

| Critère | Live (SSE / WS) | Client simulé (retenu) |
|---|---|---|
| Réveil SSE | Reviens sur ADR-001 + décision user 2026-05-14 | Aucun impact archi BE |
| Cohérence avec doctrine | viole `feedback_no_feature_flags_prelaunch` (faudrait un flag pour basculer) | Cohérent |
| Sentir le temps perçu | -40% (Groovy Web, directional, non peer-reviewed cf. findings.md §7 caveat) | Même gain — l'œil de l'utilisateur voit DES strings qui changent, pas un dot statique. La progression simulée ≠ vraie progression mais reste honnête (chaque phase **peut** correspondre à du travail réel BE) |
| Coût ingénierie | +1 channel, +1 protocole, +tests | ~150 LOC FE pure |
| Risque dérive UX vs reality | Aucun (vérité BE) | Limité — si la phase simulée diverge longuement, c'est typiquement parce que le LLM est lent (`composing` reste affiché), ce qui EST la vérité utile |
| A11y | identique | identique |

**Trade-off accepté** : la phase affichée est une heuristique, pas un état vrai. Honnêteté UFR-013 : c'est documenté dans le code (`useStatusPhase.ts` JSDoc) — la phase est un signal d'attente cosmétique honnête, pas une promesse de vérité instantanée. La phase terminale dans `metadata.phase` (R1) est, elle, la vérité — utilisable pour audit.

### 2.3 État machine FE (pure, testable)

```ts
// phases.ts
export type ChatPipelinePhase =
  | 'analyzing-image'
  | 'searching-collection'
  | 'composing'
  | 'synthesizing-voice'
  | 'done';

export const PHASE_I18N_KEY: Record<ChatPipelinePhase, string> = {
  'analyzing-image': 'chat.status.analyzing-image',
  'searching-collection': 'chat.status.searching-collection',
  composing: 'chat.status.composing',
  'synthesizing-voice': 'chat.status.synthesizing-voice',
  done: '', // never rendered — silence is success
};

// useStatusPhase.ts
export interface UseStatusPhaseInput {
  readonly isSending: boolean;
  readonly hasImage: boolean;
  readonly ttsPending?: boolean;
  /** Test seam : override the 1200 ms tick. */
  readonly tickMs?: number;
}

export interface UseStatusPhaseResult {
  readonly phase: ChatPipelinePhase | null;
}

export function useStatusPhase(input: UseStatusPhaseInput): UseStatusPhaseResult;
```

**Reducer rules** (pure, sans React, testable Node test runner) :

```
state.phase: ChatPipelinePhase | null

event START(hasImage)            → phase = hasImage ? 'analyzing-image' : 'searching-collection'
event TICK   (phase, hasImage)   → next phase along sequence, stay on 'composing' once reached
event RESPONSE_READY (ttsPending)→ phase = ttsPending ? 'synthesizing-voice' : null
event TTS_DONE                   → phase = null
event RESET                      → phase = null
```

- Pas de re-render FlashList : `<StatusIndicator>` est rendu en `ListFooterComponent` (comme `<TypingIndicator>` actuel). Les re-renders se limitent au footer.
- Aucune dépendance tierce ajoutée (`package.json` inchangé). Aucun Reanimated worklet.

### 2.6 A11y
