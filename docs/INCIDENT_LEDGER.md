# Incident Ledger — bugs échappés en aval des gates

> Registre des bugs qui ont atteint dev / main / TestFlight / prod **malgré** les gates shift-left.
> Vue humaine ; la vue machine est `team-knowledge/lessons/` (`post-complete-lesson-capture.sh`), cross-référencée par `INC-id`.
>
> **Pourquoi ce registre** — industrialiser le réflexe réactif. Aujourd'hui, après un incident, on improvise un sentinel. La colonne **`Tier-qui-l'aurait-pris`** force le post-mortem à répondre à une question mécanique : *« à quel niveau de la pyramide de test ce bug devient-il visible ? »*. Quand un run `/team` corrige un `INC-id`, `pre-complete-incident-regression-check.sh` (Gate D) exige que le `test-contract.md` du fix contienne un UC `Catégorie: regression` lié à cet INC-id, **avec un `Tier` ≥ celui de cette colonne**. On ne peut pas « corriger » un bug pg-réel avec un unit mocké.
>
> **Comment ajouter une ligne** — au post-mortem d'un bug échappé : choisir un `INC-id` (`INC-YYYY-MM-DD-slug`), décrire le symptôme, le niveau atteint, et surtout le `Tier-qui-l'aurait-pris` (le niveau de test minimal qui aurait rendu le bug visible AVANT le merge). `UC-régression` et `Fix commit` se remplissent quand le fix passe par `/team`.

> **INC-2026-06-14 — résidu storage : RÉSOLU (root cause trouvée + fix vérifié end-to-end).** Le fix `skipStorage` retire le 500 du *compare* (qui n'avait jamais besoin de S3). Le 403 OVH n'était PAS un problème de creds (bons, testés à l'aws-cli) ni un redéploiement : c'était un **bug de signature SigV4** dans `s3-operations.ts` — l'`Authorization` header joignait l'algorithme aux params avec une **virgule** (`AWS4-HMAC-SHA256, Credential=…`) au lieu d'un **espace** (`AWS4-HMAC-SHA256 Credential=…`). AWS S3 tolère, le parser strict d'OVH/Swift **rejette** → tous les writes object-store (images chat + audio TTS) 403aient avec le message trompeur « AWS authentication requires a valid Date or x-amz-date header » (d'où des heures de fausses pistes config/clock/path-vs-virtual). Reproduit + fixé (les 2 constructions auth, `s3-operations.ts:160,217`) + **vérifié de bout en bout en local avec les vrais creds** : `chat image (real S3 upload) OK (status=201)`. Garde-fous : (a) `smoke-api.cjs` envoie maintenant un message chat-image → upload S3 réel → un 500 fait échouer le deploy (plus de vert vacant) ; (b) l'assertion auth du test `image-storage.s3.test.ts` renforcée — l'ancien `.toContain('AWS4-HMAC-SHA256')` passait pour les DEUX formes (virgule comme espace) → maintenant prefix exact `/^AWS4-HMAC-SHA256 Credential=/` + absence de la forme virgule. **Tier-qui-l'aurait-pris :** e2e/smoke contre le VRAI OVH (un `http.server` local accepte l'auth malformée — seul le parser OVH la rejette), OU l'assertion unit exacte (désormais en place). Catégorie : regression. Fix commit : (ce commit).
