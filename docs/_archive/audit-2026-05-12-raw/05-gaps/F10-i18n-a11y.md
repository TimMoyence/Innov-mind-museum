## 1. Mobile i18n ÔÇö full report

### Setup

- Library : `i18next ^26.0.6` + `react-i18next ^17.0.4` + `intl-pluralrules ^2.0.1` + `expo-localization ~55.0.13`.
- Bootstrap : `museum-frontend/shared/i18n/i18n.ts:1-37` ÔÇö eight imports, `lng: 'en'`, `fallbackLng: 'en'`, `escapeValue: false`, `useSuspense: false`.
- Locale switching : `I18nContext.tsx:71-99` ÔÇö `setLanguage(lang)` checks `needsRTLReload(curr, next)`, calls `applyRTLLayout()`, then `Updates.reloadAsync()` in prod (or in-session swap in `__DEV__`).

## 7. EAA scope & compliance posture

### Is Musaium in scope?

**Yes, B2C is in scope.** Per EAA Art. 2(2)(c) :

> "consumer banking services ; e-books and dedicated software ; **e-commerce services**"

Plus Art. 2(2)(e) :

> "passenger transport services"

ÔÇŽand Annex I, Section IV defines "**website and mobile-device-based services**" as included. Musaium B2C voice-first companion + freemium = consumer service + mobile app. Once the B2B-museum revenue line opens (per `feedback_no_feature_flags_prelaunch.md` doctrine inversion), additional B2B accessibility obligations layer in via the Web Accessibility Directive (WAD) if a public-sector museum is a customer.

### Microenterprise exemption?

Defined : **< 10 employees AND (turnover < ÔéČ2 M OR balance sheet < ÔéČ2 M)**. Musaium today (pre-launch, sole-dev per project notes) **could** technically qualify, but :

1. The exemption applies only to **services** (Musaium-mobile is a service). Products are never exempt.
2. The defense weakens fast post-revenue. The CNIL / French ARCOM equivalents may still pursue based on "consumer-facing app" arguments.
3. **Building the exemption defense is more work than just complying.** Recommend conformance, not exemption.

### Disproportionate burden?

Available under EAA Art. 14 ÔÇö requires **documented cost-benefit analysis** per accessibility requirement, reviewed every 5 years. For a pre-launch app the documentation burden Ôëł implementing the feature. Not a viable shortcut.

### Penalty exposure

Per WebSearch findings :

- **France** (Musaium primary market) : ÔéČ1 500 / individual offence, ÔéČ7 500 / legal entity, doubling on repeat, aggregate ceiling **ÔéČ250 000**. Separate **ÔéČ25 000 / year** for failing to publish an accessibility statement. **First lawsuits filed November 2025** against Auchan / Carrefour / Leclerc / Picard.
- **Germany** : Bundesnetzagentur fines **up to ÔéČ100 000 / violation** under BFSG (Barrierefreiheitsst├Ąrkungsgesetz). Early enforcement targets e-commerce + banking apps. Private warning letters started flowing within weeks of June 2025.
- **EU-wide** : Member-state-set, generally proportional, repeat / systemic = aggregate ceilings ÔéČ250 K ÔÇô ÔéČ500 K.

### Required artefacts

| Artefact | EAA mandate | Musaium status |
|---|---|---|
| Public accessibility statement (per service) | Annex V + Art. 13 | **MISSING** ÔÇö confirmed in R22 ┬ž12 table |
| Conformance level declaration (full / partial / non) | Annex V | Missing |
| Limitations + alternatives explicitly listed | Annex V | Missing |
| Assessment method + date | Annex V | Missing |
| Contact / complaint mechanism | Art. 11 + Annex V | Email exists but not labeled for a11y use |
| Version history of statement | Implicit "ongoing conformity" Art. 13(2) | Missing |

### Verdict ÔÇö EAA : **NON-CONFORMANT**

Musaium is in scope, has no accessibility statement, has demonstrable gaps (MFA hard-coded EN, untested RTL, partial WCAG coverage). Risk = mise en demeure by French disability associations (APF France handicap files publicly) leading to public reputational damage > formal fine.

## 8. Keyboard navigation, screen reader, contrast

### Keyboard nav (web)

- Tab order : not explicitly audited. Next.js default flows top-bottom DOM order, generally OK on landing.
- `:focus-visible` rings : Tailwind 4 defaults ÔÇö present but contrast not verified against amber-500 / primary-700 backgrounds used in MFA section (`amber-50/amber-900` pair = approx 7:1, OK ; `amber-500 / white` button text Ôëł 2.5:1 = **FAIL** vs 4.5:1 target).
- Skip link : `Header.tsx` includes one ÔÇö `aria-label="Skip to main content"` style verified in code (line not pinned here).

### Screen reader

- Web : axe-core covers most ARIA correctness for the 6 tested routes.
- Mobile : VoiceOver / TalkBack readout depends on `accessibilityLabel`. Coverage is 58 % element-level. **High-leverage gaps** :
  - `MessageActions.tsx` (chat message context menu) ÔÇö 4 occurrences of accessibility props, but row has 5+ icons. Some unlabeled.
  - `OfflineMapsSettings.tsx` ÔÇö 1 a11y prop only.
  - `DataModeSettingsSection.tsx` ÔÇö 2 a11y props for what looks like a toggle group.
- TTS audio output (`ChatMessage.audioUrl`) **is announced via screen reader** through the play button ÔÇö that's an accessibility win (see ┬ž9).

### Color contrast

- Design system primary colors not exported to a contrast-audit script. Recommend running `pa11y` or `axe-cli` on the live preview before launch.
- Amber-500 / white in MFA "Copy all" button (`mfa/page.tsx:127`) Ôëł **2.45:1** ÔÇö fails WCAG 1.4.3 (4.5:1 normal text, AA).
- Other observed pairs (primary-700 / white on buttons) Ôëł 8:1 ÔÇö OK.

### Quick wins (ÔëĄ 1 day each)

1. Add `role="img" aria-label={t('admin.mfa.qrAriaLabel')}` to MFA QR `<div dangerouslySetInnerHTML>`.
2. Change MFA "Copy all" button bg from `amber-500` to `amber-700` (raises contrast > 4.5:1).
3. Add `accessibilityLabel` to every `IconButton` in `MessageActions.tsx`, `ChatHeader.tsx`, `OfflineMapsSettings.tsx`.
4. Bump `buttonTokens.sm` minimum height to 24 px (mobile) / add CSS `min-height: 1.5rem` (web).
5. Add `writingDirection: 'auto'` to all user-content `<Text>` (chat bubbles, names, MFA secret display).

## 9. Voice as accessibility feature

Per project doctrine (CLAUDE.md "Voice V1 (2026-04)") :

- Pipeline always-on : STT (`gpt-4o-mini-transcribe`) Ôćĺ LLM Ôćĺ TTS (`gpt-4o-mini-tts`, voice `alloy`).
- Audio MP3 persisted on `ChatMessage.audioUrl` (S3) ÔÇö replayable across sessions.
- Guardrails applied to both transcripts and LLM output before TTS.

### EAA narrative value

- Direct benefit for **low-vision** users (cannot read small text but can listen).
- Direct benefit for **dyslexic / cognitive-disability** users (audio narration of artwork descriptions).
- Direct benefit for **temporary disabilities** (eyes-on-art, hands-on-phone-not-eyes-on-screen ÔÇö the design audience).

This **is** a legitimate accessibility benefit and **must be explicitly called out** in the EAA accessibility statement under "Equivalent functionality through alternative means".

### What voice does NOT solve

- Does not replace screen-reader semantic markup (a blind user navigating the UI itself still needs `accessibilityLabel` on every button).
- Does not help deaf users ÔÇö **no captioning / transcript surfacing for TTS output**. Currently the LLM text is displayed alongside the audio (good), but there is no explicit "read transcript" affordance separate from chat history. Verify pre-launch.
- Does not help motor-impaired users (still need keyboard / switch-control compat).

### Verdict ÔÇö Voice a11y : **NET POSITIVE**

Document this prominently in the accessibility statement. Estimate +1 conformance argument for EAA Annex I ┬ž c ("provided in more than one sensory channel").
