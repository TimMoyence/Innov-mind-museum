## P0.C8 — Multi-tenant leak `/api/admin/analytics/*` + `/api/admin/stats`

### VERDICT: ⚠️ PARTIAL — RBAC/BOLA closed on `/stats`; aggregate leak on `/stats` is **REAL, documented, and untested**; `/analytics/*` are NOT multi-tenant at all (admin-only, no museumId).

### Findings (verified)

**1. RBAC entry gate — `/stats` now admits museum_manager with forced scope (CLOSED).**
- `museum-backend/src/modules/admin/adapters/primary/http/routes/admin.route.ts:246-266`.
- Line 249: `requireRole('admin', 'moderator', 'museum_manager')` — museum_manager is in the list (was admin/moderator only per the audit's original text).
- Lines 256-259: forced scope — `if (req.user?.role === 'museum_manager') scopedMuseumId = req.user.museumId ?? undefined;` → JWT claim wins over `?museumId=`. A crafted `?museumId=99` cannot smuggle another tenant's intent (BOLA negative guard).
- `requireRole` (`museum-backend/src/shared/middleware/require-role.middleware.ts:11-31`): super_admin implicit escalation (line 22), else 403 (line 27). Correct.
- JWT carries `museumId`: `museum-backend/src/shared/middleware/authenticated.middleware.ts:84` (`req.user = { id, role, museumId }`), `token-jwt.service.ts:106`/`123`. `UserRole.MUSEUM_MANAGER = 'museum_manager'` exists (`museum-backend/src/modules/auth/domain/user/user-role.ts:18`).
- `statsQuerySchema` = `z.strictObject({ museumId: z.coerce.number().int().positive().optional() })` (`admin.schemas.ts:103-105`) → extra param → 400 (via `validate-query.middleware.ts:16-17` `safeParse`→`badRequest`).

**2. Aggregate leak — `getStats.useCase.ts` is a documented NO-OP. The forced `scopedMuseumId` is THREADED but DISCARDED.** ← the unresolved half of C8.
- `museum-backend/src/modules/admin/useCase/analytics/getStats.useCase.ts:24-32`: `execute(_input)` ignores `_input.museumId` and calls `this.repository.getStats()` with no scope. Comment lines 25-30 explicitly state "the aggregate returned is the global cross-tenant snapshot until users/sessions/messages gain museum_id columns".
- `museum-backend/src/modules/admin/adapters/secondary/pg/admin.repository.pg.ts:213-257` `getStats()`: zero `WHERE museum_id`. Returns **platform-wide** `totalUsers`, `usersByRole` (ALL roles, e.g. counts of other admins/super_admins), `totalSessions`, `totalMessages`, `recentSignups`, `recentSessions`.
- **CONSEQUENCE**: a `museum_manager` calling `GET /api/admin/stats` gets `200` + the **entire platform's** user/role/session/message aggregate. The dashboard (`museum-web/src/app/[locale]/admin/page.tsx:50` fetches `/api/admin/stats`) renders these numbers to the museum operator. Real cross-tenant aggregate disclosure (OWASP API3-adjacent, GDPR aggregate-exposure). Severity = LOW-MED at V1 (demo museums only, no contracted B2B tenant per CLAUDE.md; numbers are platform totals, not another tenant's PII rows) but the roadmap "✅ faille fermée" half-claim is **misleading without the no-op caveat** — which `:139` does carry. So the roadmap text is honest.

**3. `/analytics/usage|content|engagement` are NOT multi-tenant — admin-only, no museumId param.**
- Routes `admin.route.ts:323-368`: all three are `requireRole('admin')` → museum_manager + moderator get **403** (only super_admin escalates).
- Schemas `usageAnalyticsQuerySchema`/`contentAnalyticsQuerySchema`/`engagementAnalyticsQuerySchema` are `z.strictObject` and do **NOT** include `museumId` (`admin.schemas.ts:63-79`).
- Query impls `queryUsageAnalytics`/`queryContentAnalytics`/`queryEngagementAnalytics` (`admin-analytics-queries.ts:44-309`) never accept/read a museumId — global aggregates only.
- The original audit text ("3 endpoints + `/api/admin/stats` SANS museumId scoping") conflated `/stats` with `/analytics/*`. Reality: `/stats` got the museumId plumbing (no-op at repo); `/analytics/*` got nothing and are admin-locked. Roadmap `:139` only references `/stats` for the no-op — consistent.

**4. FE/BE divergence in the analytics page (NEW finding — beyond the leaf brief, flagged for honesty).**
- `museum-web/src/app/[locale]/admin/analytics/page.tsx:108-112` defines `withMuseumScope(path)` that appends `?museumId=<id>` when a museum is selected, then applies it to all 3 analytics calls (lines 125-128, 178).
- But the BE analytics schemas are `z.strictObject` WITHOUT `museumId` → selecting a museum makes all 3 analytics calls **400** even for an `admin`. The museum-selector feature on the analytics page is **broken-by-construction** (dead UI). Not a security leak — a functional bug / FE-BE contract drift (W2.3 "stats per-museum" claim, ROADMAP `:184`).

### C8 DEBT
- **TD-C8-1 (open, roadmap-tracked)**: `getStats` no-op — needs `museum_id` on users/chat_sessions/chat_messages OR descope. ROADMAP `:139` effort 3-4h. Untested: `analytics-scope.test.ts` mocks the useCase so the leak is invisible to CI.
- **TD-C8-2 (NEW)**: analytics page `withMuseumScope` → 400 (strictObject rejects `museumId`); either add `museumId` to the 3 analytics schemas + thread to queries, or remove the museum selector from `analytics/page.tsx`. FE feature shipped against a BE contract that rejects it.
