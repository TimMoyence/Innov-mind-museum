The column is effectively a **no-op for the multi-tenant guarantee the roadmap claims** (per-museum NPS endpoint + tenant-scoped moderation both absent). **What is BROKEN / no-op (the core failure):**
- **Admin READ (reviews) unscoped** — `GET /admin/reviews` (`admin.route.ts:419-435`) → `adminReviewFacade.list` (pure pass-through `admin-review.facade.ts:17-19`) → `ListAllReviewsUseCase` (`listAllReviews.useCase.ts:30-35`): builds `filters` with **only `status` + pagination, no `museumId`**. Any admin/moderator sees ALL tenants' reviews. The repo's filter capability is never exercised here.
- **Admin UPDATE (reviews) unscoped** — `PATCH /admin/reviews/:id` (`admin.route.ts:437-462`) → `ModerateReviewUseCase` (`moderateReview.useCase.ts:59-72`) → `review.repository.pg.ts:114-123` `moderateReview` updates by `reviewId` only. **No tenant guard** — a museum-A moderator could moderate museum-B's review. `ModerateReviewUseCaseInput` has no `museumId` field at all.
- **Admin READ (tickets) unscoped** — `admin.route.ts` support list → `adminSupportFacade.list` (`admin-support.facade.ts:17-19`) → `ListAllTicketsUseCase` (`listAllTickets.useCase.ts:35-39`): filters = status + priority + pagination, **no museumId**. `ListAllTicketsInput` has no museumId field.
- **Admin UPDATE (tickets) unscoped** — `UpdateTicketStatusUseCase` (`updateTicketStatus.useCase.ts:37-42`) → `support.repository.pg.ts:166-180` `updateTicket` updates by `ticketId` only. **No tenant guard.** `UpdateTicketStatusInput` has no museumId field.
- **Public READ (reviews) unscoped** — `GET /reviews` (`review.route.ts:74-87`) → `listApprovedReviewsUseCase` passes only page/limit. Global list, no per-museum option. (Arguably acceptable for a public global catalog view, but it means no public per-museum reviews page exists.)
- **Per-museum NPS endpoint = DEAD CODE** — `aggregateNps` (`review.repository.pg.ts:88-107`) + `findByMuseum` (`:71-80`) are defined on `IReviewRepository` (`review.repository.interface.ts:35,42`) but called by **NOBODY**. Grep across `museum-backend/src` for `aggregateNps`/`findByMuseum` returns zero non-repo, non-test callers. No route, no facade, no useCase invokes them. The "KR2 NPS per-museum" the roadmap says C7 unblocks has **no API surface**.
- **museumId never assigned to a user** — grep found NO write path that sets `users.museum_id` (no admin "assign manager to museum" route). The C8 admin-stats route enforces a museum_manager scope from the JWT claim, but nothing populates that claim. Net effect for V1: every review/ticket persists `museum_id = NULL`, so even the working write-path scope is inert until a user-provisioning path exists.

**Honest reconciliation with roadmap text:** the original audit offered a binary — "scope ALL read+update paths, OR descope KR2 to NPS global." Neither was fully done. Write paths are scoped (1/2 of the requirement), but the read+update admin paths (the W3.3 moderation surface the line names) are NOT, and KR2 NPS-per-museum was NOT descoped (the aggregate is coded but unwired = worse than descoping, it's dead code masquerading as a feature). The ✅ DONE-DEV on `ROADMAP_PRODUCT.md:162` ("C7 ✅ DONE-DEV") is **over-claimed**; line 162's sibling C8 already self-flags "stats museum_id = no-op documented" — C7 deserves the same ⚠️ no-op qualifier on the read+update + NPS claims.

**Debt (C7):**
- D-C7-1 (CORRECTNESS / multi-tenant, but **LOW V1 severity**): admin moderation (reviews + tickets) is cross-tenant unscoped. Real BOLA-class gap IF B2B museum_managers ever get moderation access. For V1 (B2C-only, museum_manager unprovisioned, all rows NULL), not exploitable today. Track as "wire `museumId` into ListAllReviews/ListAllTickets/ModerateReview/UpdateTicketStatus inputs + force-scope for museum_manager role (mirror C8 admin-stats pattern at `admin.route.ts:237-254`)."
- D-C7-2 (DEAD CODE / honesty): `aggregateNps` + `findByMuseum` unwired. Either wire a `GET /museums/:id/nps` (and per-museum reviews) endpoint, or bury them (UFR-016) and mark KR2 NPS as global-only. Currently they inflate the apparent completeness of C7.
