### REAL GAP — `artwork_matches` rows not exported
`artwork_matches` (`artworkMatch.entity.ts`) is a per-message OneToMany (`chatMessage.entity.ts:80-84`) holding artworkId/title/artist/confidence/source/room — i.e. **which artworks the user's uploaded photos were AI-matched to**. Verified persisted as a *separate table row*, NOT denormalized into message `metadata` (`chat.repository.typeorm.ts:170-182`; metadata = caller-passed `input.metadata`, loose-record schema). The DSAR message projection exports `metadata` but NOT the `artworkMatches` relation → these rows are **personal data about the subject that is absent from the Art.15 export**. Not listed in the roadmap B3 remediation. Severity: LOW-MODERATE (AI-derived content tied to the user's photos; an argued-borderline "personal data" but defensibly inside Art.15 scope). **Debt: TD — add artworkMatches to DSAR export (port + message-level projection), or document explicit exclusion rationale.**

### Minor scoping notes (defensible, not hard gaps)
1. Audit export = `actorId` only. Rows where the subject is the `targetId` of an admin action (e.g. admin-suspend) are personal data about the subject but excluded. Defensible (avoids leaking admin identity); a stricter Art.15 reading would include target-scoped rows with admin PII redacted.
2. User columns omitted: `sessionsMonthCount` / `sessionsMonthStart` (quota usage counters), `mfa_enrollment_deadline`. User-attributable but derived/operational; minor.

**Verdict: ⚠️ — the 8 categories the roadmap claimed were added ARE all present and correct; the roadmap's enumeration itself missed `artwork_matches`.**
