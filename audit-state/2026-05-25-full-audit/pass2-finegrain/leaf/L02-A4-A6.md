The vision path is entirely untested.

**VERDICT A4: PARTIAL — string-path done & wired; vision/multimodal array `content` PII leaks to Langfuse. Residual P0 if `LANGFUSE_ENABLED=true` in prod (default false → not active today, but the in-museum photo flow is the core V1 UX, so any prod-enable ships the leak).**

### ⚠️ DIVERGENCE — HALF_OPEN recovery WIPES `dailySpend` (CONFIRMED, by-design but hazardous & untested)
The auditor's specific concern is **correct**: recovery resets the daily accumulator.

Trace:
- `LlmCostCircuitBreaker.recordCharge` (`llm-cost-circuit-breaker.ts:136-150`): checks `shouldTrip` FIRST (line 142) — this correctly KEEPS the breaker OPEN while still over cap (good: a daily-cap trip won't recover until spend drops below cap). Then HALF_OPEN→CLOSED via `circuit.recordOutcome('success')` (line 148).
- `ThreeStateCircuit.recordOutcome('success')` in HALF_OPEN (`three-state-circuit.ts:128-135`): transitions CLOSED **and calls `this.strategy.reset()`** (line 131).
- `CostTripStrategy.reset()` (`cost-trip-strategy.ts:63-66`): wipes BOTH `hourlyWindow = []` AND `dailySpend = { day:'', cents:0 }`.

**Exploit window — HOURLY-trip recovery zeroes the DAILY cap mid-day:**
1. Hourly spike trips breaker (e.g. $50 within 1h) while true cumulative day-spend is, say, $300/$500.
2. After cooldown + 1h, hourly window prunes below threshold → HALF_OPEN.
3. Probe succeeds (small charge), `shouldTrip` false → `recordOutcome('success')` → `strategy.reset()` → **`dailySpend` zeroed to 0**.
4. The $500/day hard cap (the very thing meant to stop the "$437/night retry loop" cited in the roadmap) now needs a FRESH $500 to trip again, same UTC day. Repeated hourly-trip/recovery cycles make the daily cap effectively unbounded.

This is INTENTIONAL per the FSM contract (interface doc `three-state-circuit.ts:33-34` "called on reset() and probe success") and the breaker docstring ("Recovers HALF_OPEN → CLOSED" at `:135`). But the daily-cap erosion side-effect is **not flagged, not documented as a tradeoff, and not tested**.

- **Test coverage gap (confirmed)**: `tests/unit/chat/llm-cost-circuit-breaker.test.ts:121-137` "HALF_OPEN probe success → CLOSED" deliberately uses `dailyBudgetCents: 100_000` (huge) and a comment (lines 122-124) explicitly engineering AROUND the daily dimension. No test asserts daily spend survives an hourly-trip recovery. `three-state-circuit.test.ts:115-136` only asserts `strategy.reset()` IS called on recovery (i.e., it locks IN the wipe behaviour as correct).
- Tests run live: `pnpm test -- --testPathPattern="...strip-free-text|cost-trip-strategy|three-state-circuit"` → 3 suites, 29 tests PASS (llm-cost-circuit-breaker.test.ts not matched by the jest project filter, but reviewed statically). Green confirms current behaviour, not absence of the hazard.

**VERDICT A6: DONE (enforcement) — `canAttempt()` wired correctly BEFORE invokeSection, fail-CLOSED, both default+walk paths; marker's old "never called/telemetry only" finding is superseded. BUT a latent daily-cap-integrity hazard exists: HALF_OPEN recovery (incl. from hourly trips) calls `strategy.reset()` which wipes `dailySpend`, eroding the $500/day hard cap within a UTC day.**

**Debt (new)**: `CostTripStrategy.reset()` should NOT zero the daily UTC counter on recovery — only the hourly window should clear (the hourly window is the trip cause being probed; the daily budget is an independent same-day invariant). Options: (a) split `reset()` into `resetHourly()` (recovery) vs `resetAll()` (UTC-day rollover / explicit `reset()`); or (b) make the FSM recovery call a strategy `onRecover()` hook that the cost strategy maps to hourly-only clear. Add a test: "daily spend survives an hourly-trip HALF_OPEN recovery within the same UTC day."
