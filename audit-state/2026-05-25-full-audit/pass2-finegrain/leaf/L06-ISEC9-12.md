## I-SEC10 — KE scraper Content-Length 2 couches → **VERDICT : ✅ DONE, vérifié**

Marqueur roadmap : `✅ DONE-DEV (#293)`, preuve `html-scraper.ts:188`.

Re-dérivation — `html-scraper.ts`, fn `readBodyWithCap` (L182-243) :
- **Layer 1 — pre-fetch Content-Length** : L188-199. `response.headers.get('content-length')` → `Number.parseInt` → si `Number.isFinite(declared) && declared > maxContentBytes` → `logger.warn('scraper_payload_too_large')` + `return { ok: false }`. Rejet AVANT consommation du body.
- **Layer 2 — streamed cumulative cap** : L213-233. `body.getReader()` + boucle ; `bytesRead += value.byteLength` (L222) ; si `bytesRead > maxContentBytes` (L223) → `logger.warn('scraper_payload_streamed_overflow')` + `await reader.cancel()` (L229) + `return { ok: false }`. Abort réel du stream, RAM bornée O(maxContentBytes).
- Câblé dans `scrape()` : L379 `readBodyWithCap(response, url, this.config.maxContentBytes)` ; L380 `if (!bodyResult.ok) return null;` avant `extractContent`.

Defense-in-depth réelle : serveur malveillant peut mentir sur Content-Length (bypass L1) mais pas L2. Docstring L168-179 documente exactement ce design (D8).
Bonus hors-scope mais présent : SSRF guard complet (private IP, IPv4-mapped IPv6 hex+dec, redirect re-validation L250-287, blocked metadata hostnames) — robuste.

**Divergence vs marqueur** : aucune. Les 2 couches existent et sont câblées. Marqueur EXACT (preuve L188 = Layer 1 ; Layer 2 = L223).
