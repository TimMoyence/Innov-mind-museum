## P0.B6 — BE consent enforced at LLM call site — VERDICT : ✅ CONFIRMÉ enforcement réel ; ⚠️ doc-claim sur le mécanisme de surfacing FAUX ; ⚠️ 2 scopes (`profile`) décoratifs par design

**Énoncé roadmap** (L112) : DONE-DEV `71f103b35` #294. Texte d'origine conservé : *"seul `location_to_llm` gated ; 8 third_party_ai scopes NOT enforced ; Art.7(3) violation symbolique. Vérifié résolu 2026-05-25 (D3)."*

**Réalité code — enforcement** :
- Gate : `consent-gate.ts:55` `checkThirdPartyAiConsent` → appelé par `prepare-message.pipeline.ts:245` (`runConsentGate`) → invoqué `prepare-message.pipeline.ts:268`, **AVANT** `ensureSessionAccess` (271), image processing (274), guardrail (283), et tout appel LLM. Bonne position (Art. 5(1)(c) — refus ne persiste rien, ne touche pas Redis/BullMQ).
- Fail-CLOSED anon : `third-party-ai-consent-checker.ts:42-44` (`userId` nullish → `false` sans toucher le repo).
- Checker injecté en prod : `chat.service.ts:695` `buildThirdPartyAiConsentChecker()` → passé pipeline `chat.service.ts:723,834`. **Pas** seulement un seam de test.
- Repo réel : `consent-checker.ts:50-51` lazy-import `userConsentRepository.isGranted(userId, scope)`.

**Quels scopes RÉELLEMENT checkés (le cœur de la question)** :
- 12 scopes au total (`userConsent.entity.ts:24-37`), dont **8** `third_party_ai_*` = 4 catégories (text/image/audio/profile) × 2 providers (openai/google).
- Le gate ne check QUE les catégories présentes dans le turn (`consent-gate.ts:62-69`) : `text` si `input.text` non vide, `image` si `input.image`. AND-intersection (refuse si l'un manque).
- La catégorie est résolue vers **le scope du provider ACTIF** via `resolveActiveProviderForScope` (`provider-resolver.ts:46`) : `LLM_PROVIDER=google` → `..._google`, sinon `..._openai`. DeepSeek/inconnu → fallback `openai`.
- **Conséquence** : à tout instant, au plus **1 scope text + 1 scope image** (du provider actif) sont enforced sur le chat pipeline ; les 4 scopes de l'autre provider sont inertes (jamais évalués tant que `LLM_PROVIDER` ne switch pas). C'est un design correct (mono-provider runtime), PAS un bug — mais ce n'est PAS "8 scopes enforced".
- **`audio`** : enforced ailleurs (route, cf. B7), PAS dans ce gate (`consent-gate.ts` ne push jamais `'audio'` — pas de champ audio dans le pipeline texte).
- **`profile` (`third_party_ai_profile_openai` / `_google`) : JAMAIS enforced nulle part.** Grep : `profile` n'apparaît que dans `consent-audit-mapping.ts:14,30` (enregistrement/audit du consentement) et l'entity. Aucun call-site `isGranted(..., 'third_party_ai_profile_*')`. `provider-resolver.DispatchChannel` = `'text'|'image'|'audio'` seulement — pas de canal `profile`. **2 scopes purement décoratifs** (collectés/auditables, jamais checkés à un point de données).

**Divergence doc CRITIQUE (honnêteté)** :
- `consent-gate.ts:23` et `:88` + `consent-checker.ts:15-16` affirment que `chat.service.ts` wrappe le refus en **`AppError({code: 'CONSENT_REQUIRED', scope})`** vers HTTP. **FAUX vérifié** : `grep CONSENT_REQUIRED museum-backend/src` → 0 hit dans `chat.service.ts`. Le vrai chemin : `chat-message.service.ts:245` `if (prep.kind === 'refused') return prep.result;` → renvoie la **bulle assistant de refus** (`consent-gate.ts:108-122`, `id: consent_refusal::<scope>`, texte localisé FR/EN), pas une erreur HTTP. Donc côté client le turn "réussit" avec un message de refus, statut 201, PAS un 403/422. Le commentaire décrivant `AppError CONSENT_REQUIRED` est stale (probablement une intention de design abandonnée). Enforcement OK, **doc trompeuse**.

**Debt** :
- DOC-B6-A (modéré, honnêteté UFR-013) — réécrire commentaires `consent-gate.ts:21-31,84-90` + `consent-checker.ts:14-17` : le surfacing chat = refusal-bubble via `prep.result`, PAS `AppError({code:'CONSENT_REQUIRED'})`. (Le 403 `CONSENT_REQUIRED` n'existe QUE sur la route audio, cf. B7.)
- DOC/GDPR-B6-B (à trancher) — `third_party_ai_profile_*` collectés mais non-enforced. Soit (a) il existe un flux "profil envoyé au LLM" non gardé (à auditer — userMemory/personnalisation ?), soit (b) le scope est mort → décider : enforcer au point d'usage OU retirer du set collecté (sinon consentement décoratif = signal de conformité-théâtre). Roadmap dit "8 scopes enforced" → **inexact**, à corriger en "scopes text+image du provider actif enforced ; audio via route ; profile non-enforced".

## P0.B7 — Audio consent sur POST /sessions/:id/audio + ⚠️ endpoint TTS — VERDICT : ✅ audio upload gated ; ❌ TTS NON GATED (gap réel)

**Énoncé roadmap** (L113) : DONE-DEV `71f103b35` #294. *"Audio consent enforced sur POST /sessions/:id/audio — ThirdPartyAiConsentChecker câblé chat-media.route.ts:42,224. Vérifié résolu 2026-05-25 (D3)."*

- `chat-media.route.ts:66-71` : `resolveActiveProviderForScope('audio')` → `third_party_ai_audio_openai` (audio hard-pinné openai, `provider-resolver.ts:51-52`), `consentChecker.isGranted(currentUser?.id, audioScope)` ; deny → **`res.status(403).json({error:'consent_required', scope})`** AVANT `postAudioMessage` (donc avant Whisper STT). - Checker injecté par défaut : `chat-media.route.ts:224` `consentChecker = buildThirdPartyAiConsentChecker()`. Route montée `chat-media.route.ts:241-252`.

**Réalité code — TTS (le point ⚠️ demandé)** : ❌ **NON CONSENT-GATED**
- Endpoint `POST /messages/:messageId/tts` : `chat-media.route.ts:271-279` → middlewares `isAuthenticated, userLimiter, sessionLimiter, llmCostGuard, createTtsHandler(chatService)`. **`consentChecker` n'est PAS passé** à `createTtsHandler` et aucun `isGranted` n'y est appelé.

**Divergence** : la fermeture B7 ne couvre QUE l'upload audio (STT entrant). Le **TTS sortant** (synthèse vocale, qui POST du texte à OpenAI) **bypass le consent gate**. Asymétrie : refuser le consent audio bloque la transcription mais PAS la lecture vocale TTS. Même scope logique (`third_party_ai_audio_openai`), un côté gardé, l'autre non.

**Debt** : **GDPR-B7-TTS (P1, gap réel non couvert)** — gater `createTtsHandler` / `synthesizeSpeech` sur `third_party_ai_audio_openai` (réutiliser `consentChecker.isGranted` + 403 `consent_required`, miroir `createAudioHandler:66-71`). One-handler fix côté route. Sans ça, claim "audio consent enforced" est partielle.
