## P0.B9 — `location_to_llm` consent — VERDICT: ⚠️ PARTIAL — FE present, BE consent gate has a RAW-COORDS BYPASS

### Roadmap claim (ROADMAP_PRODUCT.md:115)
`location_to_llm` scope present in FE `THIRD_PARTY_AI_SCOPES` (`consentScopes.ts:38`) + surfaced in `SettingsAiConsentCard`/`AiConsentSheetContent`. Original audit text: BE returns false → location SILENTLY DROPPED. Marked ✅.

### FE — CONFIRMED PRESENT
- `museum-frontend/features/chat/domain/consentScopes.ts:38` — `'location_to_llm'` is the 9th entry of `THIRD_PARTY_AI_SCOPES`. (File renamed from `thirdPartyAiConsent.ts`, C1 hexagonal 2026-05-23 — comment lines 23-27.)
- Header comment (lines 12-21) documents it as a coarse-location data-sharing scope gating BE propagation via `location-resolver.ts isGranted(userId,'location_to_llm')`.
- FE UI surfacing (`SettingsAiConsentCard`/`AiConsentSheetContent`) not byte-read in this leaf but the union type `ThirdPartyAiScope` (line 41) drives the existing grant/revoke wiring per the comment; the scope is in the canonical array, so it rides the existing render maps. **FE claim = plausible-verified at the data layer.**

### BE consent gate — CORRECT for the RESOLVED path
- `museum-backend/src/modules/chat/useCase/location-resolver.ts:188-209` `resolveLocationForMessage`:
  - line 196-200: when a `consentChecker` is supplied, requires `options.userId` AND `isGranted(userId,'location_to_llm') === true`; otherwise returns `undefined`.
  - So when consent is **denied**, `prep.resolvedLocation = undefined`. Correct.
- Wired in `prepare-message.pipeline.ts:444-447` (passes `consentChecker: this.locationConsentChecker`).
- Unit-tested: `tests/unit/chat/location-resolver.coarse.test.ts:162-177` ("emits NO location when the user has not granted location_to_llm").

### ⚠️ BYPASS — RAW FULL-PRECISION COORDS STILL REACH THE LLM WHEN CONSENT IS DENIED
The consent gate only nulls `resolvedLocation`. The **raw** `context.location` string is propagated on a SEPARATE, UNGATED channel and the prompt builder has a fallback that emits it.

Chain (all path:line verified):
1. `prepare-message.pipeline.ts:481-484` — `buildOrchestratorInput` sets `context: { location: input.context?.location, guideLevel: ... }`. This is the **raw** FE-supplied location string, copied verbatim, with **no consent check** on this field.
2. `llm-prompt-builder.ts:194-201` — `buildVisitorContextLine`:
   ```ts
   const rl = input.resolvedLocation;
   if (!rl) {
     const safeLocation = safeContextValue(input.context?.location);
     return safeLocation
       ? `<visitor_context>Visitor location: ${safeLocation}.</visitor_context>`
       : '';
   }
   ```
   When consent denied → `rl === undefined` → falls into this branch → reads raw `input.context.location`.
3. `safeContextValue` (`llm-prompt-builder.ts:22-27`) only runs `evaluateUserInputGuardrail` + `sanitizePromptInput`. The guardrail (`art-topic-guardrail.ts:216-224`) blocks only insults/prompt-injection — a coord string passes `allow:true`. `sanitizePromptInput` does Unicode-normalize/strip, **does not strip coordinates**. So `safeLocation` = the coord string verbatim.
4. **Wire format = full-precision GPS.** FE builds `context.location` as `` `lat:${latitude},lng:${longitude}` `` (`museum-frontend/features/chat/application/chatSessionLogic.pure.ts:328`). BE Zod accepts it as `optionalNonEmptyString` (`chat-session.schemas.ts:116`) — no coarsening. `parseLocationString` (`museum-backend/src/shared/utils/location.ts:1-11`) confirms the `lat:X,lng:Y` format.
5. Path is LIVE in prod: `langchain.orchestrator.ts:340` and `:528` call `buildOrchestratorMessages` → `buildVisitorContextLine`.

**Net effect**: a user who has NOT granted (or has revoked) `location_to_llm` but whose client still sends `context.location` (the FE attaches it to the message independent of consent state) gets `<visitor_context>Visitor location: lat:48.8566,lng:2.3522.</visitor_context>` injected into the LLM prompt — i.e. **raw, full street-level coordinates leak to the third-party LLM**, which is strictly WORSE than the coarse (city/country) value the gated path is so careful to enforce (the resolved path deliberately ships only `reverseGeocodeCoarse`, comment `location-resolver.ts:143-160` + `llm-prompt-builder.ts:205-210`).

### Why prior audit missed it
The regression test `tests/unit/chat/no-museum-geolocated.test.ts:67-82` constructs the "consent not granted" case as `resolvedLocation` absent **AND `context.location` absent** (explicit comment line 73). All three test cases omit `context.location` entirely, so the fallback branch (`buildVisitorContextLine:197-200`) is never exercised with a populated raw location under denied consent. The test asserts the gate "keys off resolvedLocation" but never proves the raw side-channel is closed.

### Verdict
- FE scope: ✅ present (`consentScopes.ts:38`).
- BE resolved-path gate: ✅ correct (`location-resolver.ts:196-200`).
- **BE raw-coords bypass: ❌ OPEN.** Denied/revoked consent does NOT block raw `context.location` from reaching the LLM via the `buildVisitorContextLine` fallback. GDPR Art. 6/7 (no lawful basis once consent withdrawn) + data-minimization (fine vs coarse) both implicated. **Roadmap ✅ on B9 is OVER-STATED** — the FE half is done, the BE consent enforcement is incomplete.

### Debt / fix
- DEBT-B9-RAW-LOC (P0, GDPR): gate the raw `context.location` channel on the same `location_to_llm` consent. Options:
  - (a) In `buildOrchestratorInput`, do not propagate `context.location` when `resolvedLocation` is undefined-due-to-consent (needs a "consent denied" signal distinct from "no location sent"); OR
  - (b) Remove the `buildVisitorContextLine` raw fallback (lines 197-200) entirely — there is no legitimate reason to emit raw `lat:/lng:` to the LLM (coarse-only is the documented contract); the resolved path already covers the granted case.
  - Add a RED test: denied consent + populated `context.location` → assert NO `visitor_context` / NO `lat:`/`lng:` in the user message.

## P0.B11 — EXIF strip optional fallthrough — VERDICT: ⚠️ CONFIRMED SILENT, boot-assert ABSENT

- `museum-backend/src/modules/chat/useCase/image/image-processing.service.ts:158-165` `stripExif`:
  ```ts
  if (!this.imageProcessor) {
    const buffer = Buffer.from(base64, 'base64');
    return { base64, mimeType, sizeBytes: buffer.byteLength, buffer };  // raw image, EXIF intact
  }
  ```
  The comment (lines 153-156) claims this branch is "intentionally observable so wiring regressions surface in CI" — but there is **NO log, NO metric, NO assert, NO throw**. It silently returns the raw (EXIF-bearing) image. The "observable" claim is FALSE as written.
- Prod wiring: `chat-module.ts:801` always constructs `new SharpImageProcessor()` (also `:207`, `:286` for the compare path). So in the current prod composition the processor is present and EXIF IS stripped.
- **The requested defense (boot-assert in `chat-module.ts`) is ABSENT.** `grep` of `chat-module.ts` for `assert`/`stripExif`/`EXIF` returns no boot-time guard verifying the processor was wired into `ChatService`. A future refactor that drops the processor would silently ship EXIF (GPS in EXIF = location leak via image metadata) with zero CI/runtime signal.
- Roadmap (ROADMAP_PRODUCT.md:117) already RÉTROGRADÉ ✅→⚠️ on 2026-05-25 with exactly this finding. **My independent re-derivation AGREES with the ⚠️ downgrade.** This is a defense-in-depth gap, not an active prod leak (low risk today).

### Debt / fix
- DEBT-B11-EXIF-ASSERT (P2, defense-in-depth): add a boot-time assert in `chat-module.ts build()` that `imageProcessor` is non-null when EXIF stripping is required, OR make the `stripExif` fallthrough emit a `logger.error`/metric (`exif_strip_skipped`) so the silent branch becomes genuinely observable as the comment claims.
