## P0.B14 — Langfuse DPA + SUBPROCESSORS.md

**Énoncé roadmap** (`:120`) : `❌🧑‍🔧 NEEDS-OPS-HUMAN` — Langfuse Cloud actif `.env:180-183`, mais `docs/compliance/SUBPROCESSORS.md` 20 vendors / 0 Langfuse row. Sign DPA + ajouter row + créer `docs/legal/dpa-signed/` (n'existe pas).

**Trace code vérifiée** :
- Langfuse **actif** : `museum-backend/.env:180` `LANGFUSE_ENABLED=true`, `:181-182` keys committées (`sk-lf-…`/`pk-lf-…`), `:183` `LANGFUSE_BASE_URL="https://cloud.langfuse.com"`. Code actif : `src/index.ts:50,310` (`shutdownLangfuse` + flush spans on drain), `src/shared/observability/langfuse-langchain.ts`, `langchain.orchestrator.ts`, etc. → **les données de prompt/télémétrie transitent réellement par Langfuse Cloud.**
- `docs/compliance/SUBPROCESSORS.md` (Art. 28 ledger, 20 rows `:32-51`) : `grep -i langfuse` = **0 match** (exit 1). Vendors listés : OpenAI, DeepSeek, Google, Apple, Brevo, Sentry, OTLP collector, S3, Better Stack, Wikidata, Wikipedia, Nominatim, Overpass, Unsplash, Tavily, Brave, DuckDuckGo, SearXNG, LLM-Guard, VPS-host. **Langfuse, CARTO et Expo absents.**
- `docs/legal/ROPA.md` (existe, `:13 mai`) : `grep Langfuse` = **0 match** → Langfuse absent du registre Art. 30 aussi.
- `docs/legal/dpa-signed/` : **MISSING** (`ls` → not found). Confirme l'énoncé roadmap.

**INCOHÉRENCE MAJEURE (le cœur de B14, et plus large que l'énoncé)** :
Langfuse EST listé dans **les 3 surfaces privacy PUBLIQUES** :
- `museum-web/src/lib/legal/privacy-content.canonical.json:67,242,354,529` (`name: "Langfuse"`, `jurisdiction: "Germany"`, `transferMechanism: "internal"`).
- copie BE source-of-truth `museum-backend/src/shared/legal/privacy-content.canonical.json` (byte-identique, cf B15).
- contenu FE `museum-frontend/features/legal/privacyPolicyContent.ts` (2 occurrences, en+fr).

→ **Le public déclare Langfuse comme sous-traitant ; les ledgers INTERNES de conformité (SUBPROCESSORS.md Art. 28 + ROPA.md Art. 30) ne le mentionnent NULLE PART.** C'est l'inverse de la honte habituelle (public minimise) : ici le public est plus complet que l'interne. Mais juridiquement, l'Art. 28 (contrat sous-traitant / DPA) et l'Art. 30 (registre) sont les obligations *contraignantes* — leur absence est le vrai gap, pas l'inverse.

**Sub-vérification jurisdiction** : canonical dit Langfuse `Germany / internal (EU)`. Config pointe `cloud.langfuse.com`. La région EU de Langfuse Cloud EST `cloud.langfuse.com` (US = `us.cloud.langfuse.com`), donc le claim "EU/internal" est **plausible mais non-prouvé** depuis le repo (dépend du tier de l'org Langfuse). Si l'org est sur tier US, le claim `transferMechanism: "internal"` serait FAUX (il faudrait SCC). À confirmer ops.

**VERDICT : NON FAIT (correctement marqué `❌🧑‍🔧`).** 3 actions ouvertes :
1. Signer le DPA Langfuse (ops humain).
2. Ajouter la row Langfuse à `SUBPROCESSORS.md` **et** à `ROPA.md` (dev — réalisable dans le repo, PAS strictement ops-humain ; l'énoncé sous-estime la part dev).
3. Créer `docs/legal/dpa-signed/` + y archiver le DPA signé (mixte : dossier = dev, contenu = ops).

**Debt** : GDPR Art. 28 + Art. 30 gap actif, vendor avec PII potentielle (prompts) non-couvert par contrat documenté. Bonus : confirmer région Langfuse Cloud (EU vs US) pour valider `transferMechanism: "internal"`. Note : CARTO + Expo aussi présents au public mais absents de SUBPROCESSORS.md → même gap que Langfuse, **hors périmètre B14 mais à signaler** (le scope "20 vendors" du ledger est incomplet de ≥3 vendors).
