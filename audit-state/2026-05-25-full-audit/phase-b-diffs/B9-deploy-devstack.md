## ⚠️ À améliorer

1. **[MEDIUM — latent correctness] `repo.update({ field: undefined })` is a silent no-op, comment claims it clears state.** `seed-smoke-account.ts:149-161` does `repo.update(existing.id, { …, verification_token: undefined, verification_token_expires: undefined, reset_token: undefined, reset_token_expires: undefined })` with the comment "Clear stale verification state so the smoke account cannot be poisoned by leftover tokens". This is **exactly** the TypeORM gotcha in CLAUDE.md (Pattern B `repo.update(criteria, partialEntity)` silently skips `undefined`, does NOT emit `SET … = NULL`). The clearing does not happen. The ESLint rule `musaium-test-discipline/no-typeorm-set-undefined` does NOT catch it because the file is `scripts/seed-smoke-account.ts`, outside the rule's `*.repository*.ts`/`*.repo.ts` scope. Severity bounded: the user is re-upserted `email_verified: true`, so a stale verification token doesn't block the smoke — but the comment is false (UFR-013) and a future reset-token reuse could surprise. Fix: `() => 'NULL'` or drop the misleading comment.
2. **[LOW — honesty/staleness] Stale `:223` line citation in smoke comment + commit msg.** `smoke-api.cjs:382` (and commit `294467c2a`) cite `chat-message.service.ts:223` as the refusal early-return; the actual short-circuit is line 245 (`if (prep.kind === 'refused') return prep.result;`). Line 223 is `{`. Mechanism described is accurate; line number drifted. `consent-gate.ts:73`/`:108` citations DO resolve (73 = `if (!granted)`, 108 = synthetic-id build region). UFR-024 doc-anchor sentinel does not scan `.cjs` comments, so this won't be auto-caught.
3. **[LOW — reactive, not proactive] The "5 layers" reveal a serial-discovery smoke, not a designed one.** Each deploy-prod attempt surfaced the *next* hidden break only after the prior fix unblocked the chain (env-salt → seed constraint → consent → TTS-id → Opus magic-byte → seed-time consent). Auto-rollback kept prod safe (good), but 5 consecutive red deploys means the smoke was never run against a realistic seeded+consented user locally before shipping. Now that the smoke is full happy-path, this is largely closed — but there's still no local "run the prod smoke against the dev Docker stack" target to catch layer N+1 pre-push. Consider a `pnpm smoke:api:local` wired into the dev-stack.
