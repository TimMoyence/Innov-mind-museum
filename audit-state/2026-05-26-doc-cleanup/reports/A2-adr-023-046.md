## Findings notables

### F1 — ADR-037 : index IVFFlat dans le doc, HNSW dans le code (CRITIQUE pour ops)
**Confiance : HIGH**
L'ADR-037 décrit en 3 endroits un index IVFFlat et donne des conseils de tuning IVFFlat (`lists` parameter). Le code réel (`1778406339944-AddArtworkEmbeddings.ts:72-78`) utilise un index HNSW avec `m=16, ef_construction=64` depuis le premier commit. Les conseils de tuning IVFFlat sont inapplicables à HNSW (HNSW n'a pas de `lists` — il a `m` et `ef_construction`). Risque ops : un SRE consultant l'ADR pour tuner l'index (charge accrue, recall dégradé) ferait une fausse manœuvre. Le model name est aussi légèrement incorrect (`siglip-base` vs `siglip2-base-patch16-224` réel). **Action : modifier l'ADR-037.**

### F2 — ADR-038 : cache key v1 devenue v2 (staleness post-merge)
**Confiance : HIGH**
ADR-038 (merge 2026-05-11) affirme dans sa section Neutral que la cache key est `llm:v1:...`. Le commit `d54552beb` (2026-05-19, +8 jours après merge) a bumpé la key à `v2` pour inclure `voiceMode` + `audioDescriptionMode`. ADR-036 a été mis à jour ; ADR-038 ne l'a pas été. C'est une divergence doc→code directement traçable. **Action : ajouter une note ou amendment dans ADR-038 §Neutral.**

### F3 — ADR-028 : numéros de lignes et coverage thresholds drift
**Confiance : HIGH**
Les line references dans ADR-028 ont drifté (museum:31 → :28 ; auth:54-58 → :76-80) suite à des ajouts dans les modules. Les coverage thresholds cités (89/75/87/89) ont été abaissés par la CI (88/74/86/89 dans `jest.config.ts`). Ce sont des inconsistances mineures — la décision de déférer reste valide et le code reflète bien l'intention.

### F4 — ADR-033 et ADR-046 : SUPERSEDED, correctement marqués, conservés pour traçabilité
Les deux ADR sont explicitement marqués SUPERSEDED dans leur frontmatter et dans `README.md`. Ils ont leur place (traçabilité du plan de migration zod 3→4). Pas d'action nécessaire.

### F5 — ADR-031 : endpoint `/api/config/cert-pinning-enabled` absent du backend — conforme à l'ADR
L'ADR lui-même déclare en section "Negative" que cet endpoint "does not exist yet" et que "the endpoint must ship before the env flag is flipped to `true` in production". L'env flag est `false` dans tous les `.env*`. Pas de fausse promesse — c'est un open work item documenté.
