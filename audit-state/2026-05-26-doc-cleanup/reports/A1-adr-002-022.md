## Findings notables

### F1 — ADR-003 masquerade "Deferred" (HIGH — trompe les développeurs)
**Gravité : critique sur l'utilisabilité de la doc.**
Le status de l'ADR dit "Deferred (2026-04-20)" mais le commit `b36c67b96` (2026-05-05) l'a explicitement exécuté. Le code source `auth.route.ts:11` cite même `— see ADR-003` comme justification de l'architecture actuelle. Tout développeur lisant cet ADR croit que la refactorisation reste à faire, alors qu'elle est shippée depuis 3 semaines. La README des ADR dit aussi "DEFERRED".
**Action** : status → "Implemented (2026-05-05, commit b36c67b96)".

### F2 — ADR-010 décision inversée par commit ultérieur (HIGH — la "décision" est devenue fausse)
**Gravité : l'ADR dit le contraire de ce qui existe.**
ADR-010 prend comme décision formelle : **"Keep `eslint ^10.2.0` on backend"**. Commit `42d81090a` (2026-05-17, 12 jours après la re-confirmation du 2026-05-05) prend la décision opposée : "path (b) — alignment-down → BE descend à v9.39.4". Tous les 3 apps sont maintenant uniformes sur `^9.39.4`. L'ADR est donc factuellement faux sur sa propre décision principale. ADR-032 répercute la même erreur.
**Action** : ajouter section amendment 2026-05-17 dans ADR-010 décrivant le path-b réel. (ADR-032 doit être corrigé séparément — hors lot de cet audit.)

### F3 — ADR-006 path mort dans la note d'implémentation (MEDIUM — réf cassée)
La note d'implémentation dit `...adapters/secondary/html-scraper.ts`. Le fichier réel est dans `...adapters/secondary/scraper/html-scraper.ts`. La référence `chat-message.service.ts:260` pour la chaîne d'appel pointe vers un fichier qui n'est plus l'orchestrateur (le code a migré dans `prepare-message.pipeline.ts:172`). Deux refs mortes dans le même paragraphe.
**Action** : corriger les deux paths dans la note d'implémentation.

### F4 — ADR-014 test dans le mauvais tier (MEDIUM — confusion architecture de tests)
L'ADR cite `tests/integration/auth/mfa-flow.e2e.test.ts` mais le fichier est à `tests/unit/auth/mfa-flow.e2e.test.ts`. Selon ADR-012, un fichier avec infra boundary réelle appartient en `tests/integration/`. Un test `.e2e.` en `tests/unit/` est soit mal tiérisé, soit correctement tiérisé (test unitaire malgré le nom). Hors-scope de cet audit de statuer sur le tier correct — la réf path ADR-014 est juste incorrecte.
**Action** : corriger le path dans ADR-014.

### F5 — ADR-017 livraison partielle non documentée (MEDIUM — cache un travail fait)
ADR-017 décrit 6 phases toutes déférées post-launch+30j. En réalité `app/(stack)/mfa-enroll.tsx` a été shiped via TD-SEC-02 (R8) — il monte `MfaEnrollScreen` avec un back-on-success. Phase 1 est partiellement livrée. L'ADR ne le mentionne pas, ce qui crée un risque de re-travail (un futur agent pourrait re-créer la route).
**Action** : ajouter note "Phase 1 partial — mfa-enroll.tsx shiped via TD-SEC-02" avec description de ce qui reste (mfa-challenge.tsx, login flow, settings, banner, i18n, tests).
